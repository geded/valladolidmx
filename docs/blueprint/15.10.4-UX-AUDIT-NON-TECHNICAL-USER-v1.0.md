# 15.10.4 — Auditoría UX · Experience Builder desde la perspectiva de un usuario NO técnico

**Versión:** 1.0
**Fecha:** 2026-07-01
**Autor:** Auditoría interna
**Perspectiva:** empresario turístico, sin experiencia previa con CMS o page builders
**Alcance:** rutas `/cms/experience-builder` (Studio v0) y `/cms/experience-builder/pages` (Pages Studio v2, adenda 15.10.4b)
**Método:** revisión de la superficie renderizada, textos, jerarquía visual, controles, terminología, flujos y comparación contra Blueprint 15.10.4
**Regla:** no se propone código. Sólo diagnóstico y priorización.

---

## 0. Resumen ejecutivo

El Experience Builder **funciona técnicamente**, pero **hoy NO puede ser usado por un empresario turístico sin capacitación**. La superficie está construida como un **editor para desarrolladores editoriales** (concepto de "composición", "slug", "revisión", "tipo de página", "sincronizar biblioteca", "type interno del bloque", "JSON en list/object", "SECURITY DEFINER" filtrado en tooltips) más que como una herramienta de negocio.

El motor (Block Registry, Renderer Universal, Auto-Inspector, Variables Dinámicas, Versionado, Publicación) está **arquitectónicamente completo respecto al Blueprint 15.10.4**, pero la **capa de experiencia** que traduce esa potencia a un lenguaje no técnico **no se ha construido**. Faltan: onboarding, plantillas iniciales, previsualización real, edición inline, terminología de negocio, seguridad ante errores y guía contextual.

**Veredicto:** el Experience Builder **no puede declararse oficialmente listo para usuarios no técnicos**. Requiere una adenda de **Experience Layer** antes de continuar 15.10.4 ampliada o iniciar nuevas capacidades.

## 1. Respuestas a las 10 preguntas de la auditoría

### 1. ¿Puede una persona sin conocimientos técnicos crear una página desde cero?

**No, salvo con acompañamiento.** El flujo obligatorio actual es:

1. Entrar a `/cms/experience-builder`.
2. Si la Biblioteca está vacía → aparece banner *"La Biblioteca de bloques está vacía. Sincronízala antes de editar"* con botón **"Sincronizar Biblioteca"**. Ningún empresario entiende qué es *"sincronizar biblioteca"* y por qué es su responsabilidad.
3. Clic en **"Nueva composición"** → modal pide:
   - **Slug interno (único)** con placeholder *"home, landing-verano, etc."*.
   - **Título editorial.**
   - **Tipo de página** con opciones `generic / home / landing / institutional / destination / business / product / campaign` — en inglés, sin descripción.
4. Se abre un canvas vacío con etiqueta *"Canvas · generic"* y *"Internal Draft — esta composición no se sirve al público"*.
5. Debe elegir bloques por su nombre técnico (`hero_full_bleed`, `smart_marketplace_grid`, etc.), sin miniaturas ni descripción visible en la lista.
6. Debe llenar campos del Inspector cuyo control para tipos `list` / `object` es **un textarea con JSON crudo**.
7. Debe **Guardar borrador**, luego **Crear revisión**, luego (si es admin) **Publicar** — tres verbos diferentes para tres estados que el usuario no distingue.

**Conclusión:** un empresario no completará este flujo sin un tutorial. La curva es de herramienta para editores CMS profesionales.

### 2. ¿Puede modificar una página existente sin romper el diseño?

**Riesgo alto de romperlo.** Evidencia:

- **No hay validación amigable de errores.** Si el JSON del Inspector para un campo `list`/`object` es inválido, el `catch` es silencioso: no se guarda el cambio y no aparece mensaje. El usuario cree que guardó.
- **No hay confirmación visual del cambio** salvo repintar el canvas. Sin diff, sin "antes/después".
- **La eliminación de un bloque** se hace desde una barra de herramientas superior con botón **"Eliminar"** de tamaño idéntico al resto. No hay confirmación explícita en la ruta principal, y el bloque desaparece del canvas inmediatamente.
- **No hay undo/redo visible.** El código expone reordenar/duplicar, pero no hay `Ctrl+Z` documentado ni botón dedicado.
- **No hay preview responsive real:** `Pages Studio` cambia el ancho a 390/768/full, pero el usuario no ve una "vista móvil" (marco de dispositivo), sólo un canvas más angosto.
- **Publicar sobreescribe la versión pública del `page_type`** sin advertencia clara de a qué URL afecta. El texto es *"Reemplazará la versión pública actual del tipo 'home'"* — el usuario no sabe qué URL pública sirve *home*.

### 3. ¿Comprende naturalmente la diferencia entre Página, Sección, Bloque y Componente?

**No.** La UI usa vocabulario técnico y solapado:

| Concepto en UI | Aparece como | Interpretación esperada por el usuario |
|---|---|---|
| Composición | "Composiciones", "Nueva composición" | ¿Colección? ¿Diseño? No lo asocia con "Página". |
| Página | Sólo en la ruta `/pages` y `page_type` | Debería ser el término principal. |
| Bloque | "Biblioteca" con nombres técnicos | Concepto correcto, pero sin representación visual. |
| Sección | **No existe** en la UI. El árbol es plano (`root.children`). | El usuario espera agrupar bloques en "secciones". |
| Componente | No expuesto | OK. |
| Revisión | "Crear revisión", "Historial" | Se confunde con "Publicar" y con "Guardar". |
| Slot | Presente en el contrato, oculto | OK. |

**El árbol es de un solo nivel.** No hay "secciones" que agrupen bloques. Contradice la expectativa del empresario y contradice parcialmente el Blueprint 15.10.4 (Layout Engine con contenedores).

### 4. ¿La interfaz guía al usuario o requiere capacitación previa?

**Requiere capacitación previa.** No hay:

- Onboarding / tour al abrir por primera vez.
- Estado vacío educativo ("Empieza eligiendo una plantilla").
- Tooltips explicativos sobre *slug*, *tipo de página*, *revisión*, *publicar*, *biblioteca*.
- Ayuda contextual en el Inspector (¿qué es `soporta_i18n`? ¿qué es una capacidad?).
- Ejemplos precargados por tipo de página.

La única "guía" visible es la barra de estado inferior tipo consola (*"Listo."*, *"Borrador guardado automáticamente."*, *"Error: ..."*) — patrón de IDE, no de herramienta de negocio.

### 5. ¿Qué acciones requieren demasiados clics?

| Acción | Clics/pasos actuales | Óptimo esperado |
|---|---|---|
| Publicar un cambio de texto | 1 clic en canvas → editar en Inspector → Guardar borrador → Crear revisión → Publicar → Confirmar (5+) | 1 clic → editar inline → Publicar |
| Añadir un bloque en posición específica | Clic en biblioteca (se agrega al final) → seleccionar → subir con ↑ N veces | Drag & drop con drop-zones visibles |
| Duplicar y editar una sección | Seleccionar → Duplicar → renombrar (no existe) → editar cada campo | 1 clic "Duplicar sección" con etiqueta |
| Cambiar la imagen del hero | Seleccionar hero → escribir URL a mano en un input `url` (no hay picker) | Clic sobre la imagen → media library |
| Ver cómo se verá en móvil | Cambiar breakpoint en `/pages`, no en Studio v0 | Toggle desktop/tablet/móvil siempre visible |
| Restaurar una versión previa | Abrir composición → panel Historial → Restaurar sin preview | Historial con preview y comparación |

### 6. ¿Existen conceptos técnicos expuestos innecesariamente?

**Sí, muchos:**

- **`slug` interno** — es un identificador técnico. El usuario debería nombrar la página; el slug se deriva.
- **`page_type`** en inglés (`institutional`, `campaign`, `landing`).
- **`type` del bloque** mostrado bajo cada tarjeta de la Biblioteca (`hero_full_bleed`, `smart_marketplace_grid`, `cta_banner_static`).
- **`version` del bloque** ("v1", "v2") como columna visible.
- **`Sincronizar Biblioteca`** como acción del usuario.
- **Categorías `static` / `smart`** como filtros en inglés y sin explicación.
- **`Internal Draft`** como estado.
- **`Canvas · generic`** como título del área de trabajo.
- **Capacidades** (`i18n`, `SEO`, `datos`, `personalización`, `cache`) como chips sin definición.
- **Textarea con JSON** para campos `list` y `object`.
- **`translations` polimórfico**, `SECURITY DEFINER`, `RPC` — presentes en tooltips y comentarios de código, pero conceptos filtrados en la superficie.

### 7. ¿Qué elementos generan confusión?

1. **Tres verbos de guardado**: *Guardar borrador* / *Crear revisión* / *Publicar*. El usuario espera **un solo botón "Publicar"** con "Guardar cambios" como acción secundaria.
2. **Dos rutas paralelas del Studio**: `/cms/experience-builder` (Studio v0 con "composiciones") y `/cms/experience-builder/pages` (Pages Studio con "páginas") — dos modelos coexistiendo. El usuario no sabe cuál usar.
3. **Ausencia total de una lista de "Mis páginas" que muestre la URL pública**. Sólo se ve *slug*, *page_type* y *status*, no la URL real (`/`, `/l/verano`, etc.).
4. **Bloques `smart_*` sin contenido visible en el canvas hasta que se configuran** — canvas parece "roto".
5. **La barra superior del editor** mezcla acciones destructivas (Eliminar, Despublicar) con no destructivas (↑, ↓, Duplicar, Guardar) sin agrupación ni color de alerta.
6. **El botón "← Lista"** dentro del panel Biblioteca — descubrimiento accidental, no es donde el usuario buscará "salir".
7. **Historial sin fecha legible clara** (`toLocaleString()` local, sin autor visible en Studio v0).
8. **Ningún feedback claro** de "esto ya está publicado y visible en `https://valladolidmx.lovable.app/`".

### 8. ¿Qué partes parecen hechas para desarrolladores y no para empresarios?

- Modal *Nueva composición* → parece un formulario de generador de código.
- Filtros *all / static / smart* en inglés.
- Editor de campos `list`/`object` = editor JSON.
- Botón *Sincronizar Biblioteca* con tooltip *"Vuelca el catálogo de bloques declarado en código a la base"*.
- Barra de estado tipo consola.
- Nombres técnicos de bloques bajo cada tarjeta.
- `type` y `version` como metadatos visibles.
- Copia del inspector: *"Este bloque no tiene campos editables."*
- Sección *Historial* con `revision_number` (`#3`, `#7`).
- El *VariablePicker* inserta tokens tipo `{{contexto.usuario.nombre}}` en el texto — potente, pero sin explicación.
- Autor y BEA log invisibles: el usuario no ve quién editó qué.

### 9. ¿Qué capacidades del Blueprint 15.10.4 todavía no están presentes en el Experience Builder?

El motor cumple con la mayor parte, pero **faltan capacidades declaradas** — reales o de UX:

| Capacidad Blueprint 15.10.4 | Estado actual | Gap |
|---|---|---|
| **Contenedores / secciones (Layout Engine con anidamiento)** | Árbol plano en UI, sólo `root.children` | Sin drag & drop entre secciones ni columnas visibles. |
| **Drag & Drop en Canvas** | Ausente. Añadir bloque = clic → se agrega al final. Reordenar = flechas ↑/↓ | Crítico. |
| **Preview público real (con URL destino)** | Existe `ebIssuePreviewToken`, pero no está expuesto como botón *"Ver como visitante"* claro | Alto. |
| **Responsive Preview** | `/pages` cambia ancho, sin marco device | Medio. |
| **Media Picker integrado** | Campo `media` es `input url` con placeholder *"URL temporal (selector en 15.10.5)"* | Alto. |
| **Reference Picker (empresas, destinos, productos)** | Campo `reference` es texto libre con placeholder *"ID de X (combobox en 15.10.5)"* | Alto. |
| **Editor de campos `list` / `object`** | Textarea JSON | Crítico. |
| **Traducciones i18n visibles en Inspector** | Chip `i18n` sin UI para editar traducciones | Alto. |
| **SEO editable por página** | No hay panel SEO (title, description, OG, JSON-LD) | Crítico. |
| **Scheduling (publicar/despublicar por fecha)** | Ausente | Medio (parte de 15.10.5). |
| **Roles / permisos editoriales visibles** | Sólo `admin` puede Publicar y Sincronizar, sin explicación al no-admin | Medio. |
| **Plantillas / Partials** | No expuestas al usuario | Alto (parte de 15.10.4). |
| **Undo/Redo** | Ausente en UI | Alto. |
| **Historial con diff y preview** | Sólo lista de revisiones | Medio. |
| **Onboarding y estados vacíos educativos** | Ausente | Crítico. |
| **Terminología en español consistente** | Mezcla es/en | Alto. |
| **Alux asistente editorial** (declarado en 15.10.4) | No integrado dentro del Studio | Alto. |
| **Validaciones amigables** (errores por campo, no silenciosos) | Silencio en errores JSON | Crítico. |

### 10. ¿Qué falta para declarar oficialmente que el Experience Builder puede ser utilizado por usuarios no técnicos?

Ver §3 (plan priorizado). En resumen: **una capa de Experience Layer** que envuelva el motor actual con:

1. Terminología de negocio consistente (español, verbos claros).
2. Un único Studio (colapsar `/experience-builder` y `/experience-builder/pages`).
3. Modelo mental **Página → Sección → Bloque**, no *Composición → Nodo*.
4. Drag & Drop + edición inline sobre el canvas.
5. Pickers reales (Media, Referencias, Iconos, Color).
6. Editor visual para `list`/`object` (repetidores).
7. Panel SEO por página.
8. Preview responsivo real con marco device y botón *"Ver como visitante"*.
9. Publicación de un solo paso con confirmación clara (URL de destino, diff).
10. Onboarding + plantillas iniciales por tipo de página.
11. Undo/Redo global y confirmaciones antes de acciones destructivas.
12. Validaciones y mensajes amigables (nada silencioso).
13. Ocultar `slug`, `type`, `version`, `Sincronizar Biblioteca` detrás de "modo avanzado".
14. Alux integrado como copiloto editorial dentro del Studio.

## 3. Lista priorizada de mejoras (sin implementar)

### Crítica — bloquean la usabilidad por usuarios no técnicos

| # | Mejora | Justificación |
|---|---|---|
| C1 | **Colapsar `/experience-builder` y `/experience-builder/pages` en un único Studio** llamado *"Páginas"* o *"Editor de sitio"* | Dos rutas paralelas confunden y duplican mantenimiento. |
| C2 | **Renombrar "Composición" → "Página"** en toda la superficie | Modelo mental del usuario. |
| C3 | **Ocultar `slug`, `type`, `version`, `page_type` en inglés, `Sincronizar Biblioteca`** detrás de un "Modo avanzado" o auto-generarlos | Vocabulario técnico expuesto. |
| C4 | **Editor visual (no JSON) para campos `list` y `object`** — repetidor de tarjetas con "Añadir elemento" / "Eliminar" | Hoy es un textarea JSON: cualquier error corrompe silenciosamente. |
| C5 | **Validaciones y errores amigables por campo** (nada silencioso) | Actualmente errores JSON se ignoran. |
| C6 | **Un solo botón "Publicar cambios"** con confirmación que muestre la URL pública destino y el resumen de cambios | 3 verbos actuales (Guardar, Revisión, Publicar) son ininteligibles. |
| C7 | **Panel SEO por página** (título, descripción, OG image, canonical) accesible desde un botón "SEO" en la barra del Studio | Blueprint 15.10.4 lo exige y hoy no existe UI. |
| C8 | **Estados vacíos educativos + plantillas iniciales** por tipo de página (Home, Landing, Empresa, Destino) | Sin plantillas el usuario abandona. |

### Alta — indispensables para autonomía

| # | Mejora | Justificación |
|---|---|---|
| A1 | **Drag & Drop** para añadir/reordenar bloques con drop-zones visibles | Flechas ↑/↓ es patrón de admin, no de builder. |
| A2 | **Modelo Página → Sección → Bloque** con secciones colapsables/agrupadoras | Contradice expectativa mental y limita el Layout Engine. |
| A3 | **Media Picker real** conectado a Media Library (12) para todos los campos `media` | Hoy es input `url`. |
| A4 | **Reference Picker** (combobox con búsqueda) para empresas, destinos, productos, categorías, rutas | Hoy es texto libre con ID. |
| A5 | **Undo/Redo global** con `Ctrl+Z`/`Ctrl+Y` y botones visibles | Reduce miedo a experimentar. |
| A6 | **Edición inline** de textos directamente sobre el canvas | Elimina viaje ida y vuelta al Inspector. |
| A7 | **Preview responsivo con marco de dispositivo** (desktop/tablet/móvil) desde el propio Studio v0 (hoy sólo en `/pages`) | Empresarios validan mayormente en móvil. |
| A8 | **Botón *"Ver como visitante"*** que abre el preview público con token (ya existe `ebIssuePreviewToken`) | Cerrar el ciclo antes de publicar. |
| A9 | **Terminología 100% en español** y consistente (`estático`/`inteligente` en vez de `static`/`smart`) | Barrera idiomática. |
| A10 | **Confirmaciones destructivas explícitas** (Eliminar bloque, Despublicar, Restaurar revisión) con detalle del impacto | Evita pérdidas involuntarias. |
| A11 | **Traducciones i18n editables desde el Inspector** para campos marcados `translatable` | Chip actual no hace nada. |
| A12 | **Alux integrado como copiloto editorial** ("Sugiere un título", "Mejora este texto", "Genera 3 variantes") | Declarado en Blueprint 15.10.4. |

## 5. Conclusión y recomendación

El Experience Builder cumple su **objetivo arquitectónico** (motor extensible, contratos, versionado, publicación, i18n, SEO como capacidades del contrato) pero **no cumple aún su objetivo de producto**: permitir que un empresario turístico opere su presencia digital sin acompañamiento técnico.

**Recomendación:** antes de continuar 15.10.4 ampliada u otra funcionalidad, redactar una **Adenda 15.10.4d — Experience Layer** cuyo alcance sea exclusivamente cerrar las mejoras **Críticas** y **Altas** de este documento, con un DoD verificable en cabina real con un empresario piloto.

No se propone implementar cambios en esta iteración.
