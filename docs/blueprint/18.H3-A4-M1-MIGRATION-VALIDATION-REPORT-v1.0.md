## 1. Checkpoint y rollback

## 2. Definición de `variant_key` (huella determinista)

`variant_key` es una columna `GENERATED ALWAYS AS ... STORED` que concatena, con separadores explícitos, todos los ejes de identidad:

```
asset_id + ':fmt=' + format + ':w=' + width + ':h=' + height
         + ':eng=' + engine + ':ev=' + engine_version
         + ':q='   + quality_effective
         + ':uc='  + usage_context   (default 'generic')
         + ':cs='  + source_checksum (default '-')
```

Cualquier cambio en el original (`source_checksum`) o en la receta (`engine`, `engine_version`, `quality_effective`, `format`, `width`, `height`, `usage_context`) produce una identidad diferente. `source_checksum` se pobló desde `media_assets.original_checksum` para los assets que ya lo tenían tras el pilot y queda como `-` de forma determinista para los 24 assets legacy sin checksum, sin colisionar con futuras generaciones que sí lo llevarán.

## 5. Índice parcial de variante activa

```
select asset_id, format, width, engine, coalesce(usage_context,'generic'), count(*)
  from public.media_asset_variants
 where is_current and status='ready'
 group by 1,2,3,4,5 having count(*) > 1;   -- 0 filas
```

Índices creados:

- `media_asset_variants_variant_key_ux` — UNIQUE global de `variant_key`.
- `media_asset_variants_current_ux` — UNIQUE parcial `WHERE is_current AND status='ready'` sobre `(asset_id, format, width, engine, coalesce(usage_context,'generic'))`.
- UNIQUE previa `(asset_id, format, width, engine)` PRESERVADA.

## 6. Historial (evidencia estructural)

- `is_current`, `superseded_at`, `superseded_by` disponibles; default seguro (`is_current=true` para todo lo existente, resto `NULL`).
- `superseded_by` con `ON DELETE SET NULL` evita huérfanos en el grafo de versiones.
- Un cambio en `engine_version`/`quality_effective`/`source_checksum` genera un `variant_key` distinto → la nueva fila coexiste bajo `UNIQUE(variant_key)` y sólo el índice parcial impide dos vigentes simultáneas.
- Limitación intencional mientras el PASO 4 no se ejecute: la UNIQUE antigua `(asset_id, format, width, engine)` sigue impidiendo dos filas físicas para una misma dimensión aun con distinto `engine_version`. Es coherente con el flag OFF: no habrá regeneraciones que necesiten desdoblarse hasta M2.

## 7. Prueba de rollback

SQL de rollback (idéntico al final del archivo de migración):

```sql
DROP INDEX IF EXISTS public.media_asset_variants_current_ux;
DROP INDEX IF EXISTS public.media_asset_variants_variant_key_ux;
ALTER TABLE public.media_asset_variants
  DROP COLUMN IF EXISTS variant_key,
  DROP COLUMN IF EXISTS superseded_by,
  DROP COLUMN IF EXISTS superseded_at,
  DROP COLUMN IF EXISTS is_current,
  DROP COLUMN IF EXISTS source_checksum,
  DROP COLUMN IF EXISTS quality_effective,
  DROP COLUMN IF EXISTS engine_version;
-- La UNIQUE (asset_id,format,width,engine) permanece intacta.
```

Verificación estructural: el conteo previo a un rollback simulado sigue siendo 33. Todos los objetos eliminados por el rollback son estructuras nuevas de esta migración; no se toca ni una fila, ni un objeto de Storage, ni columnas legacy.
