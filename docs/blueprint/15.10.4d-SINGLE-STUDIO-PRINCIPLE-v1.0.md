## 1. Principio (obligatorio)

Existe **un solo editor visual** para toda la plataforma:

- Un único **Experience Builder**.
- Un único **Studio**.
- Un único **editor visual predeterminado** (WYSIWYG por defecto).
- Un **modo avanzado opcional** dentro del mismo Studio (mismo shell,
  no una app aparte).

Prohibido construir editores especializados por página, por tipo de
contenido, por rol o por superficie. La Home no puede tener un editor
distinto al de una landing, y una landing no puede tener un editor
distinto al de Marketplace, Empresas, Experiencias, Hoteles,
Restaurantes, Eventos, Oriente Maya, Arma tu Viaje, Alux o páginas
creadas por el usuario.

## 2. Cobertura obligatoria

El mismo Studio debe abrir, con la misma UX, todas las páginas
públicas de la plataforma:

- Home (`/`)
- Landings
- Marketplace y sub-rutas
- Empresas (índice y ficha)
- Experiencias, Hoteles, Restaurantes, Eventos
- Oriente Maya (índice y destino)
- Arma tu Viaje
- Alux
- Páginas creadas por el usuario desde el propio Studio

## 3. Rol del "tipo de página"

El tipo de página **sólo** puede determinar:

- Plantilla inicial.
- Bloques disponibles.
- Validaciones aplicables.
- Reglas de SEO.
- Permisos de edición.

Nunca podrá determinar un editor distinto, un shell distinto, una barra
de herramientas distinta ni un flujo de publicación distinto.

## 5. Historia de unificación (US-00 · P0)

### US-00 · Unificar el Studio (WYSIWYG por defecto + modo avanzado)
**Objetivo funcional.** Que exista un solo Studio en
`/cms/experience-builder` con dos modos dentro del mismo shell:

- **Modo visual (predeterminado):** WYSIWYG con AutoInspector traducido,
  campos etiquetados en español, valores reales precargados, sin JSON.
- **Modo avanzado (opcional, mismo shell):** acceso a JSON, IDs y
  contratos técnicos para power users. Se activa con un toggle en la
  topbar del mismo Studio, no cambiando de ruta.

**Criterios de aceptación.**
- Existe **una sola ruta** de edición: `/cms/experience-builder` (index
  visual) y `/cms/experience-builder/pages/:slug` (edición de página).
- Se retiran las rutas paralelas `/paginas` y `/paginas/inicio` una vez
  el Studio único cubre el caso Home.
- La topbar del Studio expone un toggle `Visual | Avanzado`. El modo
  visual es el predeterminado en cada apertura.
- El AutoInspector, en modo visual, jamás muestra JSON, IDs de
  referencia ni slugs crudos.
- El mismo Studio puede abrir cualquier composición (`home` incluida) y
  cualquier página nueva creada por el usuario.
- Ninguna otra ruta autenticada expone un editor visual alternativo.

**Dependencias.** Ninguna nueva. Reutiliza Experience Builder,
AutoInspector, composition-renderer, block-registry.

**Componentes afectados.**
- `src/routes/_authenticated/cms/experience-builder.tsx` (topbar con
  toggle Visual/Avanzado, modo visual por defecto).
- `src/routes/_authenticated/cms/experience-builder.pages.tsx`
  (unificar edición por slug bajo el Studio único).
- `src/components/experience-builder/AutoInspector.tsx` (rama visual sin
  JSON, con valores reales precargados).
- `src/routes/_authenticated/paginas.index.tsx` y
  `src/routes/_authenticated/paginas.inicio.tsx` → **retiro** tras
  migración.
- `src/lib/workspace/definitions/index.ts` → un único ítem "Editor de
  páginas" apuntando al Studio único.

**Riesgo.** Medio. Cambia la superficie de entrada del editor y retira
una ruta previa. Mitigación: migración en un solo commit con
redirects de `/paginas*` a `/cms/experience-builder`.

**Validación manual.** Abrir `/cms/experience-builder`, seleccionar la
composición `home`, editar Hero en modo visual, publicar, verificar `/`.
Cambiar a modo avanzado, ver JSON, volver a visual. Crear una página
nueva desde el mismo Studio.

**Prueba con usuario.** Sí. Founder no técnico realiza el flujo completo
sin instrucciones adicionales.

implementarse.
