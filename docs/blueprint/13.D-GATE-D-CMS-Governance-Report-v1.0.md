# 13.D – Gate D · CMS Governance & Content Operations Report
**Versión:** 1.0  
**Estado:** Entregado · pendiente de revisión  
**Blueprint refs:** 07, 13.0, 13.0A, 13.1, 13.3, 13.4

## 1. Resumen ejecutivo

El Bloque D habilita la capa de gobernanza editorial del CMS sobre el modelo de
dominio (Bloque B) y el sistema de identidad (Bloque C). Se incorporaron las
entidades editoriales obligatorias del documento 13.4, el flujo editorial con
transiciones controladas, auditoría por contenido, SEO universal y server
functions de lectura pública para alimentar a la Fase 0 sin romperla.

No se construyó editor visual complejo (fuera de alcance). Los mocks de Fase 0
permanecen como fallback hasta que el contenido real exista en el CMS.

## 2. Migraciones

- `phase1_blockD_cms_governance` — Crea articles, pages, events,
  editorial_routes, faqs, banners, promotions, seo_metadata, content_audit_log
  y la función `public.transition_content_status`. Amplía `entity_kind` con
  `faq`, `banner`, `promotion`.

## 3. Nuevas tablas

| Tabla | Propósito |
|---|---|
| `articles` | Artículos / blog editorial. |
| `pages` | Páginas institucionales / estáticas (bloques JSONB). |
| `events` | Calendario de eventos vinculados a destino o empresa. |
| `editorial_routes` | Rutas / itinerarios sugeridos. |
| `faqs` | FAQs globales o vinculadas a una entidad. |
| `banners` | Banners de Home / landings con ventana de vigencia. |
| `promotions` | Promociones de empresa o producto. |
| `seo_metadata` | SEO polimórfico por entidad y locale. |
| `content_audit_log` | Bitácora de transiciones editoriales. |

Todas comparten campos estándar (`status`, `locale`, `cover_media_id`,
`published_at`, auditoría) y reutilizan los enums `content_status`,
`locale_code` y `hero_palette` del Bloque B.

## 4. Modelo editorial

Estados oficiales (enum `content_status`):

```
draft → in_review → approved → published → archived
```

Transiciones permitidas (validadas por `transition_content_status`):

- draft → in_review · archived
- in_review → approved · draft · archived
- approved → published · draft · archived
- published → archived · draft (despublicación)
- archived → draft (reactivación)

Al publicar, `published_at` se sella si estaba vacío. Toda transición queda
registrada en `content_audit_log`.

## 5. Flujos de publicación

1. Un autor crea contenido en `draft`.
2. Lo envía a `in_review`.
3. Un editor/admin lo lleva a `approved`.
4. Un editor/admin ejecuta la transición a `published`; el contenido queda
   disponible para lectura pública.
5. Si requiere retiro, transiciona a `archived` o vuelve a `draft`.

Permisos:

- Lectura pública (anon/auth): sólo `status = 'published'` y `deleted_at IS NULL`.
- Escritura: `is_editor_or_admin()` excepto `promotions`, donde también pueden
  editar usuarios con `has_business_access(business_id, 'editor')`.
- Función de transición: `SECURITY DEFINER`, revoke a `anon`, exige rol
  editorial.

## 6. APIs / Server Functions

Archivo: `src/lib/cms.functions.ts`. Todas usan un cliente publishable sin
sesión, respetando las políticas RLS de lectura pública:

- `listPublishedArticles({ limit?, destinationId? })`
- `getArticleBySlug({ slug })`
- `listUpcomingEvents({ limit? })`
- `listActiveBanners({ placement? })`
- `listFaqs({ category? })`
- `getPageBySlug({ slug })`
- `listPublishedRoutes({ limit? })`
- `getSeoMetadata({ entityKind, entityId, locale? })`

## 7. SEO

`seo_metadata` cubre slug, meta título/descripción, canonical, Open Graph,
Twitter card, noindex y JSON-LD por entidad y locale. Aplica a cualquier
`entity_kind` (destinos, empresas, productos, regiones, artículos, páginas,
eventos, rutas, FAQs, banners, promociones).

## 8. Traducciones

Se reutiliza la tabla `translations` del Bloque B (puente polimórfico
`entity_kind + entity_id + locale + field`). Las nuevas entidades del Bloque D
son automáticamente compatibles porque exponen `entity_kind` en el enum.

## 9. Auditoría

`content_audit_log` registra cada transición con `from_status`, `to_status`,
`actor_user_id`, `notes` y metadata. Visible sólo para editores/admins; el
log es append-only desde la función `transition_content_status`.

## 10. Integración con Home

La Home de Fase 0 se mantiene intacta. Las server functions del paso 6
entregan datos reales del CMS cuando exista contenido publicado; los mocks
permanecen como fallback. La sustitución progresiva se hará por sección
(artículos, eventos, banners, rutas) sin tocar la dirección de arte aprobada.

## 11. Evidencias QA

- Migración aplicada sin errores; lints existentes corresponden a helpers
  preexistentes del Bloque C (`has_role`, `is_admin`, etc.) que deben
  permanecer ejecutables por `authenticated` para que las políticas RLS los
  invoquen.
- Tipos de Supabase regenerados con las 9 nuevas tablas.
- Server functions tipadas y compilando (`tsgo` limpio).
- RLS verificada: lectura anon limitada a `published`; escritura limitada a
  rol editorial.

## 12. Riesgos detectados

1. **Cobertura editorial inicial vacía:** sin contenido sembrado, la Home
   seguirá renderizando mocks; deberá poblarse vía el panel administrativo
   (Fase 2) o seeds curados antes de retirar los mocks.
2. **Editor visual diferido:** la edición se hará por cliente backend o
   herramientas administrativas hasta que se construya el panel CMS.
3. **SEO sin panel:** la tabla existe pero el panel de edición SEO se
   construirá junto con el panel CMS.
4. **Promociones business-scoped:** los dueños de negocio pueden gestionar
   promociones de su empresa; esto requiere QA dedicada al activar onboarding
   de empresas.
