## 1. Resumen de implementación

Se entregó la capa de infraestructura del servicio genérico "Protected
Actions", sin consumidores, sin UI y sin cambios observables para los
usuarios finales. El sistema queda montado y listo para que I2 conecte el
hook `useProtectedAction` y el `SignInPromptSheet` sobre esta base.

Principio arquitectónico respetado: **convive, no reemplaza**. Ningún
patrón actual (`RequestConciergeButton`, `AddToTravelPlanButton`,
`FavoriteButton`, `guest-queue`, `useAuth`, `SheetStackProvider`) fue
modificado.

El servicio se diseñó como reutilizable por CUALQUIER acción futura
(no atado a Mi Viaje / Favoritos / Concierge). La forma
`ProtectedActionRequirements` deja abiertos, sin implementar, los gates
`requiredRole`, `requiredPermission`, `requiredSubscription`,
`requiredProfileCompletion`, `requiredBusinessVerification`,
`requiredTravelerProfile` y un `custom` como escape hatch.

## 4. Documentación de comportamiento del `PendingActionRegistry`

| Escenario | Comportamiento en v1 |
|---|---|
| **TTL** | 10 min por defecto (`DEFAULT_ACTION_TTL_MS`). Configurable por acción vía `ttlMs`. Al vencer, `takeLast()` emite `protected_action.expired` y descarta. |
| **Limpieza automática** | En cada `peek`/`takeLast` se filtran vencidas. Al `SIGNED_OUT`, el runner llama `clear("signed_out")`. |
| **Múltiples pestañas** | Metadatos sincronizados por `BroadcastChannel` (`protected_actions.v1`) + `storage` event como fallback. El payload (`execute`) vive sólo en la pestaña de origen. Si el login ocurre en otra pestaña, la de origen reanuda; la ajena emite `restored` sin ejecutar. |
| **Doble clic** | Cada `push()` genera `id` propio (crypto.randomUUID o fallback). El hook I2 marcará `pending=true` durante la ejecución para bloquear reintentos visuales. |
| **Acciones concurrentes** | FIFO en memoria. `takeLast()` ejecuta la más reciente vigente y descarta las anteriores emitiendo `cancelled` con `reason: "superseded"`. Limitación conocida (DFT E7). |
| **Recarga de página** | Metadatos sobreviven en `sessionStorage`. El payload se pierde → el runner emite `restored` sin ejecutar (la UI del I2 mostrará el toast "Continúa donde lo dejaste"). |
| **Pérdida de sesión** | `SIGNED_OUT` → `clear("signed_out")` emite `cancelled` por cada acción viva. |
| **Expiración** | Detectada en lectura (`takeLast`/`peek`); no requiere timer activo. |
| **Storage bloqueado (modo privado estricto)** | `try/catch` silencioso; degrada a memoria. Sin crash. |
| **SSR / prerender** | Toda la infraestructura es `window`-safe; el runner sólo se suscribe en `useEffect`. |

## 5. Observabilidad

Eventos emitidos vía `emitProtectedActionEvent` (sin PII, sólo `actionId`,
`kind`, `ttlMs`, `reason`, `errorCode`):

- `protected_action.started` (acción iniciada)
- `protected_action.gated` (a emitir por el hook I2 al abrir el Sheet)
- `protected_action.auth_requested` (a emitir por el Sheet I2)
- `protected_action.auth_completed` (ya emitido por el runner al `SIGNED_IN`)
- `protected_action.resumed` (acción ejecutada tras login)
- `protected_action.restored` (metadato recuperado sin payload → sin ejecución)
- `protected_action.dismissed` (a emitir por el Sheet I2)
- `protected_action.cancelled` (registry limpiado o superseded)
- `protected_action.expired` (TTL vencido)
- `protected_action.failed` (error en la ejecución server-side)

`subscribeProtectedActionEvents` permite conectar un sink real
(BL-3 del DFT) sin tocar consumidores.

- Storage bloqueado → no crash (rutas `try/catch`).
- Recarga con metadato vigente → runner emite `restored` sin ejecutar.
- Sesión iniciada sin acciones pendientes → runner no-op.
- Signed-out con registry vacío → no-op.

## 8. Riesgos encontrados y mitigaciones

| # | Riesgo | Mitigación aplicada en I1 |
|---|---|---|
| R-I1-1 | Doble suscripción a `onAuthStateChange` (runner + `useAuth`) | Runner filtra sólo `SIGNED_IN`/`SIGNED_OUT` y `defer` a `setTimeout(0)` como `useAuth` — sin deadlock. |
| R-I1-2 | Sync entre pestañas puede rebotar payloads | Sólo metadatos van al canal; el payload nunca se serializa. |
| R-I1-3 | React StrictMode duplica eventos | `emitProtectedActionEvent` aplica dedup por `(event, actionId)` en ventana de 50 ms. |
| R-I1-4 | Registry queda desincronizado tras `SIGNED_OUT` en otra pestaña | Canal envía `clear` cross-tab. |
| R-I1-5 | Payload sensible filtrado a storage | Storage nunca recibe `execute`, `onSuccess`, `onError`. Sólo `{id, kind, createdAt, expiresAt, reason}`. |
