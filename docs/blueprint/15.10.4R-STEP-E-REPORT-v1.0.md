# 15.10.4R · Paso E — Gestión de roles desde el Panel Fundador

**Versión:** 1.0 · **Estado:** Ejecutado

## 1. Alcance entregado

- Nueva superficie **`/admin/sistema/usuarios`** (acceso exclusivo `super_admin`).
- Tres server functions mínimas bajo `requireSupabaseAuth`:
  - `isSuperAdmin` — gating de UI.
  - `listUsersWithRoles` — lista global de cuentas y roles.
  - `assignRole` / `revokeRole` — escrituras auditadas.
- Tres RPCs `SECURITY DEFINER` (autorización dura server-side):
  - `admin_list_users_with_roles()`
  - `admin_assign_role(uuid, app_role)`
  - `admin_revoke_role(uuid, app_role)`

## 2. Garantías de seguridad

- Toda RPC valida `has_role(auth.uid(), 'super_admin')` antes de ejecutar.
- `EXECUTE` revocado a `PUBLIC` y otorgado solo a `authenticated`.
- `search_path` fijo (`public`) en las tres funciones.
- **El rol `super_admin` NO puede asignarse ni revocarse** desde esta
  superficie (prevención de privilege escalation y de lockout del Founder).
- Un usuario **no puede revocar sus propios roles**.
- **Cero uso de `SUPABASE_SERVICE_ROLE_KEY`** en flujos de usuario.
- Modelo `user_roles` / `has_role` intacto. No se crearon roles nuevos.
- RLS existente sin cambios (la mutación se hace vía RPC SECURITY DEFINER
  con autorización explícita; las políticas actuales de `user_roles` ya
  permitirían a `admin`/`super_admin` operar, pero el flujo oficial pasa
  por las RPCs auditadas para garantizar trazabilidad).

## 3. Auditoría

Cada `assign` / `revoke` inserta una fila en `permissions_audit_log` con:
`actor_user_id`, `target_user_id`, `action` (`role.assign` | `role.revoke`),
`role` y `metadata.surface = "/admin/sistema/usuarios"`.
