## 2. Clasificación

### 2.1 Universales (reutilizables con ViewModel normalizado)

Un bloque es **universal** si su UI depende únicamente de una forma común
(título, imagen, texto, lista de items) y no del origen (business / product /
event / hotel).

| Kit block              | Reemplaza                                                                          | Contrato clave |
| ---------------------- | ---------------------------------------------------------------------------------- | -------------- |
| `vmx.shell`            | `business.shell`, `product.shell`                                                  | `crumbs[]`, `title?`, `eyebrow?`, `description?` |
| `vmx.hero`             | `business.header-badges` + `product.hero`                                          | `eyebrow`, `title`, `subtitle`, `favorite?`, `badges[]` |
| `vmx.gallery`          | `product.gallery` (+ `business.gallery` cuando exista media real)                  | `cover`, `items[] {url,alt}` |
| `vmx.rich-text`        | `business.description`, `product.description`                                      | `heading?`, `body` |
| `vmx.price-cta`        | `product.price-cta`                                                                | `price?`, `mode?`, `actions: ReactNode`, `sticky?` |
| `vmx.info-table`       | `business.info`                                                                    | `rows: {label,value}[]` |
| `vmx.contact`          | Parte contacto de `product.business-context` + futuro `business.contact` real       | `contact {type,label?,value}` |
| `vmx.location`         | Parte ubicación de `product.business-context`                                       | `location {label?,address_line1,address_line2?,lat?,lng?}` |
| `vmx.reviews`          | `product.reviews`                                                                  | `reviews[] {author,rating,title?,body}` — `Stars` interno |

## 3. Propuesta Surface Kit v1

### 3.1 Principios

1. **ViewModel-only.** Cada bloque recibe *sólo* su ViewModel (props tipadas).
   Cero `useContext` interno, cero acoplamiento a `BusinessSurfaceContext` /
   `ProductSurfaceContext`.
2. **Adaptadores por entidad.** Cada superficie define un mapper `toKitVM(entity)`
   que produce los ViewModels. Los bloques `vmx.business.*` / `vmx.product.*`
   pasan a ser **shims** de 3–5 líneas que leen el contexto y renderizan el Kit.
3. **Sin cambios en `CompositionRenderer` ni `preview-registry`.** El registro
   sigue mapeando `vmx.business.hero` → `BusinessHeroBlock`; internamente delega
   en `<KitHero vm={...} />`.
4. **Registro `vmx.kit.*` opcional (fase posterior).** Bloques neutros para que
   el Builder componga landings/campañas sin entidad.

### 3.3 Contratos ViewModel (extracto)

```ts
export type HeroVM = {
  eyebrow?: string;
  title: string;
  subtitle?: string;
  favorite?: { kind: 'business'|'product'|'promotion'; id: string };
  badges?: BadgeVM[];
};

export type GalleryVM = {
  cover?: { url: string; alt?: string } | null;
  items?: { id: string; url: string; alt?: string }[];
  emptyLabel?: string;
};

export type PriceCtaVM = {
  price?: { amount: number; currency: string } | null;
  mode?: string;
  actions: React.ReactNode;   // ProductActions ya montado por el adaptador
  sticky?: boolean;
};

export type ReviewVM = { id: string; author: string; rating: number; title?: string; body: string };
export type FaqVM   = { id: string; question: string; answer: string };
export type PromoVM = { id: string; title: string; description?: string; discountPercent?: number };
export type CardVM  = {
  id: string; href?: string; eyebrow?: string; title: string; tagline?: string;
  price?: { amount: number; currency: string };
  favorite?: HeroVM['favorite'];
  actions?: React.ReactNode;
};
```

### 3.4 Adaptadores por entidad

```ts
// business-blocks.tsx (post-refactor, ~40 LOC totales)
export function BusinessHeroBlock() {
  const b = useBusiness();
  if (!b) return <EmptyHint>Hero del negocio.</EmptyHint>;
  return <KitHero vm={businessToHeroVM(b)} />;
}
```

Los mappers viven junto a la superficie (`business-view-model.ts`,
`product-view-model.ts`). El Kit **no importa** tipos de entidad y las entidades
**no importan** tipos del Kit — sólo consumen su API.

## 5. Riesgos y mitigaciones

| Riesgo | Mitigación |
| ------ | ---------- |
| Regresión visual en Business/Product durante el shim                       | Refactor por bloque, no big-bang. Contraste manual en Studio por cada bloque migrado. |
| Registros duplicados en `block-library.ts` (`vmx.kit.*` + `vmx.business.*`) | `vmx.kit.*` NO se registra en 2.5a; sólo en 2.5d cuando lo consuma el Builder. |
| Filtración de tipos de entidad al Kit                                      | Regla: `kit/**` no importa de `@/lib/marketplace/**` ni de contextos de superficie. |
| Gate por plan viviendo en el bloque (`planAllows` en promotions)           | El bloque recibe `gated: boolean` ya resuelto. `planAllows` queda en el adaptador. |
| CTA `ProductActions` acopla al Kit a producto                              | El Kit recibe `actions: ReactNode`; no conoce `ProductActions`. |
