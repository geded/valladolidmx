## 1. Resumen ejecutivo

El Bloque B implementa el **modelo de dominio completo** de Valladolid.mx
conforme a los Blueprint 11.1 y 13.1: las 4 capas (Geografía, Comercial,
Multimedia, Editorial-base) más el bridge de internacionalización, con
auditoría completa, RLS pública para contenido publicado, índices,
triggers de `updated_at` y seeds oficiales reproducibles.

No se duplicó ninguna tabla principal para soportar i18n: las
traducciones viven en una tabla puente polimórfica (`translations`).

Las escrituras de usuarios autenticados quedan **bloqueadas** hasta que
el Bloque C aporte `app_role`, `user_roles` y `has_role()`. Mientras
tanto solo `service_role` (es decir, server functions del backend)
puede escribir.

## 7. Índices (70 totales — PK + UNIQUE + 33 explícitos)

Cubren todas las columnas requeridas por 11.5 §Índices:

- **Slug**: `UNIQUE` en `countries`, `destinations`, `business_categories`, `businesses`; `UNIQUE (parent, slug)` en `states`, `tourism_regions`, `destination_zones`, `points_of_interest`, `products`.
- **Status**: en `countries`, `states`, `tourism_regions`, `destinations`, `business_categories`, `businesses`, `points_of_interest`, `products`, `reviews`, `media_assets`.
- **Claves foráneas**: índices explícitos en todas las FK no cubiertas por PK/UNIQUE.
- **Búsquedas frecuentes**:
  - `idx_products_type` (filtrado por tipo de producto)
  - `idx_reviews_subject (kind, id)` (índice compuesto para reseñas polimórficas)
  - `idx_translations_entity (kind, id)` y `idx_translations_locale`
  - `idx_*_published_at` en `destinations`, `businesses`, `products`

## 8. Constraints

- `UNIQUE` en slugs (globales y compuestos por padre).
- `UNIQUE (storage_bucket, storage_path)` en `media_assets`.
- `UNIQUE (entity_kind, entity_id, locale, field)` en `translations`.
- `UNIQUE (business_id, media_asset_id, role)` en `business_media` y equivalente en `product_media`.
- `CHECK (day_of_week BETWEEN 0 AND 6)` en `business_hours`.
- `CHECK (rating BETWEEN 1 AND 5)` en `reviews`.

## 9. Triggers

16 triggers `BEFORE UPDATE … EXECUTE FUNCTION public.set_updated_at()`
(uno por tabla con `updated_at`).

## 10. RLS (Row Level Security)

- **19 tablas** con RLS habilitada.
- **19 policies** `*_public_read` para `anon` + `authenticated`:
  - Tablas con `status`: `USING (status = 'published' AND deleted_at IS NULL)`.
  - Tablas hijas sin `status`: `USING (deleted_at IS NULL)` o `USING (true)` (su visibilidad real se controla por el join con la tabla padre).
  - `business_contacts`: además `is_public = true`.
- **Sin policies de escritura** para `anon`/`authenticated` (queda al Bloque C).
- `service_role` bypassa RLS (estándar Supabase) → server functions del backend pueden poblar y mantener el contenido.
