**Versión:** 1.0 (Propuesta — pendiente de aprobación)

**Tipo:** Extensión del CMS Studio (no implementación)

## 1. Resumen ejecutivo

Esta propuesta define una extensión del CMS Studio denominada **Home Experience Builder (HEB)** que permitirá al equipo de Marketing administrar de forma íntegra la página principal de Valladolid.mx (`/`) sin requerir cambios de código.

El HEB se construye **sobre** la arquitectura aprobada (Blueprint, Serie 13, Serie 14) y respeta de manera estricta los principios consolidados durante las Olas 1, 2 y 3:

- Cero uso de `SUPABASE_SERVICE_ROLE_KEY` en flujos de usuario.
- Cero bypass de RLS.
- Validación server-side obligatoria mediante `createServerFn` + `requireSupabaseAuth`.
- Whitelist explícita de campos.
- Workflow editorial `draft → in_review → approved → published → archived` reutilizando `src/lib/cms/workflow.ts`.
- Auditoría centralizada en `content_audit_log`.
- Permisos vía `has_role` (CMS Studio) y `has_business_access` cuando se referencian recursos del Portal.
- Aislamiento arquitectónico: el Portal Empresarial, el Marketplace y la Fase 1 no se modifican.

El documento es **únicamente una propuesta**. No autoriza implementación. La Ola 4 continúa según el Plan 14.40.

---

## 2. Objetivo

Permitir que el equipo de Marketing administre completamente la Home desde el CMS Studio, sin tocar código, cubriendo:

- Hero principal (imagen/video/título/subtítulo/CTAs).
- Mostrar / ocultar secciones.
- Cambiar orden mediante drag & drop.
- Editar textos y CTAs por sección.
- Configurar empresas, destinos, experiencias y promociones destacadas.
- Administrar banners y bloques promocionales.
- Programar campañas por fechas (start/end).
- Preview antes de publicar.
- Versionado.
- Workflow editorial.
- Auditoría de cambios.
- Permisos.
- Compatibilidad con Marketplace, Portal Empresarial, i18n y Alux Intelligence.

## 3. Alcance funcional

### 3.1 Secciones administrables de la Home (Doc 12)

| # | Sección | Tipo en HEB | Editable | Ocultable | Reordenable |
|---|---------|-------------|----------|-----------|-------------|
| 1 | Hero | `hero` | sí | no (siempre visible) | no (fijo en posición 1) |
| 2 | Destinos | `featured_destinations` | sí | sí | sí |
| 3 | Categorías | `categories_grid` | sí | sí | sí |
| 4 | Rutas sugeridas | `featured_routes` | sí | sí | sí |
| 5 | Consejo de Alux | `alux_tip` | sí | sí | sí |
| 6 | Arma tu Viaje | `cta_block` | sí | sí | sí |
| 7 | Oriente Maya EN VIVO | `social_feed` | sí | sí | sí |
| 8 | Empresas recomendadas | `featured_businesses` | sí | sí | sí |
| 9 | Reseñas | `featured_reviews` | sí | sí | sí |
| 10 | Banners promocionales | `promo_banner` | sí | sí | sí |

El **Hero** y el **Footer** son no-reordenables por contrato visual (Doc 12, 12A v2).

### 3.2 Campañas programadas

Cada **versión publicada** y cada **bloque** pueden tener ventanas temporales:

- `start_at`, `end_at` (UTC) opcionales.
- Cuando hay varias versiones programadas, la versión activa se resuelve server-side en cada SSR.
- Bloques con ventana caduca se ocultan automáticamente.

### 3.3 Preview

- Modo "Preview" renderiza la Home con el **borrador** o una **versión candidata**, sin afectar producción.
- URL firmada y caducable, scoped al usuario editorial autenticado.

## 4. Modelo de dominio propuesto

Se introducen tres tablas nuevas en `public`, todas bajo RLS y con `GRANT` explícitos. No modifican el dominio aprobado de Fase 1 ni los recursos del Portal/Marketplace.

### 4.1 `home_versions`

| Columna | Tipo | Notas |
|---|---|---|
| `id` | uuid PK | |
| `name` | text | etiqueta editorial (p. ej. "Temporada alta 2026") |
| `status` | content_status | reutiliza enum de Ola 1 |
| `start_at` | timestamptz NULL | programación |
| `end_at` | timestamptz NULL | programación |
| `locale` | text | base `es`; traducciones vía `translations` |
| `created_by` / `updated_by` | uuid → profiles | |
| `published_at` / `published_by` | nullable | |
| `created_at` / `updated_at` | timestamptz | |

Restricción: a lo sumo **una** versión `published` con ventana activa por `locale` (índice parcial único `WHERE status='published'` + validación en RPC para solapamientos).

### 4.2 `home_blocks`

| Columna | Tipo | Notas |
|---|---|---|
| `id` | uuid PK | |
| `version_id` | uuid → home_versions ON DELETE CASCADE | |
| `kind` | enum `home_block_kind` | `hero`, `featured_destinations`, `categories_grid`, `featured_routes`, `alux_tip`, `cta_block`, `social_feed`, `featured_businesses`, `featured_reviews`, `promo_banner` |
| `position` | int | orden estable; único por `(version_id, position)` |
| `is_visible` | bool default true | |
| `start_at` / `end_at` | timestamptz NULL | ventana por bloque |
| `config` | jsonb | payload validado con Zod por `kind` (whitelist) |
| `created_at` / `updated_at` | timestamptz | |

`config` no es libre: cada `kind` tiene su Zod schema en `src/lib/cms/home/block-schemas.ts` (whitelist estricta — mismo patrón que `editor-fields.ts`).

### 4.3 `home_block_features`

Relación N:M entre bloques y entidades destacadas (empresas, destinos, productos, promociones, rutas, reseñas, banners). Mantiene integridad referencial sin acoplar `config.jsonb` a IDs sueltos.

| Columna | Tipo |
|---|---|
| `id` | uuid PK |
| `block_id` | uuid → home_blocks ON DELETE CASCADE |
| `entity_kind` | enum (`business`, `destination`, `product`, `promotion`, `route`, `review`, `banner`) |
| `entity_id` | uuid |
| `position` | int |
| `pinned` | bool default false |

Único `(block_id, entity_kind, entity_id)`.

### 4.4 GRANT y RLS (resumen)

```sql
GRANT SELECT ON public.home_versions, public.home_blocks, public.home_block_features TO anon;       -- sólo filas publicadas vía vista
GRANT SELECT, INSERT, UPDATE, DELETE ON ... TO authenticated;                                       -- restringido por RLS a roles editoriales
GRANT ALL ON ... TO service_role;
```

Lecturas públicas (SSR del Home actual) se hacen a través de una **vista materializable** `home_published_current` que sólo expone la versión activa y bloques con ventana válida. Anon **no** consulta las tablas base.

### 4.5 Reutilización

- `content_audit_log` — sin cambios de esquema, se añaden `entity_kind = 'home_version' | 'home_block'`.
- `translations` — i18n de títulos/CTAs por bloque (clave compuesta `home_block:<id>:<field>`).
- `media_assets` — todas las imágenes/videos del Hero y banners se referencian aquí (no se duplica storage).
- `workflow.ts` — máquina de estados intacta.

## 5. Capa de aplicación

### 5.1 Server functions (nuevas)

`src/lib/cms/home/home-builder.functions.ts`:

- `listHomeVersions` (CMS reads)
- `getHomeVersion(id)` (con bloques + features hidratados)
- `createHomeVersion` / `updateHomeVersion` / `archiveHomeVersion`
- `transitionHomeVersion(id, to)` — reusa `assertAllowedTransition`
- `upsertHomeBlock(versionId, kind, config)`
- `reorderHomeBlocks(versionId, [{id, position}])`
- `toggleHomeBlockVisibility(blockId, visible)`
- `setBlockFeatures(blockId, [{entity_kind, entity_id, position, pinned}])`
- `previewHomeVersion(id)` — genera token de preview (HMAC, TTL 30 min)
- `publishHomeVersion(id)` — valida solapamientos, transiciona a `published`

Todas con `requireSupabaseAuth` + check `has_role('admin'|'editor')`. Whitelist por `kind`. Auditoría obligatoria.

`src/lib/cms/home/home-public-reads.functions.ts`:

- `getActiveHome(locale)` — usado por SSR del `/` actual. Lee `home_published_current`. Sin auth, sólo datos publicados con ventana válida.

### 5.2 Rutas CMS Studio (nuevas)

- `/_authenticated/cms/home` — listado de versiones.
- `/_authenticated/cms/home/$versionId` — editor visual con:
  - Lista de bloques con drag & drop (`@dnd-kit/core`).
  - Toggle visibilidad por bloque.
  - Editor por `kind` (formulario tipado por Zod schema).
  - Selector de entidades destacadas (autocomplete contra `businesses`, `destinations`, `products`, `promotions`).
  - Programación (`start_at`, `end_at`).
  - Botones: Guardar borrador, Solicitar revisión, Aprobar, Publicar, Archivar, **Preview**.
- `/_authenticated/cms/home/$versionId/preview` — render del Home en modo preview.

No se crean rutas públicas adicionales. La Home actual (`src/routes/index.tsx`) se adapta para leer `getActiveHome(locale)` en su `loader`. La adaptación del Home no es parte de esta propuesta — se documentaría en el plan de ejecución posterior.

### 5.3 Cliente

- Drag & drop con `@dnd-kit/sortable` (a evaluar; alternativa `framer-motion Reorder`).
- Estado optimista vía TanStack Query con `invalidateQueries(['home', versionId])`.

## 6. Workflow editorial

Reutiliza la máquina de Ola 1:

```
draft → in_review → approved → published → archived
         ↘ draft       ↘ draft       ↘ draft
```

- Marketing (rol `editor`) puede crear, editar y solicitar revisión.
- `admin` aprueba y publica.
- Publicación valida: ventanas no se solapan con otra `published`, todos los bloques requeridos cumplen Zod, traducciones requeridas existen (o quedan en fallback `es`).

---

## 7. Versionado

- Cada `home_version` es inmutable una vez `published` (futuras ediciones requieren clonarla → nueva versión `draft`).
- RPC `cloneHomeVersion(sourceId)` copia versión + bloques + features.
- Historial completo navegable; cualquier versión `archived` o `approved` puede re-promocionarse vía clonación.

---

## 8. Auditoría

- Toda mutación en `home_versions`, `home_blocks`, `home_block_features` registra entrada en `content_audit_log`:
  - `entity_kind`, `entity_id`, `action` (`create|update|reorder|publish|archive|toggle_visibility|set_features`), `actor_id`, `payload_diff` (whitelisted), `created_at`.
- Preview genera un evento `home_preview_issued` con `token_hash` (no token plano).

## 10. Compatibilidad

### 10.1 Marketplace (Ola 4)

- `featured_businesses`, `featured_destinations`, `featured_routes` referencian entidades publicadas por el Marketplace.
- Reutiliza índices y RPCs de búsqueda definidos en el Plan 14.40 (Etapa 2).
- Cero acoplamiento bidireccional: la Home consume; el Marketplace no conoce al HEB.

### 10.2 Portal Empresarial (Ola 3)

- Empresas destacadas se seleccionan de `businesses` con `status='published'` y `is_visible=true`.
- El Portal **no** gana capacidad de auto-destacarse: la curación es exclusiva del CMS Studio. Esto preserva la matriz de permisos de la Ola 3.

### 10.3 Internacionalización

- `translations` cubre todos los campos de texto de `home_blocks.config` whitelisted como traducibles.
- `getActiveHome(locale)` hidrata con fallback `es` cuando falta traducción aprobada.
- SEO multilenguaje: cada versión expone `meta_title` / `meta_description` traducibles y `hreflang` automático.

### 10.4 Alux Intelligence

- El bloque `alux_tip` lee desde el corpus aprobado de Alux (no inventa contenido).
- El HEB expone, vía RPC interna, el "contexto público actual de la Home" (versión activa, destacados) para que Alux contextualice sus respuestas sin acceder a tablas crudas.
- Cero permisos elevados: Alux consume `getActiveHome` igual que el SSR.

## 11. Seguridad

- Cero `SUPABASE_SERVICE_ROLE_KEY` en flujos de usuario.
- Cero cambios a RLS de tablas existentes.
- Todas las RPCs `SECURITY DEFINER` con `search_path = public` y `EXECUTE TO authenticated`.
- Lecturas anónimas exclusivamente vía vista `home_published_current` (filtra por ventana y status).
- Preview tokens HMAC firmados con secret server-side; nunca expuestos al cliente sin TTL.
- Whitelist Zod por `kind` — `config.jsonb` jamás se persiste sin validar.
- Subidas de Hero/Banners reutilizan política aditiva de Storage de Ola 3.

## 12. QA propuesto

1. Crear versión, añadir todos los `kind`, reordenar drag & drop, publicar — Home pública refleja el cambio.
2. Solapamiento de ventanas → publicación rechazada con error claro.
3. Bloque con `end_at` pasado → no aparece en SSR.
4. Preview de versión `draft` no afecta a anon ni cambia la versión activa.
5. Editor solicita revisión; admin aprueba y publica; auditoría completa.
6. Toggle visibilidad oculta sección sin perder configuración.
7. Traducción faltante → fallback `es` documentado en SSR.
8. Anon no puede leer `home_versions` ni `home_blocks` (RLS verificado).
9. Editor sin rol admin no puede invocar `publishHomeVersion` (server-side bloqueado).
10. Clonación de versión publicada produce nueva versión `draft` idéntica.

## 13. Performance

Alineado al Plan 14.40 §Performance:

- `getActiveHome` debe resolver en ≤ 80ms p95 (consulta única a vista materializable + un `IN(...)` por `entity_kind`).
- Cache HTTP `Cache-Control: public, s-maxage=60, stale-while-revalidate=300` en SSR del `/`.
- TanStack Query `staleTime` 60s en el editor CMS.
- Índices: `home_blocks(version_id, position)`, `home_block_features(block_id, entity_kind)`, `home_versions(status, start_at, end_at)`.

## 16. Riesgos y mitigaciones

| ID | Riesgo | Mitigación |
|---|---|---|
| H1 | Desincronización entre versiones publicadas y entidades retiradas (empresa unpublished referenciada). | `getActiveHome` filtra entidades por su propio `status='published'` antes de devolver. |
| H2 | Editor publica versión incompleta. | Validación Zod por `kind` requerida antes de transición a `published`. |
| H3 | `config.jsonb` deriva en campo libre. | Whitelist por `kind`; cualquier campo no listado se descarta antes de persistir. |
| H4 | Drag & drop genera reorders concurrentes. | RPC `reorderHomeBlocks` aplica con `version_id` + transacción + `position` único; última escritura validada con `updated_at` optimistic concurrency. |
| H5 | Preview tokens filtrados. | HMAC + TTL 30 min + revocación por `version_id`; nunca exponer secret en cliente. |
| H6 | Alcance crece y compite con la Ola 4. | Propuesta queda en stand-by; no se implementa hasta cerrar 14.40.Z. |
