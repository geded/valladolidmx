## 4. Server Functions y validaciones server-side

Tres server functions nuevas, todas con `requireSupabaseAuth` y
verificación adicional server-side mediante RPC `is_admin`:

| Server Function | Método | Rol requerido | Resultado |
|-----------------|--------|---------------|-----------|
| `getReviewForModeration` | POST | admin | Detalle completo de la reseña. |
| `moderateReview` | POST | admin | Transición de estado + bitácora interna. |
| `listReviewModerationHistory` | POST | admin | Historial append-only por reseña. |

Validación de transición server-side contra la máquina oficial
(Serie 14):

```
draft → in_review → approved → published → archived
         ↘ draft        ↘ draft       ↘ draft
```

Cualquier transición no permitida es rechazada con
`invalid_transition:<from>-><to>`. No existe endpoint que omita esta
validación.

## 5. Cumplimiento del workflow editorial

- Las transiciones aplicables a reseñas reutilizan exactamente la matriz
  `ALLOWED_TRANSITIONS` definida en Etapa 4 para el resto del CMS.
- `published_at` se establece automáticamente al transicionar a
  `published`, en línea con el comportamiento del resto de entidades.
- La UI sólo expone las transiciones legales desde el estado actual y
  todas pasan por la misma server function — no hay atajo client-side.

## 6. Auditoría de moderación

`content_audit_log` no admite reseñas: el enum oficial `entity_kind`
NO incluye `review`. Para no introducir cambios al modelo de dominio
durante la Ola 1, la bitácora de moderación se persiste como estructura
append-only dentro de `reviews.metadata.moderation_history`:

```json
{
  "moderation_history": [
    {
      "action": "transition",
      "from_status": "in_review",
      "to_status": "approved",
      "notes": "Cumple criterios editoriales.",
      "actor_user_id": "<uuid>",
      "at": "2026-06-29T10:15:00.000Z"
    }
  ]
}
```

Características:

- Estructura tipada y limitada a 200 entradas por reseña (rolling window).
- Notas opcionales con tope de 1000 caracteres (defensa contra abuso UX).
- Lectura expuesta mediante `listReviewModerationHistory` (solo admins).
- No requiere migración, no altera RLS, no toca el enum de dominio.

La incorporación de `review` al enum `entity_kind` para unificarlo con
`content_audit_log` queda explícitamente fuera de esta etapa y deberá
plantearse como cambio documental en una ola posterior.
