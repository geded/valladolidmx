# 15.10.4 Fase 2 — Demo mínima de `/mi-viaje`

Ruta: `/mi-viaje` (protegida por `_authenticated`, redirige a `/auth` si no hay sesión).

## Pasos de la demo

1. **Iniciar sesión** como turista (`traveler`).
   - El sistema redirige automáticamente al panel del turista. El rol se asigna por trigger al confirmar el email.
2. **Llegar a `/mi-viaje`** — se renderiza el dashboard del turista.

### 1. Perfil Inteligente visible/enlazado

- Tarjeta superior **"Perfil Inteligente"** muestra:
  - Estilo · Presupuesto · Idioma · Nº intereses (chips/valores).
  - Mensaje contextual: `"Tu perfil está activo…"` o `"Completa estilo de viaje, presupuesto e intereses…"`.
- Botón **"Editar perfil" / "Completar perfil"** → `/cuenta/perfil` (editor completo whitelisteado por `upsertMyTravelerProfile`).

### 2. Formulario "Arma tu Viaje"

- Tarjeta intermedia con `<textarea maxLength=500>` (mín. 8 caracteres) + contador.
- Botón **"Enviar solicitud"** invoca `useMutation` sobre la server fn `ccCreateCaseFromPlan`, que ejecuta la RPC `cc_case_create_from_plan(summary, items, travel_plan_id)`.

### 3. Solicitud creada correctamente

- La RPC `cc_case_create_from_plan` delega en `concierge_case_from_travel_plan`, que crea:
  - `concierge_cases (status='new', priority='normal', source='travel_plan')`
  - Entrada inicial en `concierge_case_timeline` (`case_created`).
- En `onSuccess` se limpia el textarea y se invalida `["cc","my-cases"]`.

### 4. Caso visible en la lista del turista

- Tarjeta inferior **"Mis solicitudes"** consume `ccListMyCases` (`concierge_case_list_for_role` con `_scope='traveler'`).
- RLS: `ccases_select` exige `concierge_can_view_case(id, auth.uid())`, que para el turista exige `traveler_user_id = auth.uid()`. Solo aparecen casos propios.
- Cada fila muestra `summary`, `status`, `priority`, `updated_at` y enlace **"Ver caso"** → `/cuenta/concierge/$caseId`.

### 5. Estado inicial del caso

- `status = 'new'`
- `priority = 'normal'`
- `priority_source = 'system'`
- `source = 'travel_plan'`
- `last_activity_at = now()`
- Timeline: 1 entrada `case_created` (severity `info`).

## Reglas reforzadas

- Toda la operación pasa por `cc_*` (`SECURITY DEFINER`) — el frontend nunca toca tablas directamente.
- El turista solo ve sus propios casos (RLS).
- El form valida mínimo 8 caracteres antes de enviar; el backend re-valida (`z.string().min(8).max(500)`).
- No se exponen `traveler_profiles.signals` ni otros datos sensibles — la tarjeta del Perfil Inteligente solo muestra campos visibles del propio dueño vía `getMyTravelerProfile` (whitelist explícita).
