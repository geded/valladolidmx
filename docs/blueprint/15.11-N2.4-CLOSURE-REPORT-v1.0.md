# H-03 · Épica 1 · Ola N2 · Sub-ola N2.4 — Switcher de destino y continuidad de contexto

**Estado:** Entregado · en espera de validación del Founder.
**Carril:** A · Producto.
**Alcance:** Contrato, servicio, componente y política de degradación.

---

## 1. Objetivo cumplido

Habilitar el cambio de destino preservando el nivel más profundo posible
del contexto (producto → empresa → categoría → destino), degradando de
forma predecible cuando no exista equivalencia, y sin introducir lógica
específica por ruta. Toda la resolución de URLs pasa por el Navigation
Contract v1.0 (`resolveCanonicalPath`).

## 3. Política de continuidad de contexto (contrato oficial)

Orden de intento, de mayor a menor profundidad:

```text
producto ⟶ empresa ⟶ categoría ⟶ destino
```

Regla dura: si un nivel no puede preservarse, degradar **exactamente un
paso** hacia arriba, jamás a una página arbitraria (Home, Marketplace,
Búsqueda, etc.). El "peor" resultado válido es la propia ficha de
Destino, que siempre existe si el destino objetivo está publicado.

---

## 4. Matriz de continuidad y degradación

| Nivel de origen | Condición evaluada en destino objetivo | Nivel preservado | URL resultante (canónica) | `reason` |
|---|---|---|---|---|
| Producto | Existe empresa `slug` publicada en destino con categoría propia · Existe producto `slug` publicado bajo esa empresa | Producto | `/oriente-maya/:destino/:categoria/:empresa/:producto` | `ok` |
| Producto | Empresa existe, categoría existe, producto no | Empresa | `/oriente-maya/:destino/:categoria/:empresa` | `product_not_in_destination` |
| Producto / Empresa | Empresa no publicada en destino objetivo · Categoría existe con ≥1 negocio publicado | Categoría | `/oriente-maya/:destino/:categoria` | `business_not_in_destination` |
| Empresa | Empresa no publicada · Sin categoría inferida en destino | Destino | `/oriente-maya/:destino` | `business_not_in_destination` |
| Categoría | Categoría con ≥1 negocio publicado en destino objetivo | Categoría | `/oriente-maya/:destino/:categoria` | `ok` |
| Categoría | Categoría sin negocios publicados en destino objetivo | Destino | `/oriente-maya/:destino` | `category_not_in_destination` |
| Destino (sólo) | — | Destino | `/oriente-maya/:destino` | `ok` |
| Sin contexto útil | — | Destino | `/oriente-maya/:destino` | `ok` |
| Destino objetivo no existe | — | Región | `/oriente-maya` | `destination_not_found` |

`degraded` = `true` cuando `kept` es más superficial que `requested`. La
UI (o Alux) puede leer `reason` para explicar el cambio al usuario sin
inventar copy propio por ruta.

## 5. Casos de prueba (contrato ejercido, sin regresión SEO)

| # | Escenario | `from` de entrada | Comportamiento esperado | Verifica |
|---|---|---|---|---|
| C1 | Destino → mismo tipo de categoría | `{ destination: "valladolid", category: "hoteles" }` → `to: "izamal"` | `kept="category"`, path `/oriente-maya/izamal/hoteles` si Izamal tiene ≥1 hotel publicado | Blueprint § "Cambio Valladolid → Izamal conservando categoría" |
| C2 | Empresa equivalente | `{ destination: "valladolid", category: "hoteles", business: "hacienda-selva-maya" }` → `to: "espita"` | Degrada a categoría si la empresa no opera en Espita, con `reason="business_not_in_destination"` | Regla dura de degradación un-paso |
| C3 | Producto equivalente | `{ destination, category, business, product }` con producto real | `kept="product"`, URL canónica territorial completa | Preservación máxima |
| C4 | Ausencia de equivalencia total | Categoría inventada `x-y-z` | Degrada a destino, `reason="category_not_in_destination"` | Sin páginas arbitrarias |
| C5 | Deep-link | Copiar/pegar la URL resultante de C1/C3 en pestaña nueva | Renderiza directamente sin depender del switcher | Rutas resultantes ya validadas por `resolveTerritorialPath` (N2.1) y `head()` canónico (N2.2/N2.3) |
| C6 | Legacy sin URL territorial | Página `/marketplace/hacienda-selva-maya` con `from` explícito derivado del loader | Switcher no compone URL a mano; server fn devuelve `/oriente-maya/valladolid/hoteles/hacienda-selva-maya` | Reutiliza contrato; sin lógica por ruta |
| C7 | Destino inexistente | `to: "atlantis"` | `kept="region"`, path `/oriente-maya`, `reason="destination_not_found"` | Blindaje ante datos corruptos |

## 7. Contrato aplicado (regla no negociable)

- Toda URL producida por el switcher se compone **exclusivamente** con
  `resolveCanonicalPath()` — cero interpolación manual en cliente o
  servidor.
- El componente no conoce rutas específicas; el nivel preservado se
  calcula server-side y se transporta como enum (`kept`, `reason`).
- Los campos reservados de Navigation Intelligence (`motivation`,
  `temporal`, `profile`) permanecen sin consumo en la ola actual.

## 8. Demo Pack N2.4

**Ámbito:** verificación funcional del contrato de continuidad usando
datos publicados existentes en Lovable Cloud (mismos que N2.1/N2.2).

Invocaciones representativas del server fn (equivalen a los C1–C7):

```jsonc
// C1 · Conserva categoría al cambiar de destino
POST resolveDestinationSwitch
{ "to": "izamal", "from": { "destination": "valladolid", "category": "hoteles" } }
// → { path: "/oriente-maya/izamal/hoteles", kept: "category", degraded: false, reason: "ok" }
//   (si Izamal no tiene hoteles publicados: kept="destination", reason="category_not_in_destination")

// C2 · Degrada empresa → categoría
POST resolveDestinationSwitch
{ "to": "espita", "from": { "destination": "valladolid", "category": "hoteles", "business": "hacienda-selva-maya" } }
// → kept ∈ {"category","destination"} · reason="business_not_in_destination"

// C3 · Preserva producto completo (mismo destino)
POST resolveDestinationSwitch
{ "to": "valladolid", "from": { "destination": "valladolid", "category": "restaurantes", "business": "cocina-de-dona-elsa", "product": "<slug-producto-real>" } }
// → kept="product" · path territorial completo

// C7 · Destino inexistente
POST resolveDestinationSwitch
{ "to": "atlantis", "from": { "destination": "valladolid", "category": "hoteles" } }
// → { path: "/oriente-maya", kept: "region", degraded: true, reason: "destination_not_found" }
```

**URLs congeladas de referencia** (mismas que N2.2, no requieren datos nuevos):

- `/oriente-maya/valladolid/hoteles`
- `/oriente-maya/valladolid/hoteles/hacienda-selva-maya`
- `/oriente-maya/valladolid/restaurantes/cocina-de-dona-elsa`
- `/marketplace/hacienda-selva-maya` (canonical territorial)
- `/producto/<slug-producto-real>` (canonical territorial)

**Retención de datos demo:** conservar hasta que el Founder diga
literalmente “Demo validada. Puedes eliminar los datos temporales.”

## 9. Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| Degradación silenciosa confunde al usuario | Toast explicativo con copy basado en `reason`; enum estable listo para Alux. |
| Lookup a BD en cada cambio | Máximo 3 queries indexadas (`destinations`, `businesses`, `products`/`business_categories`) sin joins pesados. Cachable en capa superior si se requiere. |
| Datos incompletos (empresa sin categoría) | El switcher lo trata como "no resoluble a nivel empresa" y degrada — no rompe. |
| Componente sin mount aún | Explícitamente diferido a N2.5 con Playwright, según objetivo "una historia a la vez". |
| Rutas legacy sin `from` derivable de URL | Componente exige `from` explícito en esos casos; documentado en el JSDoc. |
