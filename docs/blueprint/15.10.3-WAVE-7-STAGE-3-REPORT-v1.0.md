# Reporte 15.10.3 — Experience Builder & Visual CMS · Etapa 3 (Migración de la Home pública)

**Versión:** v1.0  
**Estado:** entregado, pendiente de cierre formal  
**Etapa:** 15.10.3 — Public Home Migration  
**Predecesores:** 15.10.1 (Foundations) ✅ · 15.10.2 (Studio v0) ✅  
**Sucesor:** 15.10.4 (preparación pendiente)

## 1. Alcance entregado

La Home pública (`/`) deja de depender de código hardcodeado y pasa a
servirse desde una **composición publicada** del Experience Builder,
conservando el principio CMS First, Canvas Agnóstico, Layout Engine y
Page-Type Agnostic aprobados en 15.10.1 / 15.10.2.

Se entregan exclusivamente las capacidades aprobadas:

- Estado `published` en `page_compositions` con índice único parcial
  por `(page_type, variant_key)`.
- RPCs auditadas:
  - `eb_publish_composition` — admin only, registra `Composition.Published`.
  - `eb_unpublish_composition` — admin only, registra `Composition.Unpublished`.
  - `eb_get_published_home` — read-only, accesible por `anon`/`authenticated`,
    devuelve únicamente filas con `status='published'` y `page_type='home'`.
- Server functions:
  - `publishComposition`, `unpublishComposition` (autenticadas, validan rol
    en la RPC).
  - `getPublishedHomeComposition` (pública, falla cerrada → fallback al
    Home legacy).
- Renderer Universal con mapa de producción que envuelve los 9
  componentes reales del Home (`Hero`, `DestinosSection`, `CategoriasSection`,
  `RutasSection`, `ConsejoAluxSection`, `ArmaTuViajeSection`,
  `EnVivoSection`, `EmpresasSection`, `ResenasSection`).
- Ruta `/` con loader que prefetch-ea la composición publicada (SSR) y
  cae al `<LegacyHome />` cuando no existe composición publicada o la
  lectura falla.
- Studio v0 amplia su barra con `Publicar` / `Despublicar` visibles solo
  para administradores; los editores conservan `Crear revisión` / borrador.

## 2. Ampliaciones arquitectónicas aplicadas

1. **Home Composition Profile** — la Home solo es una composición de
   bloques almacenada por el Experience Builder; el route handler no
   contiene lógica de presentación fuera del Renderer Universal.
2. **Progressive Migration** — cualquier bloque puede activarse o
   revertirse sin afectar el sitio: si la composición se despublica,
   `/` cae automáticamente al Home legacy hardcodeado.
3. **Future Personalization Ready** — la lectura pública acepta el
   parámetro `variant_key` (por ahora siempre `'default'`), y el
   Renderer Universal es indistinguible para Smart Blocks futuros.
4. **Composition Variants Ready** — el modelo agrega la columna
   `variant_key` y el índice único parcial sobre `(page_type, variant_key)`,
   habilitando futuras variantes por idioma, país, campaña, sesión,
   perfil de viajero o sugerencias de Alux **sin rediseñar el
   Experience Builder**. No se implementa selección de variantes en
   esta etapa (siempre `default`).

## 3. Cambios en base de datos

`supabase/migrations/<stamp>_public_home_migration.sql` — agrega:

- `page_compositions.status` ampliado con `published`.
- Columnas: `published_at`, `published_by`, `variant_key` (`DEFAULT 'default'`).
- Índices: `idx_page_compositions_unique_published`, `idx_page_compositions_status_page_type`.
- RPCs `SECURITY DEFINER`: `eb_publish_composition`, `eb_unpublish_composition`,
  `eb_get_published_home`.

RLS sin cambios estructurales: las nuevas RPCs delegan en `has_role()` y
`eb_get_published_home` filtra a `status='published' AND page_type='home'`,
por lo que la exposición pública es mínima y verificable.

## 4. Cambios en código

- `src/lib/experience-builder/public-reads.functions.ts` — nuevo.
- `src/lib/experience-builder/studio.functions.ts` — añade
  `publishComposition`, `unpublishComposition`.
- `src/lib/experience-builder/composition-renderer.tsx` — separa mapa
  Studio (previews neutrales) del mapa de producción (componentes
  reales del Home). Mantiene `studio=true` overlays.
- `src/routes/index.tsx` — Home pública con loader + fallback a
  `<LegacyHome />`.
- `src/routes/_authenticated/cms/experience-builder.tsx` — barra de
  publicación admin-only y badge de estado en el Canvas.

## 5. Eventos BEA emitidos

- `Composition.Published` (admin) — desde `eb_publish_composition`.
- `Composition.Unpublished` (admin) — desde `eb_unpublish_composition`.
- `Composition.PublishRejected` (cualquier rol no-admin) — desde el
  bloqueo de autorización en ambas RPCs.
- Se preservan los eventos previos: `Composition.Created`,
  `Composition.DraftSaved`, `Composition.RevisionPublished`,
  `Composition.RevisionRestored`.

## 6. Exclusiones (respetadas)

- No se migran otras rutas públicas (Marketplace, Portal, Concierge,
  Destinos, etc.).
- No se implementan Templates, Smart Blocks v1/v2, Responsive Builder,
  A/B testing, in-place editing, multi-Home ni soft delete.
- No se utiliza `SUPABASE_SERVICE_ROLE_KEY` para servir contenido público.
- Alux sigue siendo read-only; no hay nuevos endpoints de escritura.

## 7. Compatibilidad

- Marketplace, Portal Empresarial, Concierge, Alux y Olas 1–6: sin
  regresiones (los componentes públicos reutilizados son los mismos del
  Home actual, envueltos por el Renderer Universal).
- CMS Studio existente: intacto; el Experience Builder añade
  capacidades pero no toca rutas previas.
- SEO / CWV: paridad 1:1 esperada, ya que en producción el Renderer
  invoca los mismos componentes que servía la Home anterior.

## 8. Validación post-implementación recomendada

1. Crear una composición `page_type='home'`, añadir los 9 bloques del
   Home y publicar (admin).
2. Verificar paridad visual contra la Home legacy.
3. Despublicar y confirmar el fallback automático al Home legacy.
4. Verificar registro de eventos en `content_audit_log` para
   `Composition.Published` / `Composition.Unpublished` /
   `Composition.PublishRejected`.
