## 1. Objetivo

Habilitar la administración desde el Portal Empresarial de los datos de
presencia de cada empresa:

1. Contactos (`business_contacts`): teléfonos, WhatsApp, correos, web.
2. Ubicaciones (`business_locations`): direcciones y geocoordenadas.
3. Horarios (`business_hours`): franjas por día y días cerrados.
4. Redes sociales (`business_social_links`): enlaces a perfiles.

Cada operación pasa por server functions con `requireSupabaseAuth` +
`has_business_access('editor')`, whitelist estricta y auditoría.

## 3. Entregables técnicos

### 3.1 Migración

- `public.log_business_presence_audit(_business_id uuid, _action text, _notes text)` —
  `SECURITY DEFINER`, `search_path = public`, `EXECUTE` restringido a
  `authenticated`. Valida:
  - `auth.uid()` no nulo.
  - `_action` dentro del conjunto cerrado de acciones de presencia
    (`contact.*`, `location.*`, `hours.*`, `social.*`).
  - `has_business_access(_uid, _business_id, 'editor')`.
  - Inserta en `content_audit_log` con `entity_kind='business'` y
    truncado seguro de notas a 1000 caracteres.
- 0 cambios en tablas, columnas, índices, triggers ni políticas RLS.

### 3.2 Server functions (TanStack)

`src/lib/portal/business-presence.functions.ts`:

| Dominio | Funciones | Notas |
|---|---|---|
| Contactos | `listBusinessContacts`, `createBusinessContact`, `updateBusinessContact`, `deleteBusinessContact` | Whitelist: `contact_type`, `value`, `label`, `is_public`, `sort_order`. Tipos restringidos: phone, whatsapp, email, website, other. Soft-delete. |
| Ubicaciones | `listBusinessLocations`, `createBusinessLocation`, `updateBusinessLocation`, `deleteBusinessLocation` | Whitelist: `label`, `address_line1`, `address_line2`, `postal_code`, `latitude` (-90..90), `longitude` (-180..180), `destination_zone_id`, `is_primary`. Soft-delete. |
| Horarios | `listBusinessHours`, `createBusinessHour`, `updateBusinessHour`, `deleteBusinessHour` | `day_of_week` 0..6, validación HH:MM(:SS), exige open/close si `is_closed=false`. Delete físico (registro operativo). |
| Redes | `listBusinessSocialLinks`, `createBusinessSocialLink`, `updateBusinessSocialLink`, `deleteBusinessSocialLink` | Plataformas restringidas. URL exige protocolo http/https. |

Todas las server fns:

- Usan `.middleware([requireSupabaseAuth])`.
- Verifican explícitamente acceso (`viewer` para listados,
  `editor` para mutaciones) vía `has_business_access`.
- Scope-tijean por `business_id` en cada WHERE (defensa en profundidad).
- Filtran `deleted_at IS NULL` donde aplica.
- Llaman a `log_business_presence_audit` tras cada mutación.

## 7. Integridad Referencial

- Todas las operaciones scope-tijean por `business_id`; FKs
  (`business_contacts.business_id`, `business_locations.business_id`,
  `business_hours.business_id`, `business_social_links.business_id`)
  permanecen estables.
- `destination_zone_id` en ubicaciones se valida como UUID opcional;
  cualquier inconsistencia con FK la rechaza Postgres.
- Auditoría usa `entity_kind='business'` + `entity_id = businesses.id`
  conforme al contrato establecido en Fase 1 / Ola 1.

## 8. Mitigación de riesgos

| Riesgo | Mitigación |
|--------|-----------|
| RW3-1 Service role | No se usa. RPC `SECURITY DEFINER` con `search_path` fijo y EXECUTE restringido a `authenticated`. |
| RW3-2 Permisos cross-empresa | Doble check (`has_business_access('editor')`) + WHERE `business_id` en cada mutación. |
| RW3-3 Escalada a publicación | El Portal nunca toca `status`, `verified` ni `published_at` de presencia (no aplican) y no expone transiciones editoriales aquí. Aprobación/publicación de la ficha pública siguen reservadas al CMS (Etapa 3). |
| RW3-4 Campos ocultos al cliente | Whitelist explícita en validadores; auditoría sólo registra el dominio y notas seguras. |
| RW3-5 URLs maliciosas | Validador `urlOrThrow` exige http/https y longitud máxima 500. |
| RW3-6 Geocoordenadas inválidas | Validación de rango -90..90 / -180..180. |
| RW3-7 Horarios inconsistentes | Validación de formato HH:MM(:SS) y de presencia obligatoria cuando no es día cerrado. |
| RW3-8 Auditoría faltante | RPC dedicada para owners/editores; conjunto de acciones cerrado para evitar inyección de eventos arbitrarios. |
| RW3-9 Selector de empresa | El cambio de empresa activa invalida queries vinculadas al business previo (`route.tsx` Etapa 1). |
