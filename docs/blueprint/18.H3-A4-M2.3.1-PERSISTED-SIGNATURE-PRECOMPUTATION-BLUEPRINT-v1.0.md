## 0. Contexto medido (M2.3)

- Warm p95 = 1 ms · cache hit ratio = 99% · single-flight OK · fallback seguro OK.
- **Cold path total p95 = 478 ms** (dominado por `signMs ≈ 475 ms`, red a Storage).
- Cache LRU es **per-worker en memoria**: se pierde en cada cold start, no se comparte entre isolates y no sobrevive despliegues. En un runtime edge con fan-out de workers, el "cold path" no es un evento raro sino recurrente por isolate.

El bloqueo no es "firmar más rápido" — la latencia de red a Storage no es negociable — sino **evitar firmar en el camino crítico del render**.

### Recomendación: **Opción B — tabla server-only `media_asset_signed_urls`**

Razones:

1. **Separación de credenciales y metadatos.** `media_asset_variants` la leen el resolver, el pipeline y (a futuro) auditoría. Mezclar una credencial temporal amplía superficie de exposición y complica RLS (una policy laxa en variants filtraría firmas).
2. **RLS estricto y auditable.** Deny-all a `anon` y `authenticated`; sólo `service_role` (scheduler) y la server fn `evaluateMediaSourceShadow` pueden leerla.
3. **Ciclo de vida propio.** Truncar/reconstruir sin tocar variantes. Compatible con `is_current`/`status` sin duplicar semántica.
4. **Rollback trivial.** Drop de tabla + apagar scheduler; el read path vuelve a M2.3 (LRU + on-demand).
5. **Compatibilidad con contrato público.** No expone URLs derivadas como canónicas.

Opción A queda como *fallback de diseño* si el Founder prefiere no crear tabla. El resto se descartan por costo (C), riesgo de contrato (D), latencia residual (E) o incumplimiento de gate (F).

## 2. Modelo de seguridad

**Postulado:** la URL firmada es una **credencial temporal**, no un dato de contenido.

- **Prohibido:** exponerla vía Data API pública, RLS `TO anon`, logs (`console.log`, telemetría, Sentry breadcrumbs), respuestas de error, headers de debug o cualquier server fn no autorizada.
- **Creación:** sólo el scheduler server-only (`service_role`) o la server fn interna `refreshMediaSignedUrl` (autorizada por `MEDIA_SHADOW_INTERNAL_SECRET` + `requireSupabaseAuth` + rol admin).
- **Lectura:** sólo `evaluateMediaSourceShadow.server.ts` (preview + secreto interno + asset allowlisted).
- **Renovación:** scheduler; nunca desde request path del consumidor.
- **Eliminación:** `ON DELETE CASCADE` desde `media_asset_variants`; purga adicional de filas con `refresh_after < now() - 24h`.
- **Impacto de filtración:** permite descargar la variante durante la vigencia restante (máx 7 días). No permite modificar, no expone otras variantes, no otorga acceso a originales inmutables.
- **Revocación real:** rotar `variant_key` → firma queda huérfana → scheduler la reemplaza. Revocación inmediata: `DELETE` + rotación. Storage no permite revocar una firma emitida; el TTL de 7 días es el límite superior tras filtración.
- **`service_role`:** jamás persistido en tabla ni en logs. Vive sólo como env var del worker del scheduler.

## 3. Contrato de persistencia

Tabla propuesta (**no ejecutar**):

```sql
-- DISEÑO — no ejecutar.
create table media_asset_signed_urls (
  variant_id       uuid primary key references media_asset_variants(id) on delete cascade,
  variant_key      text not null,
  signed_url       text not null,
  issued_at        timestamptz not null,
  expires_at       timestamptz not null,
  refresh_after    timestamptz not null,   -- expires_at - 24h
  renewal_state    text not null default 'idle',  -- idle | renewing | failed
  renewal_lock_at  timestamptz,
  last_error       text,
  last_error_at    timestamptz,
  updated_at       timestamptz not null default now()
);
create index on media_asset_signed_urls (refresh_after) where renewal_state <> 'renewing';
create index on media_asset_signed_urls (variant_key);
```

**Reglas de uso obligatorias:**

1. Una fila **no es válida** si `variant_key` no coincide con el actual del variant (`is_current=true`, `status='ready'`).
2. Una fila **no es válida** si `now() >= refresh_after` (margen ≥ 300 s vs `expires_at`, alineado con `MIN_REMAINING_SECONDS` de M2.3).
3. Clock skew: comparar siempre en UTC; margen incluye tolerancia ±60 s.
4. Atomicidad de renovación: `UPDATE ... WHERE renewal_state='idle' RETURNING` para adquirir lock; `renewal_lock_at` con TTL de 60 s para liberar locks huérfanos.
5. `variant_key` = huella determinista completa (source_checksum + engine + transform params + width + format). Cualquier cambio invalida la firma.

## 4. Renovación asíncrona

La firma expira: **no puede** depender del momento de derivación.

- **Scheduler:** cron server-side (pg_cron o TanStack route `/api/public/hooks/media-signature-refresh` con `apikey`) cada **15 min**.
- **Query:** `WHERE refresh_after <= now() + interval '30 min' AND renewal_state='idle' ORDER BY refresh_after LIMIT 100`.
- **Renovación anticipada:** `refresh_after = expires_at - 24h`.
- **Lotes:** ≤100 firmas por corrida; paralelismo interno ≤10.
- **Locks/single-flight:** adquisición atómica; `renewal_lock_at` con TTL 60 s.
- **Reintentos:** backoff exponencial 1→2→4→8→16 s (5 intentos); tras 5 fallos, `renewal_state='failed'` + `last_error`.
- **Alertas:** `media_signed_url_failed_count > 0` durante > 30 min; `media_signed_url_stale_count > 10` durante > 15 min.
- **Recuperación tras caída prolongada:** reanudar priorizando `expires_at` más cercano; firmas ya expiradas se regeneran desde cero.
- **Scheduler caído:** el read path detecta `now() >= refresh_after` y cae al nivel 2 (LRU), luego 3 (on-demand). Servicio no se degrada; pierde beneficio de latencia hasta reanudación.

## 5. Jerarquía de fallback

Orden estricto:

1. **Firma persistida válida** (`variant_key` matchea, `now() < refresh_after`).
2. **LRU per-worker de M2.3** (warm cache).
3. **Firma on-demand** (M2.3 cold path, ≤500 ms).
4. **Legacy `file_url`** (contrato público estable, siempre disponible).

**Reglas:**

- Nunca servir firma con `now() >= expires_at - 60s`.
- Nunca servir firma cuyo `variant_key` no matchea el actual.
- Si nivel 3 tarda > 250 ms y el flag "prefer legacy on cold" está activo, se sirve nivel 4.
- Cualquier excepción en niveles 1-3 cae silenciosamente a nivel 4; nunca se propaga al cliente.

---

## 6. Ciclo de vida

- **`superseded` / `failed` / `is_current=false`:** el scheduler los excluye. Firmas se purgan tras 24 h.
- **Retiro de asset:** `ON DELETE CASCADE` elimina la firma. Sin huérfanas.
- **Rollback:** apagar scheduler → firmas expiran naturalmente en 7 d → read path cae a niveles 2-4. Drop de tabla reversible.
- **Compatibilidad UNIQUE legacy:** intacta; la tabla no toca `media_asset_variants`.
- **Sin orphan-sweeper. Sin PASO 4. Sin backfill general.**

## 9. Plan de validación

Escenarios obligatorios antes de recomendar GO de implementación:

1. Worker frío + firma persistida vigente → cold total ≤ 100 ms.
2. Firma vigente (warm) → hit p95 ≤ 15 ms.
3. Firma próxima a expirar (`refresh_after` dentro de 5 min) → scheduler la renueva antes; ninguna request cae a nivel 3.
4. Firma expirada → nunca se sirve; cae a nivel 2 y luego 3.
5. Scheduler detenido 2 h → degradado pero funcional; cero errores al cliente.
6. Storage caído → niveles 1-2 sirven mientras haya firmas vigentes; nivel 4 cubre el resto.
7. DB caída → nivel 1 falla, nivel 2 sirve, nivel 3 intenta, nivel 4 cubre.
8. Cambio de `variant_key` → firma persistida inválida; scheduler la reemplaza; read path detecta mismatch y cae.
9. 50 solicitudes concurrentes con firma persistida → 50 hits nivel 1, cero llamadas a Storage.
10. Lectura desde cliente `anon` y `authenticated` a `media_asset_signed_urls` → error de permiso, cero filas.
11. Rollback completo → drop tabla + unschedule; read path idéntico a M2.3.
