# Épica E6 · Related Collection · Completion Report v1.0

**Programa:** Discovery Layer · **Carril A** (Producto)
**Versión objetivo:** v2.5 · **Fecha de cierre:** 2026-07-05
**Dependencias:** E5 (Traveler Public Profile), H-03 Ola I3.b (bloque `vmx.experience.related-collection`), N-Blueprint v1.0

## 1. Objetivo

Habilitar la **capa editorial de overrides** (pins/hides) sobre la Related Collection ya existente (H-03 · I3.b), y dejar preparada la fuente de datos que consumirá E7 (`Recommendation Engine`) sin refactor del bloque.

La Related Collection **como bloque de presentación ya existía** desde H-03 · I3.b (contrato universal `vmx.experience.related-collection` v1.0.0 con soporte multi-fuente `manual | destination | region | category | business | product | context | alux`). E6 no crea un bloque nuevo — evoluciona el sistema con la capa editorial que faltaba.

## 2. Alcance entregado

### 2.1 Backend (una migración)

- **Tabla `public.related_overrides`** con enums `related_override_mode` (`pin | hide`) y `related_entity_kind` (`business | product | destination | event`). Índices por (`entity_type`,`entity_id`,`surface`) y por relacionado. Unicidad `(entity_type, entity_id, surface, related_entity_type, related_entity_id)`.
- **RLS**: lectura pública, escritura restringida a `super_admin | admin | editor` vía `has_role`. GRANTs correctos para `anon`, `authenticated`, `service_role`. Trigger `set_updated_at`.
- **RPC `related_get_collection(entityType, entityId, surface, context jsonb, limit)`** — SECURITY DEFINER, aplica reglas automáticas por tipo (misma categoría/destino/región/tags), respeta hides, antepone pins, con **fallback populares del destino activo** cuando no hay coincidencias suficientes. Devuelve `{ items, strategy, rationale, context }`. Contrato E7-ready.

### 2.2 Server functions (`src/lib/related/overrides.functions.ts`)

- `getRelatedOverrides` (público, publishable client) — usado por fetchers de superficie.
- `listOverridesCms`, `upsertOverride`, `deleteOverride` (`requireSupabaseAuth` + `is_editor_or_admin`).
- Helpers puros `computeHiddenIds`, `pinnedIdsFor` para consumo por otros fetchers.

### 2.3 Integración en fetchers de superficie

- `src/lib/catalog/business-related.functions.ts`: aplica **hide + pin** contra las 2 buckets (`sameCategory`, `sameDestinationOther`).
- `src/lib/catalog/product-related.functions.ts`: aplica **hide + pin** contra `sameCategoryInDestination` y filtra hides de `otherInDestination`.
- No hay regresión: cuando no hay overrides el comportamiento es idéntico al anterior.

### 2.4 Admin UI (`/cms/relacionados`)

- Página `src/routes/_authenticated/cms/relacionados.index.tsx`.
- Formulario para crear override (upsert idempotente por unicidad), lista filtrable por tipo/uuid, eliminación inline.
- La superficie se asigna automáticamente por tipo de ficha (`business-profile`, `product-detail`, `destination-detail`, `event-detail`).

## 3. Contrato Provider (E7-ready)

```
getRelatedCollection({
  entityType, entityId, surface,
  context?: { activeDestinationId?, localeId?, season?, ... },
  limit?: number,
}) → { items, strategy, rationale?, context }
```

El RPC `related_get_collection` implementa este contrato desde v1. E7 podrá reemplazar la implementación (scoring, campañas, señales de traveler, Alux) sin cambiar la firma ni tocar el bloque `vmx.experience.related-collection`. Los overrides seguirán siendo la capa editorial que ninguna estrategia debe ignorar.

## 4. Reglas automáticas activas (por tipo)

| Ficha | Regla primaria | Fallback |
|---|---|---|
| Empresa (`business-profile`) | `same_category + same_destination` | `popular-destination` |
| Producto (`product-detail`) | `same_category + same_destination` (categoría heredada de la empresa) | `popular-destination` |
| Destino (`destination-detail`) | `same_region` | items del tipo cuando la región es NULL |
| Evento (`event-detail`) | `same_destination` | `popular-destination` |

Los pins editoriales sobrescriben cualquier orden natural (posición manual) y los hides eliminan ítems del output.

## 5. Matriz de reutilización

| Superficie | Bloque EB | Fetcher | Overrides pin | Overrides hide | Fallback populares |
|---|---|---|---|---|---|
| Empresa | `vmx.experience.related-collection` (I3.b) | `getBusinessRelated` | ✅ | ✅ | ✅ (RPC / heredable) |
| Producto | mismo | `getProductRelated` | ✅ | ✅ | ✅ (RPC / heredable) |
| Destino | mismo | `getDestinationRelated` | ✅ (E6.b) | ✅ (E6.b) | ✅ (RPC) |
| Evento | mismo | `getEventRelated` (E6.b) | ✅ (E6.b) | ✅ (E6.b) | ✅ (mismo destino → global) |

La infraestructura backend (tabla + RPC + admin fns + admin UI) cubre las 4 superficies. La **conexión pin/hide dentro de los fetchers** se entregó para empresa y producto (los dos surfaces de mayor volumen y con arquitectura de fetcher ya alineada). Destino y evento consumirán la misma capa en la sub-ola **E6.b** (~1 sesión, sin migración adicional).

## 7. Demo Pack

Estado: **infra funcional, seeds mínimos**.

Ruta de gestión: `/cms/relacionados` (rol `editor` o superior).

Pasos de reproducción manual (los datos actuales no tienen 2 empresas en el mismo destino+categoría, así que el pin editorial se aprecia mejor sembrando 2 negocios hermanos):

1. Crea 2 empresas publicadas en el mismo destino (por ejemplo Valladolid) y misma categoría desde `/cms/empresas`.
2. Entra a `/cms/relacionados` y crea un override `mode=pin` sobre la ficha de una, apuntando a la otra.
3. Abre la ficha pública (`/oriente-maya/…/empresas/<slug>`) — la Related Collection anteponer el ítem fijado.
4. Crea un `mode=hide` para verificar que el ítem desaparece de la colección.

La sub-ola E6.b añadirá **datos demo semilla + integración destino/evento + pestaña embebida en cada Inspector** para acortar el flujo.
