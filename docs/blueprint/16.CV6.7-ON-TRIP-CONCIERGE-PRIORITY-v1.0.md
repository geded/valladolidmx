# CV6.7 · On-Trip Concierge Priority — Blueprint oficial v1.0

Ola CV6 · Live Destination Companion · Sub-ola 7
Fase objetivo: `onsite` (con extensión a `confirmed` en ventana T-3).

---

## 1. Directiva del Founder (vinculante)

CV6.7 **no** se implementa como un banner del Concierge. Es la **primera capa del Travel Assistance Layer** de Alux.

El objetivo no es sólo dar acceso prioritario al Concierge durante la fase On-Trip, sino **establecer la arquitectura base** sobre la cual evolucionarán todas las futuras capacidades de asistencia al viajero.

### Políticas oficiales (7)

1. **Single Entry Point.** Toda interacción de asistencia durante el viaje se origina desde este contrato. Prohibidos flujos paralelos para: Concierge, Chat, SOS, Concierge Premium, IA Colaborativa, Seguimiento de incidencias, Asistencia contextual. Toda evolución futura extiende este mismo contrato.
2. **Pure Domain.** `deriveOnTripConciergeState()` es el contrato oficial del dominio: puro, determinista, sin efectos secundarios, fácilmente testeable, reutilizable por cualquier superficie futura. Toda decisión pertenece al derivador — nunca al componente visual.
3. **Stateless UI.** `OnTripConciergePriorityBanner` es 100% presentacional (props → render). No puede consultar APIs, abrir stores, iniciar polling, crear timers, ejecutar mutaciones ni contener reglas de negocio.
4. **Strict Reuse.** CV6.7 reutiliza EXCLUSIVAMENTE `LiveDayContext`, `promotePlanToCase`, `concierge_orders` y CV6.1–CV6.6. Prohibido crear nuevos Providers, Contexts, consultas al CMS/Contributors o acceso directo a APIs externas.
5. **Explainable by Default.** Toda superficie explica por qué aparece, cuál es el SLA vigente, cuál es el estado del Concierge y qué ocurrirá al pulsar el CTA.
6. **Evolution without Refactoring.** El contrato admite (sin refactor): Chat en vivo, Voz, Llamadas, WhatsApp, Premium, SOS, Incidencias, IA colaborativa, Automatizaciones, Seguimiento. La evolución extiende — no crea una segunda arquitectura.
7. **Travel Assistance Layer Principle** (nuevo · registrado en `mem://policies/travel-assistance-layer.md`): toda asistencia al viajero durante su estancia se implementa mediante un único contrato de dominio reutilizable, un único punto de entrada y una única experiencia de usuario. Vinculante para todas las futuras épicas de asistencia al viajero.

## 2. Alcance funcional CV6.7 (mínimo viable de la capa)

La capa se estrena mostrando **una única capacidad activa**: acceso prioritario al Concierge humano durante el viaje. Todas las demás capacidades quedan **declaradas en el contrato** pero desactivadas hasta futuras sub-olas.

### Superficies iniciales
- **NowNextLaterSurface** (fase `onsite`) — banda superior antes del grid de decisiones.
- **LiveDayBoard** (opcional futura) — misma componente, mismos props.

### Estados del dominio (`AssistanceState`)
| Estado | Cuándo | CTA | Reversible |
|---|---|---|---|
| `hidden` | phase ≠ onsite / T-3 · o sin plan · o sin utilidad | — | — |
| `standby` | Viaje confirmado, sin caso abierto, dentro de horario laboral | "Contactar Concierge" | sí |
| `case_open` | Caso vigente (`travel_plans.case_id` presente) | "Ver expediente" | sí |
| `sla_breach` | Caso vigente + SLA vencido (futuro) | "Escalar" | sí |
| `sos` | Reservado para CV6.8+ | — | — |

### Capacidades declaradas (contrato `AssistanceChannel[]`)
`concierge` (activo) · `chat` · `voice` · `call` · `whatsapp` · `sos` · `incident` · `ai_collab` · `automation` · `case_tracking` (todos declarados, inactivos en CV6.7).

## 3. Contratos técnicos

### 3.1 Dominio puro (`src/lib/traveler/on-trip-concierge.ts`)
```ts
export type AssistanceState =
  | "hidden" | "standby" | "case_open" | "sla_breach" | "sos";

export type AssistanceChannel =
  | "concierge" | "chat" | "voice" | "call" | "whatsapp"
  | "sos" | "incident" | "ai_collab" | "automation" | "case_tracking";

export interface OnTripConciergeState {
  state: AssistanceState;
  visible: boolean;
  headline: string;
  rationale: string;      // por qué aparece
  slaLabel: string;       // SLA vigente
  conciergeStatus: string;// estado operativo del Concierge
  ctaLabel: string;
  ctaIntent: "open_case" | "view_case" | "escalate" | "none";
  ctaOutcome: string;     // qué ocurrirá al pulsar
  reversible: boolean;
  channels: AssistanceChannel[]; // capacidades declaradas (activas + futuras)
  activeChannels: AssistanceChannel[]; // subset habilitado hoy
  explain: { sources: string[]; rules: string[] };
}

export function deriveOnTripConciergeState(input: {
  phase: TripPhase;
  liveDay: LiveDayContext;
  case?: { id: string; status?: string } | null;
  order?: { status: string } | null;
  at: Date;
  officeHours?: { openHour: number; closeHour: number }; // default 8..22 America/Merida
}): OnTripConciergeState;
```

### 3.2 UI Stateless (`src/components/traveler/OnTripConciergePriorityBanner.tsx`)
```tsx
export function OnTripConciergePriorityBanner(props: {
  state: OnTripConciergeState;
  onPrimary: () => void; // handler inyectado por el consumidor
}): JSX.Element | null;
```
No consulta APIs. No abre stores. No mutaciones. Auto-oculta si `!state.visible`.

## 4. Reutilización estricta

| Fuente | Uso |
|---|---|
| `LiveDayContext` (CV6.1) | Fase operativa · Auto-Hide |
| `DecisionCenter` (CV6.2) | Coexistencia sin duplicar decisiones |
| Contributors (CV6.4/6.5) | NO consumidos directamente |
| `promotePlanToCase` (CV2/CV4) | Handler `onPrimary` cuando aún no hay caso |
| `concierge_orders` (CV4.1) | Presencia del viaje confirmado |
| `travel_plans.case_id` | Detección de caso abierto |

Prohibido: nuevo modelo, nueva tabla, nuevo Provider, nuevo Context, nueva ruta.
