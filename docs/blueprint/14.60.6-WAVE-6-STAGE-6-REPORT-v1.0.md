## 2. Entregables técnicos

### 2.1 Migración aditiva (`20260630_alux_concierge_assistant.sql`)

- **`concierge_alux_context_for_case(_case_id uuid) → jsonb`**
  - `SECURITY DEFINER`, `search_path = public`, `STABLE`.
  - Valida `auth.uid()` + `concierge_can_view_case`.
  - Proyecta `concierge_case_file_v1` y un slice de los últimos 30 eventos
    de `concierge_case_timeline`.
  - `REVOKE` a `public/anon`, `GRANT EXECUTE` a `authenticated`.

- **`concierge_alux_log_suggestion(_case_id uuid, _capability text, _meta jsonb)`**
  - `SECURITY DEFINER`, `search_path = public`.
  - Valida `_capability ∈ { summary, products, proposal_draft, comms_digest, risk_detection, opportunity_detection }`.
  - Inserta `Alux.Suggestion.Generated` en `concierge_case_timeline`.
  - `REVOKE` a `public/anon`, `GRANT EXECUTE` a `authenticated`.

- 0 tablas creadas. 0 columnas modificadas. 0 cambios de RLS. 0 triggers.

### 2.2 Provider Lovable AI Gateway (`src/lib/ai-gateway.server.ts`)

Helper server-only que crea el provider OpenAI-compatible apuntando a
`https://ai.gateway.lovable.dev/v1` con `Lovable-API-Key` desde
`process.env.LOVABLE_API_KEY`.

### 2.3 Server functions (`src/lib/concierge/alux.functions.ts`)

| Server fn | Capability | Notas |
|---|---|---|
| `getAluxContextForCase` | — | Envuelve `concierge_alux_context_for_case`. |
| `logAluxSuggestion` | — | Envuelve `concierge_alux_log_suggestion`. |
| `generateAluxSummary` | `summary` | Resumen del expediente. |
| `generateAluxProducts` | `products` | Productos candidatos basados en el contexto. |
| `generateAluxProposalDraft` | `proposal_draft` | Borrador de propuesta integral. |
| `generateAluxCommsDigest` | `comms_digest` | Síntesis cronológica. |
| `generateAluxRiskDetection` | `risk_detection` | Riesgos operativos (SLA, cotizaciones, pagos, agenda…). |
| `generateAluxOpportunityDetection` | `opportunity_detection` | Promociones, complementarios, mejoras, venta cruzada. |

Reglas comunes:
- `.middleware([requireSupabaseAuth])` en todas.
- El contexto siempre se obtiene vía `concierge_alux_context_for_case`
  (nunca lectura directa a tablas `concierge_*`).
- Cada invocación deja log `Alux.Suggestion.Generated` con
  `{ model, latency_ms, tokens_in, tokens_out }`.
- Modelo por defecto: `google/gemini-3-flash-preview` (Lovable AI Gateway).
- Sin EXECUTE de Alux sobre RPCs de mutación; los textos generados se
  copian o se usan como insumo de los formularios humanos existentes.

### 2.4 UI — `AluxAssistantPanel` (CaseFileView)

- Sólo se renderiza para `viewer.is_internal` y nunca en
  `/_authenticated/cuenta/concierge/$caseId` (gate `hideInternal`).
- 6 botones (uno por capacidad), indicador de modelo y latencia,
  disclaimer fijo *"Sugerencia generada por Alux. Revisa antes de
  enviar."* y botón *Copiar*.
- Las salidas son transitorias (estado local); su persistencia ocurre
  sólo si el concierge las usa vía RPC humana existente
  (`concierge_case_note_add`, `concierge_proposal_create`, etc.).

## 3. Riesgos · mitigación

| ID | Riesgo | Mitigación |
|---|---|---|
| R6.1 | Acciones autónomas de Alux | Server fns no invocan ninguna RPC de mutación; UI sólo copia/prellena. |
| R6.2 | Fuga de PII al modelo | El contexto sale de `concierge_case_file_v1` con visibilidad ya validada por RLS; sin tokens, secretos ni datos de pago. |
| R6.3 | Costos AI descontrolados | Invocación 100 % explícita por botón; sin background jobs ni `pg_cron`. |
| R6.4 | Riesgos/oportunidades convertidos en acciones | Capacidades en modo estrictamente consultivo, prompt-system lo refuerza, no hay rutas que ejecuten desde el output. |
