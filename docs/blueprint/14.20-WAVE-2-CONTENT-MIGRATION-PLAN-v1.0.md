# 14.20 — Wave 2 · Content Migration Plan v1.0

**Versión:** 1.0
**Estado:** Propuesta — pendiente de aprobación documental
**Ola:** 2 — Migración de Contenido
**Serie:** 14 (Phase 2 Foundation)
**Documentos previos:** 14.10.Z (Cierre Ola 1), 14.11 (ADR · Cobertura de Auditoría para `review`)

---

## 1. Resumen ejecutivo

La Ola 2 sustituirá progresivamente los datos de mock (`src/mocks/*`) por contenido real administrado a través del CMS Studio entregado en la Ola 1. La migración se ejecutará en el orden contractual establecido por el documento 14.0: `categorias → destinos → rutas → empresas → reseñas`. Toda inserción se realizará mediante server functions autorizadas o migraciones SQL versionadas, sin uso de `SUPABASE_SERVICE_ROLE_KEY` en flujos editoriales y sin modificar RLS aprobadas.

## 2. Alcance

**Incluye**
- Carga inicial del corpus editorial mínimo para habilitar Marketplace (Ola 4).
- Sustitución progresiva de mocks (`categorias`, `destinos`, `rutas`, `empresas`, `resenas`).
- Mapeo de slugs y reglas de deduplicación.
- Reglas de idempotencia para reejecución segura.
- Validaciones server-side de integridad del contenido.

**Excluye**
- Reescritura del shell público o de la Fase 1.
- Modificaciones a RLS, enums o políticas (salvo las aprobadas vía ADR 14.11).
- Funcionalidades de Marketplace, Concierge, Alux, Travel Plans, Translations o Dashboards.
- Onboarding del Portal Empresarial (Ola 3).

## 3. Corpus de contenido

| Dominio | Fuente | Volumen estimado | Estado destino |
|---------|--------|------------------|----------------|
| `categorias` | `src/mocks/categorias.ts` + curaduría editorial | ~10–15 registros | `published` |
| `destinations` | `src/mocks/destinos.ts` + ficha territorial | ~20–30 registros | `published` |
| `editorial_routes` | `src/mocks/rutas.ts` | ~8–12 registros | `published` |
| `businesses` (semilla editorial) | `src/mocks/empresas.ts` | ~30–50 registros | `published` (sin operador real) |
| `reviews` (semilla controlada) | `src/mocks/resenas.ts` | ~20 registros | `published` |
| `media_assets` asociados | `docs/brand-assets/photography/*` | según corpus | `published` |

El corpus definitivo debe ser confirmado por el área editorial antes del Definition of Ready.

## 4. Estrategia de migración

1. **Preparación documental**
   - Mapa de campos mock → entidad CMS.
   - Tabla de slugs canónicos (ver §7).
   - Inventario de assets a subir a Storage (sin uso de `supabaseAdmin`).
2. **Ejecución por dominio (orden contractual)**
   - `categorias` → `destinations` → `editorial_routes` → `businesses` → `reviews`.
   - Cada dominio se aborda en una sub-etapa con su propio reporte de avance.
3. **Mecanismo de carga**
   - Server functions de creación ya entregadas en Etapa 3 (writes) cuando la operación sea unitaria y revisable.
   - Migración Serie 13 dedicada (idempotente, ver §5) cuando el volumen lo justifique.
   - Prohibido: `supabaseAdmin` para insertar contenido editorial salvo migraciones controladas, documentadas y revisadas.
4. **Sustitución del mock**
   - El import del mock se elimina sólo cuando QA cierra la entidad y el loader público lee desde el dominio real.
   - Hasta entonces, el mock permanece como fuente de respaldo.
5. **Cierre por dominio**
   - Reporte de avance + visado documental antes de pasar al siguiente dominio.

## 5. Reglas de idempotencia

- Toda migración SQL debe poder ejecutarse N veces sin duplicar filas ni alterar estados.
- Patrón canónico: `INSERT ... ON CONFLICT (<clave_natural>) DO NOTHING` o `DO UPDATE` cuando aplique.
- Claves naturales por entidad:
  - `categories.slug`
  - `destinations.slug`
  - `editorial_routes.slug`
  - `businesses.slug`
  - `reviews (user_id, target_id, target_kind)` cuando exista; para semilla editorial documentar excepción.
- Nunca se asignan `id` (uuid) literales: se delega a `gen_random_uuid()` salvo en seeds con `id` derivado determinístico documentado.
- Toda función de carga server-side debe verificar existencia previa antes de insertar.

## 6. Deduplicación

- Detección por slug canónico (slugify NFKD, minúsculas, separador `-`).
- Cuando dos registros mock colisionan en slug: se fusionan manualmente antes de cargar; no se generan sufijos automáticos.
- Para `businesses`: además de `slug`, considerar `(name, destination_id)` como clave de control.
- Para `reviews` semilla: no se permiten múltiples reseñas del mismo usuario sintético sobre el mismo target.

## 7. Mapeo de slugs

- Slug oficial = slug ya presente en el mock cuando cumple convención; en caso contrario se renormaliza.
- Reglas:
  - minúsculas, sin acentos, sin caracteres no `[a-z0-9-]`.
  - colisiones se resuelven antes de la carga; no en runtime.
  - el slug es **estable** una vez publicado; cualquier cambio posterior requiere redirect 301 (fuera del alcance de Ola 2).
- Se entregará una tabla maestra `slug_map.csv` como anexo del reporte de inicio de cada sub-etapa.

## 8. Validaciones

- **Pre-carga**
  - Esquema validado con Zod a nivel de server function.
  - Slug único en el corpus a migrar.
  - Foreign keys resueltas (e.g. `destination_id` existente antes de cargar rutas).
- **Post-carga**
  - Conteo esperado vs conteo real.
  - Verificación de `content_status = 'published'` para el corpus designado.
  - Verificación de que `media_assets` referenciados existen en Storage.
  - Verificación de que no se introdujeron registros con `created_at` en el futuro.

## 9. Rollback

- Cada migración Serie 13 debe entregarse con su contraparte de rollback documentada (no necesariamente ejecutable automáticamente).
- Estrategias permitidas:
  - `DELETE` selectivo por clave natural cargada en la migración.
  - Transición a `archived` cuando el `DELETE` no sea seguro (referencias cruzadas).
- Prohibido `TRUNCATE` sobre tablas de dominio.
- Toda ejecución de rollback debe quedar registrada en `content_audit_log` (o equivalente aprobado por ADR 14.11 para `review`).

## 10. Riesgos

| ID | Riesgo | Severidad | Mitigación |
|----|--------|-----------|------------|
| R-1 | Migración duplica contenido por slug inconsistente | Alta | Tabla maestra de slugs + `ON CONFLICT DO NOTHING`. |
| R-2 | Carga rompe lectura pública por RLS demasiado restrictiva | Alta | Verificación de políticas `TO anon` previa al inicio de cada sub-etapa. |
| R-3 | Uso indebido de `supabaseAdmin` para insertar contenido | Crítica | Revisión obligatoria en code review; bloqueo en QA. |
| R-4 | Reseñas semilla generan ruido reputacional | Media | Marcar reseñas semilla con flag editorial / `metadata.seed = true`. |
| R-5 | Assets faltantes en Storage tras migración | Media | Pre-carga de buckets + verificación post-carga. |
| R-6 | Cambio de slug post-publicación rompe SEO futuro | Media | Slug congelado tras publicación; redirects fuera de alcance. |
| R-7 | Auditoría incompleta para `review` mientras ADR 14.11 no se ejecute | Media | No iniciar sub-etapa de reseñas hasta resolver ADR. |

## 11. Definition of Ready (DoR)

- [ ] ADR 14.11 aprobado y migración asociada planificada (si Alternativa A).
- [ ] Corpus editorial confirmado por el área de contenido.
- [ ] Tabla maestra de slugs entregada.
- [ ] Mapa de campos mock → CMS revisado.
- [ ] Inventario de assets confirmado.
- [ ] Plan de rollback redactado por dominio.
- [ ] Revisión de políticas `TO anon` por entidad pública afectada.
- [ ] Confirmación de que ninguna sub-etapa requiere `SUPABASE_SERVICE_ROLE_KEY` en flujo editorial.

## 12. Definition of Done (DoD)

- [ ] Corpus migrado coincide con el corpus aprobado (conteo y muestreo).
- [ ] Loaders públicos leen desde el dominio real para los mocks sustituidos.
- [ ] Mocks correspondientes eliminados del bundle (imports limpios).
- [ ] Auditoría completa en `content_audit_log` (o destino aprobado por ADR 14.11).
- [ ] Sin nuevas dependencias npm.
- [ ] Sin modificaciones a RLS no aprobadas documentalmente.
- [ ] Reporte de avance entregado por cada sub-etapa y reporte final 14.20.Z al cierre.
- [ ] Fase 1 sin regresiones verificadas.

## 13. Execution Gates

| Gate | Criterio |
|------|----------|
| Gate A — Infraestructura | Sin nuevos buckets, sin nuevos secretos, sin cambios estructurales fuera de migraciones planificadas. |
| Gate B — Dominio | Sin alteraciones a tablas/columnas no contempladas en este plan o en ADR 14.11. |
| Gate C — Identidad y permisos | Cargas mediante server functions autorizadas; sin uso indebido de `supabaseAdmin`. |
| Gate D — CMS Governance | Todo contenido cargado respeta workflow Serie 14; auditoría preservada. |
| Gate E — QA / UAT / Release | Validaciones post-carga ejecutadas y firmadas; mocks retirados sólo tras QA. |
| Gate Visual 12D | El shell público y SiteHeader/SiteFooter permanecen intactos; cualquier ajuste de cards consume tokens semánticos. |
