# H3·A4 · M2.3.1 · Persisted Signature Precomputation — Blueprint v1.2

**Estado:** Diseño · **NO-GO** para implementación hasta aprobación Founder v1.2.
**Alcance:** eliminar cold-path de red (478 ms) precomputando firmas por variante y persistiéndolas en tabla server-only, preservando invariantes de M1/M2.1/M2.2/M2.3. **Sin código, sin esquema, sin cron, sin flags, sin cambios en consumidores.**

**Cambios v1.2 (respuestas al Founder):**
1. Arquitectura completa de emisión de firma (§1) — `pg_cron` es sólo scheduler; el trabajo real corre en endpoint server privado autenticado con HMAC independiente.
2. SQL de selección atómica ejecutable con `FOR UPDATE SKIP LOCKED` (§2).
3. Estado `failed` sin terminal accidental — reintento perpetuo con backoff saturado y alerta (§3).
4. Modelo de invalidación honesto sobre `media-derived` (§4).
5. Gate estadístico corregido: N ≥ 4 000 y Wilson-95 % LB > 0.999 (§5).
6. Margen servible unificado en 6 h (§6).
7. UPSERT condicionado por `variant_key` + `issued_at` (§7).
8. Threat model sin ceros residuales (§8).

## 1 · Arquitectura de emisión de firma

**Principio rector:** el SDK de Supabase Storage no es invocable desde PL/pgSQL. `pg_cron` sólo agenda. El trabajo real corre en un endpoint server privado. El endpoint **no** es consumidor visible: sólo lo llama `pg_net` desde el propio proyecto y sólo escribe la tabla server-only.

### 1.1 Secuencia canónica

```
pg_cron (cada 1 min, advisory lock de sesión)
  └─▶ pg_net.http_post(
        url:  https://project--<id>.lovable.app/api/public/hooks/media-signature-renew
        body: { "batch_hint": 50, "nonce": <uuid>, "issued_at_ms": <epoch> }
        headers:
          x-lovable-media-hmac:  hex(HMAC-SHA256(MEDIA_SIGNATURE_RENEW_HMAC, raw_body))
          x-lovable-media-nonce: <uuid>
          x-lovable-media-ts:    <epoch_ms>
      )

Route handler /api/public/hooks/media-signature-renew (server-only):
  1. Lee flag `media_persisted_signatures_enabled`. OFF ⇒ 204 no-op.
  2. Verifica HMAC (timingSafeEqual) sobre body crudo.
  3. Verifica |now - ts| < 60 s y consume nonce en `media_signature_renew_nonces`
     (INSERT UNIQUE, TTL 10 min). Colisión ⇒ 409 anti-replay.
  4. Import dinámico de `@/integrations/supabase/client.server` dentro del handler;
     process.env.SUPABASE_SERVICE_ROLE_KEY leído sólo aquí.
  5. Reclama lote (§2) → renueva vía storage.from(bucket).createSignedUrl(path, TTL)
     con AbortController(3 s por firma) → UPSERT condicionado (§7).
  6. Devuelve JSON sanitizado { processed, failed, skipped }. NUNCA URLs/tokens/paths/ids.
  7. Timeout duro 25 s por request.
```

### 1.2 Autenticación y secreto

- Secreto: **`MEDIA_SIGNATURE_RENEW_HMAC`** (32 bytes random, `generate_secret`). **Independiente** de `MEDIA_SHADOW_INTERNAL_SECRET`; no se comparten ni se rotan juntos.
- Almacenamiento: Runtime Secret Lovable (equivalente Vault). Nunca en `cron.job.command`, nunca en logs, nunca serializado al cliente.
- Rotación: crear nuevo secreto, dual-verify 24 h, retirar viejo. Cron idempotente ante rotación.

### 1.3 Anti-replay

- Tabla server-only `media_signature_renew_nonces(nonce uuid PRIMARY KEY, issued_at timestamptz, consumed_at timestamptz default now())`.
- `REVOKE ALL FROM public, anon, authenticated`. Sólo service_role.
- Purga: `DELETE WHERE consumed_at < now() - interval '1 hour'` best-effort en cada request.
- Ventana ±60 s en `x-lovable-media-ts` complementa el nonce.

### 1.4 Permisos y superficie

- Handler en `src/routes/api/public/hooks/media-signature-renew.ts`. `/api/public/*` bypasea auth de plataforma; la autorización real es HMAC + nonce + flag.
- `supabaseAdmin` cargado con `await import("@/integrations/supabase/client.server")` **dentro** del handler (regla `tanstack-supabase-import-graph`).
- Respuesta HTTP: sólo contadores agregados.

### 1.5 pg_cron sin secretos en SQL

`cron.job.command` no contiene el HMAC. Lo calcula una función server-only `public.media_signature_renew_tick()` `SECURITY DEFINER` que:

- Lee el secreto desde un GUC de rol (`ALTER ROLE media_renew_role SET app.media_hmac = '...'`) — preferido, para que no viaje en diffs SQL.
- Emite `pg_net.http_post` con headers firmados.
- Purga `pg_net.responses` cada 15 min; cualquier body de respuesta contiene sólo contadores.

**Prohibido:** service-role key en SQL, `cron.job.command`, `pg_net.request/response` o logs de aplicación. Sólo se toca desde `process.env` en runtime Node.

### 1.6 Retorno de la firma

El handler escribe directo a `media_asset_signed_urls` con service-role. **No** viaja de vuelta por `pg_net`. `pg_net.responses` guarda `{ processed, failed, skipped }` y nada más.

## 2 · Selección atómica ejecutable

Postgres no acepta `UPDATE ... LIMIT`. Patrón CTE con `FOR UPDATE SKIP LOCKED`:

```sql
WITH candidates AS (
  SELECT id
  FROM media_asset_signed_urls
  WHERE
    ( renewal_state = 'idle'
      AND expires_at < now() + interval '48 hours' )
    OR
    ( renewal_state = 'failed'
      AND next_retry_at <= now() )
    OR
    ( renewal_state = 'renewing'
      AND lock_acquired_at < now() - interval '5 minutes' )   -- huérfanos
  ORDER BY expires_at ASC NULLS FIRST
  LIMIT 50
  FOR UPDATE SKIP LOCKED
)
UPDATE media_asset_signed_urls u
SET renewal_state    = 'renewing',
    lock_acquired_at = now(),
    lock_owner       = $1                                     -- worker uuid
FROM candidates c
WHERE u.id = c.id
RETURNING u.id, u.variant_key, u.bucket, u.path, u.attempt_count;
```

**Concurrencia entre ticks:** `SKIP LOCKED` garantiza reclamos disjuntos. Advisory lock `pg_try_advisory_lock(hashtext('media_signature_renew_tick'))` evita solapamiento; si no se obtiene, el tick retorna inmediatamente. Locks huérfanos se recuperan por la tercera rama del `WHERE`.

## 3 · Estado `failed` sin terminal accidental

Se elimina el gate `attempt_count < MAX_ATTEMPTS`. Política:

- Reintento perpetuo con backoff exponencial saturado:
  `next_retry_at = now() + least(interval '6 hours', interval '30 seconds' * 2^least(attempt_count, 9))`.
- Éxito: `attempt_count = 0`, `last_success_at = now()`, `renewal_state = 'idle'`.
- Fallo: `attempt_count += 1`, `last_error = <clase sanitizada>`, `renewal_state = 'failed'`.
- Alertas: `attempt_count = 10` ⇒ `system_alerts(severity='warning')`; `= 20` ⇒ `critical`.
- Kill-switch por fila: sólo columna administrativa `admin_disabled boolean default false`, humano-controlada.

Ninguna fila queda fuera del ciclo sin intervención humana explícita.

## 4 · Invalidación honesta sobre `media-derived`

Las variantes viven en el bucket `media-derived`, disjunto de `media` (originales) y del `file_url` legacy. Eliminar un objeto derivado:

- Invalida esa variante inmediatamente (Storage responde 404).
- No toca original ni legacy ⇒ fallback legacy sigue sirviendo.
- URLs firmadas ya emitidas responden 404 aunque el token siga vigente (revocación efectiva).

### Procedimiento de invalidación inmediata

1. `UPDATE media_asset_variants SET status='invalidated', is_current=false WHERE id=$1`.
2. `UPDATE media_asset_signed_urls SET renewal_state='idle', expires_at=now() WHERE variant_key=$1`.
3. `invalidateByVariantKey($1)` — purga LRU per-worker (M2.3).
4. Sólo si se requiere revocación criptográfica de tokens ya emitidos: `storage.from('media-derived').remove([path])`.
5. `file_url` legacy intacto.

**Distinción tabla vs URL:** `media_asset_signed_urls` es server-only y jamás se serializa al cliente. En M2.4 el navegador recibirá el contenido del campo `signed_url` (la URL), nunca la fila. Cachear la URL en el navegador es responsabilidad del `<img>`, no de este contrato.

## 5 · Gate estadístico corregido

Promoción M2.3.1 → M2.4:

- **N ≥ 4 000** evaluaciones elegibles en 24 h sostenidas.
- Éxitos/N con **Wilson 95 % lower bound > 0.999**.
- p95(read-path total, warm) < 50 ms y p95(cold, con firma persistida) < 100 ms medidos server-side.
- 0 fugas de secreto en logs (grep automatizado en pipeline).

**Prohibido:** aprobar con 1 000/1 000 (Wilson-LB ≈ 0.9963, insuficiente). Prohibido sustituirlo por tres ventanas de 1 000 con proporción cruda.

**Distinción:**
- *Carga sintética* → mide throughput; no cuenta para el gate.
- *Disponibilidad operacional real* → sólo evaluaciones del preloader en preview con tráfico natural cuentan.

## 6 · Margen servible unificado (6 h)

Regla única en todas las capas:

- Vida restante `> 6 h` ⇒ **servible** (persistida o LRU).
- Vida restante `≤ 6 h` ⇒ **no servible desde persistida**. Comportamiento:
  1. LRU per-worker puede servir sólo si su entry tiene vida > 6 h.
  2. En su defecto, `sign()` on-demand (cold ~500 ms) o fallback legacy según flag.
- Elegibilidad de scheduler (§2): `expires_at < now() + 48h` como banda amplia; ajustable a menor umbral bajo carga.

**Escenario de prueba actualizado:** firma con 6 h 1 min restantes ⇒ servible. Con 5 h 59 min ⇒ el resolver la rechaza y activa cold-path/legacy. Escenario previo (“dentro del margen de 6 h → servida”) queda derogado.

## 7 · UPSERT condicionado

```sql
INSERT INTO media_asset_signed_urls
  (variant_key, bucket, path, signed_url, issued_at, expires_at,
   renewal_state, attempt_count, last_success_at, lock_owner, lock_acquired_at)
VALUES ($1,$2,$3,$4,$5,$6,'idle',0,now(),NULL,NULL)
ON CONFLICT (variant_key) DO UPDATE SET
  signed_url       = EXCLUDED.signed_url,
  issued_at        = EXCLUDED.issued_at,
  expires_at       = EXCLUDED.expires_at,
  renewal_state    = 'idle',
  attempt_count    = 0,
  last_success_at  = now(),
  lock_owner       = NULL,
  lock_acquired_at = NULL
WHERE
  media_asset_signed_urls.variant_key = EXCLUDED.variant_key
  AND EXCLUDED.issued_at > media_asset_signed_urls.issued_at;
```

Garantías:

- Nunca reemplaza una firma más nueva por una anterior (`issued_at` monotónico).
- Si `variant_key` cambia (nueva derivación), la fila vieja queda hasta que el sweeper la marque `admin_disabled` y limpie.
- Resultados tardíos (scheduler completa después que on-demand) no alteran `renewal_state` porque `issued_at` menor los descarta.
- Single-flight per-worker (M2.3) intacto; este UPSERT es la barrera cross-worker.

## 8 · Threat model (sin ceros residuales)

| Vector | Mitigación | Riesgo residual |
|---|---|---|
| Fuga de service-role vía SQL/cron/logs | GUC de rol + REVOKE + purga `pg_net.responses` + grep CI | muy bajo |
| Replay de request al endpoint | HMAC + nonce + ventana 60 s | muy bajo |
| URL firmada copiada (7 d, contenido editorial público) | Aceptado por Founder + `remove()` para revocación | bajo, aceptado |
| LRU sirve firma stale | Regla 6 h unificada + `invalidateByVariantKey` | bajo |
| Lock huérfano tras crash | Reclamación `renewing > 5 min` en §2 | muy bajo, observable |
| Fila `failed` atrapada | §3 sin terminal + alertas 10/20 | mitigado y observable |
| Escritura concurrente pisa firma nueva | UPSERT condicionado §7 | muy bajo |
| Endpoint privado abusado desde internet | HMAC + nonce + flag OFF | muy bajo |
| `pg_net.responses` filtra datos | Body sólo contadores + purga 15 min | muy bajo |

Ningún vector se declara `0`.

## 9 · Migración y rollback

**Migración aditiva (dos pasos, ambos reversibles):**

1. Crear `media_asset_signed_urls` (cols: `id`, `variant_key UNIQUE`, `bucket`, `path`, `signed_url`, `issued_at`, `expires_at`, `renewal_state`, `attempt_count`, `next_retry_at`, `last_success_at`, `last_error`, `lock_owner`, `lock_acquired_at`, `admin_disabled`, `created_at`, `updated_at`).
   `REVOKE ALL FROM anon, authenticated`. `GRANT ALL TO service_role`. RLS ON, sin políticas ⇒ inaccesible fuera de service-role.
2. Crear `media_signature_renew_nonces` + `public.media_signature_renew_tick()` + `cron.schedule('media-signature-renew','* * * * *', 'SELECT public.media_signature_renew_tick();')`. Flag `media_persisted_signatures_enabled = false`.

**Rollback no destructivo:**

- A: `cron.unschedule('media-signature-renew')`.
- B: flag off (idempotente, ya está off).
- C (opcional): `DROP TABLE media_asset_signed_urls, media_signature_renew_nonces`. Sin efecto sobre M1/M2.x — no las referencian.

## 10 · Plan de pruebas v1.2

1. Worker frío, tabla poblada, vida > 6 h ⇒ read-path total p95 < 20 ms.
2. Firma con vida 6 h 1 min ⇒ servible.
3. Firma con vida 5 h 59 min ⇒ no servible; cold/legacy activo.
4. Firma expirada ⇒ scheduler recupera en ≤ 1 min.
5. Scheduler detenido 24 h ⇒ 0 filas terminales; backoff saturado + alertas.
6. Storage caído ⇒ `attempt_count++`, `next_retry_at` respetado, sin loop apretado.
7. DB caída ⇒ endpoint responde 5xx, cron reintenta, nada persiste.
8. Cambio de `variant_key` ⇒ nueva fila; vieja `admin_disabled` en ≤ 1 tick.
9. 50 solicitudes concurrentes al mismo asset ⇒ 1 sola llamada Storage (single-flight per-worker) + UPSERT idempotente cross-worker.
10. Cliente anónimo `select * from media_asset_signed_urls` ⇒ **permission denied**.
11. Replay de request cron ⇒ 409 nonce.
12. Rotación de `MEDIA_SIGNATURE_RENEW_HMAC` en dual-verify ⇒ 0 firmas perdidas.
13. Rollback completo (unschedule + drop) ⇒ read-path vuelve a M2.3 sin regresión.
