**Estado:** Radiografía + plan. **No se implementa código en esta fase.**
**Criterio:** Definir cómo eliminar el modelo dual sin romper Business, Product, Preview, Public pages ni Studio.

### 1.3 Consumidores de v2 (`eb_pages`)

**Sólo tres archivos:**
- `src/lib/experience-builder/eb-studio.functions.ts` (CRUD, versiones, publish, preview, themes, variants).
- `src/lib/experience-builder/eb-public.functions.ts` (`ebResolvePublicPage`).
- `src/integrations/supabase/types.ts` (tipos generados).

**Ningún componente, ruta pública, Studio, Portal ni motor de render importa `eb-studio` o `eb-public`.**
Verificado con `rg` — cero call-sites fuera del propio módulo.

### 1.4 Origen histórico del dualismo

- **v2 `eb_pages`** nació en Fase 2 del 15.10.4b como intento inicial del Studio con modelo relacional (sections/versions/themes/variants).
- **v1 `page_compositions`** se consolidó a partir de 15.10.4d como fuente única: árbol JSON por composición, adoptado por US-R3 Olas 0..2.5d y Sub-olas 2.6/2.6b.
- El Single Studio Principle (15.10.4d) ya redirige `/cms/experience-builder/pages` (editor v2) al Studio sobre v1. El código de v2 quedó como cascarón huérfano.

## 2 · Diagnóstico

| Aspecto | v1 | v2 |
|---|---|---|
| Uso productivo | **Sí** (100%) | No |
| Cobertura de superficies | Home, Marketplace, Alux, ATV, Destinos, Producto, Landings, Preview, Portal, Semillas Kit | Ninguna |
| RPC pública consumida por rutas | `eb_get_published_by_slug`, `eb_resolve_public_route` | `ebResolvePublicPage` (nadie la llama) |
| Modelo de datos | Árbol JSON (`snapshot`) + revisiones + redirects + SEO/sitemap | Relacional (pages+sections+versions+themes+variants) |
| Deuda | `saveCompositionDraft` traduce síncrono y silencia errores | Cascarón sin uso: ~15 server fns, 10 tablas |
| Riesgo de eliminar | Alto — rompe todo | **Bajo** — nadie lo importa |

**Conclusión:** No hay realmente un modelo dual en producción. Hay **un modelo vivo (v1)** y **un modelo muerto (v2)** que sólo existe como tablas vacías y tres archivos huérfanos. La "unificación" es en la práctica un **retiro controlado de v2**, no una migración de datos.

---

## 3 · Fuente de verdad futura

**Oficial y única:** `page_compositions` + `page_revisions` (v1), con:
- `page_redirects` para 301s (Ola 0).
- `composition_preview_tokens` para preview firmado.
- RPCs `eb_get_published_by_slug`, `eb_resolve_public_route`, `eb_process_scheduled_publishes`.
- `CompositionRenderer` como único ensamblador.
- Bloques `vmx.kit.*`, `vmx.business.*`, `vmx.product.*`, `vmx.surface.*` como catálogo canónico.

**Descartado:** modelo `eb_*` relacional. Cualquier capacidad genuina que aún se necesite (themes globales, variants A/B) se re-incorpora sobre v1 como columnas/tablas complementarias, no como modelo paralelo.

## 4 · Estrategia de unificación (sin código todavía)

Enfoque: **deprecación quirúrgica en tres pasos, cada uno reversible y verificable.**

### Paso A · Congelar v2 (soft freeze) — Fase 3.3a propuesta
1. Marcar `eb-studio.functions.ts` y `eb-public.functions.ts` como `@deprecated` en JSDoc; añadir `console.warn` server-side si se invocan.
2. Añadir aserción de repo (grep/CI) que falle si alguien vuelve a importar `eb-studio`/`eb-public` fuera de sí mismos.
3. No tocar tablas. Riesgo: **cero**. Rollback: revertir el commit.

### Paso B · Retirar código huérfano de v2 — Fase 3.3b propuesta
1. Borrar `eb-studio.functions.ts` y `eb-public.functions.ts`.
2. Verificación: typecheck limpio, smokes Kit/Composer/Business/Product idénticos, rutas públicas OK.
3. Rollback: `git revert`.

### Paso C · Retirar tablas v2 (hard drop) — Fase 3.3c propuesta
Pre-condiciones obligatorias:
- Confirmación por query de que las 10 tablas `eb_*` (excepto las de v1) siguen en 0 filas.
- Dump lógico previo archivado como respaldo.

Migración única:
1. `DROP POLICY` de todas las políticas RLS sobre esas tablas.
2. `DROP TABLE ... CASCADE` en orden inverso al de dependencias.
3. `DROP FUNCTION` de cualquier RPC exclusivamente `eb_*` no reutilizada (auditar antes; conservar `eb_get_published_by_slug`, `eb_resolve_public_route`, `eb_process_scheduled_publishes`, `has_role`).
4. Regenerar `types.ts`.

Rollback: aplicar dump lógico previo (esquema vacío).

### Paso D · Higiene de deuda v1 — diferido a Fase 3.4+
- Endurecer `saveCompositionDraft`: eliminar `catch` silencioso, tipar errores.
- Congelar y documentar la API pública de `CompositionRenderer`.
- Auditar RPCs `eb_*` restantes para confirmar guards (mismo patrón de 3.2).

## 6 · Riesgos y mitigaciones

| Riesgo | Probabilidad | Mitigación |
|---|---|---|
| Un consumidor externo (edge fn, cron, integración) invoca RPC `eb_pages` fuera del repo | Baja | Antes del paso C, `pg_stat_user_functions` + búsqueda cruzada de logs; conservar función 30 días con `RAISE NOTICE` antes de `DROP` |
| Semillas Kit o Studio adoptan v2 por error en fases futuras | Baja | Bloqueo por linter/aserción del paso A |
| Pérdida de capacidad real (themes/variants) que se creía muerta | Baja | Ninguna UI las usa; si aparece requerimiento, se re-diseña sobre v1 |
| Regenerar `types.ts` rompe imports a `Database["public"]["Tables"]["eb_pages"]` | Media | Los únicos consumidores desaparecen en el paso B; orden B → C es obligatorio |

## 7 · Recomendación al Founder

1. Autorizar **Paso A (soft freeze)** como Fase 3.3a — cambio mínimo, cero riesgo, deja evidencia si alguien intentara reactivar v2.
2. Ejecutar **Paso B (borrar código v2)** como Fase 3.3b tras al menos una iteración sin warnings.
3. Ejecutar **Paso C (drop tablas v2)** como Fase 3.3c sólo tras confirmar 0 filas y con dump de respaldo.

Con esto queda cerrado el modelo dual sin tocar una sola composición viva, sin renderer nuevo, sin migración de datos y sin ventana de riesgo para Business/Product/Preview/Public/Studio.
