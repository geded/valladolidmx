## 1. Resultado

La ruta autenticada `/cms/visitor-intel/decisions` convierte oportunidades
agregadas de CV8.7 y CV8.8 en una cola humana operable. Founder/Admin pueden
proponer, aceptar, asignar, diferir, descartar, validar y crear una corrección;
Editor y Concierge Lead sólo ven decisiones asignadas y avanzan su trabajo
conforme a la política server-side existente.

La superficie no ejecuta campañas, contenido, Alux, Concierge ni cambios de
catálogo. Cada interacción de negocio llama las funciones gobernadas CV8.9.2,
que añaden eventos al ledger append-only. No se agregaron tablas, snapshots,
columnas ni migraciones.

CV8.9.4 cierra el ciclo lógico con CV8.6: los resultados terminales de decisión
se recomponen desde todo el historial y se combinan por `metric_id` con los
resultados de recomendaciones dentro de la ventana. `validated` suma señal
positiva; `rejected` y `dismissed` suman señal negativa. El umbral
`MIN_FAMILY_SIGNAL = 5` y la fórmula de confianza no cambian.

## 3. Gobernanza y privacidad

| Garantía            | Evidencia                                                                              |
| ------------------- | -------------------------------------------------------------------------------------- |
| Decisión humana     | Toda propuesta requiere aceptación explícita antes de trabajo.                         |
| Owner obligatorio   | `accepted` exige un operador humano y acuerdo completo.                                |
| Permisos            | Los controles replican la política pura; cada mutación vuelve a autorizar en servidor. |
| Assigned-only       | Editor/Concierge Lead sólo reciben decisiones cuyo owner coincide con su usuario.      |
| Auditabilidad       | Actor, rol, timestamps, motivo, evidencia e historial derivan del ledger.              |
| Privacidad          | No se listan correos ni identidades de visitantes; sólo operadores y agregados.        |
| Cero automatización | No existen llamadas a campañas, Alux, Concierge, contenido o catálogo.                 |

El gate del CMS permite a `concierge_lead` únicamente la URL de decisiones; no
abre el resto del CMS para ese rol. Los endpoints siguen siendo la frontera de
seguridad.

## 4. Feedback real a CV8.6

`aggregateRecommendationValidation` conserva sus recomendaciones activas y
cerradas, pero ahora calcula `family_confidence` sobre dos fuentes append-only:

1. recomendaciones CV8.6 cerradas dentro de la ventana;
2. decisiones CV8.9 con outcome terminal dentro de la ventana.

La proyección de decisiones carga el historial completo, paginado de 1,000 en
1,000 y ordenado por `occurred_at`, para no perder el `proposed` o el acuerdo
cuando el outcome ocurre en una ventana posterior. Sólo la señal terminal se
filtra por ventana.

## 5. Métricas del workflow

| Métrica                | Definición técnica                                                                   |
| ---------------------- | ------------------------------------------------------------------------------------ |
| acceptance rate 7d/30d | decisiones propuestas en la ventana que alcanzaron `accepted` / cohorte propuesta    |
| time to accept p50     | mediana de horas `proposed → accepted`                                               |
| time to implement p50  | mediana de horas `accepted → implemented`                                            |
| validation rate        | decisiones que terminaron `validated` / decisiones que alcanzaron `implemented`      |
| bias flag rate         | `dismissed` cuyo motivo menciona sesgo o explicación alternativa / total `dismissed` |
| SLA breach rate        | activas con due date o ventana de validación vencida / activas                       |

Todas son proyecciones en memoria. No se persisten métricas ni snapshots.
