## 1. Resumen

Se habilita el cobro en línea de órdenes confirmadas (`conversion_mode =
'reservar_en_linea'` y `accepts_online_payment = true`) mediante Stripe
Checkout Sessions, detrás de una **capa de abstracción de proveedor de
pagos** (`PaymentProvider`) que permitirá incorporar Paddle, Mercado Pago
o PayPal en olas posteriores sin tocar el dominio del Marketplace, el
Portal Empresarial, el EMS ni Alux Intelligence.

## 2. Cumplimiento de los lineamientos

| Lineamiento | Cumplimiento |
|-------------|--------------|
| Payment Intents (Stripe) | Stripe Checkout Session (≡ Payment Intent server-side). |
| Webhooks con verificación de firma | `stripe.webhooks.constructEvent` con `STRIPE_WEBHOOK_SECRET`; 401 si la firma falla. |
| Idempotencia por `client_request_id` | `idempotencyKey` Stripe = `co_<order_id>_<client_request_id>`; webhook idempotente por `UNIQUE(provider, provider_event_id)`. |
| Revalidación server-side antes del cobro | `startPayment` recarga orden + ítems + productos y revalida `status='published'`, `conversion_mode`, `accepts_online_payment`, `price_amount`, `price_currency`. |
| Metadata consistente orden ↔ pago | `metadata.order_id`, `user_id`, `provider`, `client_request_id` enviados a Stripe; vuelven en el webhook vía `client_reference_id`/`metadata`. |
| Auditoría completa en `order_events` | Nuevos eventos `payment_initiated`, `payment_succeeded`, `payment_failed`, `payment_refunded`. |
| Sin `SUPABASE_SERVICE_ROLE_KEY` en flujos del viajero | `startPayment` usa `requireSupabaseAuth` (RLS aplica); `supabaseAdmin` sólo dentro del webhook tras verificar firma. |
| Payment Provider Abstraction | Contrato `PaymentProvider` neutral; UI/dominio no importan `stripe`. |

## 3. Arquitectura

```text
src/lib/payments/
├── provider.ts                  # Contrato PaymentProvider + tipos neutrales
├── registry.server.ts           # Selección por env PAYMENTS_PROVIDER
├── stripe.server.ts             # Implementación Stripe (server-only)
└── payments.functions.ts        # Server fns (startPayment, getActivePaymentProvider)

src/routes/api/public/payments/$provider/webhook.ts
    # Webhook neutral: /api/public/payments/:provider/webhook

src/routes/_authenticated/cuenta/
├── pagos.exito.tsx              # Página de éxito (neutral)
└── pagos.error.tsx              # Página de error (neutral)
```

- El registry mapea `PAYMENTS_PROVIDER` → implementación. Hoy sólo
  `stripe` está implementado; `paddle`, `mercadopago`, `paypal` quedan
  declarados como `null` para señalizar el contrato pendiente.
- `getActiveProvider()` se importa **dinámicamente** dentro de los
  handlers (`await import("./registry.server")`) para mantener
  `stripe.server.ts` fuera del bundle de cliente.
- El UI (`historial.tsx`) sólo conoce `startPayment` y `mode`; según el
  proveedor, redirige (`window.location.href = redirectUrl`) o, en el
  futuro, montará Elements/SDK del proveedor que corresponda.

## 4. Cambios de base de datos

Migración `payments_stage5_*`:

- `orders` (aditivo): `payment_provider TEXT`, `payment_intent_id TEXT`,
  `payment_status TEXT NOT NULL DEFAULT 'unpaid'` (check
  `unpaid|processing|paid|failed|refunded`), `paid_at TIMESTAMPTZ`.
- Índice único parcial `(payment_provider, payment_intent_id) WHERE
  payment_intent_id IS NOT NULL`.
- Tabla `payment_events` con `UNIQUE(provider, provider_event_id)`,
  RLS de lectura para el dueño de la orden, sin INSERT/UPDATE/DELETE
  para `authenticated`. `service_role` controla las escrituras desde el
  webhook verificado.
- RPCs `order_mark_paid(p_order_id, p_provider, p_intent_id, p_event_id)`
  y `order_mark_payment_failed(... , p_reason)`:
  `SECURITY DEFINER`, `search_path = public`, `EXECUTE` revocado de
  `PUBLIC` y concedido **sólo a `service_role`**.
  Idempotentes: si la orden ya está `paid`, retornan sin mutar.
- Enum `order_event_type`: añadidos `payment_initiated`,
  `payment_succeeded`, `payment_failed`, `payment_refunded`.

## 5. Flujo end-to-end

1. Viajero presiona "Pagar" en `/cuenta/historial`.
2. `startPayment` (`requireSupabaseAuth`):
   - Carga la orden propia, valida `status ∈ {pending, confirmed}` y
     `payment_status ≠ paid`.
   - Carga ítems + productos y revalida elegibilidad y precios.
   - Lee email desde `profiles`.
   - Llama `provider.createIntent(...)` con `idempotencyKey`.
   - Actualiza `orders.payment_provider`, `payment_intent_id`,
     `payment_status='processing'`.
   - Inserta `order_events.payment_initiated`.
   - Devuelve `{ provider, mode, redirectUrl?, clientSecret? }`.
3. Stripe redirige al checkout y de vuelta a `/cuenta/pagos/exito` o
   `/cuenta/pagos/error`.
4. Stripe llama
   `POST /api/public/payments/stripe/webhook`:
   - Lee `rawBody`, verifica firma con
     `stripe.webhooks.constructEvent`.
   - Inserta en `payment_events`; `23505` (duplicado) → `200 ok`.
   - Llama `order_mark_paid` / `order_mark_payment_failed` con
     `service_role`.
   - Marca `processed_at`.
5. Las páginas `pagos.exito` y `pagos.error` no mencionan Stripe; el
   estado real proviene de `orders.payment_status` actualizado por el
   webhook.

## 6. Compatibilidad con el documento rector 14.40.B v1.1

- **Carrito y checkout** sólo aceptan productos
  `reservar_en_linea + accepts_online_payment` (ya enforced en
  `cart_add_item`/`order_confirm`; ahora también en `startPayment`).
- **Comisiones, reembolsos, pagos parciales, conciliaciones, splits**:
  el contrato `PaymentProvider` queda abierto para añadir `refund`,
  `capture` y `split` en olas posteriores sin romper el dominio.
- **Alux Intelligence**: no se incorpora ningún ranking ni promoción
  patrocinada en esta etapa; el cobro es ortogonal a la imparcialidad
  editorial.
- **EMS / Arma tu Viaje**: sin acoplamiento; estas superficies seguirán
  invocando `startPayment` igual que el carrito el día que tengan
  productos reservables.

## 7. Pendientes de provisión (no bloquean el código)

- `STRIPE_SECRET_KEY` (secret server-only).
- `STRIPE_WEBHOOK_SECRET` (secret server-only).
- Opcional: `PAYMENTS_PROVIDER` (default `stripe`).
- Configurar el endpoint del webhook en el dashboard de Stripe:
  `https://valladolidmx.lovable.app/api/public/payments/stripe/webhook`
  con los eventos: `checkout.session.completed`,
  `checkout.session.async_payment_succeeded`,
  `checkout.session.async_payment_failed`,
  `checkout.session.expired`, `charge.refunded`.

Mientras estos secretos no estén configurados, `startPayment` falla
con `stripe_secret_key_missing` y el webhook con
`stripe_webhook_secret_missing`; el resto del Marketplace sigue
operativo.
