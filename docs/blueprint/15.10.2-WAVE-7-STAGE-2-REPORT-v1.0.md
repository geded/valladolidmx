## 0. Resumen ejecutivo

Se implementó el **primer editor visual operativo del Experience Builder**, sin tocar el sitio público ni regresar Olas 1–6. El Studio permite a roles `admin` y `editor`:

- Crear composiciones dummy con `slug`, `title` y `page_type` libres.
- Insertar, reordenar, duplicar, eliminar y configurar bloques del catálogo aprobado en 15.10.1.
- Editar configuración mediante un Inspector generado dinámicamente desde el `schema` del Block Contract.
- Guardar borradores (autosave + manual) y crear revisiones inmutables.
- Listar y restaurar revisiones previas.

El render del Canvas usa el **Renderer Universal**, que será el mismo módulo que servirá composiciones públicas a partir de 15.10.3.

## 1. Cumplimiento de los principios arquitectónicos ampliados

### 1.1 Canvas Agnóstico

`CompositionRenderer` (`src/lib/experience-builder/composition-renderer.tsx`) recibe únicamente un `CompositionTree` y, opcionalmente, un `pageType` como metadato. No contiene ninguna rama condicional por tipo de página. El mismo Canvas servirá Home, Landing Pages, Páginas Institucionales, Destinos, Empresas, Productos, Campañas y cualquier superficie futura sin modificarse.

### 1.2 Layout Engine

`layout-engine.ts` desacopla **estructura visual** y **contenido**:

- `isContainerBlock(type)` reconoce qué bloques aceptan hijos.
- `registerContainerType(type)` permite incorporar nuevos contenedores (filas, columnas, grids, tabs) sin modificar el núcleo.
- Los bloques de contenido (Hero, secciones, tarjetas) no conocen al contenedor que los aloja; el motor inyecta `renderChildren` solo a los contenedores.

Esto cumple "los layouts forman parte del motor de composición y no de los bloques individuales".

### 1.3 Page-Type Agnostic

`page_type` se almacena en `page_compositions.page_type` como **configuración** (default `generic`), nunca como lógica del editor. El Studio, el Renderer Universal y las server functions tratan `page_type` como metadato. Para incorporar un nuevo tipo de página (Campaña, Producto, Destino, etc.) basta con crear una composición con ese `page_type`; no se requiere desarrollar editor adicional.

## 2. Entregables

### 2.1 Base de datos (`supabase/migrations/…_studio_v0.sql`)

- Extensión del enum `entity_kind` con `composition` y `revision`.
- Tabla `public.page_compositions` con RLS, GRANTs explícitos a `authenticated` y `service_role`, y trigger `eb_touch_composition`.
- Tabla `public.page_revisions` con FK a composiciones, índice `composition_id, revision_number DESC`, RLS y GRANTs.
- RPCs `SECURITY DEFINER` con `SET search_path = public` y validación por `has_role`:
  - `eb_create_composition(_slug, _title, _description, _page_type)`
  - `eb_save_composition_draft(_id, _tree)`
  - `eb_create_revision(_id, _notes)`
  - `eb_restore_revision(_id, _revision_id)`
- Cada RPC registra el evento BEA correspondiente en `content_audit_log`.

### 2.2 Núcleo del Experience Builder

- `src/lib/experience-builder/composition-tree.ts` — modelo `CompositionTree` / `CompositionNode`, helpers inmutables (`appendToRoot`, `moveRootNode`, `duplicateRootNode`, `updateNodeConfig`, `removeNode`, `findNode`, `mapTree`).
- `src/lib/experience-builder/layout-engine.ts` — registro de contenedores (`isContainerBlock`, `registerContainerType`, `listContainerTypes`).
- `src/lib/experience-builder/composition-renderer.tsx` — Renderer Universal con `COMPONENT_MAP` para los 18 bloques iniciales del catálogo y placeholders `StudioErrorBlock` para bloques deprecados/desconocidos (solo visibles en modo Studio).

### 2.3 Server functions (`studio.functions.ts`)

- `listCompositions`, `getComposition`, `createComposition`, `saveCompositionDraft`, `createCompositionRevision`, `listCompositionRevisions`, `restoreCompositionRevision`.
- Todas autenticadas con `requireSupabaseAuth`. Ninguna usa `SUPABASE_SERVICE_ROLE_KEY`.

### 2.4 UI editorial

- `src/routes/_authenticated/cms/experience-builder.tsx` — Studio v0 (Library Panel + Canvas + Inspector + Toolbar + Historial).
- Filtros por categoría (`static` / `smart`) y búsqueda; etiqueta visible **"Internal Draft"** sobre el lienzo.
- Inspector dinámico que cubre `text`, `rich_text`, `number`, `boolean`, `select`, `url`, `color`, `media`, `reference` (campos no soportados se muestran como texto plano en v0).
- Entrada añadida al sidebar del CMS Studio (`/cms/experience-builder`).

### 2.5 Eventos BEA emitidos

- `Composition.Created`
- `Composition.DraftSaved`
- `Composition.RevisionPublished`
- `Composition.RevisionRestored`

Todos persistidos en `content_audit_log` con `entity_kind` ∈ {`composition`, `revision`}.

## 4. Exclusiones honradas

- Sin publicación pública de composiciones.
- Sin migración de la Home actual.
- Sin Templates / Partials / Landing Pages.
- Sin Smart Blocks operativos (categoría declarada, sin resolvers).
- Sin Responsive Builder, validador SEO bloqueante, UNC editorial.
- Sin Alux Write.
- Sin edición colaborativa multiusuario en tiempo real.

## 5. Riesgos vigilados

| # | Riesgo | Estado |
|---|--------|--------|
| R1 | Drift visual Studio ↔ producción | Mitigado: mismo Renderer Universal. |
| R2 | Pérdida editorial | Mitigado: autosave 1.2 s + guardado manual + revisiones. |
| R3 | Composiciones con bloques inválidos | Renderer marca `StudioErrorBlock` en modo Studio; producción no renderiza nada (pendiente bloqueo duro en 15.10.3). |
| R4 | Privilegios | RLS estricto + verificación en cada RPC. |
| R5 | Acoplamiento al catálogo | Library Panel consume `listBlockLibrary`; añadir bloques no toca el Studio. |
| R6 | Inspector con esquemas complejos | Campos no soportados degradan a `text`. |
| R7 | Confusión dummy vs pública | Etiqueta **"Internal Draft"** visible + ausencia de rutas públicas. |
| R8 | Crecimiento de revisiones | Conteo monitorizable; política de retención queda diferida a 15.10.5. |
