# 16.AC1 · Anonymous Travel Continuity — Blueprint v1.0

**Estado:** Emitido — pendiente de aprobación del Founder para iniciar implementación.
**Iniciativa:** Anonymous Travel Continuity (AC1).
**Depende de:** Travel Plan Contract v1.0, CV0, CV6.O1/O2, Founder Travel Companion First, Founder Daily Value, Founder Journey First.
**Política nueva:** `mem://policies/founder-anonymous-continuity.md` (Founder Anonymous Travel Continuity Principle).

---

## 1. Objetivo

Permitir que un visitante anónimo explore, marque favoritos, arme un borrador de viaje (fechas tentativas, número de viajeros, ítems) y regrese al mismo navegador para continuarlo — **sin registrarse y sin crear cuenta anónima ni fila de DB**. El registro se solicita únicamente en momentos donde aporta beneficio evidente.

## 2. Principios rectores

1. **Alux nunca pide antes de dar.**
2. **Local-first**: el borrador vive en el dispositivo del usuario.
3. **Travel Plan sigue siendo la única fuente canónica** tras el registro. El borrador NO es un modelo paralelo.
4. **Rendimiento**: 1,000+ anónimos simultáneos no aumentan linealmente la carga de DB.
5. **Privacidad**: sin fingerprinting, sin PII, sin ubicación precisa, sin datos de pago.

## 3. Contrato del borrador — `AnonymousTravelDraft` v1.0.0

```ts
// src/lib/traveler/anonymous-draft/contract.ts (previsto)
export const ANON_DRAFT_VERSION = "1.0.0" as const;
export const ANON_DRAFT_TTL_DAYS = 30;

export interface AnonymousTravelDraft {
  draftId: string;               // uuid v4 generado en cliente
  version: string;               // semver del contrato
  createdAt: number;             // epoch ms
  updatedAt: number;             // epoch ms
  expiresAt: number;             // createdAt + TTL
  source: "web" | "pwa";         // canal de origen
  travelStage: TravelStage;      // reutiliza contrato CV6.O1
  destinationIds: string[];      // máx N (ver §7)
  favoriteEntityIds: Array<{ kind: FavoriteEntityKind; id: string }>;
  plannedItems: Array<{
    kind: TravelItemKind;        // reutiliza contrato Travel Plan
    targetId: string | null;
    title?: string; slug?: string; imageUrl?: string; subtitle?: string;
    notes?: string;              // sanitizado, sin PII
    addedAt: number;
  }>;
  tentativeDates?: { from?: string; to?: string }; // ISO date
  travelerCount?: { adults: number; children?: number };
}
```

**Reglas del contrato:**
- Pequeño, serializable, migrable, reversible.
- No almacena: PII, ubicación precisa, pagos, conversaciones Concierge, tokens, cookies de terceros.
- Migraciones futuras: función pura `migrate(raw): AnonymousTravelDraft | null`.
- Validación estricta (Zod) al leer/escribir/migrar. Datos inválidos → descartar sin romper la sesión.

## 4. Arquitectura local-first

| Almacén     | Uso                                                                 |
|-------------|----------------------------------------------------------------------|
| IndexedDB   | Borrador estructurado (`AnonymousTravelDraft`), un único registro.  |
| localStorage| `draftId`, `updatedAt`, `travelStage` (metadatos ligeros).          |
| Memoria     | Cache activo del borrador, single-flight writes con debounce 400ms. |

**Módulos previstos** (todos client-only, sin server fn):
- `src/lib/traveler/anonymous-draft/store.ts` — capa IndexedDB (fallback localStorage si IDB no disponible).
- `src/lib/traveler/anonymous-draft/hooks.ts` — `useAnonymousDraft()` con selectors y writer debounced.
- `src/lib/traveler/anonymous-draft/limits.ts` — enforcement de límites.
- `src/lib/traveler/anonymous-draft/analytics.ts` — batcher de eventos.
- `src/lib/traveler/anonymous-draft/migrate-to-plan.ts` — merge determinista al Travel Plan.

**No se crean:** tablas nuevas, RPCs por interacción, canales Realtime, cookies servidor, endpoints de tracking por evento.

## 5. Estrategia de conversión a Travel Plan

Trigger: primer login exitoso tras existir borrador local válido.

1. Leer borrador desde IndexedDB.
2. Validar `version` y `expiresAt`. Si inválido → descartar y limpiar.
3. Ejecutar server fn `importAnonymousDraft` (nueva, autenticada, idempotente):
   - Reutiliza `addPlanItem` / `updatePlanMeta` del Write Contract del Travel Plan (prohibido acceso directo a `travel_plans`).
   - Deduplica por `(kind,targetId)` contra ítems existentes.
   - Fusiona `favoriteEntityIds` con `toggleFavorite` (idempotente).
   - Fusiona `tentativeDates` y `travelerCount` sólo si el Plan no los tiene.
   - Registra traza `plan.imported_from_anonymous_draft` con `draftId` (sin PII).
4. Sólo tras 200 OK: borrar el borrador local y limpiar metadatos localStorage.
5. En error: mantener borrador y ofrecer reintento manual.

**Reglas:** determinista, idempotente (safe a reintentos), sin duplicados, sin promover a caso Concierge automáticamente.

## 6. TTL, recuperación y abandono

- **TTL:** 30 días desde `createdAt`; se refresca `updatedAt` en cada mutación.
- **Recuperación:** al abrir la app, si existe borrador válido → banner "Continuar donde lo dejaste" con acciones **Continuar** / **Descartar**.
- **Expirado:** limpieza silenciosa en el próximo boot.
- **Consentimiento de contacto:** si el usuario deja voluntariamente email/WhatsApp (formulario explícito, no obligatorio), habilita estrategia de recuperación autorizada. Sin ese consentimiento explícito no se identifica al visitante.

## 7. Límites y protección

| Límite                          | Valor v1 |
|---------------------------------|----------|
| Favoritos anónimos              | 25       |
| Ítems en `plannedItems`         | 40       |
| Destinos guardados              | 8        |
| Tamaño payload total serializado| 64 KB    |
| Longitud `notes` por ítem       | 280 chars|
| TTL                             | 30 días  |
| Debounce de escritura           | 400 ms   |

Al alcanzar un límite: sheet explicativo "Guarda tu viaje para seguir agregando" con beneficio claro. Nunca se presenta como castigo.

## 8. Registro progresivo — momentos válidos

Sólo estos momentos abren el `SignInPromptSheet`:

1. Guardar el viaje permanentemente (CTA explícito).
2. Abrir el viaje en otro dispositivo.
3. Compartir el viaje.
4. Solicitar disponibilidad / cotización / promover a caso Concierge.
5. Iniciar reserva / checkout.
6. Activar recordatorios de Alux.
7. Recibir itinerario por correo / WhatsApp.
8. Alcanzar límites duros (§7).

Copy oficial (ES base, i18n-ready):

> **Tu viaje ya está tomando forma.**
> Crea tu cuenta para conservarlo, abrirlo en cualquier dispositivo y recibir ayuda de Alux antes y durante el viaje.
>
> [Guardar mi viaje] · [Continuar sin registrarme]

## 9. Experiencia de Alux

Alux reconoce el progreso anónimo sin fingir identidad:
- "Ya tienes 3 experiencias guardadas para Valladolid."
- "Podemos seguir armando este viaje sin registrarte todavía."
- "Guárdalo cuando quieras conservarlo en todos tus dispositivos."

Consumo: Alux lee el borrador vía selector cliente `useAnonymousDraft()`, jamás recibe PII, jamás envía el borrador al servidor sin registro explícito.

## 10. Analítica

Eventos anónimos (batcher, envío cada 10s o 20 eventos, `sendBeacon` en unload):

- `anonymous_draft_started`
- `anonymous_favorite_added`
- `anonymous_trip_item_added`
- `anonymous_draft_recovered`
- `registration_prompt_shown` (con `reason` enumerable)
- `anonymous_draft_converted`
- `anonymous_draft_abandoned` (TTL expirado)

Payload mínimo: `{ event, ts, draftId, stage, itemCount }`. Sin PII, sin contenido de notas, sin URLs con parámetros sensibles.

## 11. Plan de pruebas

- **Unit**: contrato Zod, migración, deduplicación, límites, TTL, sanitización.
- **Integración**: `SignInPromptSheet` NO se muestra en primer favorito ni primer "agregar al viaje"; SÍ se muestra en momentos válidos (§8).
- **Merge**: conversión determinista, idempotente, sin duplicados; borrado sólo tras 200 OK.
- **Rendimiento**: harness Playwright simula 1,000 sesiones con 20 interacciones cada una y verifica que las escrituras al servidor sean 0 (excepto analítica batched) y que la latencia p95 no degrade la home.
- **Compatibilidad**: fallback localStorage cuando IDB está bloqueado (modo privado Safari, cuota llena).
- **Regresión**: contratos CV6/CV7 y Travel Plan v1.0 intactos; typecheck limpio.

## 12. Guardrails y restricciones

- Prohibido crear un modelo persistente paralelo al Travel Plan.
- Prohibido Realtime, WebSockets o polling para anónimos.
- Prohibido fingerprinting, permisos de navegador, cookies de terceros.
- Prohibido escribir en `travel_plans` / `travel_plan_items` sin pasar por el Write Contract oficial.
- Prohibido tocar `client.ts`, `client.server.ts`, `auth-middleware.ts`, `auth-attacher.ts`, `types.ts`.
- Toda evolución del contrato: aditiva y con `migrate()` puro.
