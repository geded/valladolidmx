# CV8.9 · Action Queue & Decision Workflow — Blueprint v1.0

> Autorizado emitir Blueprint. **No implementar** hasta aprobación explícita.
> Convierte las oportunidades priorizadas (CV8.7) y segmentadas (CV8.8) en una
> agenda operativa donde Founder/Admin deciden qué atender, quién responde,
> qué acción se acordó y qué KPI se espera — cerrando el ciclo con CV8.6.

## Behavioral Change Statement

- Antes: el equipo veía rankings y hallazgos por segmento sin un lugar único
  para decidir, asignar, registrar y dar seguimiento.
- Después: cada oportunidad priorizada puede convertirse en una **Decisión**
  con responsable, acción acordada, KPI esperado y estado auditable.
- Comprobación: `/cms/visitor-intel/decisions` mostrará el backlog vivo con
  ciclo de vida completo y trazabilidad append-only.
- Por qué acerca al objetivo: transforma inteligencia en acción humana
  gobernada, sin ceder autonomía al sistema.

## Founder Human Governance Principle (aplicado)

- Recomienda la máquina, decide el humano.
- Nada se ejecuta automáticamente sobre campañas, contenidos, Alux,
  Concierge, empresas ni destinos.
- Toda decisión conserva: responsable humano, motivo, acción, KPI, evidencia
  posterior, trazabilidad completa.

## Reutilización estricta

- **CV8.0** · `JOURNEY_TRANSITIONS`, `KPI_CATALOG`.
- **CV8.1** · Event ingestion (`recommendation.lifecycle`) como único canal
  de mutación auditable.
- **CV8.2** · `projectVisitorState` — el estado de una Decisión se
  **recompone** desde eventos, nunca se persiste como snapshot mutable.
- **CV8.6** · `RecommendationValidationSnapshot.family_confidence` — la
  aceptación/validación de la decisión retroalimenta la matriz aprendida.
- **CV8.7** · `PrioritizedOpportunity` como semilla de la decisión.
- **CV8.8** · `SegmentPrioritySnapshot` para vincular la decisión a un
  segmento (locale/destination/capability) cuando aplique.
- Cero engines paralelos. Cero cache mutable. Cero automatización.

## Ciclo de vida de una Decisión

Estados canónicos (append-only vía eventos `recommendation.lifecycle`):

```text
proposed → accepted → in_progress → implemented → validated
     ↓          ↓            ↓              ↓
  deferred   dismissed   blocked        rejected
```

- `proposed` — creada desde una oportunidad priorizada (CV8.7/CV8.8).
- `accepted` — Founder/Admin la asume; declara responsable + acción + KPI.
- `in_progress` — el responsable trabaja en la acción acordada.
- `implemented` — la acción se ejecutó (evidencia observable).
- `validated` — el KPI esperado se movió en la dirección prevista dentro
  de la ventana declarada; retroalimenta CV8.6.
- `deferred` — se pospone con fecha objetivo y motivo.
- `dismissed` — se descarta con motivo (contraevidencia, prioridad, sesgo).
- `blocked` — dependencia externa; requiere motivo y responsable de
  desbloqueo.
- `rejected` — el KPI no se movió o se movió en contra; retroalimenta CV8.6
  como señal negativa.

Transiciones prohibidas: retroceder de `validated`/`rejected` a estados
anteriores. Corrección = nueva Decisión enlazada (`supersedes`).

## Contrato Action Queue (v1.0.0 — congelable en implementación)

```ts
// src/lib/visitor-intel/decisions.ts (a crear en implementación)
export const DECISION_CONTRACT_VERSION = "1.0.0";

export type DecisionState =
  | "proposed" | "accepted" | "in_progress" | "implemented"
  | "validated" | "deferred" | "dismissed" | "blocked" | "rejected";

export interface DecisionSource {
  origin: "cv87_priority" | "cv88_segment" | "manual";
  opportunity_id?: string;   // hash estable de la oportunidad de origen
  metric_id: string;
  transition: string;        // JourneyTransitionId | "aggregate"
  segment?: { dimension: "locale" | "destination" | "capability"; value: string };
  priority_score?: number;   // score CV8.7 al momento de creación
  learned_confidence?: number; // CV8.6 al momento de creación
}

export interface DecisionAgreement {
  action: string;            // qué se acordó hacer
  rationale: string;         // por qué (motivo humano)
  owner_user_id: string;     // responsable humano (obligatorio en accepted+)
  expected_kpi_id: string;   // KPI del catálogo CV8.0
  expected_direction: "up" | "down";
  expected_delta_relative: number; // magnitud esperada (0..1)
  evaluation_window_days: number;  // ventana para pasar a validated/rejected
  due_at?: string;           // fecha objetivo opcional
}

export interface DecisionEventPayload {
  decision_id: string;       // uuid estable
  from_state: DecisionState | null;
  to_state: DecisionState;
  source?: DecisionSource;   // en proposed
  agreement?: DecisionAgreement; // en accepted / in_progress / implemented
  evidence?: {                // en validated/rejected
    metric_id: string;
    observed_delta_relative: number;
    sample_size: number;
    window_start: string;
    window_end: string;
  };
  reason?: string;           // deferred/dismissed/blocked/rejected
  supersedes_decision_id?: string; // corrección de decisión previa
  actor_user_id: string;     // quién ejecutó la transición
  actor_role: "founder" | "admin" | "concierge_lead" | "editor";
  occurred_at: string;       // ISO
}
```

Persistencia: **cero tablas nuevas**. Los eventos viajan por el canal
existente `recommendation.lifecycle` de CV8.1 con `subtype = "decision"` y
`payload` conforme a `DecisionEventPayload`. El estado actual y el histórico
se **derivan** al vuelo (Founder Journey State Principle).

## Proyección (pura, sin snapshots)

`projectDecisionQueue(events, { now })` devuelve:

- `decisions[]` con `current_state`, `agreement`, `source`, `history[]`.
- `queue_buckets`: `today`, `this_week`, `waiting_validation`, `overdue`,
  `superseded`.
- `sla_flags`: `no_owner`, `no_kpi`, `overdue_due_at`, `validation_window_expired`.
- `feedback_to_cv86`: mapping decisión→familia con `outcome ∈ {validated,
  rejected, dismissed}` para alimentar `family_confidence`.

Función pura, recomputable, sin IO.

## Trazabilidad y privacidad

- Append-only 100%: toda transición = nuevo evento; el histórico se
  recompone.
- Auditoría: `actor_user_id`, `actor_role`, `occurred_at`, `reason`.
- Privacidad: la Decisión no almacena identidades de visitantes; opera
  sobre agregados de CV8.7/CV8.8 (MIN_SEGMENT_POPULATION = 25 respetado).
- Rationale y motivo son texto libre del operador; no se derivan de PII.

## Métricas del Workflow (auto-observabilidad)

- `decision_acceptance_rate` = accepted / proposed (7d, 30d).
- `decision_time_to_accept_p50` (horas desde proposed → accepted).
- `decision_time_to_implement_p50` (accepted → implemented).
- `decision_validation_rate` = validated / implemented.
- `decision_bias_flag_rate` = % dismissed con motivo "sesgo/explicación
  alternativa" (integra CV8.8).
- `decision_sla_breach_rate` = overdue / total activas.

Ninguna de estas métricas persiste — se recompone desde eventos.

## Outcome Validation

1. **¿Ayuda al Founder a decidir qué atender primero?** Sí — backlog con
   buckets `Hoy/Esta semana/Esperando/Vencidas` alimentado por CV8.7/CV8.8.
2. **¿Preserva gobernanza humana?** Sí — cero automatización, responsable
   obligatorio desde `accepted`, motivo obligatorio en
   `deferred/dismissed/blocked/rejected`.
3. **¿Es auditable?** Sí — append-only sobre `recommendation.lifecycle`,
   con `actor_user_id`, `actor_role`, `occurred_at`, historial derivable.
4. **¿Cierra el ciclo con CV8.6?** Sí — `validated/rejected/dismissed`
   inyectan señal a `family_confidence` por familia (`metric_id`).
5. **¿Respeta privacidad y segmentos frágiles?** Sí — opera sobre agregados
   (MIN_SEGMENT_POPULATION = 25 de CV8.8); no persiste identidades ni
   estados mutables; rationale humano no deriva de PII.
