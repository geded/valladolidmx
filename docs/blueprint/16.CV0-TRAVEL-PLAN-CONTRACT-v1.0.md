## 0. Filosofía del Producto

**ValladolidMX organiza viajes, no únicamente reservas.**

No somos un carrito de compras ni un motor de reservas. Somos una plataforma que acompaña al viajero durante todo su ciclo de vida:

> **Inspiración → Planificación → Decisión → Reserva → Experiencia → Seguimiento.**

El **Travel Workspace** es el expediente vivo del viajero. Nace cuando descubre el destino, crece cuando guarda experiencias, madura cuando Alux lo enriquece, se consolida cuando el concierge humano lo aterriza, se cierra cuando se paga, y sigue vivo durante y después del viaje para reseñas, seguimiento y regreso.

El **Travel Plan** es la entidad canónica dentro de ese expediente. Todo lo demás (Alux, Concierge, Drawer, Pagos, Analytics, CRM) gira alrededor de él.

## 1. Declaración de centralidad

- El Travel Plan es la entidad canónica del producto.
- Fuente única de verdad: `travel_plans` + `travel_plan_items` + `travel_plan_build_snapshot` + `traveler_profiles` + `concierge_case_links`.
- **Prohibido** crear un segundo modelo de viaje, snapshot paralelo, carrito alternativo o base de contexto adicional.

## 2. Consumer Journey Oficial

Embudo oficial del producto. Toda decisión de diseño, UI, IA y operación se alinea a este flujo:

```text
Explora → Descubre → Guarda → Alux recomienda → El Plan evoluciona
       → Concierge recibe → Propuesta → Aceptación → Pago
       → Viaje → Reseña → Regresa
```

| Etapa | Qué ocurre | Superficie |
|---|---|---|
| Explora | Visitante navega Discovery público | PublicShell + Cards |
| Descubre | Ve ficha de destino/negocio/experiencia | Micrositios EB |
| Guarda | "Agregar a Mi Viaje" desde cualquier card | Acción universal |
| Alux recomienda | Sugerencias contextuales + cercanía + clima | Concierge IA flotante + chips |
| El Plan evoluciona | Se acumulan ítems, fechas, viajeros, notas | Travel Drawer + `/cuenta/mi-viaje` |
| Concierge recibe | `promotePlanToCase` con snapshot inmutable | Workspace Concierge |
| Propuesta | Cotización, itinerario, ajustes | Concierge Case |
| Aceptación | Viajero acepta la propuesta | Mi Viaje |
| Pago | Cierre transaccional dentro del sitio | Checkout (CV4) |
| Viaje | Timeline vivo, Alux acompaña en destino | Alux + Timeline |
| Reseña | Post-canje / post-viaje | Reviews |
| Regresa | Segunda visita, memoria territorial | Alux M3/M4 |

Responsabilidad única por etapa. Evita duplicidades funcionales.

| Etapa | Owner funcional | No responsable |
|---|---|---|
| Explora / Descubre | **Discovery Layer** (PublicShell + Cards + EB) | Travel Plan, Concierge |

| Pago | **Checkout** (CV4) | Concierge, Travel Plan |
| Viaje en curso | **Alux + Timeline** | Concierge (backup) |
| Reseña | **Reviews** vinculado a canje/viaje | Alux |
| CRM / Reengagement | **CRM + Alux M4** | Concierge |
| Analytics | **Analytics** (read-only del snapshot) | — |

## 4. Ciclo de vida oficial del Plan

Estados canónicos (sin cambios de esquema, sólo declaración):

```text
draft → active → submitted → in_review → quoted → confirmed → completed
                                                            ↘ cancelled
```

| Transición | Responsable |
|---|---|
| draft → active | Viajero (primer ítem agregado) |
| active → submitted | Viajero (`promotePlanToCase`) |
| submitted → in_review | Concierge (auto al recibir) |
| in_review → quoted | Concierge (envía propuesta) |
| quoted → confirmed | Viajero (acepta) |
| confirmed → completed | Sistema (post-viaje) |
| * → cancelled | Viajero o Concierge (con nota) |

## 5. Contrato de lectura (Read Contract)

Firma única. Todos los consumidores usan estas puertas:

- `getMyActivePlan()` — viajero autenticado.
- `getPlanForConcierge(caseId)` — concierge humano.
- `getPlanContextForAlux(sessionKey | userId)` — Alux público y Traveler.
- `getPlanSnapshot(planId)` — analytics, pagos, propuestas (inmutable).

Estructura mínima expuesta: `id`, `title`, `destination`, `travelers`, `dates`, `items[]`, `notes`, `status`, `snapshot`, `links{ case_id?, proposal_id?, payment_id? }`.

---

## 6. Contrato de escritura (Write Contract)

Únicas puertas de escritura autorizadas:

- `addPlanItem` / `removePlanItem` / `updatePlanItem`
- `updatePlanMeta` (fechas, viajeros, notas, título)
- `promotePlanToCase` — **única puerta al Concierge**
- `attachAluxSuggestion` — Alux propone, viajero confirma (nunca escribe directo)
- `acceptProposal` / `cancelPlan`

**Prohibido** escribir directo a `travel_plans` / `travel_plan_items` fuera de estas funciones.

---

## 7. Contrato de señales (Signal Contract)

Eventos emitidos por el Plan que otros subsistemas escuchan:

- `plan.item_added` / `plan.item_removed` / `plan.item_updated`
- `plan.meta_updated`
- `plan.promoted_to_case`
- `plan.proposal_received`
- `plan.accepted` / `plan.paid` / `plan.completed` / `plan.cancelled`

Bus: `window` events (frontend) + `alux_public_signals` (server) + Realtime channels (concierge).

## 8. Contratos por consumidor

| Consumidor | Lee | Escribe | Señales que escucha |
|---|---|---|---|
| Arma tu Viaje (UI) | `getMyActivePlan` | add/remove/updateItem, updateMeta, promoteToCase | `plan.*` |
| Alux público | `getPlanContextForAlux` | `attachAluxSuggestion` (con confirmación) | `plan.item_added`, `plan.meta_updated` |
| Alux Traveler | `getPlanContextForAlux` | `attachAluxSuggestion` | `plan.*` |
| Alux Concierge | `getPlanForConcierge` | — (read-only) | `plan.promoted_to_case` |
| Concierge humano | `getPlanForConcierge` | crear propuesta/quote sobre plan | `plan.promoted_to_case`, `plan.accepted` |
| Travel Drawer | `getMyActivePlan` (contador + preview) | — | `plan.item_added/removed` |
| Analytics | `getPlanSnapshot` | — | todas |
| Conversión | `getPlanSnapshot` | — | `plan.promoted_to_case`, `plan.accepted`, `plan.paid` |
| Pagos | `getPlanSnapshot` | — | `plan.accepted` → checkout |
| CRM / Reengagement | `getPlanSnapshot` | — | `plan.completed`, `plan.cancelled` |

## 9. Reglas de invariancia

- Un plan tiene un solo `traveler_user_id`.
- Un plan puede tener 0..1 `concierge_case_id`.
- Ítems son idempotentes por `(plan_id, item_kind, target_id)`.
- El snapshot al promover es **inmutable** — foto del viaje al momento del handoff.
- Alux **nunca** modifica el plan sin confirmación explícita del viajero.
- El plan sobrevive al viaje (no se borra tras `completed`) — es memoria del viajero.

---

## 10. Métricas oficiales del Plan

Definición canónica (no dashboard):

- Planes creados / activos / promovidos / convertidos / cancelados.
- Ítems promedio por plan.
- Tiempo `active → submitted`, `submitted → quoted`, `quoted → accepted`, `accepted → paid`.
- Tasa de conversión por destino, categoría, motivo de viaje.
- Aporte de Alux: % de ítems agregados vía `attachAluxSuggestion`.
- Retención: % de viajeros con segundo plan.

## 13. Founder Product Axiom

> **Todo nuevo módulo que se incorpore a ValladolidMX deberá responder primero a la pregunta:**
>
> **"¿Cómo aporta valor al Travel Workspace del viajero?"**

Si una nueva funcionalidad no fortalece el Travel Workspace, deberá justificar explícitamente su existencia.

El **Travel Workspace** será considerado el **centro funcional del producto** y la referencia principal para la evolución del **Destination Operating System**.

Este axioma es permanente y aplica a toda iniciativa presente y futura.

---

## 14. Visión arquitectónica futura (no forma parte del alcance de CV0)

Registrada como **dirección estratégica**, no como compromiso de implementación.

El Travel Plan permanecerá como la **entidad canónica** del producto y su contrato técnico (este documento) continuará siendo la referencia oficial para todos los consumidores.

A nivel de **arquitectura funcional**, el producto evolucionará hacia el concepto de **Travel Workspace**, entendido como el espacio de trabajo integral del viajero. El Travel Workspace integrará progresivamente:

- Travel Plan
- Alux
- Concierge
- Travel Drawer
- Propuestas
- Pagos
- Timeline del viaje
- Seguimiento posterior al viaje

**Objetivo:** que el Travel Plan siga siendo la entidad central, mientras que el Travel Workspace represente la experiencia completa del viajero durante todo su ciclo.

Esta visión **no modifica el alcance ni el nombre** del presente documento "Travel Plan Contract v1.0".
