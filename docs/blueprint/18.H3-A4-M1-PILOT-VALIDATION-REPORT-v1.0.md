## 8. Prueba de fallo controlado y fallback

**Escenario:** invocar el CLI con un `asset_id` inexistente (`00000000-0000-4000-8000-000000000404`).

**Resultado del CLI:** el asset se marca en el reporte como `errors:["asset not found: no row"]`; **no** se modifica ningún registro de `media_assets` ni `media_asset_variants`; **no** se sube ningún binario; exit code = 0 con `failed=0` (la ausencia total de variantes no cuenta como fallo de variante).

## 10. Análisis de riesgo de colisión en la clave de idempotencia

**Clave actual (M0):** `UNIQUE (asset_id, format, width, engine)`.

**Metadatos que NO participan del key:** `engine_version`, `quality`, `usage_context`.

| Cambio | Comportamiento actual | Riesgo |
|--------|-----------------------|--------|
| Nueva `engine_version` (p. ej. `sharp@0.36.x`) | Path incluye `engine_version` ⇒ el binario nuevo se sube a un path distinto; la fila DB se sobreescribe vía `upsert`. | Binario anterior queda **huérfano** en `media-derived` (ocupa espacio; SEO no afectado — URLs internas no canónicas). Pérdida de trazabilidad histórica. |
| Cambio de `quality` (p. ej. avif 50 → 55) | Path NO incluye `quality`; el binario se sobreescribe (`upsert:true`) en la misma ruta. Fila DB se sobreescribe. | Sobrescritura silenciosa; ningún duplicado de fila. |
| Cambio de `usage_context` del asset | Path incluye `usage_context`; los binarios nuevos quedan en otra rama; DB se sobreescribe. | Binarios anteriores huérfanos. |
| Reprocesamiento con mismo semver de sharp | `engine_version` idéntico ⇒ idempotencia OK. | Ninguno. |

**Conclusión:** no hay riesgo de **duplicados** en DB (la unicidad se conserva), pero sí de **pérdida de historia** y **binarios huérfanos** en Storage cuando cambien `engine_version` / `quality`.

### 10.1 · Corrección aditiva propuesta (NO aplicada — requiere autorización)

Cambios **aditivos** (no destructivos) para robustecer el pipeline antes de M2 sin romper contratos ni el esquema actual:

1. **Nueva llave de unicidad opcional** (`ADD CONSTRAINT`, no reemplaza la actual):

   ```sql
   -- Sólo propuesta. Requiere autorización Founder.
   ALTER TABLE public.media_asset_variants
     ADD COLUMN variant_key text GENERATED ALWAYS AS (
       asset_id || ':' || format || ':' || width || ':' || engine ||
       ':' || COALESCE(metadata->>'engine_version','') ||
       ':' || COALESCE(quality::text,'')
     ) STORED;
   CREATE UNIQUE INDEX media_asset_variants_variant_key_ux
     ON public.media_asset_variants(variant_key);
   ```

   La UNIQUE actual `(asset_id,format,width,engine)` se **mantiene** para preservar la semántica "una variante activa por dimensión × engine".

2. **Política de purga de huérfanos** en `media-derived`: job programable que compara paths en Storage contra `media_asset_variants.path` y borra los no referenciados con `engine_version` retirado (ventana ≥30 días).

3. **Marca de retiro** (`status='superseded'`) al detectar cambio de `engine_version`/`quality`, en lugar de sobrescribir la fila.

**Nada de esto se implementa en M1.** Se registra como pre-requisito recomendado antes de activar el flag globalmente.
