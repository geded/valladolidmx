**Estado:** Blueprint (estrategia). No autorizado a migrar hasta validación.
**Fecha:** 2026-07-14
**Autor:** Lovable Agent · aprobado para diseño por el Founder
**Épica origen:** H3 · Image Pipeline & Visual Optimization
**Depende de:** H3·A3.b (Media Intelligence · `resolveMediaAlt` oficial)

---

## 1. Behavioral Change Statement

Hoy cada imagen se sirve en su formato original (JPEG/PNG) al tamaño en que fue subida, sin variantes ni negociación de formato. El visitante descarga píxeles innecesarios, el LCP sufre y el CLS aparece cuando faltan dimensiones. Cargar una imagen al CMS no dispara ningún trabajo derivado.

**Cambio esperado:** al cargar una imagen, el pipeline genera de forma transparente su familia de derivados (AVIF, WebP, JPEG de respaldo × múltiples anchos), preserva el original inmutable y expone URLs estables. El renderizador consume `<picture>` con `srcset`, sin conocer la mecánica interna. La operación es reversible y escala a decenas de miles de imágenes sin intervención humana.

---

## 2. Principios rectores (vinculantes)

1. **El original nunca se pierde.** Se guarda inmutable en `media-original` (bucket separado, sin sobrescritura).
2. **Las variantes son derivadas.** Cualquier variante puede regenerarse desde el original en cualquier momento.
3. **Las URLs públicas permanecen estables.** El path canónico no cambia; el pipeline resuelve el mejor formato/tamaño detrás del mismo identificador.
4. **Transparencia total.** CMS, Studio y bloques siguen usando los mismos contratos (`media_id`, `resolveMediaAlt`, `resolveMediaSource`); no conocen AVIF/WebP.
5. **Segura a gran escala.** El pipeline es idempotente, reintentable, con backpressure y observabilidad.
6. **Compatible con `resolveMediaAlt`.** Todo consumidor futuro obtiene URL + ALT desde un único `resolveMedia()` que compone ambos resolvers.

## 3. Arquitectura del Pipeline

### 3.1 Componentes

```
 Upload (CMS/Studio)
        │
        ▼
 [1] Storage: media-original/{asset_id}/{filename}     ← inmutable
        │
        ▼
 [2] Ingest hook (server fn)  →  media_asset_variants (pending)
        │
        ▼
 [3] Worker de derivación (async, por lotes)
        │        ├── AVIF q=50   × [400, 800, 1200, 1600, 2000]
        │        ├── WebP q=75   × [400, 800, 1200, 1600, 2000]
        │        └── JPEG q=80   × [800, 1600]             ← fallback
        ▼
 [4] Storage: media-derived/{asset_id}/{width}.{format}
        │
        ▼
 [5] resolveMediaSource(media, { width, dpr, formats })   ← nueva API
        │
        ▼
 [6] <PictureAsset> component  →  <picture><source …><img …/></picture>
        │
        ▼
 [7] CDN (Lovable/Supabase Storage + Cloudflare) con cache long-lived
```

### 3.2 Modelo de datos

Extender `media_assets` sin romper contrato existente:

```sql
alter table public.media_assets
  add column if not exists original_bucket text default 'media-original',
  add column if not exists original_path   text,
  add column if not exists derivation_state text
    check (derivation_state in ('pending','processing','ready','failed','skipped'))
    default 'pending',
  add column if not exists derivation_error text,
  add column if not exists derived_at timestamptz;

create table if not exists public.media_asset_variants (
  id uuid primary key default gen_random_uuid(),
  media_id uuid not null references public.media_assets(id) on delete cascade,
  format text not null check (format in ('avif','webp','jpeg','png')),
  width  int  not null,
  height int  not null,
  bytes  int  not null,
  storage_bucket text not null,
  storage_path   text not null,
  quality int,
  created_at timestamptz not null default now(),
  unique (media_id, format, width)
);
```

Grants + RLS: `select` a `anon`/`authenticated`; `insert/update/delete` sólo `service_role` (el worker).

### 3.3 Buckets

| Bucket             | Público | Contenido                          | Cache-Control                         |
|--------------------|:-------:|------------------------------------|---------------------------------------|
| `media-original`   |   No    | Original inmutable                 | `private, max-age=0`                  |
| `media-derived`    |   Sí    | Variantes por `{width}.{format}`   | `public, max-age=31536000, immutable` |
| `media` (legacy)   |   Sí    | Compatibilidad hacia atrás         | Sin cambios                            |

## 4. Estrategia de generación de variantes

### 4.1 Matriz de derivados

| Formato | Calidad | Anchos (px)                     | Uso                              |
|---------|:-------:|---------------------------------|----------------------------------|
| AVIF    | 50      | 400, 800, 1200, 1600, 2000      | Primera opción (moderno)         |
| WebP    | 75      | 400, 800, 1200, 1600, 2000      | Fallback amplio (>95% browsers)  |
| JPEG    | 80      | 800, 1600                        | Fallback universal / OG images   |

Se omiten anchos > que el original (nunca se upscale). Si `original.width < 400`, se genera sólo el tamaño real.

### 4.2 Motor

- **Preferido:** `@cf-images` (Cloudflare Image Resizing) — negociación en el edge, sin CPU propio, sin almacenamiento derivado. Bucket derived se vuelve caché opcional.
- **Alternativa self-hosted:** worker Node con `sharp` sobre Supabase Storage; procesa vía cola (`pg_boss` o cron cada 30s).
- **Decisión de arquitectura pendiente:** benchmark comparativo (§9) antes de comprometer motor.

### 4.3 Reglas

- Idempotente: mismo `(media_id, format, width)` no se recomputa si ya existe y el hash del original coincide.
- Reintentos exponenciales (3, 30, 300 s); tras 3 fallos → `derivation_state='failed'` + alerta.
- Presupuesto de CPU/tiempo por asset (skip si > 30 s) → marca `skipped` y usa original.

## 5. Estrategia de cache

1. **CDN edge:** `Cache-Control: public, max-age=31536000, immutable` en `media-derived/*`. Los paths incluyen el `asset_id` (inmutable) + `{width}.{format}`, por lo que nunca se invalida en caliente.
2. **Cache-busting sólo si el original cambia:** un nuevo upload = nuevo `asset_id` (asset inmutable por diseño). El anterior queda referenciado hasta su borrado explícito.
3. **HTTP `Accept` negotiation:** cuando la ruta es Cloudflare Image Resizing, el edge sirve AVIF/WebP según cabecera. Con self-hosted, `<picture>` hace la negociación en el cliente.
4. **Stale-while-revalidate** en el resolver de servidor (5 min) para las lecturas de `media_asset_variants`.

## 7. Compatibilidad hacia atrás

- Contrato `media_assets.storage_path` intacto → todo consumidor legacy sigue funcionando.
- Nueva API `resolveMediaSource(media, opts) → { src, srcSet, sources[], width, height }` con **fallback automático al original** cuando `derivation_state !== 'ready'`.
- URLs públicas actuales (`.../media/xxxx.jpg`) **no cambian**; los derivados viven en `.../media-derived/{id}/{w}.{fmt}`.
- OG/Twitter images: se mantiene JPEG 1600 como formato canónico (compatibilidad con crawlers).
- SEO: sin cambio de URL → sin redirects → sin pérdida de señal.

## 9. Coste estimado

**Supuestos:** 10 000 imágenes objetivo, promedio 1.2 MB original, 12 variantes × ~120 KB.

| Concepto            | Volumen              | Coste unitario           | Coste mensual estimado |
|---------------------|----------------------|--------------------------|-------------------------|
| Storage original    | ~12 GB               | $0.021/GB (Supabase)     | ~$0.25                  |

| Procesamiento inicial (sharp self-hosted) | ~10k × 2s CPU | ~5.5h CPU × $0.05/h | ~$0.30 (una vez) |

**Total operativo estimado:** < $1/mes self-hosted · $5–15/mes con Cloudflare Image Resizing. La decisión de motor se toma tras benchmark.

## 10. Plan de rollback

1. **Feature flag** `mediaPipeline.v2.enabled` (por env). Off → `resolveMediaSource` devuelve el original y `<PictureAsset>` degrada a `<img>` clásico.
2. **Rollback por asset:** `update media_assets set derivation_state='skipped' where id=…` → resolver ignora variantes.
3. **Rollback global:** apagar worker + flag off; no se toca ningún original ni ninguna URL pública.
4. **Purga segura de derivados:** `truncate media_asset_variants` + limpiar bucket `media-derived` (script idempotente). Originales intactos. Regeneración bajo demanda.
5. **Auditoría continua:** vista `v_media_pipeline_health` con conteos por `derivation_state` + alertas de `failed` > 1 %.
