## 2. Base de datos (migración única)

- **Tablas:**
  - `public.marketplace_search_metrics` — telemetría de búsqueda. Sin
    grants para `anon` / `authenticated`; RLS habilitado y sin policies
    (lectura solo vía RPC admin).
  - `public.system_alerts` — alertas técnicas y funcionales. Mismo
    modelo de aislamiento (RLS sin policies, lectura vía RPC admin).
- **Enums:** `system_alert_severity` (`info`/`warning`/`critical`) y
  `system_alert_status` (`open`/`acknowledged`/`resolved`).
- **Índices:** `idx_msm_created_at`, `idx_msm_zero_results` (parcial),
  `uq_system_alerts_open_kind` (parcial), `idx_system_alerts_status_last`.
- **RPCs `SECURITY DEFINER` con `search_path = public` y grants explícitos:**
  - Server-only (`TO service_role`):
    `record_search_metric`, `raise_system_alert`.
  - Admin (`TO authenticated`, gated por `has_role`):
    `admin_marketplace_funnel`, `admin_top_products`,
    `admin_search_metrics_summary`, `admin_list_system_alerts`,
    `admin_acknowledge_system_alert`, `admin_resolve_system_alert`,
    `admin_evaluate_functional_alerts`.

Todas las RPCs administrativas validan rol `admin` o `super_admin`
server-side; cualquier llamada sin rol responde `forbidden`.

## 3. Server functions

`src/lib/observability/observability.functions.ts` expone:

- `getMarketplaceFunnel` — embudo Marketplace → Arma tu Viaje → Carrito
  → Pago → Reserva confirmada (+ leads, reservado para futura captura).
- `getTopProducts` — top productos por `most_added` / `most_reserved` /
  `most_abandoned`.
- `getSearchMetricsSummary` — total, zero-results, rate, p50/p95 ms,
  top términos sin resultados.
- `listSystemAlerts`, `acknowledgeSystemAlert`, `resolveSystemAlert`.
- `evaluateFunctionalAlerts` — evaluación on-demand de umbrales.

Todas usan `requireSupabaseAuth` + RPC con verificación de rol; los
payloads salen serializados como string JSON para respetar las
restricciones de serialización del runtime.

---

## 4. Instrumentación

- `searchMarketplace` (Etapa 2) registra cada llamada en
  `marketplace_search_metrics` con `result_count` y `duration_ms`,
  y emite alerta crítica `api.search_marketplace.error` ante fallo
  del RPC público. La observabilidad nunca rompe el flujo principal.
- Webhook `/api/public/payments/$provider/webhook` emite alerta crítica
  `payments.webhook.<provider>.verify_failed` ante fallo de
  verificación de firma, antes de responder `401`.
- Inserciones se hacen vía `supabaseAdmin.rpc(...)` con `SECURITY DEFINER`,
  manteniendo el principio de no exponer `SUPABASE_SERVICE_ROLE_KEY`
  en flujos de usuario.

## 5. UI administrativa

Dos nuevas rutas dentro del shell CMS Studio:

- `/cms/observabilidad` — embudo de negocio, top productos
  (most_added / most_reserved / most_abandoned), métricas de búsqueda
  (p50/p95, zero-results, top términos sin resultados). Selector de
  ventana 7/14/30/90 días.
- `/cms/alertas` — listado por estado (open/acknowledged/resolved/all),
  reconocimiento, resolución, y evaluador on-demand de umbrales
  funcionales (1h).

La sidebar del CMS incorpora ambos enlaces.

## 6. Alertas funcionales cubiertas

- `payments.webhook.<provider>.verify_failed` — webhooks de pago con
  fallos de verificación (críticas, agrupadas por proveedor).
- `payments.failed_spike` — incremento de pagos fallidos en ventana
  configurable (umbral por defecto: ≥ 5 / 60 min).
- `orders.pending_stale` — órdenes `pending` con > 2 h de antigüedad
  por encima del umbral (≥ 10).
- `search.zero_results_spike` — tasa de zero-results ≥ 40 % sobre ≥ 20
  búsquedas en la ventana.
- `api.search_marketplace.error` — error crítico de la API pública de
  búsqueda.

Todas las alertas son idempotentes por `kind` mientras estén `open`
(índice único parcial), incrementando `occurrences` y actualizando
`last_seen_at` ante repetición.

## 8. Hardening

- `EXECUTE` revisado en todas las RPCs nuevas: `REVOKE ALL FROM PUBLIC`
  + `GRANT EXECUTE` explícito al rol estrictamente necesario
  (`service_role` para helpers internos, `authenticated` para RPCs
  admin con verificación de rol interna).
- `search_path = public` fijo en todas las funciones `SECURITY DEFINER`
  introducidas.
- Tablas nuevas con RLS habilitado y sin policies (acceso únicamente
  vía `service_role` y RPCs `SECURITY DEFINER` autorizadas), evitando
  exposición accidental al cliente.
- Retención / archivado de `*_events` y `*_audit_log`: lineamiento
  documental, sin ejecución automatizada en esta etapa.
