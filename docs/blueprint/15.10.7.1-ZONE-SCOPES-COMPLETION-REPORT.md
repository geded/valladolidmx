# 15.10.7.1 · Permisos por zona — Completion Report v1.0

**Estado:** ✅ Cerrada
**Etapa padre:** 15.10.7 — Empresas y productos · onboarding multiempresa
**Fecha:** 2026-07-03

## 1. Alcance entregado

Infraestructura de permisos acotados a zonas geográficas (regiones
turísticas y destinos), sin alterar los roles globales ni el sistema
`user_roles` existente. Los scopes son **aditivos**: un usuario con
`editor` global sigue teniendo alcance global; con `editor` + scope
de región X, el consumidor puede decidir acotarlo cuando aplique.

## 2. Cambios de base de datos

| Objeto | Tipo | Propósito |
| --- | --- | --- |
| `public.user_zone_scopes` | Tabla | Vincula `user_id` × (`scope_type`, `scope_id`) × `role` |
| `public.has_zone_scope(uuid, text, uuid, app_role)` | RPC | Verifica scope específico |
| `public.user_zone_scopes_for(uuid)` | RPC | Lista scopes activos de un usuario |
| `public.assign_zone_scope(uuid, text, uuid, app_role, text)` | RPC | Alta (sólo super_admin/admin) |
| `public.revoke_zone_scope(uuid)` | RPC | Baja (sólo super_admin/admin) |

RLS activo. GRANTs: `SELECT` a `authenticated`, `ALL` a `service_role`.
Escrituras exclusivamente vía RPCs SECURITY DEFINER; `EXECUTE`
revocado de `PUBLIC`/`anon`.

## 3. Cambios de código

- `src/lib/admin/zone-scopes.functions.ts` — `listUserZoneScopes`,
  `assignUserZoneScope`, `revokeUserZoneScope` (todas
  `requireSupabaseAuth`).
- `src/components/admin/ZoneScopesDialog.tsx` — diálogo de gestión
  reutilizable.
- `src/routes/_authenticated/admin/sistema.usuarios.tsx` — botón
  **Zonas** por fila de usuario en Staff/Empresas/Viajeros.

Sin rutas nuevas. Sin engines nuevos. Sin cambios en Workspace Engine,
Design System, BEA, Auth Broker ni Discovery Layer.

## 4. Invariantes respetadas

- Infrastructure Freeze: 0 nuevos engines/registries/capas.
- Workspace First: la UI vive en superficie autenticada existente.
- Roles siguen almacenándose en `user_roles`; `user_zone_scopes` es
  complemento, no reemplazo.
- BEA: cero eventos nuevos.
- Grants explícitos en la migración.

## 5. Validación

- `tsgo --noEmit`: ✅ verde.
- Migración aplicada sin errores.
- Warnings del linter Supabase (SECURITY DEFINER) heredados del
  proyecto y consistentes con el patrón oficial (REVOKE PUBLIC/anon,
  GRANT sólo a authenticated/service_role).

## 6. Adopción diferida (fuera de esta sub-adenda)

El helper `has_zone_scope` queda disponible para que futuras adendas
lo consuman en:

- Filtros por zona en listados CMS (destinos, empresas, productos).
- `eb_can_edit_scope` opcional para editores zonales.
- Portal empresarial: acotar zonas visibles por usuario.

Cada adopción irá acompañada de su propia sub-adenda para preservar
la trazabilidad y el principio "una historia a la vez".
