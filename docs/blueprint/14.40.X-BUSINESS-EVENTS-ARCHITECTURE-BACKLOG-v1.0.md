# 14.40.X — Business Events Architecture (Backlog Arquitectónico)

**Tipo:** Lineamiento arquitectónico para futuras olas.
**Origen:** Aprobación formal del Reporte 14.40.5.
**Reutiliza:** `order_events`, `payment_events`, `content_audit_log`,
`permissions_audit_log`.

## Visión

La plataforma Valladolid.mx evolucionará hacia una **arquitectura basada
en eventos de negocio**, donde toda acción relevante del dominio emite
un evento normalizado, reutilizable por:

- Auditoría regulatoria y trazabilidad.
- Analítica de producto y BI.
- CRM y automatizaciones de marketing.
- EMS (Experience Management System).
- Alux Intelligence (contexto y recomendaciones).
- Portal Empresarial (reporting al socio).

## Forma canónica del evento

```ts
interface BusinessEvent {
  id: string;                 // uuid
  type: string;               // ej. "Payment.Succeeded"
  occurred_at: string;        // ISO timestamp
  actor: {
    kind: "user" | "system" | "provider";
    id: string | null;
  };
  subject: {                  // entidad de negocio principal
    kind: string;             // "order" | "trip" | "business" | ...
    id: string;
  };
  context: Record<string, unknown>;  // metadata acotada y serializable
  source: string;             // módulo emisor (marketplace, portal, ems…)
  version: 1;
}
```

## Principios

- **Inmutables**: un evento, una vez emitido, no se modifica.
- **Idempotentes**: deduplicación por `(source, id)`.
- **Serializables**: payloads JSON estables, versionados (`version`).
- **Neutrales al proveedor**: nunca incluyen secretos ni payloads crudos
  de terceros (esos viven en `payment_events.payload`).
- **Reutilización**: el bus de eventos se monta sobre las tablas de
  auditoría existentes; no se introduce infraestructura nueva en olas
  intermedias salvo evaluación explícita.
