## 1. Resumen Ejecutivo

La Ola 3 entrega el Portal Empresarial completo, aislado bajo el subtree
`_authenticated/portal/*`, sin alterar el Blueprint, el modelo de dominio
ni la RLS aprobada en Fase 1. El Portal habilita a propietarios y editores
autorizados de cada empresa a administrar su ficha pública, presencia
(contactos, ubicaciones, horarios, redes), galería multimedia, productos
y promociones, manteniendo intacto el workflow editorial: aprobación y
publicación permanecen reservadas al CMS Studio.

Toda escritura pasa por server functions con `requireSupabaseAuth` y por
RPCs `SECURITY DEFINER` con `search_path = public` y `EXECUTE` restringido
a `authenticated`. No se usa `SUPABASE_SERVICE_ROLE_KEY` en ningún flujo
de usuario. La auditoría se consolida en `content_audit_log`.

## 3. Matriz Final de Permisos

Roles de empresa (`business_users.role`): `owner` ≥ `editor` ≥ `viewer`.
Roles globales (CMS): `admin`, `editor` (vía `has_role`).

| Acción | viewer | editor | owner | CMS editor/admin |
|--------|:------:|:------:|:-----:|:----------------:|
| Listar empresas accesibles | ✓ | ✓ | ✓ | ✓ |
| Ver ficha / presencia / galería / catálogo | ✓ | ✓ | ✓ | ✓ |
| Editar ficha (whitelist) | — | ✓ | ✓ | ✓ |
| CRUD presencia (contactos/ubicaciones/horarios/redes) | — | ✓ | ✓ | ✓ |
| Subir / actualizar / eliminar media | — | ✓ | ✓ | ✓ |
| Crear / editar / archivar productos y promociones | — | ✓ | ✓ | ✓ |
| Solicitar / retirar revisión | — | ✓ | ✓ | ✓ |
| Aprobar / publicar / despublicar | — | — | — | ✓ (sólo CMS Studio) |
| Invitar colaboradores | — | — | ✓ | — |
| Aceptar invitación (destinatario) | n/a | n/a | n/a | n/a |

`has_business_access(business_id, min_role)` gobierna toda escritura del
Portal; las RPCs SECURITY DEFINER lo revalidan internamente.

## 10. Recomendaciones para la Ola 4 — Marketplace

1. **Vitrina pública del catálogo**: exponer productos/promociones
   `published` en rutas públicas con SSR, derivando `og:image` del media
   principal. Reutilizar `public-reads.functions.ts` y nuevas vistas
   read-only con políticas `TO anon` estrictas.
2. **Búsqueda y filtros**: índices funcionales sobre `products` y
   `promotions` (categoría, destino, rango de precio, vigencia). Evitar
   joins amplios en runtime — preparar vistas materializadas si la
   cardinalidad lo amerita.
3. **Carrito y reservas**: nueva tabla `bookings`/`orders` con su propia
   serie de RPCs SECURITY DEFINER. No tocar Portal ni CMS; integrar a
   través de eventos auditados.
4. **Pagos**: integrar pasarela (Stripe) detrás de server functions con
   `requireSupabaseAuth`; webhooks bajo `/api/public/*` con verificación
   HMAC. Service role sólo en `*.server.ts` para reconciliación.
5. **Transferencia de propiedad**: cerrar R5 antes de habilitar cobros.
6. **Verificación consolidada de Storage**: extender la checklist de
   MIME/tamaño/bucket/nombre a cualquier nuevo bucket del Marketplace.
7. **Observabilidad**: dashboards de `content_audit_log` y métricas de
   uso del Portal antes de abrir el Marketplace al público.
8. **Plan documental**: emitir `14.40 — Wave 4 · Marketplace · Plan de
   Ejecución v1.0` con la misma metodología por etapas (1..N) + reportes
   por etapa + cierre `14.40.Z`.
