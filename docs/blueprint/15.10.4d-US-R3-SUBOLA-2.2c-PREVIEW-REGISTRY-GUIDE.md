# 15.10.4d · US-R3 · Sub-ola 2.2c
# Guía · Cómo agregar un nuevo Preview Provider

**Aplica a:** toda plantilla madre dinámica del Experience Builder
(Business, Region, Destination, Product, Event, Experience, Hotel,
Restaurant y futuras).

**Regla vinculante (Founder, 2.2c):** un nuevo `page_kind` debe poder
entrar al sistema **registrando su provider aquí y nada más**. Está
prohibido tocar `CompositionRenderer`, rutas públicas o `VisualStudio`
para dar de alta un provider.

---

## 1 · Requisitos previos

El `kind` ya debe existir en:

1. El enum `public.eb_page_kind` de la base de datos.
2. `src/lib/experience-builder/page-kind-registry.ts` (label, ruta
   pública, defaults SEO/sitemap).
3. Una **superficie React** con su `Provider` de contexto (ej.
   `BusinessSurfaceProvider`) que exponga el dato de dominio al árbol
   de bloques `vmx.<kind>.*`.

Si el nuevo `kind` reutiliza el modelo de datos de otro (por resolución
H-R3-2: Hotel/Restaurante/Experiencia comparten `businesses`), puedes
reutilizar el mismo `Provider` — no dupliques.

---

## 2 · Registrar el provider

Abrir `src/lib/experience-builder/preview-registry.tsx` y añadir una
entrada al arreglo `REGISTRY`:

```ts
{
  kind: "restaurant",                     // === page_kind
  label: "Vista previa con restaurante",  // barra del Studio
  placeholder: "Selecciona un restaurante…",
  async loadCandidates() {
    const items = await listMarketplaceBusinesses();
    return items
      .filter((b) => (b.category_slug ?? "").includes("restaurant"))
      .map((b) => ({ slug: b.slug, label: b.display_name }));
  },
  async loadDetail(slug) {
    return await getMarketplaceBusinessBySlug({ data: { slug } });
  },
  demoData: demoRestaurant,     // dato estable para plantilla sin datos reales
  Provider: BusinessDataProvider,
} satisfies TemplatePreviewProvider<MarketplaceBusinessDetail>,
```

Con eso, al abrir en el Studio la plantilla del kind `restaurant`,
aparece automáticamente la barra "Vista previa con restaurante" y el
canvas se hidrata con el detalle real (o con `demoData` si no hay).

---

## 3 · Contrato del `Provider`

```ts
Provider: ComponentType<{
  data: TData | null;      // resultado de loadDetail() o demoData()
  children: ReactNode;
}>
```

- Debe montar el `SurfaceContext` que consumen los bloques del kind.
- Debe tolerar `data === null` sin lanzar (los bloques hijos muestran
  su propio `EmptyHint` cuando no hay datos).

---

## 4 · Selección del preview

La selección del slug se persiste en `localStorage`
(`eb.preview.<kind>.slug`). El editor la ve idéntica la próxima vez
que abra la plantilla. Cambiar a *"Datos demo"* limpia la clave.

---

## 5 · Fallback controlado

Todo el árbol de preview está envuelto por `PreviewProviderBoundary`
en `VisualStudio.tsx`. Si el `Provider` de un kind falla en render,
el Studio muestra un aviso reversible y el canvas se pinta **sin
hidratación** — el editor puede seguir trabajando la plantilla. Si
`loadDetail()` lanza, la barra recae automáticamente en `demoData()`.

---

## 6 · Paridad Studio ↔ Preview ↔ Público

El `CompositionRenderer` es el mismo en las tres superficies:

- **Studio:** `VisualStudio.tsx` envuelve el árbol con el `Provider`
  del preview y con `PreviewProviderBoundary`.
- **Preview token** (`/preview/composition/$token`): renderiza sin
  provider; asume que la ruta pública equivalente hidrata datos.
- **Público** (`/l/$slug`, `/marketplace/$slug`, `/oriente-maya/*`…):
  la ruta pública envuelve el árbol con el mismo `Provider` de la
  superficie del kind.

Cambios visuales en un bloque `vmx.<kind>.*` impactan las tres capas
por construcción — está prohibido introducir estilos exclusivos del
Studio.

---

## 7 · Checklist para el PR

- [ ] Entrada nueva en `REGISTRY` de `preview-registry.tsx`.
- [ ] `Provider` reutilizado o nuevo, sin lógica en `CompositionRenderer`.
- [ ] `demoData()` estable y sin datos reales de producción.
- [ ] `loadCandidates()` respeta RLS del cliente publishable.
- [ ] Cero cambios en `VisualStudio.tsx`, rutas `/marketplace/*`,
      `/oriente-maya/*`, `/l/*` u otras superficies públicas.
- [ ] Screenshot Studio ↔ Público lado a lado (paridad 1:1).
