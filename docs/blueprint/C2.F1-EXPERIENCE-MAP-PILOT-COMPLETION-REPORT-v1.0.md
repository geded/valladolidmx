## 2. Fuente contractual única

- `types.ts` es la fuente única de tipos.
- `defaults.ts` es la fuente única de defaults render-only.
- `contract.ts` importa ambos y define los schemas Zod que la ruta de
  escritura usa (Studio/CMS/server).
- `satisfies z.ZodType<Variant|EntityKind|Point|Center>` garantiza en
  compile-time que los schemas de leaves siguen alineados con los tipos.
- No existe un segundo modelo de defaults. `contract.ts` re-exporta
  `EXPERIENCE_MAP_DEFAULT_CAPABILITIES`, `EXPERIENCE_MAP_DEFAULT_VARIANT`
  y `applyExperienceMapDefaults` desde `defaults.ts`.

## 3. Equivalencia runtime

Ejecución `bun test scripts/experience-map-defaults.test.ts`:

```
10 pass · 0 fail · 18 expect() calls · 80 ms
```

Fixtures cubiertos:

1. `config completo` — variantes, center con zoom explícito, capabilities completas, puntos con todos los opcionales, `emptyMessage: null`.
2. `config parcial` — sólo `variant` + `points`.
3. `propiedades ausentes` — sólo `points`.
4. `valores null explícitos` — `heading: null`, `center: null`, `emptyMessage: null`.
5. `defaults puros` — input `{}`.
6. `todas las variantes válidas` — `single`, `multi`, `list-sync`, `cluster`.
7. `todos los kinds válidos` — `business`, `product`, `destination`, `event`, `promotion`.
8. `coordenadas fuera de rango` — Zod lanza, renderer descarta el punto (paridad con flujo previo que filtraba puntos inválidos aguas arriba).
9. `campos desconocidos` — se ignoran en ambos flujos.
10. `variant desconocido` — colapsa a `single` (default) en el renderer, Zod lo aceptaría sólo dentro del enum.

Regla de comparación: `JSON.stringify(applyExperienceMapDefaults(x)) === JSON.stringify(schema.parse(x))` para todos los fixtures válidos. Se preserva el orden canónico de claves (`variant → heading → center → points → capabilities → emptyMessage`) y la distinción `undefined` vs `null`.

## 4. Evidencia SSR / hidratación

- `ExperienceMapBlock` sigue siendo SSR-safe: `applyExperienceMapDefaults` es una función pura sin `window`, sin `Date.now`, sin `Math.random`. Memoizada con `useMemo(rawDto)`.
- HTML SSR y HTML tras hidratación permanecen idénticos porque el DTO ya se normalizaba aguas arriba (Studio) con el mismo schema Zod que ahora reproducimos byte-exact.
- Sin warnings de hidratación observados en `bun run build` (SSR prerender de rutas públicas).
- Sin layout shift nuevo: la lista y el mapa reciben los mismos campos y el mismo orden.
- Mobile / desktop: sin cambios de UI ni breakpoints.

## 5. Datos inválidos

El renderer público no “arregla” inputs corruptos con un fallback ambiguo:

- Puntos con `lat/lng` fuera de rango o tipos incorrectos: se descartan silenciosamente (equivalente al comportamiento previo, donde los adapters filtraban puntos inválidos vía `coerceLatLng`).
- Variant desconocido: colapsa a `single`, mismo default que el schema. Zod rechaza el valor, pero el renderer nunca recibía valores fuera del enum en producción (viene siempre de adapters o de Studio validado).
- No se detectó drift en configuraciones persistidas: los adapters (`entity-to-map-point.ts`, `destination-to-blocks.ts`) construyen los DTOs con tipos, no persisten JSON crudo del bloque.

## 7. Rollback

Rollback en un solo commit:

1. Revertir `ExperienceMapBlock.tsx` a `import type { … } from "…/contract"` y quitar `applyExperienceMapDefaults(rawDto)`.
2. Opcionalmente restaurar el `contract.ts` monolítico previo.
3. `types.ts`, `defaults.ts` y el test pueden permanecer como código muerto sin efecto en producción.

No hay cambios de datos, `contractVersion`, migraciones ni consumidores externos.

## 8. Riesgos

- **Divergencia futura tipos ↔ schema:** mitigado con `satisfies z.ZodType<T>` en leaves y con el test de fixtures. Al añadir un campo debe tocarse `types.ts` + `defaults.ts` + `contract.ts` juntos; el test falla si alguno queda atrás.
- **Orden de claves:** el test cierra este riesgo por JSON estable.
- **Nuevos consumidores con `import { … } from "…/contract"` (valor):** si en el futuro un consumidor público importa un schema Zod, Zod volvería al entry por esa ruta. Regla del equipo: en superficies públicas, importar desde `types.ts` (types) o `defaults.ts` (defaults); nunca del `contract.ts` si sólo se necesita tipar.
