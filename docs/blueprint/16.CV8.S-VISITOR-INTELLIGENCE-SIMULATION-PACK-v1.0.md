# CV8.S · Visitor Intelligence Simulation Pack v1.0

**Estado:** Blueprint oficial · Aprobado Founder · 2026-07-14
**Épica:** Independiente. NO forma parte del producto productivo.
**Objetivo:** poblar el Visitor Intelligence Center con una simulación
realista de ~90 días de operación turística del Oriente Maya (capital:
Valladolid) para validar toda la inteligencia construida en CV8.0..CV8.9.

> Política vinculante: **Founder Reproducible Simulation Principle**
> (`mem://policies/founder-reproducible-simulation.md`).

---

## 1. Behavioral Change Statement

Hoy el Founder no puede validar Journey Funnel, Segmentación, Benchmarks,
Recommendation Validation, Decision Prioritization, Fair Opportunity ni
Opportunity Intelligence porque el Center está vacío. Al cerrar CV8.S,
el Founder recorre el Visitor Intelligence Center completo y observa una
operación turística realista de Valladolid + Oriente Maya durante 90 días
**sin distinguir visualmente que se trata de una simulación**, y puede
eliminar todo con un único comando.

## 2. Alcance

### 2.1 Destino principal
**Valladolid** como capital turística. Los visitantes navegan naturalmente
por Chichén Itzá, cenotes, Izamal, Ek Balam, Espita, Río Lagartos, Las
Coloradas — con rutas heterogéneas (no todos recorren lo mismo).

### 2.2 Volumen
~10,000 visitantes anónimos (escala **medium** por defecto). Escalas
configurables: **light** ≈1k · **medium** ≈10k · **full** ≈100k (autorizada).

### 2.3 Ventana temporal
90 días con distribución no uniforme: weekends, vacaciones, alta/baja
temporada, días lluviosos, eventos especiales.

### 2.4 Perfiles de viajero (mínimo)
pareja internacional · pareja nacional · familia · mochilero · jubilados ·
gastronómico · cultural · naturaleza · lujo · excursionista de un día ·
huésped en hotel · con auto rentado. Cada perfil con propensiones distintas
(profundidad exploración, tasa favorito→plan, aceptación Concierge, ticket,
idioma, sensibilidad al clima).

### 2.5 Comportamiento observable
Exploración → intención (favorito/comparar) → Travel Plan (algunos
abandonan, otros regresan) → Concierge → propuesta → aceptación/rechazo →
pago → viaje → reseña → embajador. Todo bajo Journey Contracts v1.0.0
(T1..T9).

### 2.6 Empresas
Distribución heterogénea: hoteles high-conversion, restaurantes muy
visitados, experiencias con alto abandono, empresas nuevas, empresas con
pocas visitas, empresas con mala respuesta al Concierge.

### 2.7 Alux
Preguntas frecuentes, recomendaciones aceptadas/rechazadas, optimizaciones
de itinerario, consultas en destino.

### 2.8 Concierge
Casos abiertos, propuestas, aceptación/rechazo, pagos, seguimientos,
cierres exitosos y tardíos.

### 2.9 Problemas plantados (para validar detección)
Abandono temprano, baja conversión, idioma incorrecto, empresas con mala
respuesta, propuestas tardías, destinos con poca exploración, cuellos de
botella. El sistema (CV8.5 Opportunity + CV8.7 Prioritization + CV8.8 Fair
Opportunity) debe detectarlos correctamente.

## 3. Arquitectura

### 3.1 Módulos (`src/lib/visitor-intel/simulation/`)
- `prng.ts` — PRNG determinístico (mulberry32) sembrado por `seed`.
- `scenario.ts` — contrato `SimulationScenario` (Zod) con todos los
  parámetros configurables.
- `profiles.ts` — catálogo de 12 perfiles con propensiones.
- `calendar.ts` — modelador temporal (weekends, temporadas, clima).
- `paths.ts` — generador de rutas territoriales heterogéneas.
- `behavior.ts` — motor que traduce perfil+contexto en secuencia de
  `VisitorEvent` (CV8.1) respetando `JourneyTransition` T1..T9.
- `alux.ts` · `concierge.ts` · `commerce.ts` — sub-motores.
- `run.functions.ts` — server function admin `runSimulation({scenario, seed})`.
- `cleanup.functions.ts` — server function admin `wipeSimulation({run_id})`.

### 3.2 Aislamiento (regla vinculante)
Todo INSERT lleva:
- `is_simulation boolean NOT NULL DEFAULT false` (add column donde falte).
- `simulation_run_id uuid NULL` con índice.
- Los `subject_id` simulados nacen con prefijo determinístico
  `sim_<run_id_short>_<n>` para trazabilidad; jamás colisionan con
  productivos.

### 3.3 Determinismo
Toda aleatoriedad pasa por el PRNG sembrado. Prohibido `Math.random`,
`Date.now()`, `crypto.randomUUID()` fuera de un helper `sim.uuid(rng)` que
deriva del PRNG.

### 3.4 Seguridad
- `runSimulation` requiere rol `super_admin` (RPC `has_role`).
- Escala `full` exige flag `confirmed: true` + banner con volumen
  estimado, coste (filas insertadas) y entorno.
- En producción, escala `full` requiere además
  `process.env.ALLOW_FULL_SIMULATION === "1"`.
- `wipeSimulation` sólo borra filas con `simulation_run_id = ?` y
  `is_simulation = true`. Cero riesgo sobre datos reales.

## 5. Outcome Validation (al cerrar CV8.S)

1. ¿El Center muestra tendencias/oportunidades/segmentos/benchmarks/
   prioridades/validaciones/recomendaciones con datos coherentes?
2. ¿La misma seed reproduce el mismo escenario byte-a-byte?
3. ¿`wipeSimulation` elimina 100% de filas simuladas y 0 filas reales?
4. ¿Los problemas plantados (§2.9) son detectados correctamente por CV8.5
   / CV8.7 / CV8.8?
5. ¿Un observador externo confunde la simulación con operación real?

---

## 6. Definition of Done

- Typecheck limpio.
- Escenario `oriente-maya-90d` corre en < 60s (medium) y produce
  ≥ 10k visitantes, ≥ 40k eventos, ≥ 500 planes, ≥ 150 casos Concierge,
  ≥ 80 órdenes pagadas, ≥ 200 reseñas.
- Center completo poblado y navegable.
- `wipeSimulation` verificado (sim → 0; real → intactos).
- Blueprint + Completion Report emitidos.
- Outcome Validation firmada por Founder.

---

## 7. Riesgos

- **Contaminación de producción** — mitigado por `is_simulation` + prefijo
  de subject_id + guardas de entorno.
- **Realismo insuficiente** — mitigado por perfiles con propensiones,
  calendario estacional, paths heterogéneos y problemas plantados.
- **No reproducible** — mitigado por PRNG sembrado y prohibición de
  fuentes de aleatoriedad implícitas.
