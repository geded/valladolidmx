# 20.00 · Governance Automation v1

**Estado:** Approved

**Autoridad:** Founder directive del 2026-07-22: construir Governance Automation v1 antes de continuar RV0.3, pilotos o V1.

## 1. Propósito

Convertir la admisión y sincronización documental en controles ejecutables y reproducibles. El sistema impide que la mera creación de un archivo lo convierta en documento canónico y elimina conteos o versiones rígidas de los validadores.

## 2. Fuente canónica y autoridad

`docs/governance/06-BLUEPRINT-MASTER-INDEX.md` es la puerta canónica de admisión. Cada archivo bajo `docs/blueprint/` debe tener exactamente una fila con estado permitido, dominio D01–D14 y evidencia de autoridad. La automatización no adjudica estados, no elige dominios y no fabrica aprobación Founder.

Un documento se admite únicamente cuando:

1. cumple las reglas canónicas vigentes y su evidencia queda registrada en 06; o
2. contiene autorización Founder explícita registrada en 06.

La ambigüedad, ausencia de dominio o falta de evidencia bloquea el proceso.

## 3. Contrato ejecutable

- `bun run governance:sync` valida admisión y sincroniza las proyecciones 07/08 y el inventario.
- `bun run governance:check` comprueba que las proyecciones estén actualizadas y ejecuta los validadores oficiales.
- GitHub Actions ejecuta el modo `--check`; nunca adjudica ni reescribe gobernanza en CI.
- Los conteos esperados se derivan del universo admitido en 06.
- La versión esperada de 07 se deriva de la versión vigente de 06.
- Las relaciones técnicas existentes se preservan; no se inventan dependencias.

## 4. Guardrails

- Cero cambios automáticos en `src/`, UI, DB, rutas, migraciones o configuración de producto.
- Ningún Blueprint huérfano puede pasar CI.
- Ninguna fila sin autoridad, estado válido y dominio canónico puede pasar CI.
- Los archivos generados se actualizan juntos y su deriva bloquea el PR.
- No se permiten excepciones artificiales para obtener verde.
- Las decisiones nuevas, conflictos con CANON/ADR o dominios ambiguos requieren Founder Review.

## 6. Criterios de aceptación

1. El universo 06/07/08 se calcula sin números rígidos.
2. Un Blueprint sin fila en 06 falla.
3. Una fila sin autoridad o dominio canónico falla.
4. Una proyección obsoleta falla en modo check.
5. Los tres validadores oficiales concluyen PASS.
6. Los gaps permanecen en el baseline previo, sin aumento por la automatización.
7. `src/routeTree.gen.ts` y todo producto permanecen intactos.

## 7. Fuera de alcance

RV0.3, pilotos, V1, cambios de producto, creación de dominios, modificación de ADRs y adjudicación automática por inferencia.
