## 1. Resumen ejecutivo

Esta propuesta define una extensión del CMS Studio denominada
**Experience Management System (EMS)** que permitirá al equipo de
Marketing administrar de forma íntegra **la experiencia visual y
comercial** de Valladolid.mx —Home, micrositios territoriales,
landings del Marketplace, banners promocionales, campañas estacionales,
empresas/productos destacados y bloques editoriales recurrentes— sin
requerir cambios de código.

El EMS se construye **sobre** la arquitectura aprobada (Blueprint,
Serie 13, Serie 14) y respeta de manera estricta los principios
consolidados durante las Olas 1, 2 y 3:

- Cero `SUPABASE_SERVICE_ROLE_KEY` en flujos de usuario.
- Cero bypass de RLS.
- Validación server-side obligatoria (`createServerFn` +
  `requireSupabaseAuth`).
- Whitelist explícita de campos por tipo de bloque.
- Workflow editorial `draft → in_review → approved → published →
  archived` reutilizando `src/lib/cms/workflow.ts`.
- Auditoría centralizada en `content_audit_log`.
- Permisos vía `has_role`.
- Aislamiento arquitectónico: Portal Empresarial, Marketplace y Fase 1
  no se modifican.

El documento es **únicamente una propuesta**. No autoriza implementación.
La Ola 4 continúa según el Plan 14.40.

---

## 2. Objetivo

Permitir que Marketing administre desde el CMS Studio:

- Páginas-experiencia (Home, landings temáticas, micrositios de destino).
- Bloques visuales reordenables (hero, grids, banners, CTAs, testimonios).
- Empresas, productos, destinos, rutas y promociones destacadas.
- Banners y campañas estacionales con vigencias.
- Slots editoriales recurrentes (consejo de Alux, "en vivo", rutas
  sugeridas).
- Programación temporal (`start_at`/`end_at`) por página y por bloque.
- Preview firmada antes de publicar.
- Versionado e historial navegable.
- Workflow editorial.
- Auditoría de cambios.
- Permisos por rol.
- Internacionalización (vía `translations`).
- Compatibilidad con Marketplace, Portal Empresarial y Alux.

## 3. Alcance funcional

### 3.1 Superficies administrables

| # | Superficie | `surface_kind` | Notas |
|---|------------|----------------|-------|
| 1 | Home `/` | `home` | Reemplaza el contenido estático actual (Doc 12 / 12A v2). |
| 2 | Micrositios de destino `/oriente-maya/$destino` | `destination` | Composición editorial sobre el destino. |
| 3 | Landings del Marketplace `/marketplace/...` editoriales | `marketplace_landing` | Vitrina curada (categorías destacadas, campañas). |
| 4 | Páginas "Arma tu Viaje", "Eventos", "Hoteles", etc. | `page` | Páginas estáticas existentes administrables. |
| 5 | Banners globales (header / footer / interstitials) | `banner_slot` | Vigencia y segmentación por superficie. |

### 3.2 Catálogo de bloques (extensible)

`hero`, `featured_destinations`, `categories_grid`, `featured_routes`,
`alux_tip`, `cta_block`, `social_feed`, `featured_businesses`,
`featured_reviews`, `promo_banner`, `featured_products`,
`featured_promotions`, `editorial_collection`, `rich_text`,
`media_gallery`, `faq_block`.

Cada `kind` posee un schema Zod en
`src/lib/cms/ems/block-schemas.ts` que **whitelista** los campos
permitidos (mismo patrón que `editor-fields.ts`). Cualquier campo no
listado se descarta antes de persistir.

### 3.3 Campañas programadas

Versiones y bloques pueden declarar ventanas `start_at`/`end_at` (UTC).
La versión activa se resuelve server-side por superficie + locale. Los
bloques caducos se ocultan automáticamente.

### 3.4 Preview

- Render del borrador con URL firmada (HMAC, TTL 30 min), scoped al
  usuario editorial autenticado. Nunca afecta la versión activa.

## 4. Modelo de dominio propuesto

Se introducen tablas nuevas en `public`, todas bajo RLS y con `GRANT`
explícitos. No se modifican dominio aprobado de Fase 1 ni los recursos
del Portal/Marketplace.

### 4.1 `experience_surfaces`

Catálogo de superficies administrables.

| Columna | Tipo |
|---|---|
| `id` | uuid PK |
| `kind` | enum `surface_kind` (`home`, `destination`, `marketplace_landing`, `page`, `banner_slot`) |
| `slug` | text único por `kind` (ej. `home`, `destinos/oriente-maya`) |
| `name` | text (etiqueta editorial) |
| `description` | text |
| `created_at` / `updated_at` | timestamptz |

### 4.2 `experience_versions`

| Columna | Tipo |
|---|---|
| `id` | uuid PK |
| `surface_id` | uuid → experience_surfaces |
| `name` | text |
| `status` | content_status (reusa Ola 1) |
| `start_at` / `end_at` | timestamptz NULL |
| `locale` | text (base `es`; traducciones vía `translations`) |
| `created_by` / `updated_by` | uuid → profiles |
| `published_at` / `published_by` | nullable |
| `created_at` / `updated_at` | timestamptz |

Restricción: a lo sumo una versión `published` con ventana activa por
`(surface_id, locale)` (validada en RPC y reforzada por índice parcial).

### 4.3 `experience_blocks`

| Columna | Tipo |
|---|---|
| `id` | uuid PK |
| `version_id` | uuid → experience_versions ON DELETE CASCADE |
| `kind` | enum `experience_block_kind` |
| `position` | int (único por `version_id`) |
| `is_visible` | bool default true |
| `start_at` / `end_at` | timestamptz NULL |
| `config` | jsonb (validado por Zod según `kind`) |
| `created_at` / `updated_at` | timestamptz |

### 4.4 `experience_block_features`

Relación N:M entre bloques y entidades destacadas.

| Columna | Tipo |
|---|---|
| `id` | uuid PK |
| `block_id` | uuid → experience_blocks ON DELETE CASCADE |
| `entity_kind` | enum (`business`, `destination`, `product`, `promotion`, `route`, `review`, `banner`, `event`) |
| `entity_id` | uuid |
| `position` | int |
| `pinned` | bool default false |

### 4.5 `experience_published_current` (vista)

Vista expuesta a `anon` que une superficie + versión activa + bloques
con ventana válida + features con entidades aún `published`. El SSR
público lee de esta vista; nunca de las tablas base.

### 4.6 GRANT y RLS (resumen)

```sql
GRANT SELECT ON public.experience_published_current TO anon;
GRANT SELECT, INSERT, UPDATE, DELETE ON
  public.experience_surfaces, public.experience_versions,
  public.experience_blocks, public.experience_block_features
  TO authenticated;
GRANT ALL ON ... TO service_role;
```

Las tablas base se protegen con RLS y solo son accesibles a roles
editoriales (`admin`, `editor`). Anon nunca toca tablas base.

### 4.7 Reutilización

- `content_audit_log` — sin cambios de esquema; nuevos `entity_kind`
  `experience_version` y `experience_block`.
- `translations` — i18n de textos de `config` whitelisted como
  traducibles (clave `experience_block:<id>:<field>`).
- `media_assets` — toda imagen/video se referencia aquí; sin storage
  duplicado.
- `workflow.ts` — máquina de estados intacta.

## 5. Capa de aplicación

### 5.1 Server functions (nuevas)

`src/lib/cms/ems/ems-builder.functions.ts`:

- `listSurfaces`, `getSurfaceVersions(surfaceId)`
- `getExperienceVersion(versionId)` (hidrata bloques + features)
- `createExperienceVersion`, `cloneExperienceVersion`,
  `updateExperienceVersion`, `archiveExperienceVersion`
- `transitionExperienceVersion(id, to)` — reusa
  `assertAllowedTransition`
- `upsertExperienceBlock(versionId, kind, config)`
- `reorderExperienceBlocks(versionId, [{id, position}])`
- `toggleExperienceBlockVisibility(blockId, visible)`
- `setExperienceBlockFeatures(blockId, [...])`
- `issueExperiencePreviewToken(versionId)` — HMAC, TTL 30 min
- `publishExperienceVersion(id)` — valida solapamientos y schemas Zod

Todas con `requireSupabaseAuth` + `has_role('admin'|'editor')`.
Auditoría obligatoria.

`src/lib/cms/ems/ems-public-reads.functions.ts`:

- `getActiveExperience(surfaceSlug, locale)` — usado por SSR de las
  superficies públicas. Lee `experience_published_current`. Sin auth.

### 5.2 Rutas CMS Studio (nuevas)

- `/_authenticated/cms/experience` — listado de superficies.
- `/_authenticated/cms/experience/$surfaceId` — versiones de la
  superficie.
- `/_authenticated/cms/experience/$surfaceId/$versionId` — editor
  visual:
  - Lista de bloques con drag & drop (`@dnd-kit/core`).
  - Toggle visibilidad.
  - Editor por `kind` (formulario tipado por Zod).
  - Selector de entidades destacadas.
  - Programación (`start_at`, `end_at`).
  - Botones: Guardar borrador, Solicitar revisión, Aprobar, Publicar,
    Archivar, Clonar, **Preview**.
- `/_authenticated/cms/experience/$surfaceId/$versionId/preview`.

Las rutas públicas existentes (`/`, `/oriente-maya/...`, etc.) se
adaptarán en su `loader` para leer `getActiveExperience(surfaceSlug,
locale)`. La adaptación de cada superficie no es parte de esta
propuesta — se documentaría en el plan de ejecución posterior por
superficie.

### 5.3 Cliente

- Drag & drop con `@dnd-kit/sortable`.
- Estado optimista vía TanStack Query con `invalidateQueries(['ems',
  versionId])`.

## 6. Workflow editorial

Reutiliza la máquina de Ola 1. Marketing (rol `editor`) crea, edita y
solicita revisión; `admin` aprueba y publica. Publicación valida:
ventanas no solapadas, Zod por `kind`, traducciones requeridas
presentes o fallback `es`.

---

## 7. Versionado

- Versiones `published` son inmutables (clonar para editar).
- `cloneExperienceVersion(sourceId)` produce nuevo `draft`.
- Historial navegable; cualquier versión puede re-promocionarse.

---

## 8. Auditoría

- Cada mutación en tablas EMS produce registro en `content_audit_log`
  con `entity_kind`, `entity_id`, `action`, `actor_id`, `payload_diff`
  whitelisted.
- Preview emite evento `experience_preview_issued` con `token_hash`
  (sin token plano).

## 10. Compatibilidad

### 10.1 Marketplace (Ola 4)

- `featured_businesses`, `featured_products`, `featured_promotions`,
  `featured_destinations` referencian entidades publicadas por el
  Marketplace.
- Reutiliza índices y RPCs aprobadas en 14.40 (incluido
  `search_marketplace`).
- Cero acoplamiento bidireccional: el EMS consume; el Marketplace no
  conoce al EMS.

### 10.2 Portal Empresarial (Ola 3)

- El Portal **no** gana capacidad de auto-destacarse: la curación es
  exclusiva del CMS Studio. Preserva la matriz de permisos de la Ola 3.

### 10.3 Internacionalización

- `translations` cubre todos los campos textuales whitelisted como
  traducibles.
- `getActiveExperience` hidrata con fallback `es` cuando falta
  traducción aprobada.
- SEO multilenguaje: `meta_title`/`meta_description` traducibles y
  `hreflang` automático.

### 10.4 Alux Intelligence

- Alux consume `getActiveExperience` para contextualizar respuestas;
  no accede a tablas crudas ni gana permisos elevados.

## 11. Seguridad

- Cero `SUPABASE_SERVICE_ROLE_KEY` en flujos de usuario.
- Cero cambios a RLS de tablas existentes.
- RPCs `SECURITY DEFINER` con `search_path = public` y `EXECUTE TO
  authenticated`.
- Lecturas anónimas exclusivamente vía vista
  `experience_published_current`.
- Preview tokens HMAC firmados con secret server-side; nunca expuestos
  sin TTL.
- Whitelist Zod por `kind` — `config.jsonb` jamás se persiste sin
  validar.
- Subidas de Hero/Banners reutilizan política aditiva de Storage
  aprobada en Ola 3.

## 12. QA propuesto

1. Crear versión en superficie `home`, añadir bloques de cada `kind`,
   reordenar drag & drop, publicar — la Home pública refleja el cambio.
2. Solapamiento de ventanas en misma superficie+locale → publicación
   rechazada con error claro.
3. Bloque con `end_at` pasado → no aparece en SSR.
4. Preview de versión `draft` no afecta a anon ni cambia la versión
   activa.
5. Editor solicita revisión; admin aprueba y publica; auditoría
   completa.
6. Toggle visibilidad oculta sección sin perder configuración.
7. Traducción faltante → fallback `es` documentado en SSR.
8. Anon no puede leer tablas base (RLS verificado).
9. Editor sin rol admin no puede invocar `publishExperienceVersion`
   (server-side bloqueado).
10. Clonación de versión publicada produce nuevo `draft` idéntico.
11. Banner global con vigencia caducada deja de servirse sin
    intervención.
12. Empresa destacada que pasa a `draft` desaparece automáticamente del
    bloque `featured_businesses`.

## 13. Rendimiento

Alineado al Plan 14.40 §Performance:

- `getActiveExperience` ≤ 80 ms p95 (consulta única a vista + `IN(...)`
  por `entity_kind`).
- Cache HTTP `Cache-Control: public, s-maxage=60,
  stale-while-revalidate=300` en SSR de superficies públicas.
- TanStack Query `staleTime` 60 s en el editor CMS.
- Índices: `experience_blocks(version_id, position)`,
  `experience_block_features(block_id, entity_kind)`,
  `experience_versions(surface_id, status, start_at, end_at)`.

## 16. Riesgos y mitigaciones

| ID | Riesgo | Mitigación |
|----|--------|------------|
| E1 | Entidades destacadas retiradas quedan referenciadas. | `getActiveExperience` filtra entidades por su propio `status='published'`. |
| E2 | Editor publica versión incompleta. | Validación Zod por `kind` requerida antes de transición a `published`. |
| E3 | `config.jsonb` se vuelve campo libre. | Whitelist por `kind`; campos no listados se descartan. |
| E4 | Drag & drop produce reorders concurrentes. | RPC `reorderExperienceBlocks` con transacción y optimistic concurrency por `updated_at`. |
| E5 | Preview tokens filtrados. | HMAC + TTL 30 min + revocación por versión. |
| E6 | Alcance crece y compite con la Ola 4. | Propuesta queda en stand-by; no se implementa hasta cerrar 14.40.Z. |
| E7 | Banners segmentados por superficie generan acoplamiento. | `banner_slot` se modela como superficie propia; reglas explícitas, sin lógica oculta. |
