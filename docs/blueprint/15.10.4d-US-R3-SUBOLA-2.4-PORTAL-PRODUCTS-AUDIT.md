# Sub-ola 2.4 · Portal Empresarial editando productos — Auditoría (G6)

> Alcance: cerrar el ciclo "empresario crea producto → publica → aparece en /producto/$slug".
> Metodología: primero auditoría quirúrgica; luego se autoriza construcción por fases.

## 1. Estado actual (qué ya existe)

### 1.1 CRUD de productos ✅
- `src/lib/portal/business-catalog.functions.ts` expone `list/create/update/archive/requestReview/withdrawReview` con `requireSupabaseAuth`.
- Todas las escrituras pasan por RPC SECURITY DEFINER (`create_business_product`, etc.) que revalidan `has_business_access('editor')` y auditan en `content_audit_log`.
- UI: `src/routes/_authenticated/portal/catalogo.tsx` (762 líneas) ya lista productos, permite crear, editar campos básicos y solicitar/retirar revisión.

### 1.2 Promociones ✅
- CRUD equivalente disponible (`create_business_promotion`, `update_business_promotion`, `archive_business_promotion`, `request_promotion_review`, `withdraw_promotion_review`).
- UI Portal ya renderiza pestaña "Promociones" en `catalogo.tsx`.

### 1.3 Media de productos ⚠️ parcial
- Existen server fns en `src/lib/cms/products-media.functions.ts`: `signProductImageUpload`, `registerProductMedia`, `listProductMedia`, `reorderProductGallery`, `removeProductMedia`.
- **Gap**: viven en el namespace `cms/` orientado a Founder (guard `is_editor_or_admin`). El Portal no las expone y probablemente no autorizan a `business_users` con rol `editor`.

### 1.4 Contacto público ✅
- `business_contacts` + editor del Portal en `/portal/ficha` (ya operativo desde Ola 3 Etapa 5).

### 1.5 Flujo de estado (draft/publicado/oculto) ⚠️ incompleto
- Estados soportados: `draft → in_review → approved → published → archived`.
- Portal sólo puede empujar hasta `in_review`. La transición `approved → published` requiere Founder/moderador.
- **Gap crítico**: no hay ruta directa "empresario publica" ni estado explícito "oculto" (podría mapearse a `archived` temporal).

### 1.6 Preview antes de publicar ❌ inexistente
- `/producto/$slug` lee via `getMarketplaceProductBySlug` filtrando `status='published'`.
- No hay endpoint autenticado que renderice la superficie Product v1 sobre un producto en `draft`/`in_review`.
- **Gap**: sin preview, el empresario publica a ciegas.

### 1.7 FAQs de producto ❌ bloqueadas para empresarios
- Tabla `faqs` con RLS: `faqs_editor_all` USING `is_editor_or_admin(auth.uid())`.
- Un business_user con rol `editor` NO puede crear FAQs de sus propios productos.
- **Gap**: hay bloque `vmx.product.faq` renderizando vacío para empresarios.

### 1.8 Superficie pública Product v1 ✅
- `/producto/$slug` con `ProductSurface`, `ProductSurfaceProvider`, 9 bloques `vmx.product.*`, preview-registry, ProductActions. Sub-ola 2.3a cerrada.

## 2. Gaps clasificados

| # | Gap | Severidad | Impacto |
|---|-----|-----------|---------|
| G1 | Media portal-scoped: server fns viven en `cms/*` con guard admin | **P0** | Empresario no puede subir fotos de sus productos |
| G2 | FAQs bloqueadas por RLS `is_editor_or_admin` | **P0** | Bloque FAQ inutilizable en Portal |
| G3 | Sin preview autenticado de producto `draft`/`in_review` | **P0** | Empresario publica a ciegas |
| G4 | Estado "oculto" no definido (publicado→retirar sin archivar) | **P1** | Empresario no puede pausar sin perder producto |
| G5 | Sin publicación directa por empresario (siempre pasa por revisión Founder) | **P1** | Depende de decisión de gobernanza |
| G6 | Validaciones pre-publicación no explícitas (portada obligatoria, precio, descripción mínima) | **P1** | Publicaciones incompletas rompen visual |
| G7 | UI Portal `catalogo.tsx` monolítica (762 líneas) sin drawer/inspector para gestión rica | **P2** | Fricción operativa, pero funcional |

## 3. Decisión de gobernanza pendiente

El Founder debe decidir el **modelo de publicación**:

- **Opción A · Publicación directa** (rápida): empresario `draft → published` sin moderación. Revisión Founder queda opcional/curatorial. Cierra G3/G5 con una RPC `publish_business_product`. Riesgo: contenido no curado en marketplace.
- **Opción B · Publicación mediada** (curada): mantener `draft → in_review → approved → published`. Founder aprueba en CMS. El empresario obtiene preview y estado, pero no publica directo. Cierra G3 pero no G5.
- **Opción C · Híbrida por plan/reputación** (recomendada): empresas con `plan >= pro` o `verified=true` publican directo; el resto pasa por revisión. Cierra G3 y da un camino para G5 sin sacrificar curación.

**Recomendación: Opción C** para no sacrificar la marca Valladolid.mx. Implementación mínima igual que A + flag por empresa.

## 4. Plan de fases propuesto (Sub-ola 2.4a)

### Fase A · Media Portal (P0)
1. Extraer las 5 server fns de `cms/products-media.functions.ts` a `src/lib/portal/portal-product-media.functions.ts` con `requireSupabaseAuth` + guard `has_business_access(uid, product.business_id, 'editor')`.
2. Reutilizar bucket `product-images` y tabla `product_media` (compartida). Sin migración.
3. UI: subcomponente `ProductMediaManager` (drop zone + drag&drop reorder + delete + set cover) reutilizando patrón de `business-media` del Portal.

### Fase B · FAQs por empresa (P0)
1. Migración SQL: RPCs `create_business_product_faq`, `update_business_product_faq`, `delete_business_product_faq`, `reorder_business_product_faqs` con guard `has_business_access('editor')` validando que `faq.entity_kind='product' AND faq.entity_id.business_id == arg.business_id`.
2. Política RLS adicional o simplemente ejecutar todo via RPC SECURITY DEFINER (patrón ya establecido). Sin ampliar `faqs_editor_all`.
3. Server fns Portal + UI dentro del editor de producto.

### Fase C · Preview autenticado (P0)
1. Nueva server fn `getPortalProductPreviewBySlugOrId` con `requireSupabaseAuth` + `has_business_access('editor')`, que ignora `status='published'` y devuelve el mismo shape que `getMarketplaceProductBySlug`.
2. Ruta `/_authenticated/portal/productos/$productId/preview` que renderiza `<ProductSurface>` con overlay "Vista previa — no publicado".
3. Reutilizar `ProductSurfaceProvider` y `__tpl_product__` desde `preview-registry`. Cero cambios a CompositionRenderer.

### Fase D · Publicación directa condicional (P0/P1 según opción)
1. Migración SQL: RPC `publish_business_product(_product_id)` que valida:
   - `has_business_access('editor')`
   - Portada seteada en `product_media` (`is_cover=true`).
   - `price_amount` no nulo O `product_type in ('servicio','artesanal')`.
   - `description` no nulo con ≥ 40 chars.
   - Empresa `verified=true` OR `plan >= 'pro'` (si Opción C).
   Setea `status='published'`, `published_at=now()`, auditoría.
2. RPC `unpublish_business_product` (setea `status='draft'`, conserva media) — cubre G4 sin nuevo estado.
3. Server fn + botón "Publicar" con checklist visible.

### Fase E · UI cohesiva (P1)
1. Refactor `catalogo.tsx`: extraer `ProductEditorSheet` con tabs (Detalles · Media · Precio · FAQ · Preview · Publicar).
2. Mobile-first, patrón sheet (Workspace Sheet Stack) — reutilizar infraestructura existente. Cero UI nueva.

## 5. Criterios de aprobación Sub-ola 2.4a

Un empresario debe poder, sin intervención Founder:
1. Crear producto → cargar 3+ fotos y ordenarlas → set portada.
2. Agregar precio, descripción, 2+ FAQs.
3. Ver vista previa idéntica a `/producto/$slug` con etiqueta "no publicado".
4. Publicar (si califica) o solicitar revisión (si no).
5. Ver producto en `/producto/$slug` con Product Surface v1 exacto.
6. Ocultar (`unpublish`) y volver a publicar sin perder data ni media.

Todo dentro de RLS por empresa, sin tocar CompositionRenderer, sin abrir checkout, sin inventario.
