# 15.10 · H-03 — Directiva Oficial · Identidad Visual del Destino & Institutional Badges

**Versión:** 1.0 (vinculante)
**Estado:** Directiva del Founder — vigente para toda la Biblioteca Oficial del Experience Builder y todas las Plantillas Madre.
**Depende de:** 12A, 12B, 12C.0, H-03 Block Design Rules, H-03 Reusability Matrix Rule.

---

## 1. Principio rector

La evolución del Experience Builder no moderniza únicamente la tecnología: **preserva y evoluciona la identidad visual del destino**.

La referencia conceptual es siempre **Valladolid.mx** — no para copiar su diseño, sino para conservar su esencia.

La plataforma debe transmitir:

- ciudad colonial · piedra · calidez · patrimonio;
- cultura maya · naturaleza;
- elegancia · tranquilidad · hospitalidad.

El objetivo es una **versión contemporánea de Valladolid.mx**, nunca un SaaS genérico.

---

## 2. Design Language (obligatorio)

Toda la Biblioteca Oficial respeta un mismo lenguaje visual:

- paleta cromática inspirada en Valladolid (tonos tierra, piedra, ocres, dorados, blancos cálidos);
- radios y sombras consistentes;
- iconografía elegante;
- animaciones discretas;
- fotografía protagonista;
- tipografía con personalidad editorial.

Cada bloque nuevo debe sentirse parte del mismo producto. El Design System (`src/styles.css` `@theme`) es la única fuente de verdad para tokens.

---

## 3. Institutional Badges — concepto oficial

Se crea el concepto **Institutional Badges** como sistema reutilizable dentro del Design System y como bloque oficial del Experience Builder.

**Reglas:**

- Los distintivos NO se colocan de forma fija ni repetitiva en las plantillas.
- Se integran mediante un **contrato de datos administrable desde el CMS**.
- Toda superficie (Destino, Empresa, Producto, Evento, Landing, Micrositio…) puede mostrar uno o varios badges configurables.
- Los badges NUNCA se hardcodean dentro de plantillas o adaptadores.
- El sistema queda preparado para incorporar futuros distintivos sin cambiar el contrato.

**Tipología inicial soportada:**

- `pueblo-magico`
- `patrimonio`
- `premio-nacional`
- `certificacion`
- `distintivo-turistico`
- `reconocimiento-oficial`
- `marca-oriente-maya`
- `marca-despierta-en-valladolid`
- `otro` (extensible por `kind` + `slug`)

---

## 4. Regla · Pueblos Mágicos del Oriente Maya

El Oriente Maya cuenta con tres Pueblos Mágicos:

- **Valladolid**
- **Izamal**
- **Espita**

El sistema de Institutional Badges asigna el distintivo **Pueblo Mágico por destino**, nunca global.

- Si el `destination.slug ∈ { "valladolid", "izamal", "espita" }` → el badge `pueblo-magico` puede mostrarse.
- En cualquier otro destino, el badge `pueblo-magico` NO se muestra.
- La lista de Pueblos Mágicos vive en un registry configurable (`pueblos-magicos.ts` o tabla equivalente) y queda preparada para nuevas incorporaciones.

---

## 5. Implicaciones para la Biblioteca Oficial

Se autoriza abrir una nueva ola dedicada:

**H-03 · Ola I3.c — `vmx.experience.institutional-badges@1.0.0`**

- Bloque oficial de 3 capas (contrato / presentación / comportamiento).
- Fuentes soportadas: `manual`, `destination`, `business`, `product`, `event` (destino y empresa en I3.c; el resto reservado).
- Variantes: `strip`, `stack`, `inline`, `hero-overlay`.
- Capacidades: `showLabel`, `showTooltip`, `compact`, `monochrome`, `linkToProgram`.
- Consumido por Plantillas Madre (BusinessSurface, DestinationSurface) vía composición, nunca vía JSX propio.

La sub-ola I3.d podrá abordar la evolución del **Design Language** (tokens Valladolid) sobre `src/styles.css` sin romper compatibilidad con los bloques ya cerrados (Regla de Compatibilidad Evolutiva).

---

## 6. Objetivo

Cuando un visitante abra cualquier página deberá reconocer inmediatamente que navega dentro del **ecosistema oficial de Valladolid y del Oriente Maya de Yucatán**:

- tecnología moderna;
- identidad colonial, elegante y atemporal;
- distintivos institucionales integrados con elegancia, sin saturar la interfaz ni competir con la marca del destino.
