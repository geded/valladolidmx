# US-R3 · Sub-ola 2.2 · Auditoría de Plantilla Business v1.0

> Estado: **Propuesta para aprobación**. No implementar hasta autorización explícita del Founder.
> Alcance: definir la plantilla oficial `kind = business` del Experience Builder que servirá como **plantilla madre** de todas las categorías de negocio de Valladolid.mx.
> Fuentes: implementación actual (`src/routes/marketplace/$slug.tsx`, `marketplace-reads.functions.ts`), esquema (`businesses`, `business_*`, `products`, `promotions`, `reviews`, `media_assets`), Blueprint 11.1, Recovery Plan 15.10.4d, Single Studio Principle, Experience Builder Vision.

---

## 1. Principio rector

**Una sola plantilla `business`, infinitas categorías.**

No se construirán plantillas independientes para Hotel, Restaurante, Cenote, Museo, Tour, Agencia, Transportista, Tienda o Servicio. Todas estas superficies son **variantes** de la misma plantilla resueltas por:

1. **Categoría** (`businesses.primary_category_id` → `business_categories.slug`).
2. **Configuración** por bloque (visibilidad, orden, densidad).
3. **Plan contratado** (Standard / Destacado / Premium → gates de bloques y capacidades).
4. **Permisos** (super_admin / admin / editor / dueño).
5. **Datos disponibles** (un bloque se auto-oculta si no hay contenido publicado, salvo que el editor lo fuerce).

URL canónica confirmada (H-R3-3): `/marketplace/{empresaSlug}`. Sin cambios en Sub-ola 2.2.

## 2. Estado actual (baseline)

La ficha viva en `/marketplace/$slug` renderiza hoy:

| Sección visible | Fuente de datos | Observación |
|---|---|---|
| Encabezado (nombre + tagline) | `businesses` | Sin logo, sin portada, sin badge de plan |
| Botón "Favorita" | `traveler_favorites` | OK |
| Descripción larga | `businesses.description` | Texto plano, sin formato |
| Grid de productos | `products` (publicado) | Sin filtro por tipo ni ordenamiento por plan |
| Grid de promociones | `promotions` (vigentes) | Sin destaque visual por plan |

**No** están conectados aún: logo/portada, galería, ubicación/mapa, horarios, contactos, redes sociales, categorías secundarias, reseñas, verificación visible, badge de confianza Alux, FAQs, artículos editoriales, eventos vinculados, integraciones (WhatsApp, reservas, menú, tarifas), CTA de conversión personalizado, seguimiento comercial.

**Deuda de datos** que la plantilla debe cubrir cuando se enciendan bloques:

- `business_media` (logo / portada / galería) sin lectura pública en `marketplace-reads`.
- `business_locations` / `business_contacts` / `business_hours` / `business_social_links` sin exposición.
- `reviews` sin superficie pública en la ficha.
- `products.visibility_level` no gobierna orden ni destaque en la ficha.

## 3. Taxonomía de bloques (contrato Business Template v1.0)

Cada bloque se registra en `block-library.ts` bajo el namespace `vmx.business.*` con `constraints.surfaces = ['business']`. Todos son **opcionales** salvo los marcados **[core]**. Todos declaran `requiredData` para ocultarse si el registro no tiene información.

### 3.1 Bloques comunes a todo negocio (columna vertebral)

| Bloque | Propósito | Plan mínimo | Notas |
|---|---|---|---|
| `vmx.business.hero` **[core]** | Portada + nombre + tagline + verificación + categoría + destino | Standard | Variantes visuales (compacto / cinemático / split) |
| `vmx.business.identity-strip` **[core]** | Logo + badges (verificada, Alux, plan) + CTA primario | Standard | CTA primario depende de la categoría |
| `vmx.business.summary` | Descripción larga + highlights | Standard | Rich text con anidados |
| `vmx.business.contact-card` | Teléfono, WhatsApp, email, sitio web | Standard | Contactos filtrables por rol público/privado |
| `vmx.business.location-map` | Dirección + mapa + cómo llegar | Standard | Multi-ubicación en Destacado+ |
| `vmx.business.hours` | Horarios semanales + estado abierto/cerrado | Standard | Requiere `business_hours` |
| `vmx.business.gallery` | Galería multimedia | Standard 5 · Destacado 20 · Premium ∞ | Lightbox, orden manual |
| `vmx.business.products-grid` | Catálogo de productos publicados | Standard | Filtros por tipo; destaque por `visibility_level` |
| `vmx.business.promotions` | Promociones vigentes | Standard | Auto-oculto si vacío |
| `vmx.business.reviews` | Reseñas verificadas + rating agregado | Standard 3 · Destacado 10+resp · Premium todas | Modera desde CMS |
| `vmx.business.social-links` | Redes sociales | Standard | Auto-oculto si vacío |
| `vmx.business.faqs` | Preguntas frecuentes | Destacado | Editables desde CMS |
| `vmx.business.related` | Otros negocios de misma categoría/destino | Standard | Smart Block |
| `vmx.business.alux-cta` | "Pregúntale a Alux sobre este negocio" | Standard | Inyecta contexto al concierge |
| `vmx.business.seo-footer` **[core]** | JSON-LD, breadcrumbs, canonical | Standard | Siempre emitido; no visible |

### 3.2 Bloques específicos por categoría (variantes)

Activados por defecto según `business_categories.slug`. Editables por super_admin/admin/editor; dueños los ordenan/ocultan pero no crean.

| Categoría | Bloques adicionales | CTA primario por defecto |
|---|---|---|
| Hotel / Hospedaje | `rooms`, `amenities`, `check-in-out`, `rates-calendar` | "Consultar disponibilidad" |
| Restaurante | `menu`, `reservations`, `cuisine-tags`, `dress-code` | "Reservar mesa" |
| Cenote / Atractivo natural | `safety-notes`, `access-info`, `entry-fees`, `what-to-bring` | "Cómo llegar" |
| Operador turístico / Tour | `tour-catalog`, `itineraries`, `included-excluded`, `pickup-points` | "Reservar tour" |
| Museo / Cultural | `exhibitions`, `entry-fees`, `accessibility`, `events` | "Ver exposiciones" |
| Agencia | `packages`, `certifications`, `destinations-covered` | "Solicitar cotización" |
| Transportista | `routes`, `fleet`, `pricing-table` | "Reservar traslado" |
| Tienda / Artesanía | `shop-grid`, `shipping-policy`, `origin-story` | "Comprar en línea" |
| Servicio | `services-list`, `pricing-table`, `coverage-area` | "Contactar" |

Todos son **variantes del mismo contrato**; comparten `props` genéricos (título, densidad, plan gate, visibilidad por dispositivo) más `data` específica.

### 3.3 Gates por plan contratado

El plan del negocio (`visibility_plan`: `standard | destacado | premium`) determina:

| Capacidad | Standard | Destacado | Premium |
|---|---|---|---|
| Bloques comunes | Todos | Todos | Todos |
| Variantes por categoría | Básicas (2) | Todas | Todas + custom |
| Galería | 5 medios | 20 | Ilimitada + video |
| Reseñas visibles | 3 últimas | 10 + respuesta | Todas + moderación |
| Ubicaciones | 1 | 3 | Ilimitadas |
| CTA personalizado | Preset | Preset + WhatsApp directo | Custom + integración |
| Reservas / pagos en línea | — | Opt-in | Nativo |
| Contenido editorial vinculado | — | 3 | Ilimitado |
| Analítica pública | — | — | Sí |
| Destaque visual en listados | — | Sí | Sí, priorizado |

Los gates son datos en el registry; los bloques bloqueados aparecen en el EB con candado y mensaje de upsell.

## 4. Contrato de datos requerido

La plantilla necesita ampliar la lectura pública (`marketplace-reads`) para exponer, con RLS `TO anon` sobre registros publicados:

- `business_media` (logo, cover, gallery) vía `media_assets`.
- `business_locations` (dirección, geo, primaria).
- `business_contacts` (canales públicos).
- `business_hours` (7 días + excepciones futuras).
- `business_social_links`.
- `business_category_links` (categorías secundarias).
- `reviews` (aprobadas, con rating agregado).
- `products` (con `visibility_level` para ordenar/destacar).
- `promotions` (vigentes).
- `faqs`, `articles`, `events` vinculados por `business_id`.

**No** se cambia el modelo en Sub-ola 2.2. Se documenta la deuda y se resuelve por bloque conforme se enciendan (adopción incremental, sin regresión).

## 5. Modelo de evolución

```text
              kind = business
        (una sola plantilla oficial)
                    |
     +--------------+---------------+
     |              |               |
  bloques        variantes       gates
  comunes        por categoría   por plan
     |              |               |
     +--------------+---------------+
                    |
           configuración editorial
          (Experience Builder Visual)
                    |
     +--------------+---------------+
     |              |               |
  Hotel        Restaurante        Cenote  ... (N)
```

Consecuencias:

- Añadir una nueva categoría ("Spa") **no requiere código nuevo**: se registra la categoría, se declaran los bloques específicos (si los hay) en `block-library.ts` con `constraints.categories`, y la plantilla los ofrece.
- Migrar Hoteles/Restaurantes en olas futuras (2.3+) reutiliza esta plantilla; sólo se activan variantes.
- Modernización visual (rediseño) se hace en la plantilla y se propaga a todas las categorías.

## 6. Reutilización de componentes existentes

| Componente actual | Rol en la plantilla |
|---|---|
| `FavoriteButton` | Reutilizado en `hero` / `identity-strip` |
| `ProductActions` | Reutilizado en `products-grid` y variantes de reserva |
| `PublicShell` | Contenedor exterior de la ruta; la plantilla vive dentro |
| `PlaceholderImage` | Fallbacks de galería/portada |
| Cards de `discovery/cards-registry` | Base visual para `related`, `products-grid`, `promotions` |
| `AddToCartButton` | Integración de compra cuando `accepts_online_payment` |

No se duplican componentes. Los bloques nuevos son wrappers declarativos sobre los existentes cuando existan.

## 7. Capacidades futuras que la plantilla debe soportar

- Reservas nativas (hotel/restaurante/tour) con motor propio o integración.
- Pagos en línea por producto/paquete (ya modelado en `products.accepts_online_payment`).
- Chat con dueño / WhatsApp Business.
- Certificaciones y sellos (sostenibilidad, accesibilidad, seguridad).
- Contenido en vivo (transmisiones, promos flash).
- Multi-idioma por bloque (i18n overlay ya existe en EB).
- Analítica pública para Premium.
- Multi-sucursal con selector.
- Vinculación con Rutas Sugeridas y Arma tu Viaje.
- Contexto para Alux (el bloque `alux-cta` inyecta el `business_id` al concierge).

Todas incorporables **sin cambiar la plantilla**: se agregan bloques o se amplían `props`.

## 8. Plan de adopción (Sub-ola 2.2)

**Principio:** adopción reproductiva. Cero regresiones. Paridad visual 1:1 en el paso 1; modernización en pasos siguientes bajo aprobación.

| Paso | Alcance | Salida verificable |
|---|---|---|
| **2.2.a — Contrato** | Registrar `vmx.business.*` (columna 3.1) en `block-library.ts`. Mapear en `composition-renderer.tsx`. Extraer `BusinessSurface` desde `/marketplace/$slug` sin cambiar UI. | Typecheck + build limpios. UI idéntica. |
| **2.2.b — Plantilla oficial** | Sembrar `page_compositions` slug `__template__:business` con los bloques mínimos que reproducen la ficha actual. Resolver por slug con fallback seguro al `BusinessSurface` extraído. | Ficha real renderizada desde composición; feature flag para revertir. |
| **2.2.c — Datos ampliados (opcional)** | Extender `marketplace-reads` para logo/portada/galería/horarios/ubicación/contacto sólo si son requeridos por bloques que se encienden ahora. Validado por RLS `TO anon`. | Bloques nuevos con datos reales o auto-ocultos. |

Los pasos 2.2.a y 2.2.b son obligatorios para cerrar 2.2. Los pasos 2.2.c–d pueden diferirse a 2.3 (Hotel/Restaurante) sin bloquear el cierre, siempre que la deuda quede documentada.

## 9. Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| Bloques específicos por categoría fragmentan el contrato | Comparten `props` base; se registran con `constraints.categories`, no como plantillas nuevas |
| Datos aún no expuestos rompen bloques | Cada bloque declara `requiredData` y se auto-oculta; feature flag por bloque |
| Regresión visual en `/marketplace/$slug` | Ruta mantiene fallback al `BusinessSurface` extraído hasta validar la composición |
| Plan gate confunde al editor | Bloques bloqueados muestran candado + tooltip; nunca desaparecen |
| Deuda de lectura pública crece | Reads se amplían por bloque, no de golpe; cada ampliación pasa por RLS `TO anon` |
