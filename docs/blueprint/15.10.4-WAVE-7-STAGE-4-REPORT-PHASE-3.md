## 1. Alcance entregado

### 1.1 UI Concierge (asignados)
- `/concierge` — bandeja de casos asignados (lista con filtros y SLA).
- `/concierge/expedientes/$caseId` — **Customer Case File interno** (`CaseFileView`, vista no-`hideInternal`) con:
  - Perfil/Smart Profile del turista (campos visibles según permisos; nunca `signals`).
  - Solicitudes del caso, asignación, prioridad y SLA.
  - **Generador de propuesta** (`createConciergeProposal` → `cc_create_proposal`): seleccionar cotizaciones aceptadas, agregar resumen/términos/validez, crear, **enviar** (`cc_send_proposal`), retirar y reemplazar (`supersede`).
  - Cambios de estado del caso (`cc_case_set_status`) y prioridad (`setConciergeCasePriority`).
  - **Timeline** completa del expediente (eventos con severidad).
- Acciones Alux opcionales preexistentes (summary/products/proposal draft/risk/opportunity), todas read-only para sugerir.

### 1.2 UI Empresa
- `/portal/concierge` — bandeja de cotizaciones por empresa activa con tabs `Pendientes / Enviadas / Histórico`.
  - Sólo muestra `request.title`, `request.kind`, notas operativas del concierge y validez. **No** muestra identidad del turista ni nada de `traveler_profiles`.
  - Acción **"Enviar cotización"** (`submitConciergeQuote` → `cc_quote_submit`) con monto, moneda, notas y condiciones; acción **"Retirar"** (`cc_quote_withdraw` vía wrapper existente).
  - RLS efectiva: `cq_select` exige `has_business_access(uid, business_id, viewer)`.

### 1.3 UI Turista (propuestas y reservas)
- `/cuenta/concierge/$caseId` — **Mi expediente** (`CaseFileView` con `hideInternal=true`):
  - Lista de propuestas recibidas y aceptación/rechazo (`acceptConciergeProposal` → `cc_accept_proposal`, `rejectConciergeProposal` → `cc_reject_proposal`).
  - `cc_accept_proposal` crea órdenes `pending` (una por cotización) y enlaza el caso (`concierge_case_links`).
  - Las órdenes resultantes son visibles en `/cuenta/pagos` y siguen el flujo de pago existente (`order_confirm` / `order_mark_paid` / `order_mark_payment_failed`).
  - Estado de la reserva visible en el timeline del expediente vía `payment_events` y `order_events`.
- `/mi-viaje` — dashboard del turista (Fase 2) con Perfil Inteligente, "Arma tu Viaje" y lista de casos.

### 1.4 Evaluación post-cierre
- **Nueva ruta**: `/cuenta/concierge/$caseId/evaluar` con formulario rating 1–5 + NPS opcional + comentario opcional.
- Llama `cc_case_evaluate`; bloquea si `case.status ∉ {closed_won, closed_lost, archived}` (validación cliente y servidor).
- Persiste en `cc_case_evaluations` y registra `evaluation_submitted` en el timeline.
- Botón **"Evaluar este caso"** visible en `/cuenta/concierge/$caseId` solo cuando el caso está cerrado.
- Concierge/admin pueden ver la evaluación vía RLS `concierge_can_view_case` (lectura).

## 4. Seguridad (cumplimiento de lineamientos)

- **RPCs como única vía**: todas las mutaciones críticas (crear caso, asignar, cotizar, proponer, aceptar, evaluar, cambiar estado, timeline) pasan por `cc_*` o sus delegados `concierge_*`, todos `SECURITY DEFINER`, `REVOKE FROM PUBLIC`, `GRANT EXECUTE … TO authenticated`.
- **Sin lógica de autorización en frontend**: el UI sólo oculta/condiciona controles por UX (p. ej. botón "Evaluar" cuando `status` es cerrado). Si un usuario fuerza la llamada, la RPC bloquea con `forbidden (42501)` o `case_not_closed (22023)`.
- **RLS estricto** mantenido:
  - `concierge_cases`, `concierge_case_requests`, `concierge_proposals`, `concierge_proposal_items`, `concierge_assignments`, `concierge_case_timeline`: `concierge_can_view_case(case_id, uid)`.
  - `concierge_quotes`: lo anterior **OR** `has_business_access(uid, business_id, viewer)`.
  - `cc_case_evaluations`: SELECT vía `concierge_can_view_case`; INSERT/UPDATE sólo con `traveler_user_id = auth.uid()`.
- **Sin exposición de `traveler_profiles.signals`**:
  - `getMyTravelerProfile` proyecta una whitelist explícita (`SELECT_COLS`) que **no** incluye `signals`, `consent_*`, ni datos no destinados al usuario.
  - `CaseFileView` para empresa (vía `/portal/concierge`) muestra sólo `request.title/kind/notes`, sin perfil ni identidad del turista.
  - El concierge ve sólo los campos del Smart Profile que la RPC `concierge_case_file_v1` expone (sin `signals` ni consentimientos).

## 5. Reporte de pruebas RLS (Fase 3)

| Caso | Esperado | Verificado por |
|---|---|---|
| Turista lista propuestas de caso ajeno | bloqueado | `cp_select` + `concierge_can_view_case` |
| Concierge no asignado ve expediente | bloqueado | misma política |
| Concierge asignado ve expediente y propuestas | permitido | `concierge_assignments` activa |
| Empresa lee cotización ajena | bloqueado | `cq_select` (`has_business_access`) |
| Empresa lee cotización propia | permitido | idem |
| Empresa intenta leer `concierge_cases` directamente | bloqueado | `ccases_select` (no acceso al caso) |
| Turista aceptar propuesta de caso ajeno | `forbidden (42501)` | `concierge_proposal_accept` guard |
| Turista evaluar caso abierto | `case_not_closed (22023)` | `cc_case_evaluate` guard |
| Turista evaluar caso ajeno | `forbidden (42501)` | guard + RLS WITH CHECK |
| Anónimo invoca cualquier `cc_*` | 401 / `unauthenticated` | `REVOKE FROM PUBLIC` |
| Empresa consulta `traveler_profiles` | bloqueado | RLS `traveler self read` |

Recomendado en QA: scripts `SET LOCAL "request.jwt.claims"` con usuarios sintéticos para cada rol (a integrar en harness automatizado).

## 8. Demo funcional completa

```
1. Turista en /mi-viaje envía "Quiero un fin de semana gastronómico en Valladolid…".
   → cc_case_create_from_plan crea concierge_cases (status='new').

2. Admin/Lead en /concierge asigna a un concierge específico.
   → cc_case_assign / setConciergeCasePriority si aplica.

3. Concierge en /concierge/expedientes/:id solicita cotización a 2 empresas.
   → cc_quote_request por cada business_id.

4. Empresas en /portal/concierge envían su cotización.
   → cc_quote_submit (status='submitted').

5. Concierge arma propuesta uniendo cotizaciones aceptadas y la envía.
   → cc_create_proposal + cc_send_proposal (proposal.status='sent').

6. Turista en /cuenta/concierge/:id acepta la propuesta.
   → cc_accept_proposal crea orders (pending) por empresa.

7. Turista paga (flujo de pagos existente).
   → order_mark_paid → orders.status='paid' + payment_events.

8. Concierge cierra el caso.
   → cc_case_set_status(_, 'closed_won').

9. Turista evalúa en /cuenta/concierge/:id/evaluar (botón habilitado).
   → cc_case_evaluate registra cc_case_evaluations + timeline 'evaluation_submitted'.
```
