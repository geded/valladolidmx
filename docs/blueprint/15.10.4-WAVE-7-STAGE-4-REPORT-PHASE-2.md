## 2. Diagrama del flujo "Arma tu Viaje"

```
 Turista (/mi-viaje)
     │  cc_case_create_from_plan(summary, items[])
     ▼
 concierge_cases (status='new')  ──►  timeline: case_created
     │
     │  (lead / admin)
     │  cc_case_assign(case, concierge_user)
     ▼
 concierge_assignments (active)  ──►  status='triaged'/'in_progress'
     │
     │  Concierge analiza y solicita cotizaciones
     │  cc_quote_request(request_id, business_id)
     ▼
 concierge_quotes (status='requested')
     │
     │  Empresa (/empresa)
     │  cc_quote_submit(quote_id, total, currency)
     ▼
 concierge_quotes (status='submitted')
     │
     │  Concierge arma propuesta
     │  cc_create_proposal(case, items[]) → cc_send_proposal(p)
     ▼
 concierge_proposals (status='sent')
     │
     │  Turista revisa en /mi-viaje
     │  cc_accept_proposal(p)   ─►  orders.* (status='pending')
     │                                    │
     │                                    │  Pago (existing payment flow)
     │                                    ▼
     │                              orders.status='paid'
     │
     │  Concierge cierra caso
     │  cc_case_set_status(case, 'closed_won')
     ▼
 Turista evalúa
 cc_case_evaluate(case, rating, nps, comment)
     ▼
 cc_case_evaluations + timeline: evaluation_submitted
```

## 4. Estados del caso

`concierge_cases.status` (constraint):
`new` → `triaged` → `in_progress`
    → `awaiting_traveler` ⇄ `awaiting_business`
    → `closed_won` | `closed_lost` | `archived`

Estados auxiliares:
- `concierge_case_requests.status`: `open` → `quoting` → `quoted` → `accepted`/`rejected`/`cancelled`.
- `concierge_quotes.status`: `requested` → `submitted` → `accepted`/`rejected`/`withdrawn`/`expired`.
- `concierge_proposals.status`: `draft` → `sent` → `viewed` → `accepted`/`rejected`/`withdrawn`/`superseded`/`expired`.
- `orders.status`: `pending` → `paid` | `failed` | `cancelled`.

## 6. Reporte de pruebas RLS

Validación lógica sobre las políticas existentes y las nuevas:

| Caso de prueba | Resultado esperado | Verificado |
|---|---|---|
| Turista lee caso ajeno → bloqueado | `concierge_can_view_case` false | ✅ política `ccases_select` |
| Concierge no asignado lee caso → bloqueado | false | ✅ misma política |
| Concierge asignado lee caso → permitido | true | ✅ vía `concierge_assignments.active` |
| Empresa lee cotización de otra empresa → bloqueado | false | ✅ política `cq_select` (`has_business_access`) |
| Empresa lee cotización propia → permitida | true | ✅ |
| Turista inserta evaluación de caso ajeno → bloqueado | RLS WITH CHECK falla | ✅ `cc_eval_insert_self` |
| Turista evalúa caso abierto → bloqueado | `case_not_closed` (22023) | ✅ guard en `cc_case_evaluate` |
| Anónimo invoca cualquier `cc_*` → bloqueado | `unauthenticated` / 401 | ✅ `REVOKE FROM PUBLIC` |

Pruebas funcionales recomendadas en QA: scripts SQL con `SET LOCAL ROLE`
+ `SET LOCAL "request.jwt.claims"` para simular cada rol (a incorporar
