# 15.10.6.3 — Sync Queue (IndexedDB) + Background Synchronization

Estado: **INFRAESTRUCTURA OFICIAL — LISTA PARA ADOPCIÓN CONTROLADA**

## Alcance
- **Cola durable** de mutaciones autorizadas sobre IndexedDB nativo
  (`src/pwa/sync-queue.ts`). Base `valladolidmx.sync`, store `queue`,
  índices `status` y `createdAt`. Cero dependencias externas nuevas.
- **Runner** en el hilo principal (`src/pwa/sync-runner.ts`) que drena
  la cola ante `online`, `visibilitychange:visible` y un intervalo
  defensivo de 60s. Backoff exponencial 2/4/8/16/32s con jitter.
- **User Awareness** vía `SyncStatusBanner` (Discovery Layer, exportado
  por `@/components/discovery`). Muestra pending / in_flight / failed /
  conflict en tiempo real; se autoexcluye en preview / iframe.
- **Observabilidad** end-to-end vía eventos `pwa:sync:*`
  (`enqueued`, `in_flight`, `done`, `failed`, `conflict`,
  `retry_scheduled`, `removed`, `purged`, `drain_start`, `drain_end`,
  `skipped`).
- **Kill Switch**: `?sw=off` desactiva el runner y el SW; `purgeQueue()`
  vacía la cola local emitiendo eventos por trazabilidad.

## Contrato de entrada (`SyncEntry`)
Todo `enqueueMutation()` requiere:
- `id` (auto UUID si no se provee), `resource`, `endpoint`, `method`
  (`POST|PUT|PATCH|DELETE`), `payload` JSON serializable, `label`
  legible y `idempotencyKey` obligatorio (≥8 chars).
- Estado del ciclo de vida: `pending → in_flight → done | failed |
  conflict | discarded`.
- El runner envía `X-Idempotency-Key` en cada intento; el backend DEBE
  respetarla (garantía de idempotencia end-to-end).

## Denylist absoluto (Security Before Offline)
`assertAuthorizedResource()` bloquea el encolado de cualquier endpoint
relacionado con: OAuth/session/tokens, `/api/**/payments`, Stripe,
Paddle, billing/invoice/payouts, roles, permissions, user-roles.
Intentar encolar cualquiera de estos recursos LANZA.

## Gobernanza 15.10.6.3 aplicada
- **Reliable Sync**: integridad > velocidad. En duda, la entrada
  permanece `pending`/`failed` y nunca se ejecuta a ciegas.
- **User Awareness**: `SyncStatusBanner` visible siempre que haya
  operaciones pendientes/fallidas/en conflicto.
- **Conflict Safety**: 409/412 → `conflict` sin reintento automático;
  requiere resolución explícita del consumidor (sub-adenda de UI de
  resolución queda para 15.10.6.3.x).
- **Idempotencia obligatoria**: `idempotencyKey` requerido a nivel
  contrato; sin él, `enqueueMutation()` lanza.
- **Reintentos acotados**: máximo 5 intentos; después → `failed`
  (permanece en cola hasta acción explícita del usuario o descarte).
- **Observability First**: cada transición emite `pwa:sync:*`.
- **No auto-adopción**: NINGUNA mutación existente se captura
  automáticamente. La adopción se hará superficie por superficie en
  sub-adendas posteriores conforme se autorice.

## Compatibilidad
- **Discovery Layer**: `SyncStatusBanner` se monta en `__root.tsx`
  junto a `OfflineBanner`, sin interferir con SEO/OG/`buildPublicHead`.
- **Workspace Engine**: la infraestructura es transversal; el runner
  corre a nivel raíz. Adopción por Portal/CMS/Cuenta requiere sub-adenda.
- **Preview Lovable**: runner y banner se autoexcluyen en iframe.
- **iOS / Safari**: no se usa Web Background Sync API (limitada); el
  runner en hilo principal cubre iOS/iPadOS de forma uniforme.

## Rollback
1. Retirar `startSyncRunner()` y `<SyncStatusBanner />` de
   `src/routes/__root.tsx`.
2. Eliminar `src/pwa/sync-queue.ts`, `src/pwa/sync-runner.ts`,
   `src/components/discovery/SyncStatusBanner.tsx` y su export en
   `src/components/discovery/index.ts`.
3. Opcional: forzar `purgeQueue()` en sesiones activas antes del
   deploy o publicar un release que ejecute la limpieza al montar.
4. `?sw=off` sigue siendo válido como kill-switch inmediato.
