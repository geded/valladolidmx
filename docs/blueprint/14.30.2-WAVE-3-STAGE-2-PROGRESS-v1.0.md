## 1. Objetivo

Habilitar el flujo de onboarding por invitación al Portal Empresarial:

1. Owners crean invitaciones para una empresa (token, rol, TTL).
2. El destinatario previsto recibe el enlace y la acepta una sola vez.
3. La aceptación crea/activa la membresía en `business_users` mediante
   una RPC `SECURITY DEFINER` que valida destinatario, expiración y
   reutilización.
4. El cambio de empresa activa invalida el contexto cacheado anterior.

## 3. Entregables técnicos

### 3.1 Migración

- `accept_business_invitation(_token text)` — `SECURITY DEFINER`,
  `search_path = public`. Valida:
  - `auth.uid()` presente;
  - `scope_type = 'business'` y `scope_id` no nulo;
  - `lower(email) == lower(auth.users.email)` (destinatario previsto);
  - `status = 'pending'` (no reutilizable si `accepted`/`revoked`);
  - `expires_at > now()` (auto-marca `expired` si vencida);
  - `INSERT … ON CONFLICT (business_id, user_id) DO UPDATE`
    (idempotencia de membresía).
  - Marca la invitación `accepted` con `accepted_at = now()`.
- `preview_business_invitation(_token text)` — `SECURITY DEFINER`,
  `STABLE`. Devuelve datos básicos SOLO al destinatario previsto. A
  terceros responde `{ found:false }` (no revela existencia).
- `REVOKE … FROM PUBLIC` + `GRANT EXECUTE … TO authenticated` en ambas.
- 0 cambios en tablas, columnas, índices, triggers o RLS.

### 3.2 Server functions (TanStack)

`src/lib/portal/invitations.functions.ts`:

| Función | Método | Middleware | Notas |
|---|---|---|---|
| `listBusinessInvitations` | POST | `requireSupabaseAuth` | RLS exige owner del business para SELECT. |
| `createBusinessInvitation` | POST | `requireSupabaseAuth` | Doble verificación con `has_business_access(_min_role='owner')`. Token 32 bytes hex, TTL 1–30 días (default 7). |
| `revokeBusinessInvitation` | POST | `requireSupabaseAuth` | Sólo `pending → revoked`; RLS limita a owners. |
| `previewInvitation` | POST | `requireSupabaseAuth` | Envuelve `preview_business_invitation`. |
| `acceptInvitation` | POST | `requireSupabaseAuth` | Envuelve `accept_business_invitation`. |

### 3.3 UI del Portal

- `src/routes/_authenticated/portal/invitaciones.index.tsx` — gestión
  para owners: crear, listar, revocar, enlace por invitación.
- `src/routes/_authenticated/portal/invitaciones.$token.tsx` — vista
  del destinatario con preview, mensajes específicos para
  `not_found / not_pending / expired`, y botón de aceptación.
- `src/routes/_authenticated/portal/route.tsx` — nuevo enlace
  "Invitaciones" en el nav; al cambiar empresa, `queryClient.removeQueries`
  elimina toda query cuyo `queryKey` contenga el `business_id` previo.

## 6. Integridad Referencial

- `business_users.business_id` y `business_users.user_id` permanecen
  como FKs existentes; sin cambios.
- Aceptación usa `ON CONFLICT (business_id, user_id) DO UPDATE`: no
  genera huérfanos ni duplica membresías.
- `invitations.scope_id` válido contra `businesses.id` por convención de
  Fase 1; la RPC no inserta filas huérfanas.

## 10. Advertencias del linter

- `0029_authenticated_security_definer_function_executable` añade 2
  ocurrencias por las nuevas RPCs (`accept_business_invitation`,
  `preview_business_invitation`).
- Son intencionales: son el único mecanismo para que un destinatario
  no-owner interactúe con su propia invitación (RLS de `invitations`
  permite SELECT/UPDATE sólo a owners). Las validaciones de
  destinatario, estado y expiración están dentro de la función.
