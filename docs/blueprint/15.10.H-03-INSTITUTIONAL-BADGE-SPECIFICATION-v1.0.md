# 15.10 · H-03 — Institutional Badge Specification v1.0

**Versión:** 1.0 (vinculante)
**Estado:** Especificación oficial del Design System — parte del **Valladolid Colonial Design System v1.0**.
**Autoriza:** Founder (H-03 · I3.d.2 · Paso M1 previo a implementación).
**Depende de:** DSL Colonial v1.0, Directiva Identidad Visual & Institutional Badges v1.0, Auditoría de Tokens I3.d.1.
**Alcance:** Toda superficie pública, autenticada, plantilla madre, micrositio, landing, componente presente y futuro.

---

## 0. Principio rector

Un Institutional Badge **transmite confianza y prestigio**. Nunca satura la interfaz, nunca compite con el contenido principal, nunca compite con la marca del destino.

Refuerza la identidad del destino y ayuda al visitante a identificar reconocimientos relevantes con elegancia editorial. Todos los distintivos —actuales y futuros— comparten **un único lenguaje visual** y **un único sistema de tokens**. No existen implementaciones divergentes por badge.

---

## 1. Catálogo institucional cubierto (v1.0)

Todos los tipos siguientes se implementan sobre la MISMA especificación (mismos tokens, mismos tamaños, mismas variantes, mismo comportamiento). Sólo cambian `slug`, `label`, `icon` y color institucional asignado desde el registry.

| `kind` | Ejemplos | Familia cromática |

| `despierta-en-valladolid` | Programa oficial Valladolid.mx | `--color-badge-despierta` |
| `oriente-maya` | Marca territorial Oriente Maya de Yucatán | `--color-badge-oriente-maya` |
| `patrimonio` | UNESCO, patrimonio nacional/estatal | `--color-badge-patrimonio` |
| `verified-business` | Empresa Verificada por Valladolid.mx | `--color-badge-verified` |
| `alux-recommended` | Recomendado por Alux | `--color-badge-alux` |

## 2. Anatomía

Un badge se compone SIEMPRE de los mismos elementos:

```
┌─────────────────────────────────────────────┐
│  [icon]  Label textual        (opcional: →) │
└─────────────────────────────────────────────┘
```

- **Icon** — Lucide 1.5 stroke, tamaño ligado al `size` (§3).
- **Label** — Inter 500 (semibold en `sm`+). Máx. 24 caracteres visibles; overflow → `shortLabel`.
- **Optional trailing chevron** — sólo cuando `linkToProgram = true`.

Nunca se muestra sin icono. Nunca se muestra sin label salvo en variante `icon-only` (§4) para saturaciones móvil, con `aria-label` obligatorio.

---

## 3. Tamaños oficiales

Tres tamaños canónicos. Cualquier bloque consumidor elige uno de estos y no inventa medidas intermedias.

| Size | Altura | Padding X | Icon | Text | Uso |
| --- | --- | --- | --- | --- | --- |
| `sm` | 20px | 8px | 12px | 11px / 1.0 | Cards densas, listas, chips inline |
| `md` (default) | 28px | 12px | 14px | 13px / 1.1 | Página, sección, hero-overlay |
| `lg` | 36px | 16px | 18px | 15px / 1.15 | Hero editorial, cabecera destino |

Tokens correspondientes (M1):

```
--badge-h-sm: 20px;  --badge-h-md: 28px;  --badge-h-lg: 36px;
--badge-px-sm: 8px;  --badge-px-md: 12px; --badge-px-lg: 16px;
--badge-gap-sm: 4px; --badge-gap-md: 6px; --badge-gap-lg: 8px;
--badge-icon-sm: 12px; --badge-icon-md: 14px; --badge-icon-lg: 18px;
--badge-text-sm: 11px; --badge-text-md: 13px; --badge-text-lg: 15px;
--radius-badge: 999px; /* pill por defecto — coherente con DSL */
--radius-badge-editorial: var(--radius-md); /* variante rectangular editorial */
```

No se permiten tamaños fuera de esta escala. En móvil, `md` se degrada a `sm` cuando el contenedor tiene < 360px de ancho útil.

## 4. Variantes

Tres variantes visuales canónicas + una utilitaria móvil:

| Variante | Uso | Definición |
| --- | --- | --- |
| `filled` | Máxima jerarquía. Hero, distintivo protagonista. | Fondo `color-badge-*`, texto `color-badge-*-foreground`. |
| `soft` (default) | Uso general. Cards, secciones, listados. | Fondo `color-mix(in oklab, var(--color-badge-*) 14%, transparent)`, texto `color-badge-*` (oscurecido AA), borde 1px `color-mix(... 22%, transparent)`. |
| `outline` | Contextos con fotografía cargada o modo denso. | Fondo `transparent`, texto/icono `color-badge-*`, borde 1px `currentColor`. |
| `icon-only` | Saturación móvil, chips en imagen. | Círculo `--badge-h-*`, fondo `filled`, tooltip obligatorio (§9). |

Modo `monochrome` (capacidad global del bloque): fuerza uso de `--color-foreground` / superficie neutra manteniendo icono; sirve para pies de página o densidades editoriales sin color.

## 5. Colores institucionales (tokens M1)

Se añaden al Design System como tokens semánticos. **Aditivos**: no reemplazan ni modifican tokens existentes. Todos calculados sobre la paleta oficial DSL v1.0.

```
--color-badge-despierta:          oklch(0.777 0.140 74.9); /* Ocre Valladolid */
--color-badge-despierta-fg:       oklch(0.227 0.017 112.9);

--color-badge-oriente-maya:       oklch(0.511 0.073 125.9); /* Verde Selva */
--color-badge-oriente-maya-fg:    oklch(0.977 0.013 86.8);

--color-badge-patrimonio:         oklch(0.548 0.096 217.9); /* Cenote Profundo */
--color-badge-patrimonio-fg:      oklch(0.977 0.013 86.8);

--color-badge-verified:           oklch(0.58 0.130 150);    /* Success */
--color-badge-verified-fg:        oklch(0.99 0.005 150);

--color-badge-alux:               oklch(0.54 0.14 295);     /* Púrpura Alux discreto */
--color-badge-alux-fg:            oklch(0.98 0.008 295);

--color-badge-award:              oklch(0.72 0.15 85);      /* Dorado premio */
--color-badge-award-fg:           oklch(0.22 0.02 85);

--color-badge-certification:      oklch(0.62 0.11 230);     /* Info institucional */
--color-badge-certification-fg:   oklch(0.99 0.005 230);

--color-badge-recognition:        oklch(0.42 0.045 129.7);  /* Selva profunda */

--color-badge-neutral:            var(--color-muted);
--color-badge-neutral-fg:         var(--color-foreground);

/* Sombra dedicada — sutil, cálida */
--shadow-badge: 0 1px 2px oklch(0 0 0 / 0.08),
                0 1px 1px color-mix(in oklab, var(--color-atardecer) 10%, transparent);
```

Modo oscuro (`.dark`): cada token se ajusta 1 paso hacia mayor luminosidad y menor croma para conservar contraste AA sobre `--surface-raised`. La foreground se invierte cuando corresponda; la variante `soft` recalcula sus mezclas contra `--surface` oscuro. Los tokens `-fg` no se hardcodean en componentes; siempre se leen del par.

**Regla:** ningún componente hardcodea colores de badge. Todos leen `--color-badge-<kind>` vía el registry.

## 6. Icono + texto

- Iconografía **Lucide 1.5** exclusivamente. Un icono por `kind` definido en el registry (no elegible por editor).
- Icono siempre a la izquierda del label. Nunca a la derecha.
- Icono heredará `currentColor`; nunca se aplican filtros ni sombras al icono.
- Label editorial: capitalización estilo título ("Pueblo Mágico", "Recomendado por Alux"), sin mayúsculas gritadas salvo siglas oficiales ("UNESCO", "SECTUR").

---

## 7. Comportamiento desktop y móvil

### Desktop
- Densidad completa (todos los badges visibles en `md` por defecto).
- Layout `strip` (fila horizontal con wrap) o `stack` (columna) según bloque consumidor.
- Hover: sube `--shadow-badge` un nivel (`0 → 1`), sin cambio de color.

### Móvil (< 768px)
- `md` se degrada a `sm`.
- Cuando existen ≥ 4 badges, mostrar los 3 primeros por prioridad (§8) + chip `+N` que abre un `Sheet` con la lista completa.
- Layout `strip` mantiene scroll horizontal con snap si el consumidor lo activa (`overflow: mobile-scroll`).
- Variante `icon-only` autorizada en overlays de hero móvil.

Nunca romper línea a mitad de un badge. Nunca truncar el label con `…` cuando existe `shortLabel` disponible.

---

## 8. Orden de prioridad

Cuando coexisten varios badges, el orden es **fijo, determinista y no configurable por editor**. Se ordena por `priority` numérico ascendente definido en el registry:

1. `pueblo-magico` (10)
2. `patrimonio` (20)
3. `oriente-maya` (30)
4. `despierta-en-valladolid` (40)
5. `award` (50)
6. `official-recognition` (60)
7. `certification` (70)
8. `verified-business` (80)
9. `alux-recommended` (90)
10. `custom` (100+)

Empates de prioridad: se ordenan alfabéticamente por `slug`. El bloque consumidor NUNCA reordena.

---

## 9. Separación y espaciado

- Gap horizontal en `strip`: `8px` (`sm`), `10px` (`md`), `12px` (`lg`).
- Gap vertical en `stack`: idéntico al horizontal correspondiente.
- Margen exterior mínimo respecto a contenido adyacente: `16px` en hero, `12px` en cards, `8px` en chips inline.
- Un badge nunca toca el borde superior o inferior de una imagen sin al menos `12px` de aire.
- Densidad máxima recomendada por superficie: **5 badges visibles** simultáneos; el resto colapsa en `+N`.

---

## 10. Accesibilidad

- Contraste texto/fondo ≥ **AA 4.5:1** para todas las variantes en light y dark.
- Variante `icon-only` requiere `aria-label` con el label completo del badge.
- `role="img"` para badges puramente decorativos; `role="link"` cuando `linkToProgram = true`.
- Tooltip accesible (`aria-describedby`) obligatorio en `icon-only` y `compact`; opcional en el resto.
- Foco visible con `--shadow-focus` estándar del DSL. No se sobreescribe.
- Respeto a `prefers-reduced-motion`: no hay animación de entrada/salida por defecto; sólo transición de sombra en hover de `160ms` que se anula bajo reduced motion.
- Todos los badges deben ser legibles a 200% de zoom sin recortes.

---

## 11. Modo claro y modo oscuro

- El sistema resuelve variantes automáticamente por `.dark` — el consumidor no elige tema.
- `soft` en dark: mezcla contra `--surface-raised` para preservar textura.
- `outline` en dark: aumenta grosor de borde a `1.5px` sólo si `size = sm` para conservar legibilidad.
- `filled` en dark: mantiene color institucional; la foreground se recalcula automáticamente vía el par `-fg`.
- Prohibido definir estilos condicionales por tema fuera del token — el token es la única fuente.

---

## 12. Reglas de uso

**Permitido**
- Mostrar badges en Hero, Header de destino, Cards de destino/empresa/producto, Sección institucional, Footer editorial.
- Combinar badges de distinto `kind` respetando §8 y §9.
- Consumir el badge desde cualquier bloque oficial vía composición.

**Prohibido**
- Hardcodear un badge dentro de una plantilla o un adaptador.
- Inventar colores, tamaños, formas o iconos fuera de esta especificación.
- Colocar `pueblo-magico` en destinos no autorizados (registry: Valladolid, Izamal, Espita).
- Usar badges como elementos decorativos sin significado institucional real.
- Reordenar badges por criterios editoriales o comerciales.
- Aplicar sombras, gradientes, animaciones o efectos ajenos a los tokens del DSL.
- Traducir automáticamente el label sin revisión editorial (los nombres institucionales son propios).

## 13. Contrato de datos (preparado para I3.c)

```ts
type InstitutionalBadge = {
  kind: BadgeKind;           // enum del registry (§1)
  slug: string;              // p. ej. "pueblo-magico:valladolid"
  label?: string;            // override editorial opcional (no traduce marcas oficiales)
  shortLabel?: string;       // fallback móvil
  programUrl?: string;       // si `linkToProgram = true` en el bloque
  issuedAt?: string;         // ISO — para tooltip "Otorgado en 2019"
  source: "manual" | "destination" | "business" | "product" | "event";
};
```

El bloque `vmx.experience.institutional-badges@1.0.0` acepta `items: InstitutionalBadge[]` + `capabilities` (`variant`, `size`, `showLabel`, `showTooltip`, `compact`, `monochrome`, `linkToProgram`).

## 14. Roadmap de adopción

1. **I3.d.2 · M1 (autorizado)** — Introducir tokens §5 en `src/styles.css` sin tocar componentes existentes. Añadir densidades §3. Validación visual sin regresiones.

4. **Integraciones futuras** — DestinationSurface, BusinessSurface, ProductSurface, EventSurface consumen el bloque vía composición.

## 15. Gobernanza

- Este documento es **la única especificación** de Institutional Badges. Cualquier badge nuevo se incorpora extendiendo el registry, **no** creando componentes paralelos.
- Cambios a §3, §4, §5, §8 requieren nueva versión (`v1.1`, `v2.0`) y validación explícita del Founder.
- Cambios en el registry (nuevos `kind`, nuevos destinos autorizados) requieren PR dedicado con evidencia oficial de la institución que otorga el distintivo.
