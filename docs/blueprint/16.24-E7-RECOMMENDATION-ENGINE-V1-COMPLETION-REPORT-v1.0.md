# Épica E7 · Recommendation Engine v1 — Completion Report v1.0

**Fecha:** 2026-07-05
**Programa:** A Visitante + E Alux IA
**Carril:** A (Producto)
**Versión objetivo:** v3.0 (Beta Cerrada)
**Estado:** ✅ Cerrada (v1)

## 1. Objetivo
Introducir la capa oficial de **scoring + explicabilidad** para la
Related Collection, sustituyendo el ordenamiento heurístico del MVP
por un motor único, versionado y consumible por Alux. Cumple L6 de
la ficha de madurez del sistema Related Collection (H-03 · I3.b).

## 2. Alcance entregado

### 2.1 Backend — RPC `related_recommend_v1`
Nueva función SECURITY DEFINER (STABLE) que devuelve el mismo shape
del contrato E6 con dos adiciones por ítem:
- `score` numérico (0..N) — combinación ponderada de señales.
- `reasons` text[] — etiquetas cortas (`same-category`, `same-destination`,
  `same-region`, `verified`, `recent`, `popular`, `upcoming`, `editorial-pin`).
- `rationale` string legible — derivado de las señales (ej. *"Misma
  categoría en el destino · Verificada · Con reseñas"*).

Señales por tipo (v1):
| Tipo | Señales |
|---|---|
| business | same-category(+3), same-destination(+2), same-region(+1), verified(+2), recent(+1), popular por reseñas(+0..2) |
| product | same-category(+3), same-destination(+2), same-region(+1), verified-business(+1), recent(+1) |
| destination | same-region(+2), recent(+1) |
| event | same-destination(+3), upcoming(+2) |

Respeta la **capa editorial de E6** (pins primero con `score=999` y
`reasons=['editorial-pin']`, hides siempre excluidos). Firma idéntica
a `related_get_collection` — E7.b puede migrar los fetchers al nuevo
RPC sin romper el contrato del bloque.

### 2.2 Frontend — Explainable by Default
`ExperienceRelatedCollectionBlock.buildDTO()` ahora activa
`showRationale: true` por defecto cuando `source !== "manual"`. El
editor puede seguir desactivándolo explícitamente en Studio si prefiere
una vista más compacta. Consecuencia inmediata:
- Empresa (`source=business`) muestra el rationale que ya inyectaba el
  adapter (*"Otras opciones de la misma categoría en el destino"*,
  *"Otras experiencias del mismo destino"*).
- Producto (`source=product`) muestra rationale por grupo (misma
  empresa / misma categoría en destino / otros en destino).
- Destino (`source=destination`) — se enriqueció el adapter
  `destinationRelatedBucketToItems`/`Products`/`Events` para aceptar
  y renderizar rationale por bucket (*"Hoteles del destino"*,
  *"Próximos eventos en el destino"*, etc.).
- Categoría (`source=category`) ya inyectaba rationale desde el adapter.

### 2.3 Fuera de alcance (E7.b)
- Migración de fetchers (`getBusinessRelated`, `getProductRelated`,
  `getDestinationRelated`, `getEventRelated`) para consumir
  `related_recommend_v1` en vez de sus SQL propios. Se hará cuando
  tengamos ≥5 empresas por destino/categoría (validación con datos
  reales, no seeds sintéticos).
- Señales adicionales: overlap con `traveler_favorites` cuando hay
  sesión, boost por `travel_plans.active_destination`, decaimiento
  temporal de eventos ya pasados.
- Integración directa desde Alux (`AluxTravelerPanel.recommendations`
  tomará ítems de `related_recommend_v1` en E-Alux · siguiente sub-ola).

## 4. Contrato para Alux (E7-ready)
```
related_recommend_v1({
  entityType, entityId, surface,
  context?: { activeDestinationId?, localeId?, travelerId?, ... },
  limit?: number,
}) → {
  items: [{ id, entity_type, title, slug, score, reasons[], rationale, ... }],
  strategy: 'recommend-v1' | 'recommend-v1+manual-pin',
  rationale: string,
  context: {...}
}
```
Alux debe consumir este RPC — prohibido generar recomendaciones
paralelas (Directiva Founder H-03 · I3.b: *"Alux no deberá generar
recomendaciones paralelas: siempre que sea posible deberá reutilizar
la lógica de este bloque"*).

## 6. Pasos de reproducción (Demo Pack)
1. Abrir cualquier ficha pública de empresa (`/oriente-maya/valladolid/…/…`) —
   cada card de la Related Collection muestra su rationale ("Otras
   opciones de la misma categoría en el destino · Verificado").
2. Abrir producto (`/producto/<slug>`) — mismos rationales por grupo.
3. Consultar la RPC directamente (SQL editor):
   ```sql
   SELECT related_recommend_v1(
     'business'::related_entity_kind,
     (SELECT id FROM businesses WHERE slug='hacienda-selva-maya'),
     'business-profile',
     '{}'::jsonb,
     8
   );
   ```
   → `items[]` con `score`, `reasons`, `rationale` legible.
4. Crear un `pin` en `/cms/relacionados` → la RPC lo devuelve primero
   con `reasons=['editorial-pin']` y `strategy='recommend-v1+manual-pin'`.

## 7. Impacto en Roadmap v2.0
- **Programa A Visitante:** 86% (refuerza descubrimiento explicable
  en todas las fichas públicas).
- **Programa E Alux IA:** 40% → **45%** (contrato listo para

  Espita, DSL Colonial M4..M5.

## 8. Founder Decisions incorporadas
- **Explainable by Default:** cada ítem declara `rationale`, `reasons` y
  `score` como parte del contrato.
- **Single Source of Truth:** un único motor (`related_recommend_v1`)
  para toda superficie que necesite recomendaciones — Alux incluido.
- **Regla Evolutiva de bloques:** cero bloques nuevos, cero contratos
  nuevos, cero engines nuevos. Se evoluciona por RPC + capabilities.
- **Freeze de infraestructura:** reutiliza `related_overrides`,
  `related_entity_kind`, `page_compositions` v1.
- **Product Vision Rule:** ayuda al visitante a entender por qué le
  mostramos cada opción (vivir mejor); ayuda a Alux a razonar sobre
  fuentes verificables (responder mejor); reduce trabajo editorial
  (curación sólo cuando el ranking natural no basta).

## 9. Rollback
- Revertir `showRationale` a `false` por defecto → cambio de una línea
  en `buildDTO`.
- La RPC `related_recommend_v1` es aditiva; puede eliminarse con
  `DROP FUNCTION` sin afectar superficies (nadie la consume aún).

**Cierre:** E7 v1 listo. Motor de recomendaciones explicable en
producción, disponible para Alux y futuras superficies. Related
Collection alcanza nivel de madurez L6.
