# H-03 · Reglas de Diseño de Bloques (Arquitectura Oficial)

**Estado:** vinculante — forma parte de la arquitectura oficial del proyecto.
**Fecha:** 2026-07-04
**Origen:** aprobación H-03 · Auditoría UX de Plantillas Madre (Founder).
**Alcance:** toda la Biblioteca Oficial del Experience Builder, presente y futura.

---

## 1. Contrato de tres capas

Todo bloque oficial del Experience Builder DEBE construirse con tres
capas claramente separadas y documentadas en su ficha:

### 1.1 Presentación
Cómo se muestra el bloque al visitante.
- Componente React presentacional puro (sin fetching, sin router).
- Recibe todo por props tipadas.
- Sirve variantes visuales declarativas (`variant`, `layout`, `density`).
- Mobile-first, Touch-first, accesible (WCAG AA), sin dependencias de red.

### 1.2 Contenido
Qué datos consume, cómo se alimenta y qué contratos utiliza.
- Contrato Zod versionado del `config` editable en Studio.
- Contrato Zod versionado del `data` runtime (DTO).
- Server function o hook TanStack Query oficial que resuelve el DTO.
- Fuentes permitidas: Lovable Cloud (Supabase), Context Engine,
  Discovery Navigator, Alux. Prohibido acoplarse a fuentes ad-hoc.

### 1.3 Comportamiento
Cómo interactúa con usuario y plataforma.
- Reglas de interacción (clicks, gestos, sticky, sheets, keyboard).
- Integración con Context Engine (lectura + emisión de eventos).
- Integración con Alux (fuentes explicables, sugerencias).
- Integración con Discovery Navigator y protected-actions.
- Extensibilidad: `extensions[]` para capacidades futuras.

El wrapper EB (`*Block.tsx`) orquesta las tres capas. La ficha del
bloque en `block-library.ts` DEBE declarar explícitamente los contratos
de las tres capas.

## 2. Regla de Compatibilidad Evolutiva

**Prohibido crear bloques duplicados** (`Gallery V2`, `Hero Pro`,
`Reviews Next`, etc.). La Biblioteca crece por evolución, no por
duplicación.

Toda nueva capacidad debe extender el bloque existente mediante:

- `config` versionado (nuevos campos opcionales, defaults seguros).
- `capabilities` opt-in (`{ video: true, ar: false }`).
- `variant` (`panel | grid | list | immersive | ...`).
- `permissions` (rol/plan requerido para activar la capacidad).
- `extensions[]` (slots tipados para módulos externos).

### 2.1 Versionado de contratos
- El schema del bloque incluye `contractVersion` (semver).
- Cambios aditivos → minor (compatibles).
- Cambios que rompen → major (requiere migración documentada en el
  bloque, nunca un bloque nuevo).
- Toda composición publicada debe seguir renderizando bajo el mismo
  `blockType`.

### 2.2 Ejemplo canónico — Galería
Un bloque `vmx.media.gallery` debe arquitectarse para incorporar en el
futuro, sin reemplazo:

- imágenes (v1),
- video (capability),
- recorridos 360° (capability),
- modelos 3D (capability),
- UGC moderado (capability + fuente Cloud),
- realidad aumentada (capability + permission),
- futuras experiencias inmersivas (extension slot).

El mismo `blockType` cubre todas las evoluciones.

## 3. Renombrado H-03 (aplicación inmediata de la regla)

La auditoría original propuso sufijos `-pro`. Se descartan por
incompatibilidad con esta regla. Nombres oficiales actualizados:

| Auditoría (v1.0)         | Nombre oficial       |
|--------------------------|----------------------|
| `vmx.media.gallery-pro`  | `vmx.media.gallery`  |
| resto de bloques (§3)    | sin cambio           |

No se crearán nunca variantes `-pro`, `-v2`, `-next`, `-lite`. Toda
diferencia se expresa vía `variant` / `capabilities` / `permissions`.

---

## 4. Checklist de admisión (Definition of Ready por bloque)

Antes de implementar cualquier bloque, la ficha en `block-library.ts`
debe responder afirmativamente a las seis preguntas oficiales:

1. ¿Puede reutilizarse en al menos dos plantillas?
2. ¿Aporta valor al visitante?
3. ¿Mejora el descubrimiento del destino?
4. ¿Mejora la conversión?
5. ¿Es compatible con móviles?
6. ¿Respeta el Context Engine?

Si alguna respuesta es negativa, el bloque se rediseña antes de
implementarse.

---

## 5. Definition of Done por bloque

- Presentación, Contenido y Comportamiento entregados y documentados.
- Contratos Zod versionados (`contractVersion`).
- Preview neutral en Studio.
- Al menos una plantilla oficial integrando el bloque.
- Fallback legacy preservado durante migración.
- Typecheck + build + smoke verdes.
- Comparativa antes/después de la superficie migrada.
- Completion Report con impacto de producto.

---

**Aplicación obligatoria a partir de H-03 · Ola I1 y para toda futura
ola, épica e iniciativa del Experience Builder.**
