## 1. Behavioral Change Statement (obligatorio · Founder Behavioral First)

### 1.1 Comportamiento actual
El equipo de producto/marketing/concierge decide sin evidencia de transiciones del Journey — se apoya en anécdotas, sesiones y clics. Alux, Concierge y Decision Center generan valor pero su efecto sobre las transiciones T1..T9 no queda registrado de forma comparable.

### 1.2 Comportamiento esperado
Cada decisión relevante de la plataforma (una recomendación de Alux aceptada, un plan creado, un caso promovido, una orden confirmada, un recap post-viaje) queda registrada como **evento canónico append-only** enlazado a una transición del Journey y a la capacidad que la influyó. El equipo consulta transiciones, no clics.

### 1.3 Por qué acerca al objetivo estratégico
Habilita la observación real del **JPR-30d** y de las 9 transiciones secundarias. Sin ingesta no hay medición; sin medición no hay optimización del avance del viajero por el Journey del Oriente Maya.

### 1.4 Cómo se comprobará
- Cobertura ≥ 80% de las 9 transiciones canónicas con al menos un evento canónico por día en producción tras CV8.2.
- 0 eventos ingeridos que no correspondan a los 4 `kind` del contrato v1.0.0.
- 0 mutaciones sobre `visitor_intel.events` (append-only verificable por trigger).
- Ratio señal/ruido monitoreado: eventos con `attributed_action` + `influencing_capabilities` presentes ≥ 90% en transiciones T4..T9.

## 3. Arquitectura

### 3.1 Esquema `visitor_intel` (dominio aislado)

Tabla única append-only `visitor_intel.events`:

| Campo | Tipo | Notas |
|---|---|---|
| `event_id` | `uuid PK` | Cliente propone; server valida unicidad (idempotencia). |
| `occurred_at` | `timestamptz NOT NULL` | Momento reportado por origen. |
| `ingested_at` | `timestamptz NOT NULL DEFAULT now()` | Marca de servidor. |
| `schema_version` | `text NOT NULL` | `'1.0.0'` — checked. |
| `kind` | `text NOT NULL` | `journey.transition` \| `intent.signal` \| `decision.offered` \| `outcome.observed`. |
| `subject_id` | `text NOT NULL` | UUID del sujeto (anónimo local o autenticado). |
| `trust_level` | `text NOT NULL` | `N0..N4` enum-checked. |
| `is_authenticated` | `boolean NOT NULL` | |
| `locale` | `text NULL` | BCP-47. |
| `destination_id` | `uuid NULL` | FK lógica (sin CASCADE en delete). |
| `surface` | `text NOT NULL` | Superficie declarada. |
| `route` | `text NOT NULL` | |
| `travel_stage` | `text NULL` | Enum lifecycle. |
| `live_day_phase` | `text NULL` | |
| `payload` | `jsonb NOT NULL` | Cuerpo tipado del evento (transition/intent/decision/outcome). Validado en server fn con Zod contra `VisitorEventSchema`. |
| `retention_bucket` | `text NOT NULL` | Derivado de `trust_level` en trigger. |

**Índices**: `(occurred_at DESC)`, `(subject_id, occurred_at DESC)`, `(kind, occurred_at DESC)`, `((payload->'transition'->>'id')) WHERE kind = 'journey.transition'`.

**Append-only garantizado por**:
- Sólo `INSERT` permitido por policy.
- Trigger `BEFORE UPDATE OR DELETE` que lanza excepción (`raise exception 'visitor_intel.events is append-only'`).
- Correcciones se emiten como nuevo evento (nunca mutando el original).

### 3.2 RLS + GRANTs
- Esquema `visitor_intel` sin `USAGE` a `anon`/`authenticated` (sólo `service_role`).
- Escrituras exclusivamente desde server functions `createServerFn` (usando `supabaseAdmin` cargado dentro del handler, autorizando la fuente).
- Ninguna lectura desde cliente. Dashboards (CV8.3) consumirán vistas materializadas dedicadas con k-anonymity ≥ 25, expuestas por server functions específicas.

### 3.3 Retención por `trust_level` (aplicación en CV8.11)
- N0 anónimo: 30 días (alineado a AC1 draft TTL).
- N1..N2: 180 días.
- N3..N4: 24 meses (auditoría transaccional).
CV8.1 sólo persiste `retention_bucket`; el job real llega en CV8.11.

### 3.4 Server functions (App)

Ubicación: `src/lib/visitor-intel/ingest.functions.ts` (client-safe, no importa `client.server` en top-level).

- `ingestVisitorEvent({ event })` — server fn autenticada (`requireSupabaseAuth`) para usuarios N1+. Deriva `subject_id`/`trust_level` desde `context.claims` (nunca confía en cliente).
- `ingestAnonymousVisitorEvent({ event })` — server fn pública, rate-limited por IP+subject_id. Sólo acepta `trust_level: 'N0_anonymous'`. Valida `event.subject.is_authenticated === false`.

Ambas:
1. `VisitorEventSchema.parse(event)` (Zod).
2. Rechazan `kind='journey.transition'` cuyo `transition.id` no sea `JourneyTransitionId` canónico (`isCanonicalTransition`).
3. Cargan `supabaseAdmin` **dentro del handler** y hacen `insert` idempotente por `event_id` (`on conflict do nothing`).
4. Devuelven `{ accepted: boolean, reason?: string }`.

## 5. Métricas de la propia ingesta (Regla de Accionabilidad)

| KPI técnico | Decisión que habilita |
|---|---|
| `ingest.accepted_rate` por `kind` | Detectar producers que emiten eventos inválidos → corregir producer, nunca aflojar contrato. |
| `ingest.canonical_transition_rate` | Detectar producers que emiten transiciones fuera de T1..T9 → bloqueo. |
| `ingest.append_only_violations` | DEBE ser 0. Cualquier valor > 0 dispara incidente. |
| `ingest.by_trust_level` | Verificar segregación real N0..N4. |

## 6. Riesgos y guardrails

- **Riesgo:** producers emiten eventos "por si acaso" → viola Signal Quality. **Guardrail:** revisión Founder de cada PR de instrumentación (CV8.2+) contra Regla de Eventos.
- **Riesgo:** el modelo se convierte en analytics genérico. **Guardrail:** rechazo automático de eventos con `kind` fuera de los 4 canónicos.
- **Riesgo:** ingesta anónima sin control → floods. **Guardrail:** rate limit por IP+subject_id en `ingestAnonymousVisitorEvent` y idempotencia por `event_id`.
- **Riesgo:** breaking changes al contrato. **Guardrail:** `schema_version` check en columna + Regla de Evolución.

## 7. Outcome Validation (obligatorio · Founder Outcome Validation Principle)

1. **¿Se implementó completamente la experiencia diseñada?**
   Sí en el alcance declarado: dominio, tabla append-only, trigger, server functions y test. Sin instrumentación de superficies (fuera de alcance).

2. **¿Existe evidencia de que puede provocar el comportamiento esperado?**
   Estructural: (a) contrato Zod compartido con CV8.0, (b) append-only verificable por trigger, (c) canal listo para producers autorizados. El cambio de comportamiento organizacional se observará cuando CV8.2..CV8.7 conecten producers reales.

3. **¿Qué métricas deberán observarse en producción?**
   `ingest.accepted_rate`, `ingest.canonical_transition_rate`, `ingest.append_only_violations = 0`, distribución por `trust_level`.

4. **¿Qué resultados demostrarían la hipótesis incorrecta?**
   - Producers necesitan eventos fuera del contrato → contrato insuficiente.
   - `append_only_violations > 0` → guardrail defectuoso.
   - Ratio de transiciones no canónicas alto → producers mal diseñados o Journey mal modelado.

5. **¿Qué ajustes se harían?**
   Extender contrato por versión minor aditiva (sin romper consumidores) y reforzar validación en `ingestVisitorEvent`. Prohibido relajar el trigger append-only.

**Outcome esperado.** Canal único y confiable de eventos de Journey, ready para que CV8.2..CV8.7 conecten producers sin fragmentación.

**Riesgos conocidos.** Presión por instrumentar rápido puede empujar producers a emitir eventos sin transición clara — mitigado por revisión Founder + Regla de Eventos.

**Resultado esperado a validar en producción.** Al cerrar CV8.7, los 9 producers canónicos alimentan `visitor_intel.events`; `append_only_violations = 0`; cobertura ≥ 80% de las 9 transiciones día a día.
