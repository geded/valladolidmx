## 1. Behavioral Change Statement

Hoy el navegador del visitante descarga el JPEG original completo (frecuentemente ≥400 KB, sin `srcset`, sin priorización por viewport). Con M2 activo sobre un asset elegible, el mismo `<img>` renderiza AVIF/WebP con el ancho más cercano al viewport y JPEG como fallback; el visitante ve la misma imagen en la misma URL canónica, pero descargando ~60–75% menos bytes y con menor tiempo hasta LCP. Si algo falla, el navegador recibe **el mismo binario legacy que hoy**, sin salto visual ni URL diferente.

## 2. Objetivo de negocio

- Reducir peso y latencia de imágenes en superficies públicas de alto tráfico (fichas de destino/negocio/experiencia) sin degradar SEO ni contrato público.
- Habilitar la palanca técnica que sostiene Core Web Vitals (LCP, CLS, INP) y el crecimiento internacional con conexiones móviles limitadas.
- Hacerlo de forma **reversible en segundos** sin migración, redeploy ni intervención humana coordinada.

## 3. Hipótesis medible

> **H2.1** — Sobre los assets del pilot servidos con `resolveMediaSource()` en modo pipeline, el peso mediano transferido por imagen bajará ≥55% vs legacy, el LCP mediano de la ficha bajará ≥15%, con **cero** incremento medible en tasa de error de imagen y **cero** cambios en URLs indexadas.

**Falsable:** si peso baja <35%, o LCP no mejora, o error rate sube >0.2 pp → rollback automático.

## 4. North Star y métricas secundarias

- **North Star:** `bytes_saved_per_impression_p50` sobre superficies elegibles.
- **Secundarias:**
  - `variant_selected_ratio` (pipeline vs legacy) por asset.
  - `fallback_ratio` (target < 1%).
  - `image_error_rate` (target ≤ baseline + 0.1 pp).
  - `lcp_p75_ms` en rutas piloto.
  - `cls_p75` (target = baseline; sin regresión).
  - `variant_resolution_latency_ms_p95` (server, target < 5 ms; SSR safe).
  - `signed_url_expiry_hits` (target = 0; ver §Riesgos).

```text
          └────────────────────────────────────────────────────────┘
                                │
                 ┌──────────────┴───────────────┐
                 ▼                              ▼
       flag ON & asset in allowlist?     NO → return LEGACY {canonical:file_url}
                 │ YES
                 ▼
       source_checksum != '-' ?          NO → LEGACY (log ineligible_no_checksum)
                 │ YES

       status='ready' AND ctx match?     0  → LEGACY (log no_ready_variant)
                 │ ≥1
                 ▼

       elegir width por sizes/targetWidth
                 │
                 ▼
       firmar URLs (TTL renovable)       err → LEGACY (log sign_error)
                 │ ok
                 ▼

                 │
                 ▼
       <picture><source .../><img .../></picture>
                 │
                 ▼
       navegador falla al decodificar?   → <img src=jpeg> ya es fallback nativo
                                          → si jpeg también falla, onError → file_url
```

## 7. Contrato propuesto de `resolveMediaSource()`

Extensión aditiva sobre el contrato M0 (no rompe consumidores existentes):

```ts
export interface ResolveMediaSourceOptions {
  context?: MediaUsageContext;
  targetWidth?: number;
  sizes?: string;                 // NEW — passthrough a <img sizes>
  priority?: "high" | "low" | "auto"; // NEW — mapea a fetchpriority/loading
  fallbackUrl?: string;
}

export interface ResolvedMediaSource {
  canonical: string;              // SIEMPRE = file_url legacy (contrato público estable)
  src: string;                    // JPEG (pipeline o legacy)
  sources: Array<{ type: string; srcSet: string; sizes?: string }>;
  width?: number;
  height?: number;
  usedPipeline: boolean;
  pipelineStatus: "disabled"|"pending"|"processing"|"ready"|"failed"|"skipped"|"unknown";
  fallbackReason?:                // NEW — observabilidad
    | "flag_off" | "not_in_allowlist" | "no_checksum"
    | "no_ready_variant" | "sign_error" | "unsupported_format";
  urlTtlSeconds?: number;         // NEW — para caches upstream
}
```

**Invariantes:**
- `canonical` **nunca** apunta a `media-derived/*`.
- `usedPipeline=true` ⇒ `sources.length ≥ 1` y `src` proviene del pipeline.
- `usedPipeline=false` ⇒ `sources=[]` y `src=canonical` (o `fallbackUrl`).
- Función pura, sincrónica sobre datos precargados; el fetch de variantes ocurre aguas arriba en el loader del route.

## 10. Métricas y umbrales automáticos de rollback

Ventana deslizante 15 min, evaluada cada 1 min:

- `fallback_ratio > 5%` → **PAUSA** (flag OFF automático).
- `image_error_rate` incremento > **0.5 pp** sobre baseline → **ROLLBACK**.
- `sign_error_rate > 1%` o cualquier `url_expired_hit` observado en cliente → **ROLLBACK**.
- `lcp_p75_ms` regresión > **10%** sostenida 30 min → **PAUSA**.
- `variant_resolution_latency_p95 > 20 ms` → **PAUSA**.

"Pausa" = flag global OFF, allowlist intacta. "Rollback" = flag OFF + vaciar allowlist a `[]`. Ambos son operaciones de una fila en `platform_settings`, sin migración ni redeploy.

## 11. URLs firmadas y caché (resolución explícita)

**Riesgo:** una URL firmada con TTL corto puede quedar cacheada en HTML SSR/CDN y expirar en el navegador → imagen rota.

**Contrato propuesto:**

- TTL de firma = **7 días**; renovación proactiva al 50% (3.5 días) en cada render server-side (idempotente por `variant_key`).
- SSR **no** cachea HTML con URLs firmadas más allá del TTL de la firma; `Cache-Control` de respuestas HTML se acota a `max-age ≤ 300, s-maxage ≤ 300, must-revalidate` en rutas que consuman pipeline. Nunca superior al 20% del TTL de firma.
- Firma por lote (una llamada Storage por render de ruta) para mantener `variant_resolution_latency_p95 < 5 ms`.
- Alternativa evaluada y **descartada** por ahora: bucket público `media-derived`. Se descarta por Founder Stable Public Asset Contract: URLs `media-derived/*` **no** son canónicas y no deben filtrarse a crawlers.
- Contrato lógico estable: `canonical = file_url`; toda URL firmada es efímera y jamás se persiste en JSON-LD/OG/sitemap.

## 12. Restricción UNIQUE anterior (deuda explícita)

`UNIQUE (asset_id, format, width, engine)` sigue vigente. Consecuencias para M2:

- Sólo existe **una** fila física por dimensión → `is_current` es informativo hasta el PASO 4.
- No es posible A/B entre `engine_version` distintos sobre el mismo width simultáneamente.
- El read path selecciona por `is_current AND status='ready'`; hoy coincide 1:1 con la UNIQUE legacy, por lo que no hay riesgo de doble activa.
- Requerido para versionado real: PASO 4 destructivo (fuera de este blueprint).

## 13. Checksums legacy `-` (deuda explícita)

24 assets tienen `source_checksum='-'`. **No son elegibles** para canary. Opciones futuras (requieren autorización aparte):

1. Backfill puntual por asset (recomputar checksum leyendo `media-original`).
2. Política "checksum-on-touch": al re-derivar un legacy se calcula y persiste.
3. Excluir permanentemente si el asset será retirado.

M2 no ejecuta ninguna.

## 14. Manejo de fallos (cobertura)

| Escenario | Detección | Acción | Log |
|---|---|---|---|
| Flag OFF | lectura `platform_settings` | LEGACY | `flag_off` |
| Asset no elegible | allowlist | LEGACY | `not_in_allowlist` |
| `source_checksum='-'` | validación | LEGACY | `no_checksum` |
| Variante inexistente / no ready | join vacío | LEGACY | `no_ready_variant` |
| Error al firmar | try/catch Storage | LEGACY | `sign_error` |
| URL expirada en cliente | `onError` del `<img>` | swap a `canonical` | `url_expired_hit` |
| Storage inaccesible | timeout Storage | LEGACY | `storage_unreachable` |
| Formato no soportado (p.ej. asset PNG con transparencia sin AVIF) | ausencia de format en pool | JPEG/PNG legacy | `unsupported_format` |

Todos los caminos de fallo devuelven **la misma URL legacy que hoy**. Nunca 404, nunca imagen distinta.

## 15. Seguridad

- Bucket `media-derived` permanece **privado**; sólo firma con `service_role` en server.
- `service_role`, `SUPABASE_SERVICE_ROLE_KEY`, allowlist y flag jamás cruzan al bundle cliente.
- Contrato: `resolveMediaSource` es puro y sólo firma URLs vía helper `src/lib/media/sign.server.ts` invocado en loaders/server fns.
- Import protection: helpers de firma viven en `*.server.ts`; consumidores importan sólo el resultado ya serializado.

## 16. Resolución SSR-safe

- Toda selección de variante ocurre en loader server-side; el cliente recibe el `ResolvedMediaSource` ya materializado.
- `<picture>` renderizado en primer HTML: cero flash, cero mismatch de hidratación.
- `width`/`height` provienen de `original_width/height` → CLS = 0.
- `sizes` explícito por contexto (`hero`, `card`, `gallery`, `thumbnail`).
- `fetchpriority="high"` sólo para `hero` above-the-fold; `loading="lazy"` por defecto en el resto.

## 19. Riesgos y deuda

| Riesgo | Severidad | Mitigación |
|---|---|---|
| HTML SSR con URL firmada expirada en CDN | Alta | TTL 7d, `Cache-Control ≤ 300s`, `onError→canonical`, alarma `url_expired_hit` |
| Crawler indexa URL `media-derived` como canónica | Alta | `canonical` = `file_url` invariante; JSON-LD/OG usan `canonical`; robots no bloquea pero contrato no expone |
| Un cambio de `engine_version` requiere dos filas simultáneas | Media | Fuera de alcance M2; requiere PASO 4 |
| Assets legacy sin checksum quedan permanentemente en legacy | Media | Aceptado en M2; backfill futuro autorizado por Founder |
| Latencia Storage.sign() en spikes | Media | Firma por lote + cache in-memory 3.5 d por proceso |
| Regresión CLS por `sizes` incorrecto | Baja | Enforcement de `width/height` obligatorios; tests visuales |
| Fuga de `service_role` a cliente | Crítica | Import guard `.server.ts`; revisión previa a merge |
