## 2. Modelo de invalidación honesto (Hallazgo 1)

Storage de Supabase (S3-compatible con URLs firmadas por HMAC + `expires`) **no soporta revocación criptográfica** de URLs ya emitidas. Cualquier URL firmada válida permanecerá funcional hasta:

- expiración natural (`expires_at`), o
- rotación del secreto de firma del bucket (afecta a TODAS las URLs, operación disruptiva), o
- eliminación del objeto subyacente (rompe también legacy y variantes vigentes).

### Distinción explícita de conceptos

| Concepto | Efecto | Alcance | Latencia |
|----------|--------|---------|----------|
| **Invalidación en aplicación** | La app deja de entregar esa firma a nuevos requests | Sólo lecturas futuras del read path | Inmediata |
| **Eliminación de caché** | Se descarta la entrada en LRU per-worker | Sólo ese worker | Inmediata local |
| **Purga de tabla** (`DELETE` de la fila) | Sumado a invalidación aplica a todos los workers | Read path completo | ≤ TTL de precarga |
| **Revocación criptográfica real** | **NO SOPORTADA** por Storage sin rotar el secreto del bucket | N/A | N/A |
| **Eliminación del objeto** | La URL responde 404 aunque siga firmada | Global | Segundos |
| **Expiración natural** | Storage rechaza la firma | Global | Determinística |

### Riesgo residual máximo

- **TTL firmado:** 7 días.
- **Ventana de exposición residual máxima:** desde el momento en que la app decide "invalidar" hasta la expiración natural, hasta **7 días**.
- **Alcance del riesgo:** una URL firmada filtrada permite descargar **una única variante derivada** (ancho/formato concreto) de un asset **ya público** por diseño. El original inmutable no se expone. El bucket derivado no lista otros objetos ni permite escritura.
- **Aceptabilidad:** ACEPTABLE mientras el asset piloto sea un asset público de contenido editorial. Si el pipeline se extendiera a assets con restricción de audiencia, se reduce el TTL a ≤24 h y se documenta explícitamente antes de habilitar.
- **Prohibido** usar el término "revocación inmediata" en código, logs, docs o UI. Terminología permitida: *invalidación*, *purga*, *expiración*, *rotación del secreto*.

## 3. Esquema endurecido (Hallazgo 6)

Tabla server-only `public.media_asset_signed_urls` (sin GRANT a `anon`/`authenticated`).

```sql
-- Enum de estados de renovación
CREATE TYPE public.media_signature_renewal_state AS ENUM (
  'idle', 'renewing', 'failed'
);

CREATE TABLE public.media_asset_signed_urls (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  variant_id          uuid NOT NULL REFERENCES public.media_asset_variants(id) ON DELETE CASCADE,
  variant_key         text NOT NULL,               -- huella determinista (§ M1)
  signed_url          text NOT NULL,               -- credencial efímera
  issued_at           timestamptz NOT NULL,
  expires_at          timestamptz NOT NULL,
  refresh_after       timestamptz NOT NULL,        -- ventana proactiva de renovación
  renewal_state       public.media_signature_renewal_state NOT NULL DEFAULT 'idle',
  renewal_lock_at     timestamptz,                 -- lock del scheduler
  renewal_lock_owner  text,                        -- identificador del worker que tomó el lock
  attempt_count       integer NOT NULL DEFAULT 0,
  next_retry_at       timestamptz,
  last_success_at     timestamptz,
  last_error          text,                        -- sanitizado, ver reglas abajo
  created_at          timestamptz NOT NULL DEFAULT now(),
  updated_at          timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT chk_expires_after_issued  CHECK (expires_at > issued_at),
  CONSTRAINT chk_refresh_window        CHECK (refresh_after > issued_at AND refresh_after < expires_at),
  CONSTRAINT chk_attempts_nonneg       CHECK (attempt_count >= 0),
  CONSTRAINT chk_last_error_bounded    CHECK (last_error IS NULL OR char_length(last_error) <= 500),
  CONSTRAINT uq_current_signature      UNIQUE (variant_id)  -- 1 firma vigente por variante
);

CREATE INDEX idx_msu_refresh_after  ON public.media_asset_signed_urls (refresh_after) WHERE renewal_state = 'idle';
CREATE INDEX idx_msu_next_retry_at  ON public.media_asset_signed_urls (next_retry_at) WHERE renewal_state = 'failed';
CREATE INDEX idx_msu_renewal_lock   ON public.media_asset_signed_urls (renewal_lock_at) WHERE renewal_state = 'renewing';

-- Superficie mínima
REVOKE ALL ON public.media_asset_signed_urls FROM PUBLIC, anon, authenticated;
GRANT SELECT, INSERT, UPDATE, DELETE ON public.media_asset_signed_urls TO service_role;

ALTER TABLE public.media_asset_signed_urls ENABLE ROW LEVEL SECURITY;
-- Sin políticas: RLS deny-all salvo bypass de service_role.
```

**Reglas de `last_error`:**

- Se guarda un código de familia + mensaje corto (`"storage_timeout"`, `"storage_5xx:502"`, `"auth"`, `"unknown"`).
- Nunca URL, token, cabeceras, secretos o cuerpos completos.
- Longitud máxima 500 (enforced por CHECK); si excede, se trunca y sufija `"…"`.

**Timestamps consistentes:** trigger `update_updated_at_column` en cada UPDATE. `issued_at`/`expires_at`/`refresh_after` sólo se escriben durante `INSERT` o transición `renewing → idle` con éxito.

## 4. Máquina de estados de renovación (Hallazgo 2)

```
                  ┌──────────────────────────────┐
                  ▼                              │
     ┌────────┐  refresh_after ≤ now       ┌───────────┐  success
     │  idle  │ ─────────────────────────► │ renewing  │ ─────────► idle
     └────────┘        (lock adquirido)    └───────────┘
         ▲                                       │
         │                                       │ fail
         │                                       ▼
         │                                  ┌────────┐
         │  next_retry_at ≤ now             │ failed │
         └──────────────────────────────────┴────────┘

     Lock huérfano: renewing con renewal_lock_at + LOCK_TTL < now
     ⇒ transición forzada renewing → failed (attempt_count += 1)
```

### Transiciones autorizadas

| Origen | Destino | Condición | Efectos |
|--------|---------|-----------|---------|
| `idle` | `renewing` | `refresh_after <= now()` y lock atómico (`UPDATE … WHERE renewal_state='idle' RETURNING`) | set `renewal_lock_at=now()`, `renewal_lock_owner=<workerId>` |
| `renewing` | `idle` | firma nueva emitida OK | set `issued_at`,`expires_at`,`refresh_after`, `renewal_state='idle'`, `attempt_count=0`, `last_success_at=now()`, `last_error=NULL`, `renewal_lock_at=NULL` |
| `renewing` | `failed` | error de firma / timeout | `attempt_count += 1`, `last_error=<sanitizado>`, `next_retry_at = now() + backoff(attempt_count)`, `renewal_lock_at=NULL` |
| `failed` | `renewing` | `next_retry_at <= now()` y `attempt_count < MAX_ATTEMPTS` | lock atómico igual que desde `idle` |
| `failed` | `idle` | recuperación manual o `attempt_count >= MAX_ATTEMPTS` con intervención operativa | requiere flag operativo, no automático |
| `renewing` (huérfano) | `failed` | `renewal_lock_at + LOCK_TTL < now()` | barrido cada ciclo; `attempt_count += 1` |

### Parámetros

| Parámetro | Valor | Justificación |
|-----------|-------|---------------|
| TTL firmado | 7 días | Techo de riesgo residual (§2) |
| `refresh_after` | `issued_at + 6 días` | 1 día de margen de proactividad |
| Margen mínimo servible | 6 h | Nunca entregar firma con < 6 h de vida |
| `LOCK_TTL` | 60 s | Ampliamente > tiempo esperado de una firma (< 1 s) |
| `MAX_ATTEMPTS` | 8 | Cubre ~24 h de reintentos con backoff |
| Backoff | exponencial con jitter: `min(2^attempt · 30 s + rand(0..15 s), 6 h)` | Evita thundering herd tras caídas |
| Reset de intentos | tras éxito o tras `now() > expires_at - 24h` sin renovar (alerta) | Evita quedarse bloqueado en `failed` para siempre |
| Alerta de proximidad | `expires_at - now() < 24 h AND renewal_state != idle` | Página al operador antes de que el fallback tome el read path |
| Alerta de disponibilidad | tasa de firmas con `< 6 h` de vida > 1% en 15 min | Señal temprana de scheduler degradado |

### Recuperación tras caída prolongada

- Al arrancar el scheduler ejecuta primero un **barrido de locks huérfanos** (`renewing` con `renewal_lock_at + LOCK_TTL < now()` → `failed`).
- Después re-selecciona tanto `idle` con `refresh_after <= now()` como `failed` con `next_retry_at <= now()`.
- Ninguna fila queda excluida por diseño: la selección cubre los 3 estados productivos.

## 5. Scheduler seguro (Hallazgo 5) — elección única

**Elección productiva: `pg_cron` interno.**

Se descarta el endpoint HTTP como implementación de v1.1 para minimizar superficie de ataque, evitar un secreto adicional y evitar exposición de red. `pg_cron` invoca directamente una función SQL/PLPGSQL (o una server fn a través de `pg_net` dentro del mismo cluster) que abre un cursor sobre la tabla y realiza las transiciones.

### Requisitos del job

- **Frecuencia:** cada 5 minutos. Con TTL de 7 días y `refresh_after` a 6 días, la ventana es holgada.
- **Selección atómica** por lote pequeño (≤ 50 filas) usando `UPDATE … WHERE renewal_state IN ('idle','failed') AND (refresh_after <= now() OR next_retry_at <= now()) LIMIT 50 RETURNING id` para tomar el lock sin race.
- **Barrido previo de locks huérfanos.**
- **Firma real** ejecutada por una función server-only autorizada. La credencial de firma vive sólo en el server; nunca vuelve al cliente.
- **Kill switch:** lee `platform_settings.media_persisted_signatures_enabled`; si es `false`, el job termina sin tocar nada (§7).
- **Rate limit implícito:** tamaño de lote + frecuencia acotan el gasto máximo.
- **Logs sanitizados:** sólo `variant_id`, `attempt_count`, familia de error. Nunca URL, token, headers.
- **Idempotencia:** re-ejecutar el mismo ciclo no genera firmas duplicadas (unique por `variant_id`).

### Si en el futuro se prefiere endpoint HTTP

Requisitos vinculantes para ese caso — **NO se implementan en v1.1**:

- Método `POST` exclusivamente.
- Autenticación HMAC (`X-Media-Sched-Signature`) sobre `timestamp + body`, secreto **independiente** de `MEDIA_SHADOW_INTERNAL_SECRET` (nuevo secreto `MEDIA_SCHEDULER_HMAC_SECRET`).
- Protección anti-replay: `timestamp` con ventana ±5 min y caché de nonces.
- Rate limit por IP y por firma HMAC.
- Respuesta genérica `404 Not Found` ante cualquier fallo de autenticación (nunca `401` con detalle).
- Logs sanitizados idénticos al job SQL.
- **Prohibido** usar `apikey` como única protección.

## 6. Deadline y abandono seguro del nivel on-demand (Hallazgo 4)

Cuando la tabla no tiene firma vigente para una variante candidata, el read path **no** debe esperar a que la firma on-demand concluya para decidir servir legacy.

### Contrato

```ts
const DEADLINE_MS = 250;

async function evaluateWithDeadline(variant): Promise<Resolution> {
  const ac = new AbortController();
  const deadline = new Promise<Resolution>((resolve) => {
    setTimeout(() => {
      ac.abort();                                // 1) intento real de cancelación
      resolve({ kind: 'legacy', reason: 'deadline_exceeded' });
    }, DEADLINE_MS);
  });

  const attempt = signOnDemand(variant, { signal: ac.signal })
    .then((url) => ({ kind: 'derived', url }))
    .catch((err) => ({ kind: 'legacy', reason: 'sign_error', family: familyOf(err) }));

  // Adjuntar catch para evitar unhandled rejections si attempt pierde la carrera
  attempt.catch(() => { /* metric: abandoned_after_deadline++ */ });

  return Promise.race([attempt, deadline]);
}
```

### Reglas

- **Deadline real** = 250 ms medido desde el instante en que la firma se agenda, no desde que retorna.
- **AbortController** si el SDK de Storage lo soporta; si no, `Promise.race` con descarte seguro.
- **Unhandled rejections prohibidas:** toda promesa perdedora lleva `.catch()` propio antes del `race`.
- **Métrica obligatoria:** `media_shadow.on_demand.abandoned_after_deadline` (contador) + `media_shadow.on_demand.completed_after_deadline` (contador secundario para diagnosticar SDK sin `AbortSignal`).
- **Single-flight anti-contaminación:** el resultado tardío de una firma abandonada **no** puede entrar al caché ni a la tabla si su `variant_key` ya no coincide con el vigente. Cada firma verifica `variant_key` contra la fila antes de escribir.
- **Escritura oportunista:** si la firma completa **antes** del deadline y es válida, se escribe la fila `idle` para acelerar futuras lecturas. Si completa **después**, se descarta.

## 7. Rollback no destructivo + kill switch (Hallazgo 3)

`DROP TABLE` **no** es rollback operativo. El código desplegado podría seguir consultándola y romper.

### Kill switch independiente

Setting server-only:

```
platform_settings.media_persisted_signatures_enabled: boolean (default false)
```

Efectos cuando está `false`:

- El read path **ignora** completamente la tabla y usa el camino M2.3 (LRU + on-demand con deadline §6).
- El scheduler sale sin trabajo en cada tick.
- Ninguna escritura nueva.
- La tabla se conserva intacta para auditoría e inspección.

### Secuencia de rollback operativo

1. `platform_settings.media_persisted_signatures_enabled = false`.
2. Confirmar por logs que el siguiente tick del scheduler no realiza selección.
3. Verificar que la telemetría del read path muestra `fallbackReason ∈ {lru_hit, on_demand, legacy}` y **ningún** `persisted_hit`.
4. Ejecutar benchmark M2.3 abreviado para confirmar equivalencia funcional.
5. Conservar la tabla `media_asset_signed_urls` como evidencia; **NO** dropear en el rollback.

`DROP TABLE` sólo puede ejecutarse en una **limpieza posterior separada**, con autorización explícita, tras verificar que ningún deploy vivo la consulta.

## 8. Validación estadística (Hallazgo 7)

El gate "disponibilidad de firma ≥ 99.9%" requiere volumen y ventana para ser significativo.

### Definición precisa

- **Métrica primaria:** `signature_availability = firmas_vigentes_servidas / evaluaciones_con_variante_elegible`. Sólo cuenta evaluaciones donde el pipeline es candidato; excluye assets no elegibles.
- **Ventana:** 24 h móviles.
- **Volumen mínimo:** ≥ **1 000 evaluaciones** en la ventana. Con menos, el gate se declara **inconcluso** (no falla, no aprueba); se extiende la observación hasta acumular volumen.
- **Ventanas sin tráfico:** no se emite verdict; se registra como `insufficient_volume` sin penalizar.
- **Criterio operacional:** aprobar sólo si el **límite inferior del intervalo de confianza de Wilson al 95%** para la proporción supera 0.999. Alternativa aceptable: ≥ 99.9% en 3 ventanas consecutivas de 24 h con ≥ 1 000 evaluaciones cada una.

### Separación de dos disponibilidades

| Métrica | Qué mide | Objetivo |
|---------|----------|----------|
| `signature_availability` | Firma persistida vigente presente cuando se necesitaba | ≥ 99.9% (gate) |
| `read_path_availability` | Usuario recibió alguna URL válida (derivada o legacy) | ≥ 99.99% (gracias al fallback) |

El fallback legacy garantiza la segunda métrica aunque la primera degrade. El gate M2.3.1 mide la primera; el compromiso público con el usuario mide la segunda.

## 9. Exposición: tabla vs URL firmada en M2.4 (Hallazgo 8)

Dos conceptos que v1.0 mezclaba:

| Objeto | Superficie | Estado v1.1 | Estado en M2.4 (futuro) |
|--------|------------|-------------|-------------------------|
| **Tabla `media_asset_signed_urls`** | Base de datos | Server-only, RLS deny-all, sólo `service_role` | **Igual: server-only.** El navegador nunca consulta la tabla, ni siquiera indirectamente. |
| **URL firmada individual (string)** | Respuesta HTML/JSON del server render | No sale del server (flag OFF, sin consumidor) | Sale al navegador como atributo `src`/`srcSet` de una imagen concreta, sólo cuando M2.4 esté aprobado |

**Regla:** que en M2.4 la URL firmada llegue al navegador para descargar la imagen es correcto y esperado (es el objetivo del pipeline). Lo que **nunca** llega al navegador es:

- la tabla ni ninguna proyección de sus columnas,
- listas de URLs firmadas de otras variantes,
- metadatos de renovación (`attempt_count`, `renewal_state`, `last_error`),
- el secreto de firma del bucket.

El server entrega, por cada `<img>`, exactamente la(s) URL(s) firmada(s) que ese elemento va a solicitar — nada más.

## 10. Threat model actualizado

| Amenaza | Superficie | Mitigación | Riesgo residual |
|---------|------------|------------|-----------------|
| Filtración de URL firmada individual | HTML servido al usuario legítimo | TTL 7 d, alcance = 1 variante pública, sin write | Aceptable (§2) |
| Filtración masiva de firmas | Tabla `media_asset_signed_urls` | RLS deny-all, sin GRANT a `anon`/`authenticated`, sólo `service_role`; server fn autorizada por `MEDIA_SHADOW_INTERNAL_SECRET` | Requiere compromiso de service_role o server fn |
| Compromiso del secreto de firma del bucket | Config de Storage | Fuera del alcance de este módulo; documentado como dependencia | Rotación manual invalida todas las firmas |
| Replay contra scheduler HTTP | N/A en v1.1 (scheduler es `pg_cron`) | — | 0 |
| Escritura maliciosa en la tabla | service_role únicamente | Auditoría de uso de service_role | Bajo |
| Envenenamiento de caché por firma tardía | Read path on-demand | Verificación de `variant_key` antes de escribir (§6) | 0 |
| Locks huérfanos permanentes | Scheduler | Barrido por `LOCK_TTL` (§4) | 0 |
| Filas atascadas en `failed` | Scheduler | `next_retry_at` + backoff + `MAX_ATTEMPTS` + alerta a 24 h de expiración (§4) | Bajo, observable |
| PII o secretos en `last_error` | Tabla | Sanitización a familia + CHECK de longitud (§3) | 0 |
| Exposición de URL como canónica en SEO | Contrato público | Read path la usa sólo como `src`/`srcSet`; la URL canónica indexada sigue siendo la legacy establecida | 0 |

## 12. Plan de pruebas obligatorias (previo a GO de implementación)

Se ejecutan **después** de que el Founder autorice implementar M2.3.1. No forman parte de v1.1.

1. Worker frío con tabla poblada → warm path desde primera lectura (sin cold de firma).
2. Firma vigente (`refresh_after > now`) → sin tocar Storage.
3. Firma próxima a expirar dentro del margen de 6 h → servida + programada a renovación.
4. Firma expirada → transición `idle → renewing`; fallback legacy mientras dura la renovación (con deadline §6).
5. Scheduler detenido 24 h → al reanudar barre huérfanos, procesa `idle` y `failed`, ninguna fila queda excluida.
6. Storage caído → firmas caen a `failed` con `attempt_count` incremental; read path usa legacy; recuperación automática al volver Storage.
7. DB caída para el scheduler → read path sigue funcionando (LRU + on-demand con deadline).
8. Cambio simulado de `variant_key` (nueva derivación) → fila anterior queda desalineada; se detecta y se re-inserta con nuevo `variant_key`.
9. 50 solicitudes concurrentes al mismo variant con firma ausente → una sola firma on-demand (single-flight §6), resto espera o cae a legacy por deadline.
10. Intento de lectura desde cliente anónimo (`anon`) a la tabla → error de permisos, cero filas devueltas.
11. Rollback completo (§7) → paridad funcional con M2.3 verificada.
12. Firma tardía llega tras deadline con `variant_key` obsoleto → descartada, no envenena caché ni tabla.
