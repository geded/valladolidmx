## Behavioral Change Statement

El Founder puede ahora ensayar la operación del Oriente Maya con simulaciones
realistas sin poner en riesgo datos reales: elige escenario, escala y seed,
previsualiza el volumen, ejecuta con confirmación, audita la corrida y la
elimina íntegramente en cualquier momento — con el banner visible en todo
momento cuando la consola muestra datos simulados o combinados.

## Experience

- Consola oficial en **`/cms/simulation`** (admin/super_admin).
- Configurador: escala (light/medium/full), seed textual, preview de
  visitantes · eventos · almacenamiento estimado antes de ejecutar.
- Doble confirmación reforzada para `full` (+ guard runtime
  `SIMULATION_ALLOW_FULL`).
- Historial de corridas con estado (`pending` · `running` · `completed` ·
  `failed` · `wiped`), eventos insertados, seed y timestamp.
- Acciones por run: **Ver simulación** · **Combinado** (abren
  `/cms/visitor-intel` con el run activo) · **Wipe** (modal con frase
  `BORRAR SIMULACION`).
- Consola Visitor Intelligence muestra banner persistente y no ambiguo
  cuando `mode !== "real"`, con salida directa a modo real.

## Architecture

### Persistencia (server functions)

`src/lib/visitor-intel/simulation/persistence.functions.ts` — canal único.

| Server fn | Rol |
|---|---|
| `previewSimulationRun` | Estimación de volumen sin efectos secundarios. |
| `executeSimulationRun` | Crea `simulation_runs` (running), ejecuta `runScenario`, inserta eventos en chunks de 500 con `upsert onConflict=event_id ignoreDuplicates=true`, cierra run como `completed` / `failed`. |
| `listSimulationRuns` | Historial ordenado (máx. 200). |
| `wipeSimulationRun` | `DELETE FROM events WHERE simulation_run_id = :id` + `status='wiped'` + `wiped_at=now()`. Requiere frase literal `BORRAR SIMULACION`. |
| `getDefaultScenarioInfo` | Metadatos del escenario oficial + `allow_full`. |

Todas exigen `admin` o `super_admin` (`requireAdmin`). Todas cargan
`supabaseAdmin` dentro del handler (module-graph safe).

### Escenario oficial

`src/lib/visitor-intel/simulation/scenarios/oriente-maya-90d.ts` — fábrica
determinística (`buildOrienteMayaScenario`) con Valladolid como base (60%) y
excursiones canónicas (Chichén Itzá, Ek Balam, Izamal, Río Lagartos, Las
Coloradas, Espita). Mezcla de 12 perfiles. Locales `es/en/fr`.

### Modo real / simulación / combinada

`aggregateJourneyIntel` acepta `mode` (`real` por defecto) y
`simulation_run_id`. Filtrado en SQL vía `.eq("is_simulation", false/true)`
y `.or(...)` para combinado. La consola sincroniza el modo por search params
(`?mode=&run=`).

**Nota (Founder Evolution Principle):** oportunidades, segmentos y
validación de recomendaciones todavía leen sin filtrar en esta iteración.
El banner lo advierte explícitamente; la migración de esos tres agregados
se cerrará como sub-tarea posterior sin cambiar contratos.

### UI

`src/routes/_authenticated/cms/simulation.tsx` — 100% cliente sobre
`useServerFn` + TanStack Query. Enlace desde `/cms` hub.

## Outcome Validation

1. **¿Se cumplió la promesa al usuario?** — Sí: el Founder puede ejecutar,
   ver, comparar y eliminar simulaciones sin tocar datos reales.
2. **¿Se puede regresionar accidentalmente el aislamiento?** — Sólo si un
   PR marca `is_simulation=false` al persistir o elimina por otro criterio.
   Ambas rutas están cubiertas por la política vinculante.
3. **¿Qué pasa si una corrida falla a mitad?** — El run queda como `failed`
   con `error_message`; los eventos parcialmente insertados se pueden
   eliminar con Wipe (siguen ligados al `simulation_run_id`).
4. **¿Puede alguien no autorizado ejecutar?** — No: `requireAdmin` en cada
   server fn valida `has_role('admin' | 'super_admin')`.
5. **¿Podemos comparar dos escenarios sin ambigüedad?** — Sí: cada run se
   abre en la consola con `?mode=simulation&run=<id>`. El banner muestra el
   `run` activo para evitar confusión con datos reales.

## Riesgos & follow-ups (no bloqueantes)

- Cerrar el filtro `mode` en `opportunities`, `segments` y
  `recommendations` en una sub-ola siguiente para consistencia total.
- Escala `full` en Cloudflare Workers puede rebasar límites de memoria;
  requiere ejecución en entorno dedicado con `SIMULATION_ALLOW_FULL=true`.
