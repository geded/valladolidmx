## 1. Resumen ejecutivo

La Fase 1 entregó la línea base técnica (infra, dominio, identidad, CMS governance, storage, RLS, server functions, Home). La Fase 2 deberá convertir esa base en producto: CMS administrable, Portal Empresarial, Marketplace, Concierge y Alux. Esta preparación define el orden de construcción, dependencias críticas, riesgos, componentes reutilizables, servicios compartidos y migraciones futuras necesarias, sin tocar el Blueprint ni la línea base.

Conclusión rectora: la Fase 2 debe construirse en olas concéntricas alrededor del CMS administrable. El CMS es el habilitador de toda sustitución de mocks (Fase 0) por datos reales y de toda experiencia downstream (Marketplace, Concierge, Alux).

## 3. Secuencia óptima de construcción (olas)

Ola por ola; ninguna ola se inicia sin Visual DoD + Gates 12D + QA aprobados de la anterior.

### Ola 1 — CMS Admin UI (habilitador transversal)
Panel `/admin/*` bajo `_authenticated/`, gate por `is_editor_or_admin`. Listas, editores y workflow para las 9 entidades editoriales ya existentes. Carga a buckets, gestor de `media_assets`, editor de `seo_metadata`, gestor de `translations`. Sin esta ola no hay forma legítima de poblar datos reales.

### Ola 2 — Sustitución progresiva de mocks (orden contractual)
`categorias → destinos → rutas → empresas → reseñas`. Cada sustitución pasa por loader de ruta pública leyendo Data API con políticas `TO anon` ya existentes; el mock se elimina sólo cuando la fuente real cubre el contrato visual y QA aprueba.

### Ola 3 — Portal Empresarial (`/empresa/*`)
Onboarding de `businesses`, `business_users` (roles `owner/manager/editor/viewer`), `business_locations`, `business_hours`, `business_contacts`, `business_social_links`, `business_media`, `business_category_links`, `products` + `product_media`. Aprovecha `has_business_access`. Flujo de `invitations` para multi-usuario.

### Ola 4 — Marketplace público
Listados y detalle de `businesses`, `products`, `events`, `editorial_routes`, `points_of_interest`. SEO por entidad (consume `seo_metadata`). Filtros por geografía (`countries/states/tourism_regions/destinations/destination_zones`).

### Ola 5 — Reseñas y reputación
`reviews` con moderación, antiabuso, vinculación a `businesses` y `products`.

### Ola 6 — Concierge
`concierge_profiles`, asignación de `concierge_requests`, panel `/concierge/*` con `is_concierge*`. Bandeja, conversación, vinculación a `travel_plans`.

### Ola 7 — Arma tu Viaje (planner real)
`travel_plans` persistidos, vinculados al usuario autenticado, exportables, integrables con Concierge.

### Ola 8 — Alux (IA)
Conectado a Lovable AI Gateway. Consume `knowledge_sources`, `embeddings_queue`, `ai_summaries`. Sólo publica vía flujo editorial.

### Ola 9 — Traducciones automáticas
Motor que rellena `translations` para entidades publicadas; revisión humana antes de publicar idioma.

### Ola 10 — Dashboards y observabilidad
Vistas a partir de `content_audit_log`, `permissions_audit_log`, métricas operativas; sin recolectar PII fuera de RLS.

## 4. Dependencias entre módulos

```text
Identity (C) ──┬─► CMS Admin (Ola 1) ──┬─► Sustitución mocks (Ola 2) ─► Marketplace (Ola 4)
               │                       │
               ├─► Portal Empresarial (Ola 3) ─► Productos ─► Marketplace
               │                                            │
               │                                            ├─► Reseñas (Ola 5)
               │                                            │
               └─► Concierge (Ola 6) ◄── Travel Plans (Ola 7)
                                       ▲
                                       │
                              Alux (Ola 8) ── Translations (Ola 9) ── Dashboards (Ola 10)
```

Dependencias duras:
- Marketplace **requiere** Ola 1 (datos editables) y Ola 3 (empresas reales).
- Reseñas **requiere** Marketplace.
- Concierge **requiere** Identity (`is_concierge`) y Travel Plans.
- Alux **requiere** corpus en `knowledge_sources` poblado por CMS y Marketplace.
- Translations **requiere** entidades en estado `published` con campos canónicos completos.

## 5. Riesgos técnicos

| ID | Riesgo | Probabilidad | Impacto | Mitigación |
| --- | --- | --- | --- | --- |
| R-01 | RLS demasiado restrictiva rompe lectura pública del Marketplace | Media | Alto | Revisar políticas `TO anon` y proyecciones seguras; tests por entidad. |
| R-02 | Fuga del cliente admin (`client.server`) al bundle del browser | Baja | Crítico | Importación dinámica dentro de handlers; smoke CI. |
| R-03 | Sobrecarga del CMS por edición concurrente | Media | Medio | `updated_at` + `transition_content_status`; UI optimista. |
| R-04 | Mocks olvidados tras sustitución (deuda visual) | Alta | Medio | Checklist por ola: eliminar import del mock sólo cuando QA cierre la entidad. |
| R-05 | OAuth Google con `redirect_uri` apuntando a rutas protegidas | Media | Alto | `redirect_uri = window.location.origin`; navegación post-sesión vía estado guardado. |
| R-06 | Server fns protegidas invocadas desde loaders públicos (401 SSR) | Media | Alto | Protegidas sólo desde componente o bajo `_authenticated/`. |
| R-07 | Alux publicando fuera del flujo editorial | Baja | Crítico | 12D cap.10: IA no publica; salida entra como `draft`. |
| R-08 | Translations inconsistentes entre idiomas publicados | Media | Medio | Bloquear publicación de idioma hasta cobertura mínima. |
| R-09 | Crecimiento de `media_assets` sin ciclo de vida | Media | Medio | Política de limpieza para `temporary`; límites por empresa. |
| R-10 | Divergencia visual al introducir nuevas pantallas | Media | Alto | Visual Gates Checklist obligatorio por pantalla. |

## 10. Estrategia de integración CMS ↔ Portal Empresarial ↔ Marketplace ↔ Concierge ↔ Alux

- **CMS** es la fuente editorial.
- **Portal Empresarial** es la fuente operativa. El CMS no edita estos datos; los aprueba para publicación.
- **Marketplace** es la capa pública de lectura. Lee datos publicados vía políticas `TO anon` y server fns públicos (publishable key, nunca admin).
- **Concierge** es la capa de servicio humano. Lee Marketplace + travel_plans; escribe en `concierge_requests`. RLS por asignación.
- **Alux** es la capa IA. Consume `knowledge_sources` poblados por CMS + Marketplace; produce sugerencias que entran al CMS como `draft`. Nunca publica directo (12D cap.10).

Patrón de contrato compartido:
- Toda escritura pasa por `createServerFn` o, para callbacks externos, server route bajo `/api/public/*` con verificación de firma.
- Toda lectura pública pasa por server fn con cliente publishable + proyección segura, no por `supabaseAdmin`.
- Toda transición de estado de contenido pasa por `transition_content_status`.
- Toda acción privilegiada queda registrada en `content_audit_log` o `permissions_audit_log`.
