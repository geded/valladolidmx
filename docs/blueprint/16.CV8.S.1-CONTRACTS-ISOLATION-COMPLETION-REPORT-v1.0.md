## 1. Behavioral Change Statement

Antes de CV8.S.1 no existía forma segura de introducir datos simulados en
`visitor_intel` sin riesgo de contaminar la operación real. Al cerrar CV8.S.1,
cualquier sub-ola siguiente puede sembrar eventos simulados garantizando que:
cada fila es inequívocamente identificable, cada corrida es catalogada con
seed/escala/escenario, y toda simulación puede eliminarse por completo con un
único comando sin tocar filas productivas.

## 2. Founder Directives aplicadas

- **Founder Simulation Isolation Principle** (nueva) — `is_simulation` +

## 3. Entregables

### 3.1 Base de datos (`visitor_intel`)
- Tabla `simulation_runs` (catálogo de corridas con status, seed, scale,
  scenario_payload, rows_inserted, wiped_at).
- Columnas `is_simulation boolean NOT NULL DEFAULT false` y
  `simulation_run_id uuid REFERENCES simulation_runs ON DELETE CASCADE` en
  `visitor_intel.events`.
- Índices: `visitor_intel_events_simulation_idx` (parcial WHERE simulation)
  y `visitor_intel_events_real_idx` (parcial WHERE real).
- Trigger `validate_simulation_flag` — impide inconsistencias
  (`is_simulation=true` sin `run_id`, y viceversa).
- Función `visitor_intel.wipe_simulation_run(uuid)` `SECURITY DEFINER` —
  borra sólo filas simuladas del run indicado y marca la corrida como
  `wiped`. EXECUTE revocado; sólo `service_role`.

### 3.2 Contratos TypeScript (`src/lib/visitor-intel/simulation/`)
- `prng.ts` — `createPrng(seed)` (mulberry32 + xmur3) con API
  `next / int / bool / pick / weighted / uuid / timestamp` y `deriveSeed`.
- `scenario.ts` — Zod:
  - `SimulationScenarioSchema` v1.0.0 (scale, calendar, destinations,
    profile_mix con propensiones, planted_issues, locales).
  - `SimulationRunSchema` (espejo del catálogo DB).
  - `SimulatedEventEnvelopeSchema` (`is_simulation=true` +
    `simulation_run_id`).
  - `IntelligenceQueryModeSchema` (`real | simulation | combined`) — Capa 4.
  - `simulatedSubjectId(runId, n)` con prefijo `sim_<runShort>_<n>`.
- `index.ts` — barrel.

## 5. Outcome Validation

1. **¿Toda fila simulada es identificable?** Sí — columnas + trigger + FK.
2. **¿La misma seed produce el mismo estado?** Sí — PRNG mulberry32
   sembrado por xmur3 sin fuentes implícitas de aleatoriedad.
3. **¿El wipe elimina 100% simulación y 0% real?** Sí — DELETE con doble
   filtro `simulation_run_id = ? AND is_simulation = true`; ON DELETE
   CASCADE desde `simulation_runs` como refuerzo.
4. **¿Las decisiones bloquean la evolución al Simulation Engine?** No —
   contratos serializables (Zod), PRNG puro, catálogo de runs generalizable
   a otros dominios (concierge, commerce) por sub-olas siguientes.
5. **¿Un observador confunde simulación con operación real?** No visualmente
   (Capa 4 exige selector); sí a nivel de fila si el frontend ignorara el
   flag — mitigado en S.4 con selector obligatorio.
