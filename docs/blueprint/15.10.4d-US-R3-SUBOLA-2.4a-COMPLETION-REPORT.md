### Fase D · Publicación híbrida + Validator system (P0)

**Sistema de validadores reutilizable** (`src/lib/portal/publish-validators.ts`):
- Contrato `PublishValidator<Snapshot>` con `kind` y `validate(snapshot) → PublishIssue[]`.
- Registry (`registerPublishValidator`, `getPublishValidator`, `runPublishChecks`).
- Severidades: `block | warn | info`. Cada issue lleva `code`, `message`, `fixHint?`, `field?`.
- **`ProductPublishValidator`** implementa checklist de Product: business publicada, autorización híbrida (`verified || can_self_publish`), portada, galería ≥ 2 (warn), descripción ≥ 40 chars (block) / ≥ 120 (warn), precio requerido para tipos comerciales, FAQs (info), nombre válido.
- Añadir un nuevo tipo (Promotion, Business, Event) = registrar un objeto. Cero cambios al motor.

**Gate híbrido (Modelo C)**:
- Columna nueva `businesses.can_self_publish boolean not null default false` — **independiente del plan comercial**. Founder puede activarla/retirarla por empresa vía SQL (o UI Founder futura).
- RPC `publish_business_product(_product_id) returns jsonb`:
  - Autoriza por `has_business_access('editor')`.
  - Rechaza si empresa no publicada.
  - Gate `verified OR can_self_publish` → retorna `reason='not_authorized_to_self_publish'`.
  - Valida reglas duras (cover, description ≥ 40, price cuando aplica) → retorna `reason='validation_failed', errors=[…]`.
  - Éxito → setea `status='published'`, `published_at=now()`, auditoría con `via='portal_self_publish'`.
- RPC `unpublish_business_product(_product_id)` → devuelve a `draft`, conserva media/FAQs, auditoría.

**Server fns portal** (`src/lib/portal/portal-product-publish.functions.ts`):
- `getPortalProductPublishSnapshot` — lee producto + business + media + faqCount, corre `runPublishChecks('product', snapshot)` y devuelve `{ snapshot, check }`.
- `publishPortalProduct` / `unpublishPortalProduct`.

**UI** (`PublishSection` en `ProductAdvancedPanel`):
- Checklist visible (block rojo, warn ámbar, info gris).
- Botón "Publicar producto" deshabilitado hasta `check.ok`.
- Botón "Retirar (ocultar)" visible sólo si `status='published'`.
- Feedback contextual según `reason` devuelto por la RPC.

## Criterio de aprobación (Sub-ola 2.4a)
Un empresario:
1. ✅ Crea producto (ya existente).
2. ✅ Sube portada + galería desde `ProductAdvancedPanel → Media`.
3. ✅ Agrega FAQs desde `ProductAdvancedPanel → FAQs`.
4. ✅ Ve checklist en tiempo real desde `PublishSection`.
5. ✅ Abre vista previa privada en `/portal/productos/$productId/preview` con Product Surface v1 idéntica.
6. ✅ Publica (si empresa `verified` o `can_self_publish`) o recibe mensaje "no autorizado" — puede caer al flujo existente `Solicitar revisión`.
7. ✅ Oculta con "Retirar" sin perder data ni media.
8. ✅ El producto publicado aparece en `/producto/$slug` con la misma composición `__tpl_product__`.
