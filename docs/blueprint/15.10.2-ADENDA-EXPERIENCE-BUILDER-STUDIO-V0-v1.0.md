## 0. Naturaleza del documento

Esta adenda detalla la **Etapa 2** del Plan 15.10 — *Experience Builder Studio v0* — primer editor visual operativo sobre las Fundaciones aprobadas en 15.10.1.

- **No modifica** el Blueprint vigente.
- **No introduce** nuevos dominios.
- **No autoriza** implementación hasta aprobación formal.
- **Respeta** CMS First, Composición por Bloques, BEA, UNC, IAC, Customer Case File y Alux read-only.
- **No utiliza** `SUPABASE_SERVICE_ROLE_KEY`.
- **Mantiene** compatibilidad total con Marketplace, Portal Empresarial, Cuenta del Viajero, Payment Lifecycle, Concierge Workspace y Olas 1–6.
- **Hereda** el Block Contract único, el Capability System y el Block Registry extensible aprobados en 15.10.1.

---

## 1. Objetivo de la etapa

Habilitar un **editor visual mínimo** que permita a roles administradores **componer páginas dummy** seleccionando, ordenando y configurando bloques del catálogo aprobado, generando **composiciones JSON versionadas** persistidas en backend, sin afectar todavía la Home pública ni ninguna ruta en producción.

Studio v0 es el primer eslabón operativo del Experience Builder: introduce el **lienzo editorial**, la **biblioteca de bloques**, el **inspector de propiedades** y el **ciclo borrador → revisión** sobre el contrato establecido en 15.10.1.

---

## 2. Alcance funcional

### 2.1 Incluido

1. **Ruta editorial protegida** `/cms/experience-builder` bajo `_authenticated`, con verificación server-side de roles `admin` / `content_editor` (vía `has_role` de 13.3).
2. **Modelo canónico de composición**:
   - Tabla `page_compositions` (entidades editoriales dummy, no rutas públicas en esta etapa).
   - Tabla `page_revisions` (historial de revisiones, snapshot JSON inmutable por revisión).
   - Árbol JSON canónico: `{ root: { children: [{ block_id, type, version, config, slots? }] } }`.
3. **Lienzo editorial (Canvas)**: render en vivo de la composición mediante el **Renderer Universal** (introducido en esta etapa como módulo `composition-renderer` consumiendo el Block Registry).
4. **Biblioteca de bloques (Library Panel)**: listado lateral consumiendo `listBlockLibrary` (15.10.1); filtrable por categoría (Static / Smart) y por capacidades.
5. **Inspector de propiedades**: formulario auto-generado a partir del `schema` declarado en el Block Contract; respeta capacidades `soporta_i18n`, `soporta_seo`, `soporta_datos_dinamicos`.
6. **Operaciones editoriales mínimas sobre el árbol**:
   - Insertar bloque (drag desde Library al Canvas o click).
   - Reordenar bloque (drag dentro del Canvas).
   - Duplicar bloque.
   - Eliminar bloque.
   - Editar configuración del bloque (Inspector).
   - Undo / Redo a nivel sesión (in-memory).
7. **Ciclo de persistencia**:
   - Guardar borrador (autosave debounced + guardado manual).
   - Crear nueva revisión inmutable.
   - Listar revisiones de una composición.
   - Cargar revisión previa en el Canvas (sin publicación).
8. **Server functions** (TanStack `createServerFn`, autenticadas vía `requireSupabaseAuth`):
   - `listCompositions` — listado paginado para el rol editorial.
   - `getComposition` — composición + revisión activa.
   - `createComposition` — crea entidad dummy.
   - `saveCompositionDraft` — actualiza borrador.
   - `createCompositionRevision` — snapshot inmutable.
   - `listCompositionRevisions` — historial.
9. **Validación de integridad de composición**:
   - Cada bloque referenciado debe existir en el Block Registry y resolverse a una versión vigente (no `deprecated`).
   - El `config` debe validar contra el `schema` del Block Contract.
   - Composiciones con bloques inválidos no pueden guardarse como revisión publicable, pero sí como borrador con marcadores de error.
10. **Eventos BEA editoriales** (sobre `content_audit_log`):
    - `Composition.Created`
    - `Composition.DraftSaved`
    - `Composition.RevisionPublished` (en esta etapa equivale a "snapshot creado", sin publicación pública).
    - `Composition.RevisionRestored`
11. **Preview interno**: render dentro del Canvas idéntico al que usaría el sitio público (Renderer Universal compartido), garantizando paridad visual 1:1.

### 2.2 Excluido explícitamente

- Publicación pública de composiciones (no se sirve ninguna composición desde rutas públicas).
- Migración de la Home actual (es objeto de la Etapa 15.10.3).
- Templates, Partials, Landing Pages (Etapa 15.10.4).
- Páginas institucionales, campañas, scheduling (Etapa 15.10.5).
- Páginas de destinos (15.10.6), empresas/productos (15.10.7).
- Smart Blocks v1 / v2 (15.10.8 / 15.10.10).
- Responsive Builder avanzado, validador SEO bloqueante, UNC editorial (15.10.9).
- Personalización, Alux Recommendations, JSON-LD avanzado (15.10.10).
- Edición colaborativa multiusuario en tiempo real.
- Comentarios editoriales, workflow de aprobación multinivel.
- Soft delete / papelera (no requerido en v0; se cubre con auditoría).

## 3. Modelo de datos (propuesto)

### 3.1 `public.page_compositions`

Entidad editorial canónica. En 15.10.2 representa composiciones dummy (sin ruta pública asociada).

- `id uuid` (PK)
- `slug text` (único, scope editorial — no expone ruta pública en esta etapa)
- `title text`
- `description text` (nullable)
- `status` ∈ { `draft`, `internal_review` } (no se introduce `published` hasta 15.10.3)
- `current_draft jsonb` (árbol de composición en edición)
- `active_revision_id uuid` (FK a `page_revisions`, nullable)
- `created_by uuid` (FK `auth.users`)
- `updated_by uuid`
- `created_at`, `updated_at`

### 3.2 `public.page_revisions`

Snapshot inmutable.

- `id uuid` (PK)
- `composition_id uuid` (FK)
- `revision_number int` (incremental por composición)
- `snapshot jsonb` (árbol congelado)
- `notes text` (nullable)
- `created_by uuid`
- `created_at`

### 3.3 RLS y GRANTs

- Ambas tablas con RLS habilitado.
- Políticas: solo roles `admin` y `content_editor` (vía `has_role`) pueden leer/escribir.
- GRANTs: `SELECT, INSERT, UPDATE, DELETE` a `authenticated`; `ALL` a `service_role`. Sin acceso a `anon`.
- No se otorgan permisos públicos en esta etapa (ninguna composición es servible al público todavía).

### 3.4 RPCs (SECURITY DEFINER)

- `eb_create_composition(_slug, _title, _description)` → uuid
- `eb_save_composition_draft(_id, _tree)` → void
- `eb_create_revision(_id, _notes)` → uuid
- `eb_restore_revision(_id, _revision_id)` → void

Todas validan rol mediante `has_role(auth.uid(), 'admin' | 'content_editor')` y registran evento en `content_audit_log`.

## 4. Renderer Universal (introducción)

Módulo `src/lib/experience-builder/composition-renderer.tsx`:

- Recibe un árbol de composición.
- Para cada nodo, resuelve el componente vía Block Registry (`getBlock(type, version)`).
- Pasa `config` validado contra el `schema` declarado.
- Maneja bloques no resueltos / deprecados con un placeholder de error explícito (visible solo en Studio, jamás en producción pública).
- Es el **mismo componente** que servirá las composiciones públicas a partir de 15.10.3 → garantía visual 1:1.
- Es **agnóstico** del origen (Studio o servidor público) y no introduce estilos propios fuera de los tokens 12D.00 / 12D.01.

---

## 5. UI · Studio v0

Layout de tres paneles:

- **Izquierda · Library Panel**: catálogo de bloques (consume `listBlockLibrary`), filtros por categoría y capacidades, búsqueda por nombre.
- **Centro · Canvas**: render en vivo de la composición mediante Renderer Universal, con overlays editoriales (selección, hover, handles de reordenar).
- **Derecha · Inspector**: formulario dinámico generado desde el `schema` del bloque seleccionado.

Toolbar superior: undo / redo, guardar borrador, crear revisión, abrir historial, alternar preview limpio (sin overlays).

Restricciones visuales:

- Sin estilos hardcodeados; toda la UI editorial respeta el sistema de tokens.
- Sin gradientes genéricos ni colores fuera de la paleta semántica.
- Responsive del propio Studio: desktop primero; tablet aceptable; móvil fuera de alcance v0.

## 6. Integraciones

- **Block Registry / Block Library (15.10.1)**: única fuente de bloques renderizables.
- **Block Contract (15.10.1)**: única fuente del `schema` consumido por el Inspector.
- **Capability System (15.10.1)**: define qué campos muestra el Inspector (i18n, SEO, datos dinámicos).
- **Permisos 13.3**: `has_role` para `admin` / `content_editor`.
- **`content_audit_log` + BEA**: eventos editoriales.
- **Media Library (12)**: campos `media` del Inspector la consumen vía picker existente.
- **Internacionalización**: campos `i18n` editan traducciones sobre la tabla `translations` polimórfica.
- **Alux Intelligence**: sin integración en esta etapa (Smart Blocks no operativos todavía).

## 9. Riesgos y mitigaciones

| # | Riesgo | Mitigación |
|---|--------|------------|
| R1 | Drift visual entre componentes originales y render por composición | Renderer Universal único; QA visual obligatorio sobre los 18 bloques. |
| R2 | Pérdida de trabajo editorial por fallo de red | Autosave debounced + guardado manual + revisiones inmutables. |
| R3 | Composiciones con bloques deprecados o esquema inválido | Validación bloqueante en `eb_create_revision`; marcadores de error en borrador. |
| R4 | Filtración de capacidades editoriales a roles no autorizados | Verificación server-side en cada RPC y server function; RLS estricto. |
| R5 | Acoplamiento del Studio al catálogo actual (impide extensibilidad) | Toda la UI se alimenta exclusivamente del Block Registry; nada del catálogo hardcodeado en Studio. |
| R6 | Sobrecarga del Inspector con esquemas complejos | Renderer del Inspector limita campos por tipo soportado; campos no soportados se marcan como "no editable en v0". |
| R7 | Confusión entre composiciones dummy y páginas reales | Etiqueta visible "Internal Draft" + ausencia total de publicación pública en esta etapa. |
| R8 | Crecimiento descontrolado de revisiones | Conteo y tamaño monitorizados; política de retención se definirá en 15.10.5 (scheduling). |

## 10. Exclusiones reforzadas

Esta etapa **no**:

- Sirve ninguna composición al sitio público.
- Reemplaza la Home actual ni ninguna ruta existente.
- Introduce Smart Blocks operativos (solo categoría declarada, sin resolvers activos).
- Introduce templates, partials, landing pages, scheduling, validador SEO bloqueante ni responsive builder avanzado.
- Modifica el Block Contract ni el Block Registry aprobados en 15.10.1.
- Otorga capacidades de escritura a Alux.
- Usa `SUPABASE_SERVICE_ROLE_KEY`.
