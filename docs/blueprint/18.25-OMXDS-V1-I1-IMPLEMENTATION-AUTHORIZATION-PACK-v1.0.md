## 4. Entorno reproducible autorizado a solicitar

### 4.1 Baseline obligatorio

- repositorio y commit base registrados antes de editar;
- rama técnica nueva desde la base autorizada, nunca trabajo directo en `main`;
- Bun exactamente `1.3.14`, coincidente con `packageManager`;
- `bun.lock` obligatorio y `bun install --frozen-lockfile`;
- Node no se utilizará para sustituir silenciosamente a Bun;
- variables de entorno reducidas al mínimo del build; ninguna evidencia expondrá secretos;
- misma secuencia en local/sandbox y GitHub Actions.

### 4.2 Preflight inmutable

Antes de editar debe guardarse salida y código de retorno de:

```bash
bun --version
git rev-parse HEAD
git status --short
bun install --frozen-lockfile
bun run typecheck
bun run build
bun run governance:check
```

Si instalación, typecheck, build o gobernanza ya fallan en la base, se documentará el fallo como baseline y se detendrá I1. No se autoriza arreglar deuda ajena dentro del slice.

### 4.3 Reproducción en CI

El job de I1 deberá usar checkout completo, `oven-sh/setup-bun@v2` con `bun-version: 1.3.14`, instalación congelada y los mismos comandos. No se autoriza omitir gobernanza, regenerar inventarios para esconder drift ni aceptar diferencias entre entorno local y CI.

## 5. Harness mínimo definitivo

### 5.1 Decisión

El harness inicial reutilizará el runner nativo de Bun para módulos puros y scripts sin DOM. No se autoriza añadir Vitest, Jest, Playwright, Cypress, Testing Library, jsdom, axe u otro framework en el primer cambio.

Motivo: el repositorio observado no declara esas dependencias, aunque dos scripts importan `vitest`. I1 no legitimará ese drift instalando una suite completa sin necesidad demostrada. Esos scripts se registran como deuda preexistente, fuera del alcance.

### 5.2 Capas de evidencia

| Capa | Método | Cobertura I1 |
|---|---|---|
| contratos puros | `bun test` y `bun:test` | tema, tiempo, registry y valores inválidos |
| integración estática | scripts Bun/JS | rutas, archivos, consumidores, flag, inventario y gobernanza |
| contraste | script determinista sobre tokens computables + casos reales | Sol/Luna, texto, controles, estados y badges |
| build/SSR | build + preview controlado + consola | primer tema, hidratación y baseline OFF |
| navegador | recorrido manual reproducible en 390 y 1440 | tema, skip-link, foco, touch y reduced-motion |
| accesibilidad humana | teclado y lector de pantalla real | selector, skip-link, badge, Alux y primitive representativo |
| visual | capturas antes/después y matriz de comparación | Sol/Luna, OFF/ON y rollback |

### 5.3 Scripts candidatos

La futura implementación podrá añadir, con nombres ajustados a la convención real:

- `test:i1`: ejecuta únicamente pruebas I1;

Los scripts deben fallar con código distinto de cero, no modificar fuentes ni regenerar gobernanza. Si una prueba DOM automatizada resulta indispensable, el equipo se detendrá y presentará una adenda de dependencia antes de instalarla.

- El bootstrap previo a pintura leerá únicamente preferencia local válida, `prefers-color-scheme` y fallback Sol. No esperará red.
- Con flag `OFF`, no se mostrará el selector público y el baseline seguirá siendo la referencia de rollback.
- Con flag `ON`, el selector podrá persistir `sol | luna`; ningún otro valor será válido.
- La consulta remota del flag no decidirá tardíamente la primera pintura. Si la arquitectura no puede satisfacerlo sin flash, I1 se detiene.
- El helper de tiempo y contratos puros podrán existir sin consumidores públicos.

### 6.3 Creación y activación

Crear la fila, helper server-safe y pruebas pertenece a la futura implementación I1. Activarla en preview/staging será una autorización posterior dentro del mismo cierre; producción requerirá autorización independiente. No se autoriza tocar el flag en este paquete.

## 8. Contraste y accesibilidad

El script de contraste calculará ratios desde los tokens efectivos de Sol/Luna y fallará ante pares inferiores a WCAG 2.2 AA. Las transparencias y overlays deberán medirse sobre su fondo real; no se aprobarán por comparar colores aislados.

La automatización no sustituye la verificación humana. Selector, skip-link, badge, Alux y primitive representativo deberán probarse con:

- teclado completo, foco visible y orden lógico;
- lector de pantalla real, nombre/estado/propósito;
- zoom de texto al 200%;
- target táctil y orientación móvil;
- `prefers-reduced-motion`;
- Sol/Luna y flag OFF/ON.

Un fallo en contraste esencial, teclado o lector es P1 y bloquea G1.

## 9. Badges sin expansión de schema

I1 podrá:

- cerrar la taxonomía en TypeScript;
- exigir fuente, verificación y vigencia dentro del registry;
- tratar registros legacy como no elegibles para hechos nuevos;
- ocultar de forma fail-closed cualquier badge desconocido, no verificado o vencido;
- localizar etiqueta y nombre accesible;
- añadir pruebas unitarias del contrato.

I1 no podrá añadir columnas, tablas, migraciones, seeds ni formularios Builder. Si la fuente real requiere persistencia inexistente, el badge afectado permanecerá oculto y se abrirá una propuesta separada; jamás se inventará evidencia para hacerlo visible.

## 15. Evidencia gobernada de implementación I1

La implementación autorizada por `PCA-2026-002` endurece Institutional Badges
sin ampliar su capacidad: `contract.ts` interpreta contenido legado sin
verificación explícita como no verificado, e
`institutional-badges.registry.ts` conserva el registry único y aplica
degradación *fail-closed* cuando faltan fuente, vigencia o verificación. Las
pruebas I1 demuestran que un distintivo no verificable no se presenta como
respaldo institucional.

La trazabilidad exacta de los artefactos I1 queda registrada en
`docs/governance/generated/07-BLUEPRINT-DEPENDENCY-MAP.json` y proyectada en
`docs/governance/generated/08-KNOWLEDGE-GRAPH.json`. Esta evidencia no autoriza
capacidades Builder, superficies, rutas, datos, activación del flag ni
producción.
