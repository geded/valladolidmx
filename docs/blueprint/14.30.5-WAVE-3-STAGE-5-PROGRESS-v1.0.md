## 3. Entregables técnicos

### 3.1 Migración (aditiva)

- Nuevas storage policies (sólo añaden, no modifican existentes):
  - `portal business write logos gallery` (INSERT)
  - `portal business update logos gallery` (UPDATE)
  - `portal business delete logos gallery` (DELETE, requiere `manager`)
  En las tres, `bucket_id IN ('logos','gallery')` y la primera carpeta del
  path es `<business_id>` (validado vía `storage.foldername(name)[1]::uuid`
  y `has_business_access`). Los buckets `companies` y `products` ya
  estaban cubiertos por Fase 1.
- Tres RPCs `SECURITY DEFINER`, `search_path = public`, EXECUTE restringido
  a `authenticated`:
  - `register_business_media(_business_id, _role, _bucket, _path, _mime,
    _size_bytes, _width, _height, _alt_text, _caption, _sort_order)` —
    Inserta en `media_assets` + `business_media`. Valida acceso editor,
    bucket esperado por rol, MIME en lista blanca (jpeg/png/webp/avif/gif),
    tamaño ≤ 10 MB, path bajo `<business_id>/`. Para `logo` y `cover`
    reemplaza el existente. Audita `media.create`.
  - `update_business_media_meta(_business_media_id, _alt_text, _caption,
    _sort_order)` — Actualiza alt/caption del asset y `sort_order` del
    vínculo. Audita `media.update`.
  - `remove_business_media(_business_media_id)` — Desvincula; soft-delete
    del `media_assets` si quedó sin vínculos (status → `archived`,
    `deleted_at` poblado). Audita `media.delete`.
- 0 cambios en tablas, columnas, índices, triggers o RLS de `public.*`.

### 3.2 Server functions (TanStack)

`src/lib/portal/business-media.functions.ts`:

| Función | Rol mínimo | Notas |
|---|---|---|
| `listBusinessMedia` | viewer | Devuelve cada `business_media` activo con `signed_url` (1 h). |
| `createBusinessMediaUploadTicket` | editor | Genera signed upload URL con `createSignedUploadUrl`, en bucket esperado y path `<business_id>/...`. MIME y tamaño validados antes de firmar. |
| `registerBusinessMedia` | editor | Llama a `register_business_media` tras la subida con metadatos whitelisted. |
| `updateBusinessMediaMeta` | editor | Llama a `update_business_media_meta`. |
| `removeBusinessMedia` | editor | Llama a `remove_business_media`. |

Reglas comunes:

- `.middleware([requireSupabaseAuth])` en todas las server fns.
- Validador de input estricto antes del handler (rechaza con mensaje
  específico). Whitelist explícita: el cliente nunca propaga `business_id`
  del registro, `created_by`, `deleted_at`, `metadata`.
- Defensa en profundidad: la server fn revalida `has_business_access`
  antes de tocar storage; las RPCs revalidan dentro del `SECURITY DEFINER`.

## 6. Integridad Referencial

- `business_media.business_id → businesses.id` y
  `business_media.media_asset_id → media_assets.id` permanecen estables.
- Restricción única `(business_id, media_asset_id, role)` respetada por la
  RPC al eliminar vínculo previo antes de reinsertar para roles unitarios
  (`logo`, `cover`).
- `content_audit_log` mantiene contrato Fase 1/Ola 1
  (`entity_kind='business'`, `entity_id = businesses.id`).
- Storage objects: el path se ata a `<business_id>/...`, y la política
  derivada de `storage.foldername(name)[1]::uuid` impide cross-tenant.

## 7. Mitigación de riesgos

| Riesgo | Mitigación |
|--------|-----------|
| RW3-1 Service role en cliente | No se usa. Token de subida creado server-side; upload usa `uploadToSignedUrl` (ata token + path). |
| RW3-2 Cross-empresa | Server fn revalida `has_business_access('editor')`; RPC revalida; storage policy ata el path a `<business_id>`. |
| RW3-3 Escalada editorial | El Portal no toca `status` de la empresa; sólo `media_assets.status` se setea a `published` (lectura pública controlada por RLS existente `media_public_read`). |
| RW3-6 Bucket equivocado / MIME no validado | Server fn + RPC validan bucket-por-rol y MIME en lista blanca antes de firmar e insertar. Tamaño máximo 10 MB. |
| RW3-7 Selector de empresa | Heredado de Etapa 1: `removeQueries` por predicate al cambiar empresa. |
| RW3-Audit Auditoría inyectable | Las RPCs son las únicas vías de inserción; acciones limitadas a `media.create|update|delete`. |

## 9. Advertencias del linter

- `0028/0029_*_security_definer_function_executable` suma 3 ocurrencias
  adicionales (una por RPC). Es intencional: son la única vía por la que
  un owner/editor de empresa, sin rol editorial global, puede insertar en
  `media_assets` y en `content_audit_log`. La validación de acceso, la
  whitelist y los conjuntos cerrados de bucket/MIME/acción están dentro de
  las funciones.
- No se introducen warnings nuevos fuera de este patrón ya aprobado en las
  Etapas 2–4.
