## 1. Resumen ejecutivo

La Etapa 5 implementa el cobro en línea de órdenes del Marketplace mediante
una **Capa de Abstracción de Proveedor de Pagos** (`PaymentProvider`).
El primer proveedor habilitado es **Stripe** (Checkout Sessions). El
Marketplace nunca importa el SDK de Stripe directamente; toda interacción
ocurre detrás del contrato neutral. Esto permite incorporar Paddle,
Mercado Pago, PayPal u otros sin modificar la lógica de negocio.

El código está listo y compila sin errores. Las llaves API se inyectan
como secretos de plataforma y aún no han sido cargadas — el flujo
end-to-end se activa al cargar `STRIPE_SECRET_KEY` y
`STRIPE_WEBHOOK_SECRET`.

## 4. Confirmación: no se exponen llaves ni secretos

✅ **Confirmado.**

- `STRIPE_SECRET_KEY` y `STRIPE_WEBHOOK_SECRET` se leen exclusivamente en
  el runtime de servidor (`process.env.*`) dentro de:
  - `stripe.server.ts` (`.server.ts` → bloqueado del bundle cliente).
  - El handler del webhook (route server-only).
  - El handler de `admin.functions.ts` (sólo reporta `boolean`, nunca el
    valor).
- El panel `/cms/pagos` muestra "✓ Configurado / ⚠ Pendiente" como flag.
  Nunca devuelve el valor de la llave al cliente.
- El prefijo (`sk_test_` vs `sk_live_`) se evalúa server-side y sólo se
  devuelve la etiqueta `"test" | "live" | "unknown"`.
- El payload completo de eventos webhook NO se expone en la UI; sólo
  `event_type`, `provider_event_id`, `received_at`, `processed_at`.
- Búsqueda en cliente: `rg "STRIPE_SECRET_KEY|STRIPE_WEBHOOK_SECRET" src`
  sólo arroja matches en archivos `*.server.ts` y `admin.functions.ts`
  (server-only).

## 5. Confirmación: autorización server-side admin/super_admin

✅ **Confirmado.**

Todas las server fns del panel admin de pagos aplican doble validación:

1. `requireSupabaseAuth` middleware → exige bearer token válido y
   resuelve `userId`.
2. `assertAdmin(context)` → RPC `has_role(_user_id, 'super_admin')` OR
   `has_role(_user_id, 'admin')`. Si ambos son `false` → `throw
   Error('forbidden')`.

Funciones protegidas: `getPaymentProviderStatus`,
`listRecentPaymentEvents`, `getPaymentsSummary`.

Las server fns del viajero (`startPayment`) usan `requireSupabaseAuth`
y validan que `order.user_id === context.userId`.

## 7. Webhook URL y eventos requeridos

**URL del webhook (Stripe):**
`https://valladolidmx.lovable.app/api/public/payments/stripe/webhook`

**URL estable (preview):**
`https://project--fd89b51f-9afc-4e15-8ee2-21fe468f6aa9.lovable.app/api/public/payments/stripe/webhook`

**Eventos requeridos a suscribir en Stripe Dashboard → Developers →
Webhooks:**

| Evento | Propósito |
|--------|-----------|
| `checkout.session.completed` | Pago síncrono exitoso → `order_mark_paid` |
| `checkout.session.async_payment_succeeded` | Pago asíncrono (OXXO, SPEI) exitoso |
| `checkout.session.async_payment_failed` | Pago asíncrono fallido |
| `checkout.session.expired` | Sesión expirada sin pago |
| `charge.refunded` | Reembolso aplicado (Etapa 6) |

El handler verifica firma HMAC vía `stripe.webhooks.constructEvent`
antes de procesar cualquier evento.

## 8. Eventos auditados

Doble registro de auditoría:

**A) Tabla `payment_events`** (idempotencia y traza del proveedor):
- `id`, `provider`, `provider_event_id` (UNIQUE), `event_type`,
  `order_id`, `payload` (JSON), `received_at`, `processed_at`.
- Inserción con `ON CONFLICT (provider, provider_event_id) DO NOTHING`
  → entrega duplicada de Stripe es no-op.

**B) Tabla `order_events`** (traza por orden):
- `payment_initiated` — al crear Checkout Session.
- `payment_succeeded` — al marcar orden como pagada.
- `payment_failed` — al fallo o expiración.
- Incluye `provider`, `intent_id`, `event_id`.

El panel `/cms/pagos` muestra los últimos 50 eventos con timestamp,
tipo, orden y estado de procesado.

## 12. Confirmación: cero uso de `SUPABASE_SERVICE_ROLE_KEY` en viajero

✅ **Confirmado.**

Auditoría manual de imports:

- `cart.functions.ts` → `requireSupabaseAuth` (RLS como usuario).
- `payments.functions.ts` (`startPayment`) → `requireSupabaseAuth`.
- `marketplace-reads.functions.ts` → cliente publishable (anon).
- `admin.functions.ts` → `requireSupabaseAuth` + `has_role` RPC.
- `webhook.ts` → carga `supabaseAdmin` SÓLO dentro del handler y SÓLO
  después de verificar la firma HMAC del proveedor. El viajero nunca
  invoca este endpoint directamente; lo llama Stripe server-to-server.

`SUPABASE_SERVICE_ROLE_KEY` se usa exclusivamente en:
1. Webhook firmado (caller verificado = Stripe).
2. RPCs `order_mark_paid` / `order_mark_payment_failed` (SECURITY DEFINER,
   grant a `service_role` only).

Ningún flujo iniciado por el viajero (carrito, confirmación, inicio de
pago) escala privilegios.
