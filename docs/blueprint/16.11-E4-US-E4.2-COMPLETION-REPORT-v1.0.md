## 5. Reutilización estricta

- `travel_plans` + `travel_plan_items` (sin cambios de esquema).
- `auth.users` + triggers existentes (`handle_new_user`,
  `handle_new_traveler_role`) — desactivados vía
  `SET LOCAL session_replication_role='replica'` sólo durante la
  creación del usuario demo para evitar la colisión conocida entre
  ambos triggers y el índice único parcial de `user_roles`
  (`user_roles_user_role_global_uniq`, `WHERE scope IS NULL`), que no
  está cubierto por la cláusula `ON CONFLICT (user_id, role,
  scope_type, scope_id)` de ambos triggers. Se replicaron manualmente
  las 3 inserciones (`profiles`, `traveler_profiles`, `user_roles`).
- Cero engines nuevos, cero rutas nuevas, cero server fns nuevas, cero
  cambios de RLS, cero cambios de esquema.

## 6. Nota técnica: hash de contraseña

Se usó un hash bcrypt precalculado en lugar de `crypt(..., gen_salt())`
porque la extensión `pgcrypto` no está instalada en este proyecto y su
instalación queda fuera del alcance de una historia de producto.
