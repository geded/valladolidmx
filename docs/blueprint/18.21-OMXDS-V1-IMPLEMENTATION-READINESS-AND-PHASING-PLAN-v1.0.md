## 1. Decisión propuesta

Valladolid.mx adopta un plan de implementación progresivo, gobernado y reversible para OMXDS Visual V1. El trabajo futuro se divide en fases acotadas con inventario previo, dependencias explícitas, criterios de entrada y salida, feature flags, pruebas, evidencia, rollback y autorización Founder independiente.

La aprobación de este plan no autoriza código, schema, rutas, datos, Storage, flags, despliegues ni cambios en producción. Sólo fija el orden y las condiciones bajo las cuales podrán solicitarse autorizaciones posteriores.

No se construirá una nueva landing de Zazil Tunich, un segundo Builder ni componentes privados por empresa. V1 evolucionará `Surface Kit + Experience Builder + renderizadores existentes` y validará Empresa Premium sobre la ruta, entidad y composición actuales.

## 2. Behavioral Change Statement

El equipo pasará de intervenir superficies completas mediante encargos amplios a ejecutar incrementos pequeños que conocen de antemano sus consumidores, riesgos, pruebas y forma de reversión.

El Founder pasará de aprobar “construir V1” como una caja negra a autorizar una fase concreta con alcance, evidencia esperada y condición de parada inequívocos.

El visitante recibirá mejoras graduales sin rutas duplicadas, datos inventados, cambios abruptos de navegación ni pérdida de continuidad de Guardar, Agregar a mi viaje, Alux o Concierge.

## 3. Objetivos del plan

1. Confirmar el inventario real antes de editar código.
2. Resolver primero las dependencias transversales que bloquean múltiples gates.
3. Implementar contratos compartidos antes de migrar superficies.
4. Separar apariencia, verdad de datos, capacidades editoriales y comercio.
5. Validar G1–G3 antes de habilitar el piloto Premium.
6. Usar Zazil Tunich como prueba de sistema, no como excepción arquitectónica.
7. Publicar sólo con G4, autorización separada, observabilidad y rollback probado.
8. Escalar únicamente después de cerrar el aprendizaje del piloto.

## 4. Límites vinculantes

Este plan no permite:

- crear rutas, funciones, componentes, bloques, fields, tokens, iconos, CTA o fuentes no aprobados;
- modificar schema, RLS, APIs, permisos, Storage o analítica sin alcance autorizado;
- duplicar entidad o URL por Sol/Luna, locale, Estándar/Premium o piloto;
- escribir rating, premios, precios, horarios, geografía o disponibilidad como copy libre;
- activar checkout, comisión, ranking pagado o permiso comercial;
- resolver C1–C10 por declaración documental;
- publicar Zazil Tunich Premium;
- mezclar una fase autorizada con mejoras oportunistas;
- abrir trabajo de Governance Automation, GitHub, ramas, commits o PR dentro de este expediente sin una instrucción separada.

Una necesidad fuera de contrato se registra como propuesta de evolución y no entra silenciosamente en la fase activa.

## 6. Inventario obligatorio previo a código

La primera fase deberá producir un inventario verificable, no estimaciones de memoria. Para cada consumidor real registrará:

| Campo | Contenido obligatorio |
|---|---|

| Rol | renderer, contrato, fuente, editor, acción, SEO, analytics o infraestructura |
| Consumidores | rutas y superficies que lo usan actualmente |
| Propietario de verdad | editorial, catálogo, reputación, operación, comercio o sistema |

| Estado | reusable, extendible, reemplazable, legacy o bloqueado |
| Riesgo | P0–P3 y efecto sobre G1–G6 |
| Prueba existente | unit, integration, E2E, visual, a11y, crawl o ninguna |
| Cambio previsto | ninguno, extensión contractual, migración o retiro posterior |

## 7. Estrategia de flags

Los nombres exactos se obtendrán del inventario y requerirán autorización; este documento no inventa flags productivos. La estrategia mínima separa:

| Capa | Propósito | Estado inicial |
|---|---|---|
| contratos V1 | habilitar nuevos contratos compartidos en entornos internos | OFF |
| tema/tokens V1 | probar Sol/Luna y componentes sin migrar superficies | OFF |
| Builder V1 | habilitar capacidades por rol y entorno | OFF |
| Premium business | permitir elegibilidad, nunca autoactivar | OFF |
| piloto Zazil Tunich | limitar audiencia y superficie exacta | OFF |
| publicación | separar preview/staging de exposición pública | OFF |

Reglas:

- default seguro `OFF`;
- evaluación server-safe cuando afecte SSR/SEO;
- sin bifurcar URL, entidad ni fuente de verdad;
- apagado conserva Estándar y composición publicada vigente;
- flags de piloto no conceden permisos comerciales;
- eliminación del flag sólo después de G6 y autorización de simplificación.

## 9. Fase R0 · Inventory & Impact Map

### Alcance

- inventario del apartado 6;
- mapa `contrato → consumidor → dato → prueba → gate`;
- diff de capacidad actual contra `18.17–18.20`;
- identificación de cambios sin schema y cambios que sí requerirían propuesta separada;
- baseline reproducible de rutas/superficies representativas;
- propuesta de nombres de flags conforme a convenciones reales;
- matriz de riesgos y orden refinado de I1–S1.

### No incluye

Código, refactor, instalación, migración, seed, flag real, cambio visual, reparación ni publicación.

### Salida

- Implementation Inventory;
- Dependency & Impact Map;
- Gap Matrix;
- Test Baseline Index;
- Flag Proposal;
- Phase Authorization Pack para R1/I1.

### Exit gate

G0 `PASS`: alcance trazado, consumidores confirmados, cero rutas/funciones inventadas y cada gap asignado a una fase o propuesta separada.

## 10. Fase R1 · Critical Readiness Decisions

R1 convierte hallazgos críticos en decisiones aprobables antes de construir fundamentos. Por defecto es documental; una remediación técnica sólo puede ejecutarse mediante autorización propia.

Prioridades:

1. C1: separar SI-04, MI-01, MI-02 y MA-01 con riesgo, propietario y cierre exigido.
2. C8: aprobar política transversal `America/Merida`, límites UTC y helper mínimo.
3. C9: cerrar matriz de contraste de tokens Sol/Luna y estados.
4. C10: aprobar registry de badges, semántica y nombre accesible.
5. Confirmar política de evidencia y ruta canónica sin inventarla.
6. Confirmar ownership de fuentes gobernadas y permisos de publicación.

R1 no necesita esperar C3, C4, C6 o C7 cuando éstos no bloqueen el primer componente aislado, pero sí debe dejarlos asignados antes de G2/G4/G5.

## 11. Fase I1 · Shared Visual Foundations

### Alcance candidato

- tokens semánticos Sol/Luna ya aprobados;
- persistencia y precedencia del tema sin cambiar arquitectura;
- landmarks y skip-link compartidos;
- registry de iconos/badges y Alux oficial;
- primitives de media, foco, movimiento reducido, estados y skeleton;
- harness de regresión visual/accesibilidad para G1.

### Dependencias

C5, C9 y C10 con decisión/evidencia aplicable; inventario R0 aprobado.

### Salida

Un conjunto mínimo de primitives compartidos probado de forma aislada. No migra todavía la landing completa ni habilita Premium.

### Rollback

Flag OFF y renderer/tema vigentes intactos; ninguna migración de datos destructiva.

## 12. Fase I2 · Card Contracts & Primitives

### Alcance candidato

- anatomía compartida y contratos tipados/validados;
- implementación incremental de las siete familias de `18.18`;
- estados loading, vacío, error, offline, sin medio y negocio;
- Guardar, Agregar al viaje y CTA como acciones independientes;
- bindings de confianza sólo desde fuentes gobernadas;
- eventos analíticos contractuales sin PII;
- variantes Sol/Luna y responsive.

### Orden interno propuesto

1. una tarjeta representativa de bajo riesgo para validar el contrato;
2. `BusinessCard`, por ser dependencia del piloto;
3. familias restantes en lotes autorizados, no las siete de golpe.

### Exit gate

G1 por componente: PASS sin P0/P1, con evidencia de contrato, estados, teclado, lector, touch, analytics y regresión visual.

## 13. Fase I3 · Surface Contracts

### Alcance candidato

- regiones y composición de las ocho superficies;
- renderer compartido y variantes Estándar/Premium;
- breadcrumbs, metadata, canonical, OG y schema por entidad;
- información práctica, confianza y colecciones semánticas;
- responsive, rendimiento, offline/error y recuperación;
- degradación Premium→Estándar sobre misma entidad y URL.

### Dependencias

- G1 de los componentes consumidos;
- C7 antes del release cuando existan rutas legacy;
- C8 para fechas;
- C3 para mapas interactivos en Custom Domain;
- C4/C6 para medios Premium o degradación explícita.

### Exit gate

G2 por superficie en preview/staging. Ninguna superficie obtiene PASS con canonical falso, fuente simulada, ruta nueva o datos gobernados en copy.

## 14. Fase I4 · Editorial Builder Enablement

### Alcance candidato

- catálogo/allowlist de bloques y campos de `18.19`;
- edición, orden, visibilidad y variantes permitidas;
- bindings de sólo lectura para datos gobernados;
- RBAC y aislamiento por entidad;
- workflow, preview, validación, versionado, concurrencia y auditoría;
- rollback de composición con revalidación de fuentes vigentes;
- IA sólo como propuesta editorial revisable.

### Estrategia

Implementar primero un slice vertical mínimo: un bloque editorial, una fuente gobernada de sólo lectura, preview y flujo completo Draft→Published en entorno no público. Ampliar sólo cuando ese slice pase G3.

### Exit gate

G3 PASS con pruebas positivas y negativas de permisos, publicación, conflicto, auditoría y rollback. Preview nunca equivale a publicación.

## 15. Fase P0 · Zazil Tunich Data & Media Readiness

P0 prepara la verdad del piloto antes de vestirla de Premium.

### Alcance

- verificar entidad, territorio y composición `biz-zazil-tunich`;
- inventariar y autorizar contenido editorial existente;
- obtener/validar cover, hero, galería, OG y productos con derechos, créditos, alt y focal point;
- verificar premios, reconocimientos, reputación, ubicación, horarios y demás datos operativos;
- confirmar CTA comercial permitido y estados sin disponibilidad;
- validar mapas y fallback;
- probar pipeline y derivados elegibles;
- decidir degradación cuando un dato o medio no cumpla.

### Condición no negociable

No se usan stock ni placeholders como si fueran evidencia propia. Si faltan medios o datos, se degrada a Estándar o se omite el módulo.

### Dependencias

C1, C3, C4, C6 y la porción de C2 relativa a fuentes/medios.

## 16. Fase P1 · Zazil Tunich Premium Migration

### Alcance exacto

- ruta existente `/oriente-maya/valladolid/cenotes/zazil-tunich`;
- entidad/composición `biz-zazil-tunich`;
- `BusinessSurface` o renderer compartido confirmado por R0;
- contratos `18.17–18.20`;
- quince escenarios ZTA-01–ZTA-15.

### Prohibiciones

- no nueva URL;
- no componente privado para Zazil Tunich;
- no contenido paralelo fuera del Builder/fuentes gobernadas;
- no activación comercial implícita;
- no publicación por completar la migración.

### Exit gate

G4 PASS en preview/staging, cero P0/P1, C1–C10 aplicables cerrados o mitigados mediante decisión expresa y rollback/degradación probados. La primera publicación Premium no admite PASS WITH CONDITIONS.

## 17. Fase L1 · Controlled Publication

L1 requiere autorización separada incluso con G4 PASS.

### Precondiciones

- build/version exacta y paquete de evidencia completo;
- feature flag y audiencia inicial definidos;
- SEO/indexación revisados;
- dashboards y alertas activos;
- smoke y rollback ensayados;
- responsables y ventana operativa confirmados;
- backup/versiones disponibles;
- decisión Founder de publicar.

### Observación inicial

Medir errores, Core Web Vitals, navegación, confianza, Guardar, Agregar al viaje, CTA, degradaciones y eventos sin PII. El periodo exacto se fijará antes de L1 según tráfico suficiente; no se inventa una duración sin baseline.

### Stop conditions

Fuga o PII, ruta/canonical roto, dato crítico falso, publicación no autorizada, accesibilidad esencial perdida, CTA principal roto, duplicación de acciones, composición corrupta o regresión severa definida por runbook.

### Salida

G5 PASS o decisión de mantener, pausar, degradar o revertir. No se escala por ausencia de quejas.

## 21. Rollback por capas

1. **Componentes:** flag OFF; renderer anterior permanece disponible.
2. **Superficies:** degradación a Estándar sobre la misma URL/entidad.
3. **Builder:** restaurar composición versionada y revalidar fuentes actuales.
4. **Datos/medios:** retirar binding/asset inválido sin borrar historial ni revivir datos vencidos.
5. **Piloto:** desactivar sólo audiencia/superficie piloto.
6. **Release:** revertir build o flag conforme al runbook aprobado.

No se acepta rollback que dependa de una migración destructiva, pierda auditoría o reintroduzca datos falsos.

10. plan de rollback y stop conditions;
11. definición de terminado;
12. texto exacto de autorización Founder.

La autorización sólo cubre ese paquete. Un archivo cercano no es una invitación abierta.

## 23. Definition of Done por fase

Una fase termina cuando:

- el alcance autorizado está completo y nada adicional se coló;
- criterios aplicables tienen evidencia reproducible;
- no quedan P0/P1 abiertos en su gate;
- inventario y dependencias permanecen actualizados;
- flags, observabilidad y rollback fueron probados cuando aplican;
- deuda P2/P3 está asignada con vencimiento;
- Closure Report registra versión, resultados y desviaciones;
- Founder aprueba el cierre cuando el gate lo exige.

“Se ve bien” y “compila” no son Definition of Done; son apenas dos testigos, no el jurado.
