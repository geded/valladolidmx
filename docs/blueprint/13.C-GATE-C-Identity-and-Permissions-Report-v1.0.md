## 1. Resumen ejecutivo

El Bloque C deja operativo el sistema de identidad de Valladolid.mx:

- Autenticación email/contraseña + Google OAuth gestionada por Lovable Cloud (HIBP activo, anónimos deshabilitados, confirmación de correo).
- Modelo de identidad completo: `profiles`, `user_roles`, `traveler_profiles`, `business_users`, `concierge_profiles`, `admin_profiles`, `invitations`, `permissions_audit_log`.
- Seis roles oficiales operativos: `traveler`, `business_owner`, `concierge`, `editor`, `admin`, `super_admin`.
- Funciones de autorización SECURITY DEFINER alineadas al Blueprint 11.2: `has_role`, `is_admin`, `is_editor_or_admin`, `is_concierge`, `is_concierge_assigned`, `has_business_access`.
- RLS endurecida en todas las tablas del Bloque B con políticas de escritura/edición/borrado por rol y por organización.
- Trigger `on_auth_user_created` que crea perfil + `traveler_profile` + rol `traveler` automáticamente.
- Auditoría de cambios de roles vía `audit_user_roles_changes` → `permissions_audit_log`.
- Storage cableado a roles: editorial (admins/editores), companies/products (dueños), documents (solo admins), temporary (carpeta por usuario).
- UI: `/auth` con sign-in/sign-up/forgot, `/reset-password`, `UserMenu` reactivo a sesión, `AuthProvider` global con `onAuthStateChange`.

## 6. Políticas RLS

- Lectura pública anónima: ya establecida en Bloque B (sobre contenido publicado y no eliminado).
- Escritura/edición/borrado: añadidas en Bloque C, expresadas mediante `is_editor_or_admin`, `is_admin`, `has_business_access`, `auth.uid()=user_id` según corresponde.
- Identidad: cada tabla de perfil expone únicamente el registro propio (SELECT/UPDATE) salvo admins.
- Storage: políticas por `bucket_id`; en `companies`/`products` la pertenencia se infiere del primer segmento del path (`{business_id}/...`); en `temporary` del `{auth.uid}/...`.

## 7. Wiring del Bloque B

- `reviews.author_user_id` → FK a `auth.users(id) ON DELETE SET NULL`.
- `created_by` / `updated_by` / `deleted_by` permanecen como UUID sin FK estricto (intencional: preservar trazabilidad si el usuario se elimina); se llenarán desde la capa de aplicación bajo `requireSupabaseAuth`.

## 10. Riesgos detectados

1. **Warnings 0029 del linter** sobre funciones SECURITY DEFINER ejecutables por `authenticated`. Es **necesario** mantener `GRANT EXECUTE TO authenticated` porque las RLS las invocan en el contexto del usuario. Se acepta como falso positivo documentado (mismo patrón que recomienda la guía oficial de `user_roles`).
2. **`is_concierge_assigned` es stub**: la tabla de asignaciones (`concierge_requests` / `assignments`) llega en bloques posteriores; hoy valida solo rol + parámetro no nulo. Las RLS dependientes deberán reescribirse cuando aparezca.
3. **`created_by` sin FK estricto** en tablas del Bloque B (decisión: conservar trazabilidad post-delete). Si se requiere integridad referencial, añadir FK ON DELETE SET NULL en una migración futura.
4. **OAuth Google managed**: las credenciales son las gestionadas por Lovable; migrar a credenciales propias antes de producción real si se quiere control de branding.
5. **Email transaccional (recuperación/confirmación)**: usa plantillas por defecto de Lovable Cloud; personalización queda fuera del alcance del Gate C (puede tratarse como tarea independiente).
6. **Trigger `on_auth_user_created`**: si falla, bloquea el signup completo. Se confía en `ON CONFLICT DO NOTHING` para idempotencia.
