# US-R1 · Modelo Unificado — Completion Report

**Estado:** ✅ Entregado

## 1 · Capacidad incorporada

El Experience Builder pasa a tener un **modelo de página unificado** sobre `page_compositions`:

- Columna `kind` (`eb_page_kind`) canónica y obligatoria.
- Columna `is_template` boolean para plantillas reutilizables.
- Columna `template_of_kind` para restringir plantillas a un kind.
- Estado ampliado a `draft | published | archived` (CHECK constraint).
- Backfill 100 %: 5 home + 4 landing existentes migradas sin pérdida.
- Nuevo enum values: `home`, `marketplace`, `experience`, `hotel`, `restaurant`, `route`, `alux`, `trip_builder`, `custom` (aditivos, idempotentes).
- **Page Kind Registry declarativo** (`src/lib/experience-builder/page-kind-registry.ts`) con 21 kinds, patrón de slug, singleton, roles requeridos y ruta pública — fuente única de verdad para US-R2..R10.

## 2 · Flujo de usuario completo

Ningún flujo cambia todavía. El Studio actual sigue operando idéntico. La capacidad queda **lista para consumir** por US-R2 (panel multi-kind), sin exponer UI nueva en esta historia — respeta Product Vision Rule (no funcionalidad sin consumidor).

## 3 · Auditoría de no-regresión

- Migraciones aditivas: `ADD COLUMN IF NOT EXISTS`, `ADD VALUE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, CHECK envuelto en DO-block.
- Cero cambios en `page_type` (queda como legacy, sigue leyéndose).
- Cero cambios en RLS, GRANTs, políticas, triggers, RPCs ni server functions.
- Cero cambios en Studio, renderer, Block Registry, Layout Engine, Discovery Layer, PWA, IAC, UNC, BEA.
- Typecheck (`tsgo`) verde.
- Warnings del linter Supabase son **preexistentes** (206 issues, mismos que antes de la migración).

## 4 · Cero experiencia paralela

- No hay ruta nueva.
- No hay editor nuevo.
- No hay componente de UI nuevo.
- El Registry es un archivo declarativo puro (data + tipos), no un engine.

## 5 · Mejora vs historia anterior

El equipo editorial ahora puede razonar sobre **qué tipo de página** es cada composición desde el propio dato, no desde texto libre (`page_type`). Es el prerrequisito para: panel de páginas agrupadas por kind (US-R2), diálogo "Nueva página" con plantilla (US-R3/R4), filtro de biblioteca por `allowedKinds` (US-R5), permisos por kind (US-R7) y adopción por superficie (US-R10).

## 7 · Rollback documentado

```sql
-- Revertir US-R1 (no destructivo):
ALTER TABLE public.page_compositions DROP CONSTRAINT IF EXISTS page_compositions_status_chk;
DROP INDEX IF EXISTS public.idx_page_compositions_kind;
DROP INDEX IF EXISTS public.idx_page_compositions_is_template;
DROP INDEX IF EXISTS public.idx_page_compositions_status;
ALTER TABLE public.page_compositions
  DROP COLUMN IF EXISTS template_of_kind,
  DROP COLUMN IF EXISTS is_template,
  DROP COLUMN IF EXISTS kind;
-- Los enum values añadidos permanecen (drop de enum values requiere recrear el tipo; no procede).
-- Borrar archivo src/lib/experience-builder/page-kind-registry.ts.
```

El campo `page_type` intacto garantiza que el rollback no pierde información.
