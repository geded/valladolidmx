## Reglas transversales

1. Toda escritura crítica viaja por `createServerFn` + `requireSupabaseAuth` y una RPC `SECURITY DEFINER` que valida el rol con `public.has_role`.
2. Ninguna escritura desde cliente directo a `traveler_profiles.signals`, `concierge_proposals.internal_*` ni a tablas de auditoría.
3. `anon` sólo lee superficies públicas explícitamente expuestas (composiciones publicadas, catálogo, regiones, destinos, FAQ).
4. Asignación o revocación de rol → registro automático en `permissions_audit_log`.
5. Transición de estado del flujo Concierge → escritura en `concierge_case_timeline` con `actor_role` y emisión de evento BEA.

## Destinos de redirección por rol (Single Login `/auth`)

| Rol presente (prioridad) | Destino    |
| ------------------------ | ---------- |
| `super_admin` o `admin`  | `/admin`   |
| `business_owner`         | `/empresa` |
| `concierge_lead` o `concierge` | `/concierge` |
| `editor`                 | `/cms`     |
| `traveler` (o sin rol)   | `/mi-viaje` |

Un usuario con múltiples roles cae al primer destino aplicable en este orden. Si llegó con `?redirect=` válido y compatible con su rol, se respeta.
