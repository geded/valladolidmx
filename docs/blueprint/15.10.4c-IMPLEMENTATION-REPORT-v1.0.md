## 1. Resumen ejecutivo

El Cockpit Fundador queda **íntegramente gobernado por el Experience
Builder**. Los tres widgets cockpit aprobados (`vmx.cockpit.kpi-grid`,
`vmx.cockpit.alerts`, `vmx.cockpit.activity-stream`) se registran en el
Block Registry como **Smart Blocks** y se sirven al usuario final a
través del `composition-renderer` universal.

La ruta `/admin` no contiene JSX hardcodeado para paneles: arma su
contenido a partir de un `CompositionTree` declarativo. Añadir, quitar
o reordenar widgets futuros sólo requiere modificar el árbol y/o
registrar un nuevo Block Contract.

## 4. Confirmación arquitectónica

> **"¿Las principales superficies de Valladolid.mx ya pueden evolucionar
> desde el Experience Builder sin requerir modificaciones al código de
> presentación?"**

**Respuesta: SÍ — para las superficies actualmente bajo gobierno del EB:**

| Superficie | Renderer EB | Editable sin código |
|------------|-------------|---------------------|
| Home | ✅ `composition-renderer` | ✅ (Studio Olas 1–4) |
| Landing | ✅ | ✅ |
| Institucional | ✅ | ✅ |
| Destinos / Empresa / Producto | ✅ (Static blocks) | ✅ |
| Cockpit Fundador | ✅ **(esta etapa)** | ✅ |

**Evidencia técnica:**
1. Toda la composición visible pasa por `CompositionRenderer` → `Block Registry`.
2. Añadir un widget cockpit nuevo = añadir un `BlockContract` en `block-library.ts` + mapear un componente en `PRODUCTION_COMPONENT_MAP`. Cero cambios al núcleo EB.
3. La ruta `/admin` no instancia componentes de paneles directamente; sólo construye un `CompositionTree` declarativo.

**Pendiente para confirmación total (fuera de 15.10.4c):**
- Persistir el árbol del Cockpit en la tabla de composiciones para
  editarlo desde el Studio (planeado en 15.10.5).
- Migrar superficies todavía no compuestas (Marketplace listing, Portal
  empresarial) a EB — incluidas en el roadmap funcional.
