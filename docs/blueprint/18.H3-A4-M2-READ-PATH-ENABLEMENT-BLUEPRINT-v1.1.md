## 1. Behavioral Change Statement (recordatorio)

El visitante debe percibir **la misma imagen** (mismo encuadre, mismo canonical, mismo alt, misma posición) pero con menos peso y decodificación más rápida. El sistema debe poder revertir esa mejora en segundos sin afectar SEO, sin URLs rotas y sin cambiar el HTML indexado. Un evaluador puede leer que la decisión "hubiera sido servir la variante X" sin que el visitante reciba jamás esa variante hasta que M2.1 se autorice explícitamente.

## 2. Objetivo de negocio

Demostrar en un canary controlado y auditado que el pipeline reduce bytes y mejora LCP en imágenes turísticas críticas, sin regresión SEO, sin rupturas de contrato público y con rollback verificable. M2 no busca cobertura; busca **confianza cuantificable** para justificar cobertura futura.

## 3. Hipótesis medible (H2.1)

> Sobre un conjunto elegible de assets pilot, servir variantes derivadas reduce el peso transferido por impresión en ≥55% (p50), mejora LCP ≥15% (p75) sobre las mismas rutas y en el mismo tipo de dispositivo, sin incrementar la tasa de errores de imagen ni afectar señales SEO (canonical, sitemap, JSON-LD, OG).

La hipótesis se declara **sólo sobre assets elegibles** (§4) para no contaminar la evidencia con los assets sin checksum.

## 5. Discrepancia de los 24 checksums (corrección Founder #2)

**Hallazgo:** El Migration Validation Report v1.0 (§4) afirmó que el backfill dejó "24 legacy en `-`". La consulta real muestra `0` filas con `source_checksum='-'` y `24` filas con `source_checksum IS NULL`. El reporte anterior fue **incorrecto en ese punto**.

**Diagnóstico:** La columna `source_checksum` se añadió con default `NULL` y la migración §10 PASOS 1–3 no incluyó un `UPDATE ... SET source_checksum = '-' WHERE source_checksum IS NULL`. Sólo el default aplica a filas nuevas. Adicionalmente, `media_assets.original_checksum` es NULL en los 3 assets legacy, por lo que aunque se hubiera intentado propagar desde ahí no habría datos. El `variant_key` sigue siendo único y no-null porque su expresión generada usa `coalesce(source_checksum, '-')`; la unicidad se preservó por casualidad determinista, no por integridad del dato subyacente.

**Descartes:**
- No hubo pérdida de datos: el campo `original_checksum` nunca se calculó para esos 3 assets en el pilot v1.0 (se calculaba después del ingest en el asset vertical añadido en v1.1).
- El checksum no estaba únicamente en `metadata` — inspección de la columna `metadata` de esos 3 assets no contiene `sha256`/`checksum`.
- La migración leyó la columna correcta; simplemente no había fuente que propagar.

**Corrección puntual propuesta (NO ejecutada):**

```sql
-- 5.1 Recalcular original_checksum leyendo el binario de storage por asset.
--     Se ejecuta desde un job server-only, un asset a la vez, con verificación
--     de tamaño y mime antes de escribir. Idempotente.
UPDATE public.media_assets
   SET original_checksum = :sha256_hex
 WHERE id = :asset_id AND original_checksum IS NULL;

-- 5.2 Propagar sólo a variantes cuyo variant_key seguiría siendo válido tras
--     el cambio. En la práctica: marcar las variantes actuales como
--     superseded y re-derivar (ver §4.1). NUNCA hacer UPDATE en línea de
--     source_checksum sobre variantes existentes: rompería la identidad
--     reproducible declarada por variant_key.
```

Esta corrección queda documentada, no autorizada. Requiere aprobación separada; no forma parte de M2.1.

## 6. North Star y métricas

- **North Star:** `bytes_saved_per_impression_p50` — mediana de bytes ahorrados por impresión canary vs impresión control **del mismo asset** en la **misma ruta** y **misma clase de dispositivo** durante la misma ventana horaria. Definición operacional en §8.
- **Secundarias:** LCP p75 (canary vs control), `image_error_rate`, `fallback_reason_distribution`, `pipeline_resolution_latency_ms` (server), `signed_url_hit_ratio`, `signed_url_expiry_incidents`.
- **Guardias:** cero incremento en `image_error_rate` vs control (>+0.5pp aborta), cero cambios en canonical/sitemap/JSON-LD (chequeo diff), cero URLs `media-derived` en HTML estático publicado (crawler diff).

## 7. C0 Dark Launch — Shadow Evaluator (corrección Founder #3)

Con `media_pipeline_enabled=false` global el resolver actual retorna legacy inmediatamente y no evalúa variantes. C0 no puede aprender nada útil con esa lógica. Se rediseña como **modo sombra server-side**, independiente del flag público:

### 7.1 Contrato

- Nueva función interna `evaluateMediaSourceShadow(asset, options, ctx)` en `src/lib/media/shadow-evaluator.server.ts` (server-only, no exportada al bundle cliente).
- **No** modifica `resolveMediaSource()`. El render sigue siendo idéntico byte-a-byte al de hoy.
- Se invoca desde el mismo call site que ya llama a `resolveMediaSource()`, después del cálculo real y sólo si:
  - `ctx.env` es `preview` (no `production`), **y**
  - `ctx.request.headers` contiene un token de allowlist interno (`x-vmx-shadow: <secret>`), **o** la petición proviene de un endpoint de canary interno explícito, **y**
  - `asset.id` pertenece a la allowlist declarada (§7.3).
- Ejecuta la selección hipotética (formatos, ancho, `variant_key`, URL firmada que **no** se emite), calcula latencia, y registra un evento `media_shadow_decision` (§8).
- **Jamás** propaga la URL derivada al render, ni al HTML, ni al `srcSet`, ni a headers. El navegador nunca recibe una URL `media-derived` durante C0.

### 7.2 Aislamiento

- Archivo `.server.ts` protegido por import guard (patrón vigente en el proyecto).
- Sin caché compartido con el resolver de producción.
- Sin efectos secundarios en `platform_settings`, `media_asset_variants` ni Storage (sólo lectura).

### 7.3 Allowlist

- Tabla ya existente `platform_settings` con clave `media_pipeline_shadow_allowlist` (array de `asset_id` UUID). Vacía por defecto. Sólo super_admin puede modificarla. Al iniciar M2.1 contendrá exactamente `[642cb15f-0a13-410c-8027-c4ab92034bf5]`.

### 7.4 Qué mide C0

- Firma de decisión (`variant_key` seleccionado, formato-preferido, ancho, `usedPipeline` hipotético).
- Latencia p50/p95 del evaluador.
- Distribución de `fallbackReason` hipotético (§12).
- Detección de asimetría entre lo que serviría el pipeline y lo que sirve legacy hoy (bytes, dimensiones).

No mide LCP ni bytes transferidos reales (§8 explica por qué eso corresponde a RUM en C1+, no a C0).

## 8. Métricas reales del navegador (corrección Founder #4)

### 8.1 Telemetría server-side (C0+)

Evento `media_shadow_decision` / `media_resolve_decision` (JSON):
`{ ts, asset_id, route_pattern, viewport_bucket (xs|sm|md|lg|xl), decision (pipeline|legacy|shadow_only), variant_key?, format?, width?, latency_ms, fallback_reason?, env }`. Sin IP, sin user_id, sin cookies. Almacenado en tabla de telemetría de bajo cardinalidad con retención 30 días.

### 8.2 RUM client-side (C1+ únicamente; no C0)

Componente `<PipelineImage>` (introducido en M2.2) monta un `PerformanceObserver` con:
- `PerformanceObserver({type:'largest-contentful-paint', buffered:true})` para LCP del elemento.
- `PerformanceObserver({type:'resource', buffered:true})` filtrando por la URL renderizada — obtiene `transferSize`, `encodedBodySize`, `decodedBodySize` y `responseEnd - startTime`.
- `onerror` del `<img>` → evento `media_image_error` con `naturalWidth=0` como firma de fallo.

Payload RUM: `{ ts, route_pattern, viewport_bucket, cohort (control|pipeline), asset_id, variant_key?, format, transferSize, decodedBodySize, lcp_ms?, error?:true }`. Sin IP, sin cookies, sin identificador estable de usuario. Muestreo 100% durante canary (bajo volumen), 5% al escalar. Retención 30 días.

### 8.3 Evento de fallback del componente

`<PipelineImage>` emite `media_pipeline_fallback` cuando dispara `onError` sobre la fuente moderna y cae al `<img>` legacy (browser resuelve `<picture>` naturalmente). Payload: `{ ts, asset_id, format_attempted, fallback_reason, transferSize? }`.

### 8.4 Métricas sólo estimables

- Bytes ahorrados **por usuario individual**: no se mide (no hay ID).
- LCP atribuible a la imagen específica en páginas con LCP no-imagen: se descarta la muestra.
- Ancho de banda ahorrado en absoluto: se estima multiplicando `bytes_saved_per_impression_p50` por conteo de impresiones canary del asset.

### 8.5 Cálculo operacional de `bytes_saved_per_impression_p50`

```
Para cada impresión canary (cohorte=pipeline) con transferSize_p reportado
  emparejar con la mediana histórica reciente (últimos 7 días, ventana móvil)
  de transferSize_c del control equivalente:
    mismo asset_id, mismo route_pattern, mismo viewport_bucket
  savings_i = transferSize_c - transferSize_p
bytes_saved_per_impression_p50 = percentile_50(savings_i)
```

Si no hay control equivalente con ≥30 muestras en la ventana, la impresión no aporta a la métrica.

### 8.6 Privacidad

- No se registra IP (el edge la descarta antes de logging).
- No se registran cookies, referer completo (sólo `route_pattern` normalizado), ni user-agent completo (sólo `viewport_bucket`).
- Ningún identificador estable por usuario.
- Cumple con la política de Journey State: sólo eventos de decisión de sistema, no de comportamiento personal.

## 9. Control y baseline (corrección Founder #5)

No se depende de "datos históricos" globales. Se define un control **coetáneo**:

### 9.1 Cohortes

- Asignación **por impresión**, no por usuario, no por sesión. En cada request a una ruta canary elegible con el asset en la allowlist, el servidor calcula `cohort = (hash(asset_id + route_pattern + viewport_bucket + minute_bucket) mod 100) < N ? 'pipeline' : 'control'`, donde `N` inicia en 10 y escala.
- Anónima, estable dentro del `minute_bucket` (evita flapping en la misma sesión de usuario que refresca), sin cookies, sin cross-request identity.
- Crawlers (UA conocidos: Googlebot, Bingbot, etc.) → **siempre `control`** para blindar SEO. Se registra `cohort_reason='crawler'`.

### 9.2 Protección SEO

- El HTML servido a cualquier cohorte usa el **mismo `canonical`, mismo `<img src>` legacy en fallback, mismo alt, mismo JSON-LD**. La única diferencia entre cohortes es la presencia de `<source>` moderno dentro de `<picture>`. Los crawlers no ejecutan la selección de `<picture>` de forma que afecte a la URL indexada — de todos modos se les fuerza cohorte control.
- Diff automatizado por microfase: `curl` a la ruta con UA Googlebot vs UA navegador → HTML byte-diff debe ser cero salvo la línea del `<source>` cuando cohorte=pipeline.

### 9.3 Comparación

Consulta agregadora en warehouse interno cada hora: `savings_i` (§8.5) segmentado por cohorte y guardias (§6). Publica reporte `M2 · Read Path · Canary Digest`.

## 10. Rollback automático (corrección Founder #6)

### 10.1 Estructura de `platform_settings` (existente)

Tabla `public.platform_settings` con columnas `(key text primary key, value jsonb, updated_at, updated_by, ...)`. Claves involucradas en M2:

| key | value (jsonb) | Rol |
|---|---|---|
| `media_pipeline_enabled` | `{"enabled": false}` | Kill switch global. |
| `media_pipeline_canary` | `{"percent": 0, "asset_ids": [...], "route_patterns": [...], "started_at": ..., "updated_by": ...}` | Configuración del canary. |
| `media_pipeline_shadow_allowlist` | `{"asset_ids": [...]}` | Allowlist del shadow evaluator (§7). |
| `media_pipeline_rollback_state` | `{"tripped": false, "reason": null, "tripped_at": null}` | Historial del último rollback. |

### 10.2 Atomicidad

El rollback se materializa como un **único** `UPDATE public.platform_settings SET value = jsonb_set(value, '{enabled}', 'false'), updated_at = now(), updated_by = ... WHERE key = 'media_pipeline_enabled' AND (value->>'enabled')::boolean = true;`. Es atómico a nivel de fila y evaluado bajo `REPEATABLE READ`. Retorna `rowcount`. Si `rowcount=0`, ya estaba OFF y no se hace nada (idempotente).

### 10.3 Evaluador de métricas

- Job server-only `media-pipeline-rollback-evaluator.server.ts`, invocado por cron cada **5 minutos** durante C1+.
- Ventana móvil de 15 min, exige mínimo 200 muestras canary; si no hay volumen suficiente, no evalúa (no oscilación).
- Dispara rollback si **cualquiera**: `image_error_rate_pipeline − image_error_rate_control > 0.5pp` (α=0.05 sobre proporciones), o `lcp_p75_pipeline > lcp_p75_control * 1.10`, o `signed_url_expiry_incidents > 0` en la ventana, o `crawler_html_diff_bytes > 0`.
- Ventana de gracia de 10 min entre disparos: si el estado ya es `tripped=true` no vuelve a escribir.

### 10.4 Permisos

- El evaluador corre como service role restringido (sólo `SELECT` sobre telemetría y `UPDATE` sobre las 4 claves de `platform_settings` listadas).
- Cualquier escritura queda auditada en `content_audit_log` con `actor='system:rollback-evaluator'` y `reason` estructurado.

### 10.5 Comportamiento si el evaluador falla

- Fallo silencioso NO es aceptable. Todo error del evaluador genera `system_alerts` sev=high y **no** modifica flags.
- Si el evaluador está caído >20 min, el sistema pasa a modo defensivo: el resolver server-side deja de servir cohorte pipeline (fail closed) hasta que el evaluador vuelva. Este modo defensivo se implementa vía lectura de `updated_at` de `media_pipeline_rollback_state` (heartbeat).

### 10.6 Durante M2.1

El rollback automático **queda observable pero sin escritura**: el evaluador se despliega en modo `dry-run=true`, emite el evento `rollback_would_trip` y **no** hace `UPDATE`. Se autoriza la escritura sólo a partir de M2.3, con validación explícita del Founder.

## 11. URLs firmadas (corrección Founder #7)

Aplicable al bucket privado `media-derived` en C1+ (en C0 no se emiten al cliente).

### 11.1 Metadatos por URL

Cada URL firmada se representa internamente como `{ url, issuedAt: epoch_ms, expiresAt: epoch_ms, variant_key, asset_id }`. `expiresAt = issuedAt + TTL_SIGN`, con `TTL_SIGN = 3600s` (1 h) — igual que hoy en `studio-media`.

### 11.2 Margen mínimo antes de reutilización

Una entrada de caché sólo se reutiliza si `expiresAt − now() >= SAFETY_MARGIN`, con `SAFETY_MARGIN = 300s` (5 min). Entradas por debajo del margen se descartan y se re-firma.

### 11.3 Caché

- Clave: `variant_key` (contiene checksum + receta; es la identidad reproducible).
- Alcance: **por request-scope** dentro del handler SSR + un LRU en memoria del worker con `max=500`, TTL efectivo `min(TTL_SIGN − SAFETY_MARGIN, 3000s)`.
- **Sin caché compartido entre workers.** Cada instancia serverless firma de forma independiente. Es aceptable: `createSignedUrl` es barata y las variantes son estables.

### 11.4 Comportamiento serverless / restart

- Al reiniciar el worker, el LRU se pierde — comportamiento esperado; el próximo request re-firma. Sin degradación funcional.
- Ninguna entrada puede sobrevivir a su firma porque se comprueba `expiresAt − now() >= SAFETY_MARGIN` antes de servir; el margen actúa como colchón frente a skew de reloj (±60s asumido) y latencia CDN (≤240s).

### 11.5 Impacto de `Cache-Control: 300`

- El proxy `/api/public/studio-media/*` fija `s-maxage=300`. Si `media-derived` adoptara el mismo patrón, una CDN podría almacenar durante 5 min un `302 Location: <signedUrl>`. Con `TTL_SIGN=3600s` y `SAFETY_MARGIN=300s`, la URL firmada emitida tendrá **al menos 55 min de vida** al momento de firmar → sobra margen para la ventana CDN de 5 min. Aun así, para M2 **no se enruta `media-derived` por un proxy con `Cache-Control` largo**: las URLs firmadas viajan directamente en el `srcSet` de `<picture>`, y el navegador cachea el binario descargado (no la URL) bajo sus headers habituales de Storage.

### 11.6 Métrica de seguridad

`signed_url_expiry_incidents` = cuenta de `image_error_rate` cuya `fallback_reason='url_expired_hit'` (§12). Umbral duro: **>0 en ventana de 15 min dispara rollback** (§10.3).

## 12. Matriz de fallos y decisiones (corrección Founder #8)

### 12.1 Tipo formal de `fallbackReason`

```ts
export type FallbackReason =
  | 'pipeline_disabled'          // flag global OFF
  | 'asset_not_ready'            // pipeline_status ≠ 'ready'
  | 'no_variants_for_context'    // filtro usage_context sin resultados
  | 'variant_key_missing'        // integridad rota (nunca debería ocurrir)
  | 'signed_url_error'           // error al firmar en tiempo de resolución
  | 'storage_unreachable'        // NEW — Storage devolvió 5xx / timeout
  | 'url_expired_hit'            // NEW — img.onError con firma ya vencida
  | 'cohort_control'             // el request fue asignado a control
  | 'crawler_forced_control'     // UA identificado como crawler
  | 'shadow_only';               // C0 — nunca se sirvió al cliente
```

### 12.2 Aclaración sobre AVIF/WebP

La selección real de formato la hace el **navegador** al evaluar el elemento `<picture>` según el orden y `type` de los `<source>`. El resolver SSR **no puede** determinar con certeza qué formato consumirá el cliente: sólo puede publicar la lista ordenada AVIF→WebP→JPEG. Consecuencias:

- La telemetría server-side registra `format_preferred = 'avif' | 'webp' | 'jpeg'` como el **primero disponible**, no como el "elegido".
- El `format_effective` real sólo puede reportarse desde RUM (§8.2) leyendo `entry.name` en `PerformanceObserver` y comparándolo con las URLs del `<picture>`.
- Los reportes distinguen `format_preferred` (server) de `format_effective` (client). No confundir.
- Un `Accept: image/avif` en el header del request puede usarse como *hint* server-side para logging, pero **no** se usa para decidir el `srcSet` (romperia el modelo `<picture>`).

### 12.3 Matriz condensada

| Escenario | Server decisión | Cliente | fallback_reason | Telemetría |
|---|---|---|---|---|
| Flag OFF | legacy | `<img src=legacy>` | `pipeline_disabled` | server |
| Asset ready + cohort=pipeline + variantes OK | `<picture>` completo | Navegador elige moderno | — | server + RUM |
| Asset ready + cohort=control | legacy | `<img src=legacy>` | `cohort_control` | server |
| Sign error al resolver | legacy | `<img src=legacy>` | `signed_url_error` | server + alerta |
| Storage 5xx durante sign | legacy | `<img src=legacy>` | `storage_unreachable` | server + alerta |
| URL firmada vencida al llegar al navegador | `<picture>` intentado, img.onError cae a legacy | `<img src=legacy>` (fallback en `<picture>`) | `url_expired_hit` | RUM + rollback trigger |
| Crawler | legacy | idem | `crawler_forced_control` | server |
| C0 shadow | legacy servido | legacy | `shadow_only` (evaluación registrada aparte) | server (evento `media_shadow_decision`) |
