# Iniciativa 3 · Fase 3.2 · Completion Report

**Alcance único:** cerrar el riesgo **R1** identificado en la auditoría 3.1 —
la RPC `public.eb_process_scheduled_publishes()` era ejecutable por cualquier
usuario con rol `authenticated`, lo que habilitaba una publicación masiva no
autorizada.

**Estado:** ✅ Cerrado — cerradura aplicada sin remodelación.

---

## 1 · Cambios aplicados

Migración SQL mínima (`supabase/migrations/*_fase-3-2-r1-*.sql`):

1. `REVOKE EXECUTE ... FROM authenticated` sobre
   `public.eb_process_scheduled_publishes()`.
2. `REVOKE EXECUTE ... FROM anon` (reafirmación defensiva).
3. `GRANT EXECUTE ... TO service_role` (único rol autorizado).
4. Guard interno en la propia función (defensa en profundidad):
   sólo `service_role` (por `session_user` / `current_setting('role')`) o
   usuarios con `has_role(auth.uid(), 'admin' | 'super_admin')` pueden ejecutar.
   Cualquier otro rol recibe `RAISE EXCEPTION` con `ERRCODE = 42501`
   (`insufficient_privilege`).

**Lógica de publicación intacta.** El cuerpo (`FOR ... LOOP`, revisiones,
`content_audit_log`, despublicación de hermanos) es idéntico al original.

---

## 2 · Verificación de permisos

```sql
SELECT
  has_function_privilege('authenticated', 'public.eb_process_scheduled_publishes()', 'EXECUTE'),
  has_function_privilege('anon',          'public.eb_process_scheduled_publishes()', 'EXECUTE'),
  has_function_privilege('service_role',  'public.eb_process_scheduled_publishes()', 'EXECUTE');
```

| Rol             | EXECUTE |
| --------------- | :-----: |
| `anon`          |   ❌    |
| `authenticated` |   ❌    |
| `service_role`  |   ✅    |

ACL final de la función:

```
postgres=X/postgres
service_role=X/postgres
sandbox_exec=X/postgres
```

`authenticated` ya **no aparece** en el ACL.

## 3 · Pruebas

### Prueba negativa — usuario `authenticated` no puede ejecutar

`has_function_privilege('authenticated', ...)` devuelve `false`, por lo que
PostgREST rechaza la invocación antes de llegar al cuerpo con
`permission denied for function eb_process_scheduled_publishes`. Adicionalmente,
aunque el chequeo de GRANT se saltara, el guard interno lanza
`insufficient_privilege: eb_process_scheduled_publishes requires service_role or admin`.

### Prueba positiva — `service_role` sí puede ejecutar

`has_function_privilege('service_role', ...)` devuelve `true`. El endpoint
`src/routes/api/public/hooks/eb-process-scheduled-publish.ts` ya construye
un cliente con `SUPABASE_SERVICE_ROLE_KEY`, así que el flujo operativo de
`pg_cron` sigue funcionando sin cambios.
