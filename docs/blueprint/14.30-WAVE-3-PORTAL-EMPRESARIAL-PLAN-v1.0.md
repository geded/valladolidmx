## 1. Objetivo

Habilitar a las empresas semilladas en la Ola 2 (y a futuras altas) para
que sus dueños y colaboradores administren — desde un Portal Empresarial
autenticado — su perfil público, contactos, ubicaciones, horarios, redes
sociales, galería, productos y promociones, respetando el workflow editorial
aprobado en la Ola 1 y la matriz de permisos aprobada por Gate C
(`business_users` + `has_business_access`).

## 2. Principios rectores (heredados de Ola 1 y Ola 2)

1. **Blueprint inalterado** (Series 11, 12, 13, 14).
2. **Modelo de dominio inalterado** salvo ADRs aprobados.
3. **RLS inalterada**, salvo añadir políticas estrictamente más restrictivas
   o nuevas tablas que sigan el patrón aprobado.
4. **Sin uso de `SUPABASE_SERVICE_ROLE_KEY`** en flujos de usuario; sólo
   admisible en jobs administrativos auditados.
5. **Server functions tipadas** (`createServerFn` + `requireSupabaseAuth`),
   verificación server-side de pertenencia/rol vía
   `has_business_access(_user_id, _business_id, _min_role)`.
6. **Workflow editorial 14.10** vigente: `draft → in_review → approved →
   published → archived`, con `content_audit_log` cubriendo transiciones.
7. **Idempotencia y aditividad** en cualquier migración necesaria.
8. **Reportes de avance por etapa** y **aprobación documental** antes de
   avanzar entre etapas.
9. **Mocks** retenidos como fallback hasta superar QA y Execution Gates.

## 5. Arquitectura técnica

### 5.1 Ubicación de rutas

- `src/routes/_authenticated/portal/route.tsx` — layout con selector de
  empresa y gating client-side adicional sobre `business_users`.
- `src/routes/_authenticated/portal/index.tsx` — landing del portal.
- `src/routes/_authenticated/portal/$businessSlug.*.tsx` — secciones por
  empresa.
- **No** se crea ruta pública nueva; la edición vive 100% bajo
  `_authenticated`.

### 5.2 Server functions

- Nuevo archivo `src/lib/portal/portal-reads.functions.ts` y
  `src/lib/portal/portal-writes.functions.ts`.
- Patrón obligatorio:

  ```ts
  createServerFn({ method: "POST" })
    .middleware([requireSupabaseAuth])
    .inputValidator(/* zod */)
    .handler(async ({ data, context }) => {
      // 1) Resolver business_id
      // 2) Verificar has_business_access(userId, business_id, 'editor'|'manager'|'owner')
      // 3) Operar con context.supabase (RLS aplica como user)
    });
  ```

- Lecturas listadas usan `requireSupabaseAuth` (no cliente publishable),
  porque el Portal es 100% autenticado.
- Transiciones de estado usan **exclusivamente** la RPC
  `transition_content_status`.

### 5.3 Permisos

- Matriz mínima por acción:

  | Acción | Rol mínimo |
  |--------|------------|
  | Ver dashboard de empresa | `viewer` |
  | Editar ficha / contactos / horarios / media | `editor` |
  | Crear productos / promociones | `editor` |
  | Enviar a revisión | `editor` |
  | Invitar usuarios a la empresa | `manager` |
  | Cambiar dueño / dar de baja empresa | `owner` |
  | Acciones cross-empresa | `admin` plataforma (no portal) |

### 5.4 Storage

- Buckets existentes (privados): `logos`, `companies`, `products`,
  `gallery`, `documents`, `temporary`.
- Subidas firmadas vía server function (no client-side service role).
- Política a documentar en Etapa 5 antes de cualquier upload real.

### 5.5 Cliente / UI

- Reutilizar `CmsEntityPage`, `EntityEditor`, `StatusBadge` cuando aplique
  (componentes ya aprobados Ola 1).
- Selector de empresa persistido en `localStorage` con clave dedicada
  (no en cookies, no en URL como query param sensible).

## 6. Cambios de base de datos previstos

**Hipótesis principal:** ninguno estructural. Las tablas
`businesses`, `business_users`, `business_contacts`, `business_locations`,
`business_hours`, `business_social_links`, `business_media`,
`business_category_links`, `products`, `product_media`, `promotions`,
`invitations` y `media_assets` existen y están listas.

Si alguna etapa detecta un faltante real (p.ej. una columna `is_primary`

introducirá ningún cambio estructural sin ADR aprobado.

## 10. Riesgos previstos

| ID | Riesgo | Severidad | Mitigación |
|----|--------|-----------|------------|
| RW3-1 | Subidas de archivos privilegiadas filtran service role al cliente. | Alta | Subida firmada server-side; cliente nunca toca `SERVICE_ROLE`. Gate C audita. |
| RW3-2 | Permisos cross-empresa mal aplicados (usuario A edita empresa B). | Alta | `has_business_access` obligatorio en cada handler; tests dedicados en Etapa 1 y Etapa 7. |
| RW3-3 | Edición directa sin pasar por workflow editorial. | Media | Toda transición vía RPC; auditar en Gate D. |
| RW3-4 | Faltante de columnas detectado tarde (precedente: Ola 2 Etapa 4). | Media | Verificación de columnas como DoR; ADR previo si falta algo. |
| RW3-5 | Divergencia Mock ↔ BD si los mocks se retiran sin QA cruzado. | Media | Etapa 8 decide retiro coordinado. |
| RW3-6 | Subidas de imágenes a buckets equivocados o con MIME no validado. | Media | Validación server-side + límites de tamaño + listas blancas de MIME. |
| RW3-7 | Selector de empresa expone IDs de empresas a las que el usuario no pertenece. | Media | Lista devuelta por server function filtrada por `business_users.user_id = auth.uid()`. |
