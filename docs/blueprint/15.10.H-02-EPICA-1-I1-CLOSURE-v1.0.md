## 2. Arquitectura implementada

```text
src/lib/context-engine/

                          # RouteContextDeclaration, InheritanceRule,
                          # ContextEngineEvent(+Meta), listener.
  inheritance-rules.ts    # Whitelist estricta (previousKind → currentKind → slots).
  previous-store.ts       # Persistencia SSR-safe del recorrido previo
                          # (sessionStorage + TTL 5 min).
  resolver.ts             # Función pura resolveContext() sin side effects.
  events.ts               # Bus ligero pub/sub para context_engine.*.
  provider.tsx            # <ContextEngineProvider>, useResolvedContext(),
                          # defineRouteContext() (SSR-safe, useSyncExternalStore).
  index.ts                # Barrel público.

src/routes/lovable/
  context-engine-preview.tsx   # Playground interno, noindex/nofollow.
```

### 2.1 Flujo de resolución

```text
  defineRouteContext({ current, ancestors?, inherit?, kindDefaults?, canonical })
                            │
                            ▼
  <ContextEngineProvider>   ── lee previous (sessionStorage, cliente)
                            │
                            ▼
  resolveContext()          ── whitelist de herencia + fallback kind
                            │
                            ▼
  ResolvedContext           ── emite eventos context_engine.*
                            │
                            ▼
  useResolvedContext()      ── consumidores (breadcrumb, SEO, Alux, …)
```

### 2.2 Decisiones de diseño clave

1. **Canonical intocable:** `ResolvedContext.canonical` = la ruta actual, nunca heredada. La herencia sólo afecta ancestros/related — cero riesgo de contaminar `<link rel="canonical">` o `og:url`.
2. **Herencia por whitelist estricta** (`inheritance-rules.ts`) + opt-in por ruta (`inherit: [...]`). Default = sin herencia.
3. **SSR-safe:** `previous` es `undefined` en servidor; `useSyncExternalStore` con `getServerSnapshot` evita hydration mismatch.
4. **Provider aditivo:** si nadie lo monta, `useResolvedContext()` retorna `null` y los consumidores caen a su comportamiento actual. Retrocompatibilidad total.
5. **Namespace `context-engine`** (no `navigation`): refleja la visión ampliada del Founder — la navegación es sólo uno de los consumidores.
6. **Cero dependencias nuevas.** Sólo React + tipos internos.

## 5. Eventos definidos

Namespace reservado: **`context_engine.*`**.

| Evento | Emitido cuando |
|---|---|
| `context_engine.resolved` | El provider resuelve un contexto (una vez por `canonical`). |
| `context_engine.inherited` | Al menos un slot se hereda del recorrido previo. |
| `context_engine.fallback_kind_default` | Se usan los `kindDefaults` de la ruta (no había ni declaración ni herencia). |
| `context_engine.previous_saved` | Se persiste el contexto actual como próximo `previous`. |
| `context_engine.previous_expired` | (reservado) TTL vencido al leer. |
| `context_engine.declaration_missing` | (reservado) Se solicitó contexto sin declaración de ruta. |

Metadata común: `{ kind?, currentHref?, previousHref?, source?, inheritedSlots?, at }`.

## 7. Riesgos encontrados y mitigaciones

| # | Riesgo | Mitigación aplicada en I1 |
|---|---|---|
| R1 | Herencia mal aplicada contamina UX | Whitelist estricta (`inheritance-rules.ts`) + opt-in por ruta. Default = sin herencia. |
| R2 | JSON-LD BreadcrumbList inconsistente con canonical | Canonical se mantiene siempre self-referencial en el `ResolvedContext`; herencia sólo afecta ancestros. |
| R3 | Hydration mismatch por leer `sessionStorage` en render | `useSyncExternalStore` con `getServerSnapshot` explícito devuelve `undefined` en SSR; sólo hidrata en cliente. |
| R4 | TTL o storage lleno rompen el flujo | `try/catch` en todas las lecturas/escrituras; fallback silencioso a `undefined`. |
| R5 | Filtraciones cross-app en `sessionStorage` | Key namespaced `vll:ctx-engine:previous:v1`. |
| R6 | Provider olvidado en producción → `null` visible al usuario | Contrato explícito: consumidores DEBEN tratar `null` como "sin contexto" y caer al comportamiento actual. En I1 nadie es consumidor todavía. |
| R7 | Multi-pestaña comparte `sessionStorage` de la misma pestaña sólo | Aceptado por diseño; documentado. |

## 8. Limitaciones conocidas

- **Sin persistencia entre recargas** más allá del TTL de `sessionStorage` (5 min). Aceptado por diseño.
- **`previous` es único** — no hay stack de recorrido. Suficiente para la herencia inmediata; ampliar sólo si un consumidor futuro lo requiere.
- **`kindDefaults` se declara por ruta** en I1. En I2/I3 se puede extender `page-kind-registry` con `defaultAncestors` para centralizar defaults por kind.
- **JSON-LD BreadcrumbList** no se emite todavía — es trabajo de I7 en el plan aprobado.
- **`context_engine.previous_expired` y `context_engine.declaration_missing`** están reservados pero no emitidos en I1 (no hay consumidor real que los dispare).

## 10. Plan recomendado para I2

**I2 — Integración no-op con `PublicShell` + `BreadcrumbTerritorial`.**

Objetivo: preparar la infraestructura para consumidores reales sin activar ningún cambio observable todavía.

Alcance propuesto:

1. `PublicShell` acepta prop opcional `contextDeclaration?: RouteContextDeclaration`. Si se provee, monta el `<ContextEngineProvider>` para su subárbol.
2. `BreadcrumbTerritorial` acepta prop opcional `useContextCrumbs?: boolean` (default `false`). Cuando `true` **y** `useResolvedContext()` retorna contexto, deriva las migas desde el contexto en vez de la prop `crumbs`.
3. Ampliar el playground: montar `PublicShell` real con `contextDeclaration` y demostrar que el breadcrumb refleja ancestros + herencia.
4. Cero rutas reales modificadas: ninguna pasa `contextDeclaration` todavía → comportamiento actual intacto.

**Criterios de aceptación I2:**

- Ninguna de las 25+ rutas del inventario cambia render.
- `PublicShell` sin `contextDeclaration` = pixel-perfect igual a hoy.
- Playground demuestra el flujo completo destino → categoría → detalle con herencia visible en el breadcrumb.
- Typecheck y build verdes.

**Riesgo I2:** Muy bajo. Sigue siendo puramente aditivo y opt-in.
