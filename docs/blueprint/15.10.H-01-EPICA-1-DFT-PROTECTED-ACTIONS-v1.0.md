## 1. Objetivo de negocio

### 1.1 Problema que resuelve
Hoy la plataforma expone acciones que **requieren identidad** (guardar en Mi Viaje, marcar favorito, agregar al carrito, solicitar Concierge, dejar reseña, activar notificaciones, contactar empresa) usando **tres patrones distintos** heredados de olas anteriores:

1. **Redirección dura** a `/auth` (ej. `RequestConciergeButton`): el visitante pierde el contexto de dónde estaba y qué quería hacer.
2. **Fallback silencioso a `localStorage`** (ej. `AddToTravelPlanButton` + `guest-queue.ts`): la acción "parece" ejecutarse pero no persiste en servidor hasta un evento posterior no observable.
3. **Bloqueo sin explicación** (ej. algunos favoritos y CTA de Marketplace): el botón no responde o muestra un toast genérico.

Consecuencias de negocio:
- **Pérdida de conversión** por fricción de login sin retorno al contexto.
- **Inconsistencia percibida**: la misma acción se comporta distinto según la pantalla.
- **Datos huérfanos**: la cola de invitado no siempre migra y las acciones ejecutadas "en vacío" no quedan trazadas.

### 1.2 Importancia estratégica
Es la **base transversal** de las Épicas 2 (Mi Viaje flush), 4 (Perfiles) y 6 (Push) — todas dependen de un único punto donde la plataforma decide "requiere sesión → intercepto → reanudo". Sin este sistema, cada épica reinventaría la interceptación y multiplicaría deuda.

### 1.3 Impacto para el usuario
- **Visitante (guest):** puede intentar cualquier acción; el sistema le pide identificarse **en un Sheet contextual** (sin abandonar la página), y al terminar **la acción se ejecuta sola** con confirmación visible.
- **Miembro (autenticado):** ningún cambio de comportamiento — el sistema es transparente para él.
- **Empresario / staff:** hereda automáticamente el mismo patrón para acciones que además requieran rol específico (extensión futura).

### 1.4 Impacto para el sistema
- Elimina 3 patrones divergentes → 1 contrato único (`useProtectedAction` + `SignInPromptSheet`).
- Reduce superficie de bugs (cada patrón actual tiene sus propios edge-cases).
- Habilita observabilidad: cada intento de acción protegida emite un evento estándar (sin PII).
- No requiere cambios de base de datos ni migraciones.

## 2. Alcance

### 2.1 Incluye (v1)
1. **Hook `useProtectedAction`** — API única para envolver cualquier acción que requiera identidad.
2. **`SignInPromptSheet`** — Bottom Sheet reutilizando el Sheet Stack del Workspace (15.10.5b); presenta login/registro con el contexto de la acción pendiente.
3. **`PendingActionRegistry`** — Registro en memoria + `sessionStorage` de la acción interrumpida, con TTL corto (10 min) y limpieza automática.
4. **Resume-on-login** — Ejecución automática de la acción pendiente al detectar `SIGNED_IN` desde `onAuthStateChange`.
5. **Migración de call-sites v1** — tres consumidores actuales pasan a usar el hook:
   - `AddToTravelPlanButton` (traveler)
   - `FavoriteButton` (marketplace)
   - `RequestConciergeButton` (concierge)
6. **Contrato de observabilidad** — evento `protected_action.*` para analytics/debug (sin PII).

### 2.2 No incluye (fuera de v1)
- Migración exhaustiva de todos los CTAs (por lotes en épicas posteriores).
- Cambios visuales al login/registro (usa el `/auth` actual, embebido en Sheet).
- Gate por **rol** (`requiredRole`): contemplado como extensión futura; v1 sólo cubre "requiere sesión".
- Rate limiting o CAPTCHA (queda para SEC-2).
- Deep-link resume tras cerrar el navegador (sólo misma sesión de pestaña).

### 2.3 Queda para futuras épicas
- **Épica 2** consumirá el resume-on-login para el flush de `guest-queue`.
- **Épica 4** añadirá `requiredRole`/`requiredProfileCompletion` sin romper consumidores.
- **Épica 6** usará el mismo Sheet para el opt-in de push cuando requiera sesión previa.

## 3. Arquitectura propuesta

### 3.1 Principio rector
**Convive, no reemplaza.** Todas las capas existentes (Workspace Engine, Sheet Stack, `useAuth`, `guest-queue`, `/auth`) permanecen intactas. Este sistema es una **capa de coordinación** delgada que las orquesta.

### 3.2 Ubicación en el árbol
```
src/lib/protected-actions/
  ├── types.ts                    # Contratos: ProtectedAction, PendingAction, Result
  ├── registry.ts                 # PendingActionRegistry (memoria + sessionStorage)
  ├── use-protected-action.ts     # Hook público consumido por componentes
  └── resume-runner.ts            # Suscriptor a onAuthStateChange que reanuda
src/components/protected-actions/
  └── SignInPromptSheet.tsx       # Contenido del Sheet (form embebido de /auth)
```

Única integración cross-cutting:
- **`src/routes/__root.tsx`**: montar `<ProtectedActionResumeRunner />` una única vez, al mismo nivel que otros providers globales. Componente sin UI, suscrito al `AuthProvider`.

### 3.3 Dependencias reutilizadas (obligatorio)
| Capacidad | Fuente reutilizada | Razón |
|---|---|---|
| Bottom Sheet | `SheetStackProvider` / `useSheetStack` (15.10.5b) | Interaction Consistency Policy |
| Sesión | `useAuth` + `onAuthStateChange` | Single Source of Truth de identidad |
| Login UI | Ruta `/auth` existente (extraer `AuthPanel` reutilizable para el Sheet) | Cero divergencia visual/funcional |
| Toasts | `toast-bus` del Workspace | Consistent confirmations |
| Persistencia guest | `guest-queue.ts` **sin modificar** (Épica 1 sólo lo respeta; Épica 2 lo consolida) | Aislamiento de responsabilidades |

### 3.4 Coexistencia con `guest-queue`
`guest-queue` sigue siendo el **almacén** para acciones "coleccionables" (Mi Viaje). `useProtectedAction` es el **orquestador** de la interacción. Un botón puede tener dos modos:
- **`mode: "gate"`** (default): bloquea la acción hasta que exista sesión, luego la ejecuta contra el servidor. Sin cola.
- **`mode: "queue"`** (opt-in, sólo Mi Viaje v1): si no hay sesión, escribe en `guest-queue` y no abre Sheet; ofrece un CTA discreto "Guardar en tu cuenta" que sí abre el Sheet.

Ambos modos conviven; los componentes eligen. Ningún módulo pierde su comportamiento actual.

## 4. Diseño del flujo

### 4.1 Flujo canónico (mode "gate")
```
Visitante (sin sesión) → click en botón envuelto por useProtectedAction
  ↓
Hook detecta !user
  ↓
Registra PendingAction { id, kind, payload, createdAt, ttl: 10min } en registry
  ↓
Abre SignInPromptSheet vía sheetStack.push({
    title: "Necesitas iniciar sesión",
    description: "para guardar «Cenote Zací» en tu viaje",
    content: <SignInPromptSheet pendingActionId={id} />
  })
  ↓
Usuario elige método (email / Google / magic link) → completa auth
  ↓
onAuthStateChange dispara SIGNED_IN
  ↓
ResumeRunner lee registry, encuentra PendingAction vigente
  ↓
  ├── Cierra el Sheet
  ├── Ejecuta payload.action() (server fn original)
  ├── Elimina entrada del registry
  └── Emite toast "«Cenote Zací» agregado a tu viaje ✓"
```

### 4.2 Escenarios y estados
| # | Escenario | Comportamiento |
|---|---|---|
| E1 | Usuario ya autenticado | Hook ejecuta la acción directamente, sin Sheet. |
| E2 | Visitante completa login | Acción se reanuda; toast de éxito. |
| E3 | Visitante cierra el Sheet sin autenticarse | Se elimina PendingAction. Sin efecto secundario. |
| E4 | Autentica pero la acción falla en server (403/409/red) | Toast de error con reintento manual. PendingAction ya consumida. |
| E5 | Login exitoso pero TTL vencido (>10 min) | ResumeRunner descarta la acción y muestra toast informativo "Vuelve a intentarlo". |
| E6 | Otra pestaña autentica | `storage` event sincroniza registry; la pestaña original reanuda al recibir `SIGNED_IN` propagado por Supabase. |
| E7 | Múltiples acciones pendientes | Registry FIFO; ResumeRunner ejecuta sólo la **última** y descarta el resto (política v1, limitación conocida). |
| E8 | Acción `mode: "queue"` (Mi Viaje) | Se guarda en `guest-queue`, NO abre Sheet. Toast discreto ofrece "Guarda en tu cuenta" con acción secundaria que sí abre el Sheet. |
| E9 | Sesión expira entre gate y ejecución | Server fn responde 401; capa de reintento pide re-login vía el mismo Sheet, reencolando la acción. |
| E10 | SSR/prerender | Hook es no-op en server (window-safe); nunca abre Sheet en render inicial. |

### 4.3 Contrato UX del Sheet
- Snap inicial: `half`.
- Título dinámico según acción (i18n).
- Descripción explícita del **qué se guardará** (nombre/preview del target).
- Botón secundario "Ahora no" → cierra y descarta.
- No bloquea navegación (dismiss por overlay permitido).
- La copy de confirmación (toast) es idéntica a la del flujo autenticado — no hay "camino segundo".

## 5. API interna

### 5.1 Hook público
```
useProtectedAction<TPayload, TResult>(config: {
  kind: string;                          // "travel.add" | "favorite.toggle" | "concierge.request" | ...
  mode?: "gate" | "queue";               // default "gate"
  action: (payload: TPayload) => Promise<TResult>;
  onSuccess?: (result: TResult) => void;
  onError?: (err: unknown) => void;
  gateCopy?: { title?: string; description?: string };
  queueFallback?: (payload: TPayload) => void; // sólo mode "queue"
}): {
  run: (payload: TPayload) => void;
  pending: boolean;
}
```

### 5.2 Servicios internos
- `PendingActionRegistry.push(entry)` / `.take(id)` / `.peek()` / `.clear()`.
- `ResumeRunner` — módulo con `init(supabase, registry, sheetStack, toastBus)`; único suscriptor a `SIGNED_IN` para este propósito.
- `emitProtectedActionEvent(name, meta)` — hoya de observabilidad (no PII); v1 emite a `console.debug` + hook para analytics futuro.

### 5.3 Componentes consumidores (v1)
- `AddToTravelPlanButton` — envuelve su `addItem` server fn (`mode: "queue"` respetando cola actual).
- `FavoriteButton` — envuelve `toggleFavorite` (`mode: "gate"`).
- `RequestConciergeButton` — envuelve `createConciergeCase*` (`mode: "gate"`), reemplazando `navigate("/auth")`.

### 5.4 Eventos emitidos
| Evento | Cuándo |
|---|---|
| `protected_action.gated` | Se abre el Sheet |
| `protected_action.dismissed` | Usuario cierra sin autenticar |
| `protected_action.resumed` | Se ejecuta tras login |
| `protected_action.expired` | TTL vencido |
| `protected_action.failed` | Error de server post-resume |

### 5.5 Reutilización futura
- Cualquier CTA nuevo se envuelve con `useProtectedAction` en 3 líneas.
- Épicas 2/4/6 extienden con `requiredRole`, `requiredProfileFields`, `requiredConsent` sin cambiar consumidores existentes.

### 6.2 Herencia automática
Cualquier componente que en el futuro consuma `useProtectedAction` obtiene: Sheet contextual, resume-on-login, observabilidad, TTL, sincronización cross-tab.

### 6.3 APIs y contratos
- No se modifica ninguna server function existente.
- No se cambia la firma pública de `useAuth`.
- No se cambia el contrato de `guest-queue`.
- No se altera routing.

## 7. Riesgos y mitigaciones
| # | Riesgo | Tipo | Mitigación |
|---|---|---|---|
| R1 | Doble ejecución (resume + click manual) | Técnico | Registry con `id` único + `take()` atómico; `pending=true` durante resume. |
| R2 | Fuga entre usuarios (guest A cambia a user B tras login accidental) | Seguridad | TTL 10 min + limpieza en `SIGNED_OUT` + RLS server-side valida `userId`. |
| R3 | Sheet en SSR causa hydration mismatch | Técnico | Hook 100% cliente (`typeof window`); ResumeRunner sólo post-hidratación. |
| R4 | Payloads sensibles en `sessionStorage` | Seguridad | Storage sólo guarda `{id, kind, ts}`; `payload` vive en memoria del módulo (Map). Cross-tab reconstruye o descarta. |
| R5 | Sheet percibido como intrusivo | UX | Copy transparente, botón "Ahora no", nunca bloqueante. |
| R6 | Migración de call-sites genera regresión visual | Funcional | Migración one-by-one con test comparativo antes/después; commit atómico por componente. |
| R7 | `guest-queue` y registry se pisan | Técnico | Modos `gate` vs `queue` mutuamente excluyentes por call-site; documentado. |
| R8 | Analytics falsos por duplicación | UX/Obs | Cada evento incluye `actionId`; dedup cliente. |
| R9 | OAuth redirige fuera de la pestaña y pierde registry | Técnico | `sessionStorage` sobrevive redirección OAuth (misma pestaña); test manual E2E. |
| R10 | i18n incompleta | UX | Copy centralizada bajo `protected_actions`; fallback `es`. |

## 9. Plan de pruebas

### 9.1 Unitarias
- `PendingActionRegistry`: push/take/expiración/limpieza en SIGNED_OUT.
- `useProtectedAction`: rama gate vs queue; autenticado vs guest; TTL.
- `ResumeRunner`: idempotencia; múltiples pendientes → sólo última.
- Serialización cross-tab: no fuga de payload.

### 9.2 Integración
- Hook + Sheet Stack: apertura, cierre por overlay, cierre por acción secundaria.
- Hook + `useAuth`: reanudación en `SIGNED_IN`; limpieza en `SIGNED_OUT`.
- `guest-queue` intacto tras migración de `AddToTravelPlanButton`.

### 9.3 E2E (Playwright)
- E2E-1: Guest agrega favorito → login email → favorito aparece en `/cuenta/favoritos`.
- E2E-2: Guest solicita concierge → login Google → aterriza en `/cuenta/concierge/:caseId`.
- E2E-3: Guest add a Mi Viaje → cierra Sheet → recarga → item aún en cola → login → item persistido.
- E2E-4: TTL vencido → toast informativo, sin ejecución.
- E2E-5: Cross-tab: autenticar en pestaña B → pestaña A reanuda o descarta según política.

### 9.4 Casos negativos
- Server fn falla post-resume → toast error + acción NO reencolada.
- `sessionStorage` bloqueado (modo privado estricto) → hook degrada a memoria; sin crash.
- Sesión expira entre gate y ejecución → 401 dispara re-gate.

### 9.5 Recuperación
- `SIGNED_OUT` limpia registry.
- Reload durante Sheet abierto: registry en memoria se pierde; storage sólo tiene `{id, kind}` → toast "Continúa donde lo dejaste" sin ejecución silenciosa.

## 12. Recomendación arquitectónica

Este DFT es **compatible al 100 %** con la arquitectura vigente (Constitution, Workspace Engine, Discovery Layer, PWA). No introduce nuevos engines ni rompe el Infrastructure Freeze reforzado: reutiliza Sheet Stack, `useAuth`, `guest-queue`, `/auth` y `toast-bus`. La única capa nueva es una **coordinación delgada** (hook + registry + runner + sheet content) que centraliza un patrón hoy triplicado.

Se recomienda aprobar el DFT y proceder al Incremento I1 (infraestructura sin consumidores) como primera entrega verificable.
