
# 13.1 – Domain Model
**Versión:** 1.0  
**Estado:** Aprobado para ejecución  
**Blueprint:** Valladolid.mx v2.0

## Propósito

Este documento define el modelo de dominio oficial de Valladolid.mx. Su objetivo es traducir la arquitectura conceptual aprobada durante la Fase 0 a un modelo implementable. Lovable deberá implementar este modelo sin reinterpretar el negocio.

## Objetivos

- Crear un modelo reutilizable para múltiples regiones turísticas.
- Separar claramente territorio, operación, contenido y comercialización.
- Permitir crecimiento sin rediseñar la base de datos.

## Principios

1. El Blueprint es la única fuente de verdad.
2. Ninguna entidad se crea fuera de este documento.
3. Las relaciones deben ser escalables y normalizadas.
4. El dominio nunca se acoplará exclusivamente a Valladolid.

# Capas del dominio

## Capa Territorial
- País
- Estado
- Región Turística
- Destino
- Zona
- Punto de Interés

Relación:
País → Estados → Regiones → Destinos → Zonas → POI

## Capa Comercial
- Empresa
- Producto / Experiencia
- Categoría
- Etiquetas
- Horarios
- Tarifas

Relaciones:
- Una Empresa pertenece a un Destino.
- Una Empresa ofrece múltiples Productos.
- Un Producto puede pertenecer a varias Categorías.

## Capa Editorial
- Página
- Artículo
- Evento
- Ruta
- Galería
- Fotografía
- Video
- Documento

Todo contenido deberá ser multilenguaje y versionable.

## Capa de Identidad
- Usuario
- Organización
- Rol
- Permiso

Los permisos nunca se asignarán directamente a funcionalidades; se asignan a roles.

# Reglas de negocio

- Ningún Producto existe sin Empresa.
- Ninguna Empresa existe sin Destino.
- Ningún Destino existe sin Región.
- El contenido puede relacionarse con múltiples entidades.
- Todas las entidades deberán soportar estado (borrador, publicado, archivado).

# Capacidades obligatorias

El modelo deberá soportar:

- CMS editorial
- Marketplace turístico
- Alux IA
- Arma tu Viaje
- SEO
- Traducciones
- APIs públicas futuras
- Analítica

# Requisitos de datos

Cada entidad deberá contemplar cuando aplique:
- UUID
- slug
- nombre
- descripción
- estado
- timestamps
- auditoría
- SEO
- idioma

# Restricciones

- No duplicar información territorial.
- No usar valores hardcodeados para Valladolid.
- No crear dependencias circulares.
- Mantener compatibilidad con futuras regiones.

# Entregables

- Modelo lógico.
- Modelo físico.
- Migraciones.
- Diccionario de datos.
- Diagrama ER.
- Evidencias de QA.

# Definition of Done

El bloque concluye cuando:
- El modelo implementa todas las entidades descritas.
- Las relaciones son consistentes.
- Existen migraciones reproducibles.
- QA funcional aprobado.
- Documentación sincronizada con el Blueprint.
