# 11.2 – USERS, ROLES & SECURITY
## Autenticación, Roles, Permisos y Seguridad

**Versión:** 1.0  
**Depende de:** START-HERE-FIRST, 00–10, 11.0, 11.1  
**Estado:** Oficial para Blueprint técnico

---

# Objetivo

Definir la arquitectura de usuarios, perfiles, roles, permisos y seguridad de la plataforma Valladolid.mx.

Este documento establece cómo Lovable debe estructurar la autenticación y la seguridad antes de construir módulos como CMS, empresas, concierge, Arma tu Viaje o Alux.

---

# Principios de seguridad

1. La autenticación no es suficiente; todo acceso debe validarse por rol.
2. Row Level Security debe activarse desde el inicio.
3. Los usuarios no deben acceder a datos que no les pertenecen.
4. Las empresas solo administran sus propios perfiles y productos.
5. El concierge solo accede a solicitudes asignadas o autorizadas.
6. El administrador puede moderar, aprobar y configurar.
7. El super admin tiene control total.
8. Ninguna regla crítica debe depender únicamente del frontend.

---

# Modelo base de usuarios

La autenticación puede gestionarse mediante Supabase Auth o el sistema equivalente de Lovable Cloud.

La información operativa del usuario debe almacenarse en tablas propias.

---

# Tabla: profiles

Representa el perfil general de cada usuario autenticado.

## Campos sugeridos

- id UUID primary key
- user_id UUID references auth.users
- display_name text
- first_name text
- last_name text
- email text
- phone text
- avatar_url text
- country text
- preferred_language text default 'es'
- timezone text
- status text
- created_at timestamp
- updated_at timestamp

## Reglas

- Cada usuario autenticado debe tener un profile.
- El usuario solo puede editar su propio profile.
- Administradores pueden ver y editar perfiles cuando sea necesario.

---

# Tabla: user_roles

Permite asignar uno o varios roles a un usuario.

## Roles iniciales

- traveler
- business_owner
- concierge
- editor
- admin
- super_admin

## Campos sugeridos

- id UUID primary key
- user_id UUID references auth.users
- role text
- scope_type text nullable
- scope_id UUID nullable
- created_at timestamp
- created_by UUID

## Reglas

- Un usuario puede tener múltiples roles.
- Los roles pueden ser globales o por alcance.
- Ejemplo: un usuario puede ser business_owner solo de una empresa específica.
- Solo admin y super_admin pueden asignar roles.

# Row Level Security

RLS debe activarse en todas las tablas sensibles.

## Reglas generales

### Visitante no autenticado

Puede leer únicamente contenido público publicado.

No puede modificar datos.

### Viajero autenticado

Puede:

- editar su profile
- editar su traveler_profile
- crear y editar sus travel_plans
- enviar solicitudes al concierge
- escribir reseñas
- guardar favoritos

No puede:

- ver datos privados de otros viajeros
- modificar empresas
- acceder al CMS

### Empresa

Puede:

- editar empresas donde esté autorizado
- administrar productos de esas empresas
- responder reseñas de sus empresas
- ver métricas propias

No puede:

- editar otras empresas
- aprobarse a sí misma si requiere moderación
- alterar su trust_score manualmente

### Concierge

Puede:

- ver solicitudes asignadas
- crear notas
- actualizar estados
- preparar propuestas

No puede:

- ver solicitudes no asignadas, salvo permiso especial
- modificar contenido CMS
- cambiar planes de visibilidad

### Editor

Puede:

- crear contenido
- editar contenido
- enviar a revisión

No puede:

- cambiar roles
- acceder a pagos
- modificar configuración global

### Admin

Puede:

- administrar contenido
- moderar empresas
- aprobar reseñas
- asignar concierge
- revisar métricas

### Super Admin

Acceso completo.

# Seguridad para datos personales

Los siguientes datos deben tratarse como sensibles:

- teléfono
- correo
- país
- preferencias de viaje
- restricciones alimentarias
- necesidades de accesibilidad
- historial de solicitudes
- conversaciones con Alux

Regla:

Solo deben estar disponibles para el usuario, concierge asignado y administradores autorizados.

---

# Criterios de aceptación

La arquitectura de seguridad será válida si:

- RLS está activado desde el inicio.
- Cada usuario tiene permisos claros.
- Las empresas solo ven su información.
- El concierge solo ve expedientes asignados.
- Los viajeros controlan sus propios datos.
- Los administradores pueden operar sin romper la privacidad.
- Ninguna regla crítica depende únicamente del frontend.
