# Valladolid.mx Platform Blueprint

# 11.5 – DATABASE STANDARDS
## Estándares de Base de Datos y Calidad

**Versión:** 1.0
**Depende de:** 11.0 – 11.4

---

# Objetivo

Definir las reglas obligatorias que deberán seguirse durante el diseño, implementación y evolución de la base de datos.

---

# Convenciones de nombres

## Tablas

- snake_case
- plural
- nombres descriptivos

Ejemplos:

- businesses
- destinations
- travel_plans
- concierge_requests

## Columnas

- snake_case
- sin abreviaturas ambiguas

Ejemplos:

- created_at
- updated_at
- business_id
- destination_id

---

# Llaves primarias

- UUID en todas las entidades principales.

---

# Llaves foráneas

Toda relación debe declararse explícitamente mediante foreign keys.

---

# Auditoría

Toda tabla de negocio debe incluir cuando aplique:

- created_at
- updated_at
- created_by
- updated_by
- deleted_at
- deleted_by

---

# Soft Delete

Eliminar lógicamente los registros importantes.

No borrar información histórica salvo procesos de mantenimiento autorizados.

---

# Índices

Crear índices para:

- slug
- status
- created_at
- updated_at
- claves foráneas
- búsquedas frecuentes

Evaluar índices compuestos según consultas reales.

---

# Rendimiento

- Evitar duplicación de datos.
- Preferir tablas puente para relaciones muchos-a-muchos.
- Mantener consultas simples y escalables.
- Usar paginación en listados.

---

# Versionado

Todas las migraciones deberán:

- ser reversibles cuando sea posible
- estar documentadas
- mantener compatibilidad con datos existentes

---

# Seguridad

- RLS habilitado.
- API Keys fuera del frontend.
- Validación de permisos en backend.

---

# Calidad de datos

- Validar integridad referencial.
- Evitar registros huérfanos.
- Mantener slugs únicos.
- Normalizar información reutilizable.

---

# Escalabilidad

La arquitectura debe soportar:

- nuevos países
- nuevos idiomas
- nuevos módulos
- nuevos proveedores de IA
- nuevas integraciones

sin rediseñar el modelo principal.

---

# Criterios de aceptación

La base de datos será considerada lista cuando:

- siga toda la serie 11.0–11.5
- soporte CMS, IA y multilenguaje
- sea modular y escalable
- tenga seguridad basada en roles
- esté preparada para crecimiento internacional
